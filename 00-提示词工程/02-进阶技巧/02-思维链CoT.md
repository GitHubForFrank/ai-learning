# 思维链 Chain-of-Thought（CoT）

> 让模型在给出最终答案前，**显式输出中间推理步骤**。对复杂推理任务（数学、多跳问答、规划）提升显著。

---

## 1. 核心思想

LLM 的"思考"本质是**自回归生成**——一旦最终答案被 token 决策出来，就无法回头。  
CoT 的价值在于**把推理过程外化为 token 流**：
- 中间步骤本身也是输入，后续 token 能基于这些步骤继续推导
- 等价于给模型提供了一个"草稿纸"

实验观察：CoT 在**参数量 ≥ 60B** 的模型上涌现，小模型使用 CoT 反而可能更差。

---

## 2. 两种触发方式

### 2.1 Zero-shot CoT
一句魔法咒语：

```
Let's think step by step.
请一步一步思考。
```

适合**通用推理任务**，不改变 Prompt 结构，零成本。

### 2.2 Few-shot CoT
在示例中**展示推理过程**：

```
问题：餐厅有 23 个苹果，用掉 20 个做午餐，又买了 6 个。现在有多少？
推理：起初 23 个，用掉 20 个剩 3 个，又买 6 个，所以是 3 + 6 = 9 个。
答案：9

问题：{新问题}
推理：
答案：
```

比 Zero-shot CoT **准确率更高**，但占上下文。

---

## 3. 结构化 CoT

让推理过程更**可控、可解析**：

```
请按以下结构回答：
<analysis>
分析问题的关键信息
</analysis>
<plan>
列出解决步骤
</plan>
<execution>
执行每一步
</execution>
<answer>
最终答案
</answer>
```

好处：
- 程序可以只提取 `<answer>` 字段
- 调试时能定位是哪一步推错
- 便于 Agent 拿 `<plan>` 做后续调度

---

## 4. 进阶变体

### 4.1 Self-Consistency
同一问题采样 N 次（temperature > 0），对**最终答案投票**取多数。细节见 `04-Self-Consistency.md`。

### 4.2 Tree-of-Thoughts (ToT)
不是线性一条链，而是**搜索树**：每一步生成多个候选分支，用评估函数剪枝。适合规划、证明类任务。成本高。

### 4.3 Least-to-Most
先让模型**把难题拆成子问题**，再逐个解决，每个子问题的答案作为下一步输入。对多跳推理比纯 CoT 更稳。

---

## 5. 什么时候**不**用 CoT

| 场景 | 原因 |
|------|------|
| 简单分类 / 抽取 | 推理步骤反而引入噪声和幻觉 |
| 延迟敏感场景 | CoT 输出 token 数 ×3~10，响应变慢 |
| 结构化输出（JSON） | 推理 token 和结构 token 混在一起难解析 → 用 `<reasoning>` / `<answer>` 标签隔离 |
| 小模型（< 10B） | CoT 可能反向退化 |

---

## 6. 避坑

- **推理不等于正确**：模型可以输出看似合理但前提错误的链。链越长越要警惕"自信的胡说"
- **答案藏在推理里**：直接返回全文给用户会很长。用结构化标签或后处理截取最终答案
- **与 JSON 模式冲突**：强 JSON 约束会抑制自由推理。解决方案：
  - 两次调用：第一次自由推理，第二次基于推理产出 JSON
  - 或 JSON schema 里留一个 `reasoning` 字段

---

## 7. 与 Agent 的关系

- Agent 内部的 `<thinking>` 标签本质就是 CoT
- CoT 是**单轮**的思考链；Agent 是**多轮**的思考+工具+观察循环
- ReAct = CoT + 工具调用（详见 `03-ReAct推理与行动.md`）
