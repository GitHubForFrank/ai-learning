# Few-shot 学习（示例驱动提示）

> 在 Prompt 中**内嵌若干输入-输出示例**，让模型通过"模仿"完成新任务，不需要微调。

---

## 1. 三种粒度

| 模式 | 示例数量 | 适用场景 |
|------|---------|---------|
| **Zero-shot** | 0 | 任务本身常见（翻译、摘要、改写），指令清晰即可 |
| **One-shot** | 1 | 任务有固定格式要求（JSON schema、特殊标签） |
| **Few-shot** | 2~8 | 任务新颖、输出格式复杂、需要风格对齐 |

经验值：**超过 8 个示例收益递减**，且开始挤占上下文窗口、拉高成本。

---

## 2. 核心结构

```
<指令>
任务说明一句话

<示例>
输入：...
输出：...

输入：...
输出：...

<待处理>
输入：{真实输入}
输出：
```

三要素：**指令 → 多组 (输入, 输出) → 新输入**，结尾留空让模型续写。

---

## 3. 示例设计原则

### 3.1 多样性 > 数量
示例要**覆盖不同类型的输入**，而不是同一类型重复。

```
❌ 反例：3 个都是"表扬型"评论 → 分类"正面"
✅ 正例：正面 / 负面 / 中性各 1 条，且话题分散
```

### 3.2 难度递增
把**边界案例**放在最后。模型对 Prompt 末尾的示例敏感度更高，放易混淆的示例能强化判别。

### 3.3 格式严格一致
所有示例的分隔符、字段名、标点必须**完全相同**。一旦混用，模型会在输出时"创新"格式。

```
❌ 示例1 用 "输出："，示例2 用 "答案：" → 模型输出随机选一种
✅ 统一用 "输出：" 或全部用 JSON
```

### 3.4 顺序效应（Order Effect）
- **分类任务**：标签顺序会影响偏向。如果所有示例都是 "正面→负面→中性"，模型倾向高频预测第一个标签。**打散顺序**。
- **生成任务**：把**风格最接近目标**的示例放在最后。

## 4. 实战示例

### 任务：从用户反馈中抽取"问题类型 + 严重度"

```
你是反馈分类助手。请按示例格式抽取「问题类型」和「严重度（P0/P1/P2）」。

---
反馈：支付按钮点了没反应，钱扣了但订单没创建。
问题类型：支付异常
严重度：P0

反馈：App 启动图标有点模糊，影响不大。
问题类型：视觉瑕疵
严重度：P2

反馈：切换城市后，首页推荐还是原城市的。
问题类型：状态同步
严重度：P1
---

反馈：{用户输入}
问题类型：
严重度：
```

---

## 5. Few-shot 失效的信号

- 输出**忽略指令**、只模仿示例的表层格式 → 指令被示例淹没，把指令**前置**并加强语气
- 输出**复读示例内容** → 示例太少、输入与示例过于相似，增加多样性
- **成本爆炸** → 每次调用都带 6 个示例 × 高频场景，考虑微调或 RAG 动态检索示例

---

## 6. 与其他技巧的组合

| 组合 | 效果 |
|------|------|
| Few-shot + CoT | 示例里把推理过程也写出来，模型会模仿链式思考 |
| Few-shot + Role | "你是 X 专家，参考以下案例..." 风格一致性更强 |
| Dynamic Few-shot（RAG） | 根据输入相似度从示例库检索最匹配的 N 条 |
