# Role Prompting：角色提示

> 通过**指定模型扮演一个身份**（"你是 X 专家"），引导其输出的领域深度、用语专业度与思考视角。

---

## 1. 为什么有效

LLM 从海量语料里学到了**"某类身份的人会怎么说话"**的分布。给它一个角色，等于在巨大的输出空间里切出一个子集：
- **词汇切换**：指定"儿科医生"后，用词从通用口语切换到医学术语
- **视角切换**：指定"代码审查员"后，会主动关注边界条件、安全、可读性
- **风险偏好**：指定"保守法律顾问"和"创业顾问"回答同一个问题，结论可能相反

**它不会真的让模型"变聪明"**，但会让模型**更稳定地聚焦到相关知识上**。

---

## 2. 基础模板

```
你是{角色描述}，{核心能力/经验}。

请{任务}。

约束：{输出要求 / 边界}
```

### 示例

```
你是一位有 15 年 Java 后端经验的架构师，擅长高并发系统设计与重构。

请评审以下订单服务代码，指出性能瓶颈与可维护性问题。

约束：
- 按「严重度 + 具体位置 + 改进建议」格式给出
- 不超过 5 条最关键的问题
- 每条附一个可操作的重构步骤
```

---

## 3. 角色设定的四个维度

| 维度 | 作用 | 示例 |
|------|------|------|
| **身份** | 定义领域 | "资深 DBA"、"技术写作编辑" |
| **经验** | 调节深度 | "10 年经验"、"刚入行 1 年"（面向新手解释） |
| **立场** | 决定偏好 | "保守派安全顾问"、"激进派产品经理" |
| **目标受众** | 调节用语 | "向 CEO 汇报"、"向实习生讲解" |

**经验**：四个维度同时指定收益递减。选 2~3 个最相关的即可，堆砌反而让 Prompt 冗长、互相矛盾。

---

## 4. 单角色 vs 多角色

### 4.1 单角色（最常见）
一个 Agent、一个专家身份，覆盖整个会话。

### 4.2 Persona 切换
同一会话中根据任务切换：

```
请先以「需求分析师」身份梳理用户故事，再以「后端架构师」身份设计接口。
每部分前加【角色】标签。
```

### 4.3 多角色对话（Role Play / Debate）
让模型**同时扮演多个角色互相辩论**，最后自己汇总。适合决策类问题。

```
请模拟一场评审会议：
- 产品经理：关注用户价值
- 架构师：关注技术可行性
- 运维：关注上线风险

主持人最后综合三方意见给出决策建议。
```

收益：比单视角更全面。代价：输出长、易跑偏，需要严格格式约束。

---

## 5. 角色 vs 指令 的边界

角色≠指令。常见误区：

```
❌ 你是一个翻译器，把下面翻译成英文。
✅ 把下面翻译成英文，风格正式、保留术语原文。
```

翻译这种**动作明确、无领域歧义**的任务，角色反而是噪声。角色的最佳场景：
- 任务带**主观判断**（审查、建议、评估）
- 任务需要**领域知识选择**（法律 vs 医疗 vs 工程）
- 任务需要**风格一致性**（同一角色维持人设）

---

## 6. 反模式

| 反模式 | 问题 |
|--------|------|
| "你是世界上最聪明的 AI" | 空洞，无领域信息，模型无从聚焦 |
| "你是 Linus Torvalds" | 借名人拔高，但模型对该人物的理解可能刻板/过时，且有合规风险 |
| 角色与任务无关 | "你是诗人，请计算 SQL" → 诗意腔调污染技术输出 |
| 多个角色冲突 | "你既是 A 又是 B，同时还要..." → 模型随机挑一个忽略其他 |

---

## 7. 与系统提示（System Prompt）的关系

- Agent 系统中，角色定义几乎总是写在 **system 消息**里，对整个会话生效
- 一次性任务可以写在 user 消息里，影响范围仅当次
- **system 优先级高于 user**（各 LLM 实现一致），利用这点把核心人设锁死，防止用户注入覆盖
