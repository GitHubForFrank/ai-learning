# Prompt 调优方法论

> 把 Prompt 当成**一段有"版本"和"回归测试"的代码**来管理。拒绝"改一句，手测一次，上线"的玄学迭代。

---

## 1. 调优闭环

```
┌─────────────┐
│ 1. 明确指标 │ ← 先定义"什么叫好"，再开始调
└──────┬──────┘
       ▼
┌─────────────┐
│ 2. 构建评测集 │ ← 20~200 条带标注的真实输入
└──────┬──────┘
       ▼
┌─────────────┐
│ 3. 基线测试  │ ← 跑一次初版 Prompt，记录分数
└──────┬──────┘
       ▼
┌─────────────┐
│ 4. 假设 + A/B│ ← 只改一个变量，对比两版差异
└──────┬──────┘
       ▼
┌─────────────┐
│ 5. 归因分析  │ ← 看变差的样本在哪类场景
└──────┬──────┘
       ▼
   循环回 4
```

---

## 2. 先定义"好"

| 任务类型 | 可量化指标 |
|---------|-----------|
| 分类 / 抽取 | 精确率、召回率、F1 |
| 问答 | 与标准答案 Exact Match / 语义相似度 |
| 代码生成 | 单元测试通过率 / 编译通过率 |
| 摘要 / 改写 | ROUGE / BLEU + 人工打分 |
| 开放生成 | 人工打分（按维度评分卡） / AI 裁判 |

**反模式**：没有指标就开始改 Prompt——改完自己觉得"好像好了"，其实是安慰剂。

---

## 3. 构建评测集

### 3.1 规模
- **冷启动**：20~50 条，手工挑，覆盖主要场景
- **稳定期**：100~500 条，含**真实用户日志**
- **高要求**：1000+ 条，分层抽样（高频 + 长尾 + 边界）

### 3.2 数据来源
- **线上日志**（脱敏后）——最有代表性
- **用户反馈**中的 bad case
- **合成数据**——用大模型生成，用于冷启动或扩展覆盖
- **对抗样本**——专门构造越狱、模糊指令、语言混合

### 3.3 标注
每条带：
- 输入
- **期望输出**（或期望特征）
- **评分维度**（如正确性/格式/语气分列）

---

## 4. 基线与版本管理

每次改动产出一个版本：

```
prompts/
├── v1.0_baseline.md         # 初版
├── v1.1_add_few_shot.md     # 加示例
├── v1.2_tighten_format.md   # 收紧格式
├── changelog.md             # 每版改了什么、分数多少
└── eval_results/
    ├── v1.0.json
    ├── v1.1.json
    └── v1.2.json
```

**原则**：
- 一次只改**一个变量**（加示例 / 换角色 / 改温度 …），否则无法归因
- 每版跑**同一套评测集**，数字才可比
- **保留历史版本**，回归时能退回

---

## 5. A/B 归因分析

跑完两版后看**逐样本对比表**：

| 样本 ID | 输入 | v1 输出 | v2 输出 | v1 分 | v2 分 | 变化 |
|---------|------|---------|---------|-------|-------|------|
| 1 | ... | ... | ... | 1 | 1 | = |
| 2 | ... | ... | ... | 0 | 1 | ↑ |
| 3 | ... | ... | ... | 1 | 0 | ↓ |

关注**下降样本**：新版修了 A 但破坏了 B。典型原因：
- 加示例后，非示例类型场景退化
- 收紧格式后，长/复杂输入被截断
- 加角色后，非该角色擅长的任务退化

**对策**：
- 扩展评测集覆盖这些退化场景
- 或回退此次改动，换思路

---

## 6. 变量维度清单

每轮只动**一个**：

| 维度 | 示例 |
|------|------|
| **角色/人格** | 加/改/去 |
| **指令措辞** | 命令式 vs 请求式，肯定 vs 否定 |
| **示例数量** | 0 → 1 → 3 → 5 |
| **示例内容** | 换更有代表性的 / 加边界案例 |
| **输出格式** | 自由文本 → 结构化标签 → JSON |
| **CoT 开关** | 加/去 "step by step" |
| **约束位置** | 开头 / 结尾 / 头尾都放 |
| **温度** | 0 / 0.3 / 0.7 |
| **模型版本** | Haiku / Sonnet / Opus |

## 6.5 提示词与大模型适配性

**核心观点**：没有"万能提示词"——同一提示词在不同模型上表现差异可达 30%~50%，必须针对目标模型调优。

### 6.5.1 为什么需要适配？

不同模型的**训练数据分布**、**指令跟随能力**、**输出偏好**差异显著：

| 维度 | Claude 系列 | GPT-4 系列 | Llama 3 系列 |
|------|------------|-----------|-------------|
| **提示词结构偏好** | 喜欢详细的 System Prompt + 清晰的分步指令 | 指令放在 User 消息中效果更好，接受更简洁的 Prompt | 对 Few-shot 示例更敏感，需要更明确的格式约束 |
| **指令风格** | 偏好"协作式"措辞（"让我们一起..."） | 接受"命令式"和"请求式"均可 | 更依赖明确的角色设定 |
| **推理能力触发** | "Let's think step by step" 效果显著 | "Think through this carefully" 或 CoT 标签效果好 | 需要显式的推理步骤引导 |
| **输出格式** | XML 标签 `<think></think>` 原生支持 | JSON 模式更可靠 | 需要在 Prompt 中反复强调格式 |

### 6.5.2 适配策略清单

**策略 1：模型专属 System Prompt**
```
✅ Claude 优化版：
<system>
你是一位资深数据分析师。
请严格按照以下步骤执行：
1. 先理解问题背景
2. 提取关键数据点
3. 给出结论
</system>

✅ GPT-4 优化版：
你是一位资深数据分析师。请严格按照以下步骤执行：
1. 先理解问题背景
2. 提取关键数据点
3. 给出结论
```

**策略 2：调整 Few-shot 数量**
- Claude/Sonnet：1~3 个示例足矣
- GPT-4o：3~5 个示例最佳
- Llama 3 70B：5~8 个示例才能稳定

**策略 3：利用模型原生特性**
- Claude：优先用 `<thinking>` 标签放推理过程
- GPT-4：开启 JSON Mode / Structured Outputs
- Llama 3：用 `<|begin_of_solution|>` 标记输出边界

**策略 4：A/B 测试模板**
```
版本 A（Claude 优化）：
<system>
角色：产品经理
任务：写PRD
约束：用中文，分5个章节
</system>

版本 B（GPT-4 优化）：
角色：产品经理
任务：写PRD
约束：用中文，分5个章节
```

### 6.5.3 跨模型迁移 Checklist

当你把一套 Prompt 从模型 X 迁移到模型 Y 时，检查以下项：

- [ ] 把 XML 标签改成模型偏好的格式（或保留但增加说明）
- [ ] 调整 Few-shot 数量（+2 或 -2）
- [ ] 把"不要做 X"改成正向表述（视模型而定）
- [ ] 在目标模型上跑一遍评测集，看哪些样本退化
- [ ] 针对退化样本单独优化

**反模式**："我在 GPT-3.5 上调得好好的，直接放到 GPT-4o 应该更好"——实际上可能更差。

## 7. AI 裁判（LLM-as-a-Judge）

人工打分贵且慢。用另一个 LLM 按评分卡打分：

```
请按以下维度对回答评分（1-5）：
- 正确性：{标准}
- 格式符合度：{标准}
- 语气一致性：{标准}

标准答案：...
待评答案：...

输出 JSON：{ correctness: X, format: Y, tone: Z, reason: "..." }
```

**注意事项**：
- 裁判模型**不能和被评模型同一个**（否则自我偏好）
- 裁判 Prompt 本身也要评测（元评测）
- 用于**相对对比**（v1 vs v2）比**绝对打分**更可靠

---

## 8. 生产级调优流程

```
开发期：手工评测集，快速迭代
    ↓
灰度期：小流量上线，采集真实日志
    ↓
监控期：
  - 采样 N% 流量做 AI 裁判自动评分
  - 用户反馈（赞/踩）回流评测集
  - 定期（周/月）跑全量评测集检测回归
    ↓
报警：核心指标下降 > 阈值 → 自动回滚到上一版
```

## 10. 调优 Top-5 心法

1. **先定指标，再动手**——没有指标的调优都是玄学
2. **一次改一变量**——否则归因即赌博
3. **保留旧版 Prompt**——回归永远在路上
4. **真实数据 > 合成数据**——线上日志是金矿
5. **知道何时停手**——边际收益递减时，换模型/换架构比抠 Prompt 更划算
