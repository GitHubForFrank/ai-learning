# 常见提示词陷阱（反模式集）

> 一份**以翻车案例驱动**的避坑清单。按出现频率排序，前几条是 90% 翻车的根源。

---

## 1. 指令模糊

```
❌ 帮我优化这段代码
```

"优化"是什么？性能？可读性？结构？模型只能猜。

```
✅ 请从以下维度评审并改进：
   1) 减少数据库往返次数
   2) 去除重复逻辑
   3) 用更清晰的函数命名
   不改变外部行为。
```

**原则**：越开放的词（优化、改进、重构、更好）越要**列维度**。

---

## 2. 否定指令失效

```
❌ 不要使用 Markdown 格式
❌ 不要输出解释，只要代码
```

LLM 对"不要做 X"的响应经常**反向激活 X**（提到"不要想粉红色大象"）。

```
✅ 请直接输出纯文本，不含任何符号前缀
✅ 请只输出代码块内的内容，代码块外留空
```

**原则**：**用"做什么"代替"不做什么"**。必须否定时，搭配一个正向描述。

---

## 3. 冲突约束

```
❌ 请简短回答，同时提供详细的步骤说明和完整的示例代码
```

"简短"和"详细+完整"互斥。模型会随机偏向一端，或给出**四不像**的中间态。

```
✅ 首答≤3 句话概述；如果用户追问"展开"，再给详细步骤与完整代码
```

**原则**：冲突约束用**分阶段**或**分字段**化解，而不是堆在一起。

---

## 4. 上下文过长淹没指令

Prompt 前放了 5000 字的资料，指令夹在末尾一句"请总结上面"。模型容易：
- 把指令当成资料的一部分
- 遗漏资料中段的内容（**lost in the middle**）

```
✅ 模板：
   # 任务
   {指令写在最前面 + 最后复述一遍}

   # 资料
   {长资料}

   # 再次强调的任务
   {复述}
```

**原则**：**指令三明治**——头尾各放一次，长资料夹中间。

---

## 5. 过拟合示例

Few-shot 给了 3 个"英译中"的示例，用户输入"Hello"期待简单翻译。模型输出一大段解释，因为示例里都带了解释。

**原则**：示例**必须与真实使用场景完全对齐**。示例不足或格式不匹配时，宁可改示例也别指望模型"举一反三"。

---

## 6. Role 与任务错配

```
❌ 你是一位诗人，请帮我优化这段 SQL
```

诗意腔调会污染技术输出。角色要**匹配任务领域**。

**原则**：纯技术/工具性任务，角色反而是噪声，**不如不加**。

---

## 7. 只给正例，不给反例

```
❌ 输出：符合规范的代码
```

模型不知道"不符合规范"长什么样。

```
✅ 合格输出示例：...
   不合格输出示例（及原因）：...
```

**原则**：对**判别类任务**，正反例都要给，模型才学得到边界。

---

## 8. 温度与任务不匹配

| 任务类型 | 推荐 Temperature |
|---------|------------------|
| 代码生成、结构化抽取、数学 | 0 ~ 0.3 |
| 客服/问答（稳定但不死板） | 0.3 ~ 0.7 |
| 创意写作、头脑风暴 | 0.7 ~ 1.2 |
| Self-Consistency 采样 | 0.7 ~ 1.0 |

**反模式**：结构化任务用 T=1.0 → 输出格式随机；创意任务用 T=0 → 每次一模一样、机械感强。

---

## 9. 指令与输出格式冲突

```
❌ "请思考一步步推理，然后只输出 JSON { answer: ... }"
```

推理文字和 JSON 要求互相挤压，模型要么忽略推理、要么输出无效 JSON。

```
✅ JSON schema 里留一个 reasoning 字段：
   { "reasoning": "...", "answer": "..." }
```

或**两次调用**：第一次推理，第二次基于推理产 JSON。

---

## 10. 把 Prompt 当"说明书"

```
❌ 堆砌 20 条规则，每条都说"请一定要..."
```

超过 10 条时模型开始**选择性忽略**（前几条+最后几条保留，中间段遗忘）。

**原则**：
- 合并同类规则
- 把"必须项"和"可选项"分层
- 把"通用规则"上浮到 system prompt，"任务规则"留在 user

---

## 11. 假设模型"记得"

同会话内多轮对话，以为"刚才说过"模型就记住。但：
- 上下文可能被压缩 / 滑窗截断
- 模型对**早期消息的遵循度随轮数衰减**

**原则**：关键约束**每次关键决策前复述一次**，或放 system prompt。

---

## 12. 幻觉诱因 TOP 3

| 诱因 | 场景 | 对策 |
|------|------|------|
| **超出知识截止** | 问最新的库版本、最新事件 | RAG / 工具调用 |
| **小众领域细节** | 内部 API、公司黑话 | RAG + 示例 |
| **需要引用来源** | 学术/法律 | 强制"必须给出可验证的来源，否则说不知道" |

**原则**：涉及事实 → **让模型能访问真实数据**（RAG / 工具），而非相信参数里的记忆。

---

## 14. 假设"万能提示词"跨模型通用

```
❌ "这个 Prompt 在 GPT-3.5 上完美，直接用在 Claude 上肯定没问题"
```

同一提示词在不同模型上表现差异可达 30%~50%，因为：
- 训练数据分布不同 → 对同一句话的理解偏差
- 指令跟随能力不同 → 有的模型忽略 System Prompt
- 输出格式偏好不同 → XML vs JSON vs 自由文本

**原则**：换模型必须重跑评测集，针对退化样本单独调优。

---

## 15. 速查表

| 翻车信号 | 去哪找原因 |
|---------|-----------|
| 输出格式飘忽 | 示例格式不一 / 温度过高 / 指令冲突 |
| 忽略部分指令 | 规则太多 / 指令在中段被淹没 |
| 编造事实 | 缺少外部数据通道 / 没要求"不知道就说不知道" |
| 语气跑偏 | 人格在 user 消息里，被覆盖 / 长对话漂移 |
| 解析失败 | 没用结构化输出 API / 自由推理污染 JSON |
| 换模型后效果暴跌 | 没做模型适配，Prompt 是针对旧模型优化的 |
