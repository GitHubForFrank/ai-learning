## 1. 推荐目录骨架

```
my-mcp-server/
├── pom.xml                                       # 依赖 / Spring AI BOM / 插件
├── mvnw / mvnw.cmd / .mvn/                       # Maven Wrapper
└── src/
    ├── main/
    │   ├── java/com/example/mcp/
    │   │   ├── McpServerApplication.java         # @SpringBootApplication 入口
    │   │   ├── tools/                            # @McpTool 方法集合
    │   │   │   └── *.java
    │   │   ├── resources/                        # [软] @McpResource handler
    │   │   ├── prompts/                          # [软] @McpPrompt 模板
    │   │   └── core/                             # 业务/领域纯逻辑，零 Spring AI 依赖
    │   │       └── *.java
    │   └── resources/
    │       └── application.yml                   # MCP 协议 / 端口 / 元信息
    └── test/
        └── java/com/example/mcp/
            └── *Test.java                        # JUnit + Spring Boot Test
```

按规模可裁剪：

- **单工具小项目**：`tools/` 一个文件，`core/` 可省略
- **中大型项目**：`tools/<domain>/` + `core/<domain>/` 按子领域拆分

## 2. 各层职责（一句话）

| 层 | 职责 | 不该做的事 |
|----|------|-----------|
| `pom.xml` | Spring Boot parent / Spring AI BOM / starter / 插件 | 内嵌业务相关 plugin 配置 |
| `application.yml` | `spring.ai.mcp.server` 协议、端口、name/version 元数据 | 业务配置 |
| `McpServerApplication.java` | `@SpringBootApplication` + `main()` 一行启动 | 注册工具 |
| `tools/` | `@Service`/`@Component` 类内方法标 `@McpTool`；校验入参 + 调 `core/` | 写业务计算 |
| `core/` | 纯 Java 类、领域逻辑；与 `org.springframework.ai.mcp.*` 包**零耦合** | 引用 Spring AI MCP 类型 |

`core/` 与 SDK 解耦是关键约束：业务逻辑可独立单测，未来切换 Transport（webmvc/webflux/stdio）/ SDK 大版本时不被牵连。

## 3. 注册约定

### 3.1 注解式自动发现（推荐）

Spring 组件扫描发现 `@Component` / `@Service` Bean，Spring AI 再扫描 Bean 中的 `@McpTool` 等注解方法，**无需手动注册**。

| 注解 | 用在 | 作用 |
|------|------|------|
| `@McpTool(name=..., description=...)` | 方法 | 注册为 MCP Tool |
| `@McpToolParam(description=..., required=...)` | 参数 | 写入 JSON Schema |
| `@McpResource(uri=..., name=..., description=...)` | 方法 | 注册为 MCP Resource |
| `@McpPrompt(name=..., description=...)` | 方法 | 注册为 MCP Prompt |

约束：

- 工具类必须被 Spring 扫描到——确保 `@Service`/`@Component` 与 `@SpringBootApplication` 启动类**同包或子包**，否则 Bean 不会注册
- 方法可同时声明 `CallToolRequest` / `McpSyncRequestContext` 参数，Spring AI 自动注入

### 3.2 回调式（可选）

已有 Spring AI `ChatClient` 工具想复用为 MCP Tool 时，用 `@Tool` + `MethodToolCallbackProvider` Bean。**选它当**：本地工具同时被 ChatClient 和外部 MCP Client 调用，避免写两遍。

`@McpTool` 的 `description` 是 LLM 选工具的关键线索——写法见 [`../mcp_develop_guideline.md`](../mcp_develop_guideline.md)。

### 3.3 类型映射

参数类型由 Spring AI 自动映射到 JSON Schema（基本类型、`List<T>`、Pojo 都支持）；返回值可直接是基本类型（自动包装为 TextContent）或 `CallToolResult`（多段内容 / 显式 `isError`）。具体行为以 Spring AI 文档为准。

## 4. Stdio vs HTTP：源码层差异

源码层**几乎无差异**——切换由**依赖选型**决定，业务代码不变：

| Transport | Starter | 进程模型 |
|-----------|---------|---------|
| stdio | `spring-ai-starter-mcp-server` | Client 拉起子进程，stdin/stdout 通信 |
| http (Servlet) | `spring-ai-starter-mcp-server-webmvc` | 长驻 HTTP 服务（同步） |
| http (响应式) | `spring-ai-starter-mcp-server-webflux` | 长驻 HTTP 服务（响应式） |

三者**互斥**——`pom.xml` 只引一个 starter；配合 `application.yml` 的 `spring.ai.mcp.server.protocol: STDIO|STREAMABLE`。

**强约束**：

- stdio 模式下**所有日志禁止写 stdout**（关闭 banner：`spring.main.banner-mode=off`；console pattern 置空或重定向到文件/stderr），否则会污染协议
- 冷启动慢是 Java MCP Server 在 stdio 场景下的痛点——必要时启用 GraalVM Native Image，详见 [`03-CICD.md`](03-CICD.md)
