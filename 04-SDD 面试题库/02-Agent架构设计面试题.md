# 02 — Agent 架构设计面试题

> 覆盖 ReAct / Plan-Execute / 记忆机制 / Function Call / MCP / Tool Use / 多模态

---

## 一、Agent 基础概念（⭐）

### Q1：什么是 AI Agent？和传统 Chatbot 的核心区别？
**参考答案：**
- **Chatbot**：接收输入→生成回复，无自主行动能力，无工具调用
- **AI Agent**：具备**自主感知、推理、规划、行动**能力，能调用外部工具（API/数据库/代码执行），动态生成执行方案而非套用固定模板

核心差异：**Agent 有行动能力（take actions）**，不只是生成文本。

### Q2：Agent 的核心组件有哪些？
**参考答案：**
标准 Agent 四组件：

| 组件 | 职责 | 例子 |
|------|------|------|
| **LLM 大脑** | 理解、推理、决策 | GPT-4 / Claude / Gemini |
| **规划模块（Planning）** | 任务分解、路径规划 | ReAct / Plan-Execute / Tree-of-Thought |
| **记忆模块（Memory）** | 上下文保持、经验积累 | 短期（上下文窗口）/ 长期（向量库） |
| **工具模块（Tools）** | 外部能力调用 | Function Call / MCP / API / 代码执行 |

### Q3：Agentic Loop 是什么？画出流程
**参考答案：**
```
Think（思考下一步做什么）
  → Act（执行动作/调用工具）
    → Observe（观察执行结果）
      → 回到 Think（基于结果重新思考）
        → 直到目标达成 → Finish
```

这是 Agent 区别于单次 LLM 调用的核心——它是**循环迭代**的。

## 二、ReAct 框架（⭐⭐）

### Q4：ReAct 框架的工作原理？为什么有效？
**参考答案：**
ReAct = **Reasoning（推理）+ Acting（行动）**交替进行：

1. **Thought**：当前需要什么信息？下一步是什么？
2. **Action**：执行具体操作（搜索/计算/调用 API）
3. **Observation**：获取执行结果
4. 循环直到能给出最终答案

**为什么有效**：
- 将推理过程外化，每一步有据可查
- 行动结果作为新信息注入推理，减少幻觉
- 可中断、可纠错——某步错了下一步可以修正

### Q5：ReAct vs Plan-Execute 的区别与选择？
**参考答案：**

| 维度 | ReAct | Plan-Execute |
|------|-------|-------------|
| 规划方式 | 边做边想 | 先做完整计划再执行 |
| 灵活性 | 高（每步可调整） | 低（计划锁定后不易变） |
| 效率 | 可能绕路 | 路径直接 |
| 适用场景 | 不确定性高的探索任务 | 步骤明确的确定性任务 |
| 代表 | ReAct / Toolformer | Plan-and-Solve / LLMCompiler |

**选择判据**：任务边界清晰→Plan-Execute；任务需要边探索边调整→ReAct。

### Q6：ReAct 有哪些常见变体和改进？
**参考答案：**
- **Reflexion**：加入自我反思机制，失败后分析原因、动态调整
- **ReWOO**：分离推理和工具调用，减少 token 消耗
- **SwiftSage**：复杂任务用 Plan-Execute，简单步骤用 ReAct
- **AutoGPT 风格**：ReAct + 长期记忆 + 子目标分解

## 三、记忆机制（⭐⭐）

### Q7：Agent 的记忆机制如何设计？
**参考答案：**

| 类型 | 存储 | 用途 | 实现 |
|------|------|------|------|
| 短期记忆 | 上下文窗口 | 当前会话的历史 | LLM context |
| 长期记忆 | 向量数据库 / 知识图谱 | 跨会话的经验和知识 | ChromaDB/Pinecone/Mem0 |
| 工作记忆 | 结构化存储 | 当前任务的中间状态 | JSON/结构化变量 |

**关键设计考量**：
- 记忆的写入时机（什么值得记）
- 记忆的检索策略（什么时候查、查什么）
- 记忆的淘汰机制（防止膨胀）

### Q8：上下文窗口有限时，如何管理 Agent 的记忆？
**参考答案：**
1. **摘要压缩**：对旧消息自动生成摘要
2. **滑动窗口**：只保留最近 N 轮对话
3. **向量检索**：重要信息存入向量库，按需检索
4. **分层记忆**：核心事实→知识图谱；会话细节→摘要；临时状态→短期

## 四、工具调用（⭐⭐⭐）

### Q9：Function Call / Tool Use 的实现原理？
**参考答案：**
1. **定义阶段**：用 JSON Schema 定义工具的参数、类型、描述
2. **推理阶段**：LLM 根据用户意图判断是否需要调用工具，选择工具并生成参数
3. **执行阶段**：实际调用工具（由 Agent 框架或应用层执行）
4. **反馈阶段**：将工具返回结果注入 LLM 上下文，继续推理

**关键设计点**：
- Tool Description 质量决定调用准确率
- 参数 Schema 要足够精确（enum 约束、类型约束）
- 必须处理调用失败（重试/降级/换工具）

### Q10：MCP（Model Context Protocol）是什么？解决什么问题？
**参考答案：**
MCP 是 Anthropic 提出的**标准化工具调用协议**，类比"AI 应用的 USB-C 接口"。

**解决的问题**：
- 每个 AI 工具都自己写适配层（M×N 集成复杂度）
- 工具发现机制不统一
- 权限和安全边界不清晰

**核心概念**：
- **MCP Server**：工具提供方，暴露资源/工具/提示词
- **MCP Client**：AI 应用（如 Claude Code），发现并调用 server 的工具
- **标准化传输**：stdio / HTTP SSE

### Q11：在 SDD 中 MCP 的正确使用边界是什么？
**参考答案：**
MCP 是**开发期辅助工具**，不是回归测试本身。

| 能做（开发期） | 不能做（CI/回归期） |
|------|------|
| 让 AI 查测试库验证写入数据 | 替代 CI 自动化测试 |
| 失败用例排查定位 | 当作回归断言源 |
| 评审 PR 时对照 schema | 替代 testcontainers/fixture 体系 |

**铁律**：MCP 应出现在排查阶段（步骤 4），不是 CI 阶段（步骤 2）。

## 五、Agent 评估（⭐⭐）

### Q12：如何评估一个 Agent 的质量？
**参考答案：**

| 维度 | 指标 | 评估方法 |
|------|------|----------|
| 任务成功率 | 完成率% | AgentBench / WebArena |
| 效率 | 平均步数、耗时 | 与基线对比 |
| 工具使用准确率 | 正确调用% | 手工标注 + 自动校验 |
| 鲁棒性 | 异常输入下的表现 | 对抗测试 |
| 成本 | Token 消耗、API 费用 | 单任务成本核算 |
| 用户满意度 | NPS/CSAT | 人类评估 |

### Q13：Agent 的幻觉有哪些典型表现？如何防范？
**参考答案：**

| 幻觉类型 | 表现 | 防范措施 |
|---------|------|---------|
| 工具调用幻觉 | 调用不存在的工具/参数 | Schema 校验 + 重试机制 |
| 事实幻觉 | 编造数据/引用 | RAG 检索增强 + 事实核查 |
| 规划幻觉 | 制定了不可执行的计划 | 可执行性校验 + 依赖检查 |
| 边界幻觉 | spec 未覆盖处自由发挥 | AGENTS.md 硬约束 + 反问机制 |

## 六、架构设计实操（⭐⭐⭐）

### Q14：从零设计一个 SDD Agent，核心架构怎么搭？画出模块图
**参考答案：**
```
用户需求
  ↓
[需求澄清 Agent] → 需求摘要
  ↓
[Spec 生成 Agent] → spec.md draft
  ↓
[评审 Agent] → 评审意见 → spec v1.0 [APPROVED]
  ↓
[规划 Agent] → 实现计划 + 任务分解
  ↓
[实现 Agent × N] → 并行写代码（前端/后端/DB）
  ↓
[测试 Agent] → AC 对应测试
  ↓
[归档 Agent] → CHANGELOG + 索引更新
```

**关键设计**：
- Agent 之间通过**冻结的 spec 文件**通信，不共享 raw context
- 不同阶段用不同档位模型（澄清→重推理档，实现→平衡档，归档→快速档）
- 评审 Agent 必须独立会话、最好换模型

### Q15：LangChain 手写 Agent vs 框架 Agent vs 原生 Tool Use 怎么选？
**参考答案：**

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| 原生 Tool Use | 无框架依赖、轻量 | 需自己写编排逻辑 | 简单场景 |
| LangChain | 生态丰富、抽象好 | 抽象泄漏、版本不稳定 | PoC / 快速原型 |
| 框架 Agent（CrewAI/AutoGen） | 多 Agent 协作开箱即用 | 黑盒、定制化受限 | 中等复杂度 |
| 自研编排 | 完全可控、可优化 | 开发成本高 | 生产级系统 |

**推荐**：PoC 用框架，生产用原生 Tool Use + 自研编排。

## 七、前沿趋势（⭐）

### Q16：2025 年 Agent 领域最重要的技术趋势是什么？
**参考答案：**
1. **从单 Agent → 多 Agent 系统**：CrewAI / AutoGen / LangGraph 等编排框架成熟
2. **MCP 协议标准化**：统一工具调用接口，降低集成成本
3. **Agent 操作系统化**：Agent 作为底层运行时，而非上层应用
4. **SDD + Agent**：规范驱动开发的 Agents 化——Agent 不再需要人手写 spec，而是 Agent 帮你写 spec、审 spec
5. **上下文工程**：Prompt Caching / 分层加载 / 子 Agent 隔离等上下文管理技术

### Q17：多 Agent 协作中，Agent 之间的通信协议如何设计？
**参考答案：**
- **内容格式**：Markdown + YAML/JSON block（厂商中立）
- **版本锁定**：引用 spec 版本号而非 raw 内容
- **交接清单**：每个 Agent 完成后的产出物路径 + 待办清单
- **异步通信**：通过 Git PR / 文件而非内存共享（多个 Agent 可并行）
- **铁律**：不使用某厂商专属格式（如 Claude 的 `<thinking>` 标签在其他模型行为不同）
