# 03 — SDD 流程与实践面试题

> 覆盖六阶段流程、跨栈协作、Task 管理、feature flag、Expand-Contract、契约层、环境晋级

---

## 一、六阶段流程（⭐⭐）

### Q1：SDD 六阶段流程及各阶段产出物？
**参考答案：**

| 阶段 | 产出物 | 检查点 |
|------|--------|--------|
| 1. 需求分析 | 需求摘要（5-10 行） | 非目标已写？成功指标可度量？ |
| 2. 规范定义 | spec.md（9 模块） | 模糊词已替换？AC 可机械验证？ |
| 3. 评审确认 | 评审意见清单 + spec 新版本 | 至少发现 ≥1 个问题？ |
| 4. 实现编码 | 代码 + 单元测试 | 所有 AC 有对应测试？ |
| 5. 验证测试 | 测试报告 | AC 全绿？边界已测？ |
| 6. 归档沉淀 | 归档 spec + CHANGELOG + ADR | 半年后能回答"为什么这么做"？ |

### Q2：各阶段的进入/退出条件是什么？
**参考答案：**

| 阶段 | 进入条件 | 退出条件 | 失败处理 |
|------|----------|----------|----------|
| 需求分析 | 有原始需求 | 摘要≤10行、非目标已写 | 补充澄清 |
| 规范定义 | 需求摘要已确认 | 9模块齐全 | 回分析补信息 |
| 评审确认 | spec draft 存在 | 所有评审意见已处置 | 重写 spec |
| 实现编码 | spec approved | 代码通过自测 | 回规范定义修spec |
| 验证测试 | 代码已提交 | AC 全绿 | 修代码/修spec分流 |
| 归档沉淀 | 测试已通过 | spec入库、索引已更新 | 补齐材料 |

### Q3：哪些场景可以跳过某阶段？判断标准是什么？
**参考答案：**

| 场景 | 可跳过 | 不能跳 |
|------|--------|--------|
| 个人玩具项目 | 评审确认 | 自审 |
| Bug 紧急修复 | 完整 spec→5行 mini-spec | 验证 |
| 小型 UI 调整 | 规范定义（事后补） | — |
| 原型探索 | 规范定义+实现编码+评审 | 需求分析+归档 |

**判断标准**：是否存在信息丢失的风险——工作可能被其他人/未来的自己/AI 误解，此阶段不该跳。

## 二、跨栈协作（⭐⭐⭐）

### Q4：一份 spec 怎么管前端+后端+DB 三层？
**参考答案：**
推荐 **"主 spec + 契约 + 子 spec"** 结构：

```
docs/specs/0002-phone-login/
├── 0002-main.md        # 主 spec：端到端行为 + E2E AC
├── contracts/
│   ├── openapi.yaml    # API 契约
│   ├── schema.sql      # DB 变更
│   └── events.json     # 事件契约（可选）
├── 0002-frontend.md    # 前端子 spec
├── 0002-backend.md     # 后端子 spec
└── 0002-db.md          # DB 子 spec
```

**铁律**：子 spec 引用契约，不复制字段定义——复制一定会漂移。

### Q5：跨栈场景的 AC 怎么写才不是"半吊子 E2E"？
**参考答案：**
必须包含**三段式期望**：

```
[AC-E2E-N] <用户动作描述>
前端: <UI 应呈现什么状态>
后端: <哪个接口 + 状态码 + 响应体关键字段>
数据: <哪张表/哪些字段/期望值/不应存在的记录>
```

**铁律**：三类断言缺一条就是"半吊子 E2E"。最危险的是"前端绿 + 后端绿 + DB 红"。

### Q6：契约层为什么是跨栈 SDD 的命门？
**参考答案：**
- API 契约（OpenAPI）→ 前后端对齐的单一事实源
- DB 契约（schema/migration）→ DDL 变更的唯一真源
- 事件契约（events.json）→ MQ 上下游的通信约定

**铁律**：契约变更走**独立 PR**，先合入契约 PR，再并行开发实现 PR。这使三端真正解耦。

### Q7：多仓 SDD 的适用条件？不适用时强行上的代价？
**参考答案：**
**适用条件**（满足任一条）：
- 前后端/DB 由不同团队拥有，且有独立 release cadence
- spec 评审周期与代码评审周期差异显著
- 跨团队 spec 变更的"隐性传播"已成故障源
- 项目周期 ≥1 年且预期 ≥3 次破坏性变更

**不适用时强上的代价**：平白增加 5 倍 PR 编排成本，spec 独立成仓成为组织瓶颈而非加速器。

## 三、Task 管理（⭐⭐⭐）

### Q8：Task / Epic / spec tag 三个概念的区别？
**参考答案：**

| 概念 | 定义 | 关系 |
|------|------|------|
| spec tag | spec 仓的语义化版本号（vMAJOR.MINOR.PATCH） | 一个 tag 可含多个 Task |
| Epic | 跨仓 PR 的父协调问题 | 一个 Epic 可含 1 或多个 Task |
| Task | 最小可部署单元 | 简单: 1 Epic = 1 Task；破坏性: 1 Epic ≥ 2 Task |

### Q9：Task 编号体系怎么设计？为什么必须"全位置一致"？
**参考答案：**
- 格式：`Task<NNN>`（三位数字，001 起连续递增，不复用）
- 出现位置：issue 标题、PR 标题、迁移脚本文件名/头注释、代码追溯注释、feature flag 名、commit 消息行首

**好处**：从任何工件（脚本/代码/flag/commit）都能反查到 Task-ID，再反查到 spec 章节和讨论。

### Q10：什么时候需要 Task 轨迹文档？它和 issue tracker 的分工？
**参考答案：**
**必须留轨迹**的条件（满足任一）：
- 涉及 DDL / 跨 ≥2 仓 / 引入或移除 feature flag / 走 Expand-Contract / 改 BR 或契约

**分工**：
- Issue tracker：承载日常讨论
- Task md：承载最终决策与轨迹（决策了什么、为什么、偏差记录）

## 四、Expand-Contract 与 Feature Flag（⭐⭐⭐）

### Q11：什么是 Expand-Contract 模式？什么场景必须走？
**参考答案：**
**定义**：破坏性 DDL 拆成至少 2 个 Task 的发布模式。

```
[Expand] 加新结构(可空) → [过渡期] 双写+回填 → [Contract] 清理旧结构
```

**必须走的场景**：
- 加 NOT NULL 列（无可接受默认值）
- 改列名/类型（缩范围）
- 删列/删表
- 改主键/唯一键
- 加唯一约束（已有数据可能违反）

### Q12：为什么不能"单步 RENAME"？
**参考答案：**
- 部署窗口内"旧代码+新 DDL"或反向半态，旧代码找不到旧列直接崩溃
- 主从复制期间可能短时不一致
- 回滚成本极高

**Expand-Contract 的本质是用时间换安全**。

### Q13：Feature Flag 的 5 条核心纪律是什么？
**参考答案：**
1. 命名含 Task-ID：`feat_task<NNN>_<snake_case>`
2. 默认关闭（代码默认 false）
3. DDL 只负责"能力存在"，flag 控制"功能可见性"
4. flag 移除必须是独立 Task
5. 最晚 2 个 spec 版本内移除已全开的 flag

### Q14：选择性发布的三种打法？主推哪种？
**参考答案：**

| 打法 | 做法 | 评价 |
|------|------|------|
| α. flag + 共同上 PROD | DDL additive，都合 main 上 PROD，flag 只开需要的 | ⭐⭐⭐ 主推 |
| β. DDL 留分支 | 不合 main，长期 rebase | ⚠️ 分支腐、版本冲突 |
| γ. 环境分叉目录 | migration/prod/ vs migration/dev/ | ❌ 禁用 |

## 五、环境晋级（⭐⭐）

### Q15：标准环境阶梯是什么？各环境的 flag 策略？
**参考答案：**

| 环境 | flag 状态 | 职责 |
|------|----------|------|
| dev | 全开 | 个人开发/本地联调 |
| SIT | 当前 main 全部 Task flag 全开 | 跨仓集成/QA 全量验证 |
| UAT | 本轮晋级被批准的 flag 开 | 业务方验收 |
| PROD-mirror | 与 PROD 计划一致 | 灰度前最后验证 |
| PROD | 本次发布批准的最小子集 flag 开 | 终态 |

### Q16：环境晋级的五条不变量是什么？
**参考答案：**
1. 所有环境跑**同一个 build artifact**
2. 所有环境跑**同一组迁移脚本**
3. 所有环境 spec 真源指向**同一个 tag**
4. 唯一变化的是**环境配置**（feature flag 为主）
5. **回滚靠关 flag，不回滚 DDL/build**

## 六、渐进落地（⭐⭐）

### Q17：团队刚起步，SDD 怎么渐进式落地？
**参考答案：**
```
Week 1:   给自己的任务写 mini-spec（别人看不看都写）
Week 2:   拉一位同事 peer review spec
Week 3-4: 试点模块把 spec 作为合入门槛
Month 2:  写入 AGENTS.md，让 AI 督促全员
Month 3:  复盘数据（有 spec 的 feature vs 没 spec 的 feature）
```

**关键原则**：试点→数据→推广；避免行政命令铺开。

### Q18："spec 写得不合格"的 5 个信号？
**参考答案：**
1. 读完不知道边界在哪
2. AC 数量少于功能项
3. 出现"等"、"等等"、"诸如此类"
4. 接口只写正常路径、无错误码
5. 非功能需求全是定性词（"快"、"稳"、"好"）
