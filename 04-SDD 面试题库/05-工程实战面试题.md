# 05 — 工程实战面试题

> 覆盖 Token 成本、测试策略、Brownfield 改造、AC 设计、MCP 边界、反模式与常见误区

---

## 一、Token 成本与决策（⭐⭐）

### Q1：SDD 会不会很费 Token？完整成本怎么算？
**参考答案：**
**单次会话 token 确实更高，但全生命周期未必更贵。**

| 阶段 | Vibe Coding | SDD |
|------|-------------|-----|
| 需求澄清 | 低（隐藏在多轮"其实我还要"中） | 中（一次性做透） |
| 初次实现 | 低 | 高（带 spec 生成） |
| 返工/跑偏重写 | **高**（反复打补丁） | 低（spec 锁定方向） |
| 二次维护 | 高（读代码猜意图） | 低（读 spec） |
| 跨人协作 | 高（每个人重新教 AI） | 低（spec 一份多人复用） |
| **合计** | 隐性累积，易超支 | 前期集中，后期衰减 |

### Q2：Token 优化的 8 条实战清单是什么？
**参考答案：**
按 ROI 从高到低：
1. **启用 Prompt Caching**——缓存 AGENTS.md+稳定 spec，命中后成本约 1/10
2. **分阶段开会话**——澄清/评审/实现拆开，每个只载入必要章节
3. **引用文件不粘贴内容**——让 AI 自己 Read 文件
4. **分层加载 spec**——需求阶段看 §1-3，实现阶段看 §5-9，验收阶段看 §9
5. **AGENTS.md 精简**——只留硬约束，教程外链
6. **子 Agent 隔离跑评审**——不污染主会话
7. **规模分档**——个人脚本用 10 行 mini-spec
8. **版本稳定期集中产出**——spec approved 后批量生成实现

### Q3：什么场景下 SDD 的 Token 投入反而不划算？
**参考答案：**
- 一次性脚本 / 个人快速 PoC
- 纯探索性实验（边界都没想清楚）
- <10 行的独立小改动
- 预计后续不会再有 ≥2 次编辑

**判据**：若预计后续至少还有 2 次对这块代码的编辑/扩展，或至少 2 个人会协作，SDD 就能回本。

## 二、测试策略（⭐⭐⭐）

### Q4：SDD 下如何用 AC 编号区分测试类型？
**参考答案：**

| 前缀 | 含义 | 对应测试层级 |
|------|------|-------------|
| AC-N | 功能 AC | 单元/集成测试 |
| AC-E2E-N | 端到端 AC | E2E 测试 |
| AC-NFR-N | 非功能/性能 AC | 压力/基准测试 |

**铁律**：测试方法名必须含 AC 编号（如 `@DisplayName("AC-NFR-1: ...")`），CI 失败报告才能直接定位到 spec 章节。

### Q5：L1~L4 验收层级是什么？为什么需要分层？
**参考答案：**

| 层级 | 名字 | TC 前缀 | 跑在哪 |
|------|------|--------|--------|
| L1 | 单元 | TC-U-* | mvn/npm test |
| L2 | 集成 | TC-IT-* | testcontainers |
| L3 | 前端 | TC-FE-* | Vitest/Jest |
| L4 | E2E | TC-E2E-* | Playwright/Cypress |

**分层价值**：E2E AC 必须至少有 L4，不能用一堆 L1+L2 凑数；层级失衡（E2E>30%）说明下层 AC 没写清楚。

### Q6：三维追溯矩阵（BR × 模块 × TC）是什么？怎么用？
**参考答案：**

```
                模块（FE-01/FE-02/BE-01/DB-01...）
         ┌────────┬────────┬────────┬────────┐
BR-01    │ TC-FE-1│ ——     │ TC-IT-1│ ——     │
BR-02    │ TC-FE-2│TC-E2E-1│ TC-IT-2│ TC-U-5 │
BR-03    │ ——     │ ——     │ TC-IT-4│ ——     │
         └────────┴────────┴────────┴────────┘
```

- **行**：业务规则编号（BR-*）
- **列**：模块编号（FE-*/BE-*/DB-*）
- **单元格**：覆盖的 TC 编号
- **空白单元格 = 覆盖率漏洞**，review 时一眼看到

### Q7：压力测试在 SDD 下怎么写？AC-NFR 的四要素？
**参考答案：**
每条 AC-NFR 必须四要素齐全：**指标 + fixture 规模 + 工具 + 判定方式**

```
[AC-NFR-1] 100 万行 user 表 fixture 下，查询接口 P95 < 500ms（JMH，3 次平均）
[AC-NFR-2] 1000 QPS 持续 10 分钟，错误率 < 0.1%（Gatling 场景 scenario-login）
```

**不合格→合格改写**：
- "响应要快" → "P95 < 500ms，P99 < 1s"
- "支持高并发" → "1000 QPS 持续 10 分钟，错误率 < 0.1%"

### Q8：回归测试的 AC 池怎么管理生命周期？
**参考答案：**

| spec 状态 | AC 是否入池 | 说明 |
|----------|-------------|------|
| [DRAFT] | 否 | AC 会变 |
| [APPROVED]/[RELEASED] | **是** | 每次 CI 必跑 |
| [DEPRECATED] | 移除 | 留着变噪音 |
| [RELEASED] v2.0 | 精选保留 | 向后兼容 AC 保留，被破坏的显式清理 |

**反模式**：AC 池只增不减、性能 AC 与功能 AC 混跑、破坏性变更没清理老 AC。

## 三、Brownfield 改造（⭐⭐⭐）

### Q9：Brownfield（存量项目）SDD 化的三种切入模式？
**参考答案：**

| 模式 | 做什么 | 成本 | 适合 |
|------|--------|------|------|
| A. 只约束新功能 | 存量不动，新 feature 走 SDD | 低 | 团队刚起步 |
| B. 契约层先固化 | 导出 OpenAPI/DB schema/事件 schema | 中 | 跨团队协作 |
| C. 借重构做考古 | 结合已计划的重构倒推 spec | 高 | 本来就要动的模块 |

**推荐顺序**：A → 2 周后加 B → 3 个月后遇契机做 C。同时推 B+C 是最常见翻车姿势。

### Q10：Spec 考古的 5 步法是什么？它的价值与局限？
**参考答案：**
1. 抽入口/API 清单
2. 读测试（测试覆盖就是隐性 AC，一级金矿）
3. 翻 PR 与访谈（git blame→PR→原作者对话）
4. 对照线上行为（监控/日志中的真实调用模式）
5. 产出"现行行为 spec"（先还原现状，不在此步优化）

**价值**：拦截下一次误改。**局限**：只能还原 WHAT，很难还原 WHY。

**铁律**：考古 spec 必须标 `状态:[REVERSE-ENGINEERED]`，与绿场 spec 区分权威性。

### Q11：什么时候不值得做 Spec 考古？
**参考答案：**
- 模块 3 个月内会被重写或下线
- 代码量 < 500 行且语义明显
- 原作者还在团队能随时答疑（人本身就是 spec）
- 预估考古成本 > 重写成本

## 四、MCP 边界与安全（⭐⭐）

### Q12：DB MCP 接入的安全约定有哪些？
**参考答案：**
必须写入 AGENTS.md：
1. 只连测试/本地库，严禁连生产或共享 staging
2. 默认只读；写权限仅限白名单场景
3. 禁止 DROP/TRUNCATE/无 WHERE 的 DELETE（在 MCP server 层白名单拦截）
4. 敏感字段必须先脱敏再进上下文
5. MCP 查询结果不写回 spec 作为事实源
6. 每次 session 只查当前任务相关的表

### Q13：MCP 在回归流程的正确位置？
**参考答案：**
```
1. 改动触发 CI
2. CI 跑自动化测试         ← 真正的回归
3. 结果：绿→合入 / 红→进 PR
4. 排查阶段 → MCP 出场    ← MCP 的正确位置
5. 修复 → 回到 1
```

**MCP 应出现在步骤 4，不是步骤 2。弄反了就是用 AI 代替 CI——目前不该这么干。**

## 五、反模式与常见误区（⭐⭐）

### Q14：SDD 的 4 类典型反模式？
**参考答案：**
1. **过度设计**：未来扩展、5 种备选、预留 8 个参数
2. **写得太模糊**：等、等等、诸如此类、快、稳、好
3. **写了不遵守**：spec 和代码渐行渐远
4. **只有 AI 看 spec**：人懒得看，SDD 退化

### Q15：人机协作的两类极端 + 边界模糊各是什么？
**参考答案：**

| 极端 | 症状 | 后果 |
|------|------|------|
| 人过度代理给 AI | "我没主意你看着办"、AI 写完不审就合 | 方向跑偏、责任真空 |
| 人过度保留 | 凡事先自己做再让 AI"检查" | 失去 SDD 收益，AI 沦为 IDE 补全 |
| 边界模糊 | spec [APPROVED] 不明确谁批、AI 改 spec 但不显式说 | 出事追不到人、团队拿着新版但不知道 |

### Q16：人机协作的 5 条铁律？
**参考答案：**
1. 决策必有人——任何 trade-off/scope 变更/[APPROVED] 要落到具体人
2. AI 不能审 AI 自己写的代码——评审独立会话、最好换模型
3. AC 是人定的契约——AI 永远不能改 AC 让测试通过
4. 契约变更必须人发起——API/schema.sql/events 的修改 PR 由人提出
5. 责任落到人头上——commit author/PR reviewer/release approver 都是人

## 六、项目级工程约束（⭐⭐）

### Q17：AI 代码可读性的 5 层防线是什么？SDD 能覆盖哪几层？
**参考答案：**

| 层 | 机制 | SDD 能帮？ |
|---|------|-----------|
| 1 | AGENTS.md 编码约束 | ✅（约束文件本身就是 SDD 的一部分） |
| 2 | Linter/Formatter | ❌ |
| 3 | Spec + AC 测试 | ✅（需求层可读性兜底） |
| 4 | 人工 Code Review | ❌（SDD 不免除 review） |
| 5 | ADR/决策注释 | ✅（记录关键"为什么"） |

**核心认知**：SDD 最大的价值是让未来维护者永远有一份高于代码层的"原始意图"可参照。干净代码要靠约束文件+Linter+Review 的组合拳。

### Q18：AGENTS.md 中对 AI 代码可读性必须写死的几条？
**参考答案：**
- 不要为假想未来做抽象
- 不要在内部代码加防御性校验（只在系统边界校验）
- 函数超 N 行或嵌套超 M 层时，停下来讨论
- 命名沿用项目已有模式
- 新增依赖前必须问
- 注释只写"为什么"，不写"做什么"
- 不改与当前任务无关的代码（no drive-by refactor）

### Q19：约束 spec（conventions）9 支柱是什么？什么时候需要？
**参考答案：**

| 支柱 | 文件名 | 核心内容 |
|------|--------|---------|
| 前端 | fe-conventions.md | 目录/组件命名/HTTP 客户端 |
| DB | db-conventions.md | 表/列/索引/迁移命名 |
| 测试 | test-conventions.md | TC 编号体系 |
| 安全 | security-conventions.md | 输入校验/脱敏/审计 |
| 监控 | monitoring-conventions.md | Metrics/Logs/Traces/SLO |
| API 版本化 | api-versioning.md | 路径策略/deprecation |
| 异步任务 | async-job-conventions.md | 幂等/重试/超时 |
| 国际化 | i18n-conventions.md | 文案 key/时区 |
| 依赖策略 | dependency-policy.md | 锁版本/CVE 响应 |

**需要条件**：团队 ≥3 人或项目 ≥6 个月→至少上 fe/db/test/security 四份。
