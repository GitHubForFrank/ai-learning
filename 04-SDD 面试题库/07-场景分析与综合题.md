# 07 — 场景分析与综合题

> 开放场景题、架构设计题、故障分析题，适合高级/架构师岗的深度面试

---

## 一、架构设计题（⭐⭐⭐）

### Q1：设计一个"手机号登录"功能的完整 SDD Agent 工作流
**场景**：用户说"我们要支持手机号验证码登录"，你作为 SDD Agent 架构师，设计从需求到上线的完整 Agent 工作流。

**参考答案要点**：
1. **需求澄清 Agent（重推理档）**
   - 问出：验证码有效期？位数？短信渠道？并发限制？失败锁定策略？
   - 产出：需求摘要（场景/目标/非目标/成功指标）

2. **Spec 生成 Agent（平衡档）**
   - 产出主 spec + 契约：openapi.yaml（send-code/verify 两个接口）+ schema.sql（user_auth_phone 表 + verification_code 表）
   - 跨栈拆分子 spec：frontend（登录页/倒计时/错误态）、backend（短信集成/校验逻辑）、db（迁移策略/索引）

3. **评审 Agent（重推理档 × 3）**
   - 杠精找漏洞、安全专家审短信通道防刷、新人找模糊点

4. **实现 Agent × 3（平衡档，并行）**
   - 前端实现（按前端子 spec + openapi.yaml）
   - 后端实现（按后端子 spec + openapi.yaml）
   - DB 迁移脚本（按 db 子 spec）

5. **测试 Agent（平衡档）**
   - 翻译 AC→测试代码（L1 单元+L2 集成+L4 E2E）
   - 三段式 E2E：UI 断言（倒计时/错误提示）+ 接口断言（200/429）+ DB 断言（verification_code 表新行）

6. **归档 Agent（快速档）**
   - CHANGELOG + spec 索引更新

**关键决策点**：
- 契约 PR 先合入（openapi.yaml + schema.sql），三端再并行
- 所有 Agent 之间通过冻结 spec/契约文件交接，不传 raw context

### Q2：设计一个存量电商系统上 SDD 的 6 个月路线图
**场景**：200 万行 Java 代码，15 人团队，从未用过 SDD，每月 5-10 个 feature。

**参考答案要点**：

| 时间 | 动作 | 里程碑 |
|------|------|--------|
| Week 1-2 | 全栈盘点（前端路由/后端接口/DB 表/基础设施） | audit 报告成型 |
| Week 3-4 | 导出 OpenAPI + DB schema baseline + 事件 schema | 契约层固化成文 |
| Month 2 | 选一个**新 feature** 走完整 SDD | 首份标准 spec + 模板 |
| Month 3 | 挑**一个高价值模块**做 spec 考古（试点） | 首份 REVERSE-ENGINEERED spec |
| Month 4 | AGENTS.md 沉淀团队约束，AI 开始主动拦截 | 约束文件入库 |
| Month 5 | 契约变更走独立 PR，与实现 PR 解耦 | 契约 PR 成为常态 |
| Month 6 | 复盘：哪些 spec 救场、哪些是文档债 | 第二版落地策略 |

**关键决策**：
- 第一天别贪大，A（新功能 SDD）→B（契约层）→C（考古）顺序
- 不是所有模块都值得考古（判据：代码量/原作者是否在/是否即将重写）
- 最大的风险不是做得不完美，而是半途而废留一堆过期 spec

## 二、故障分析题（⭐⭐⭐）

### Q3：线上 Bug 复盘——spec 有，代码也有，为什么还是出了事故？
**场景**：一个"优惠券领取"功能，spec/代码/测试三全，但上线后出现：用户可以重复领取同一张优惠券，前端按钮置灰但 API 未做幂等校验。

**问题分析**：
1. spec §5 接口定义只写了"POST /coupons/claim 返回 200"，**没写幂等性**
2. AC 只写了"点击领取→弹窗提示成功"，**缺并发/重复请求场景**
3. 测试只有单次请求的用例，**没覆盖"同一用户快速双击按钮"的边界**
4. 代码按 spec 实现了，但**spec 本身有漏洞**

**反思方向**：
- SDD 不能消除所有 bug，只能把 bug 的根因从"实现偏差"变成"spec 不完整"
- 接口定义必须含幂等性说明
- AC 必须覆盖异常路径和边界场景
- 评审阶段应该能抓到——杠精角色审接口时应该追问"同一用户重复请求会怎样"

---

### Q4：Spec 和代码冲突，线上告急，怎么处理？
**场景**：一个支付接口的 spec 说"金额字段为 BigDecimal，精确到分"。但代码实现时用了 Double 类型（实现者 AI 没严格按 spec），生产环境出现 0.01 元误差，客户投诉。

**处理步骤**：
1. **先止血**：关 feature flag（如果有），或紧急修复代码
2. **定位根因**：代码用了 Double（AI 的实现偏差），spec 是正确的
3. **修复**：按 spec 重写（改为 BigDecimal），加精度测试
4. **预防**：
   - AGENTS.md 加"金额类字段必须用 BigDecimal"
   - AC 加"金额精度校验：1.01+2.02=3.03（不是 3.0300000000002）"
   - PR review 时关注金融计算类型

**SDD 原则**：以 spec 为准，代码回退；若代码实现更合理→先改 spec 再改代码，顺序不可颠倒。

## 三、方案对比题（⭐⭐⭐）

### Q5：单仓 SDD vs 多仓 SDD 的选择
**场景**：一个前后端分离项目，前端 Vue 3（3 人），后端 Spring Boot（5 人），DB 由后端团队兼管。项目已跑 2 年，每月 3-5 个 feature。现在要上 SDD。

**分析**：
- 团队虽然前后端分离，但**不满足多仓 SDD 的任一判据**：
  - 没有不同 release cadence（需求一起上线）
  - spec 评审和代码评审周期没有显著差异
  - 跨团队隐性传播不是故障源（本来就是同一团队）
- **结论**：单仓"主 spec + 契约 + 子 spec"结构足够，不要上多仓 SDD
- 不满足判据强行多仓 = 平白增加 5 倍 PR 编排成本

---

### Q6：同一需求，Vibe Coding vs SDD——3 个月后回头看
**场景**：一个"用户修改邮箱"功能，假设你用两种方式各做一次，3 个月后需要加"邮箱变更通知"功能。

| 维度 | Vibe Coding | SDD |
|------|-------------|-----|
| 原代码在哪 | `grep "email"` 找到 3 个文件、1 个散落函数 | `grep "spec 0003"` 找到 spec 文件 + 所有引用代码 |
| 原需求是什么 | 翻 PR 描述"fix email stuff" | 读 spec §1-3 |
| 有哪些边界 | 不知道，靠读代码猜 | 读 spec §9 AC + 对应的测试 |
| 改造成本 | 中高（要读代码+猜意图+怕踩旧坑） | 低（读 spec+AC→新需求→改 spec→让 AI 实施） |
| 总时间 | 约 4h | 约 1.5h |

## 四、角色冲突题（⭐⭐⭐）

### Q7：产品经理说"这个需求很急，别写 spec 了直接让 AI 写代码"
**场景**：周五下午 5 点，产品经理说周一要给老板 demo 一个新功能，要求跳过 SDD 直接上 Vibe Coding。

**你的回答思路**：
1. **判断场景**：这是 demo 还是上线？demo→可以 Vibe（探索性原型不在 SDD 范围内）；上线→不能跳
2. **压缩方案**：如果确实紧急且要上线，走"mini-spec 5 行"（bugfix 级别），而不是跳过 spec
3. **事后补**：如果只能先上线，承诺周一补 spec，纳入 review
4. **SDD 原则**：可以压缩 spec 的长度，不能压缩 spec 的存在——**验证绝不能省**

---

### Q8：团队成员"不信 AI 写的代码"，全部推翻自己重写
**问题分析**：
- 这是"人过度保留"的反模式
- 失去 SDD 大半收益——AI 沦为 IDE 补全
- 浪费 Token + 人力，团队疲惫

**建议对治**：
1. 数据说话：做一次对照实验（AI 写+人审 vs 人全写，对比效率和质量）
2. 分档信任：非核心→AI 写+轻 review；核心→AI 起草+人重审
3. 接受 70/30 原则：接受 AI 70% 的输出+修订 30%，比 100% 自写快

## 五、综合权衡题（⭐⭐⭐）

### Q9：一个 Task 涉及前端+后端+DB，spec 里发现一个矛盾怎么办？
**场景**：实现 Agent 在写代码时发现 spec §5 接口返回字段是 `phoneNumber`（驼峰），但 contracts/openapi.yaml 里写的是 `phone_number`（下划线）。

**正确处理流程**：
1. **停手**——不猜测，不"先按一个写着"
2. **反问**：把矛盾反馈给 spec owner，"§5 和 openapi.yaml 不一致，以哪个为准？"
3. **修 spec 先**：哪个错了改哪个，改完后打新版本号
4. **再继续实现**：拿到修正后的 spec/契约，继续写代码

**错误处理**：
- "我猜是 openapi 为准，直接按 openapi 写了"→ 可能猜错，返工
- "两个都写上吧"→ 数据漂移，未来有人以 spec 为准就崩
- "先按 spec 写，等 review 再说"→ review 可能没发现，bug 上线

### Q10：评估一个候选人的 SDD/Agent 综合能力，你会出什么面试题？
**参考答案**：
"给你一个真实场景：一个 3 人团队维护的 5 万行 Java 电商后端（订单/库存/支付），没有 spec，没有测试。现在要加'部分退款'功能（原来只支持全额退款）。请设计一套从需求到上线的方案，包括：

1. 用什么策略切入 SDD（三种切入模式选哪种？）
2. 需要哪些 Agent 角色、各用哪档模型？
3. spec 怎么拆（主 spec/子 spec/契约层）？
4. 哪些场景需要 AC-E2E，怎么写成三段式？
5. 如果涉及 DDL 变更（退款表加列），走什么模式？
6. 上线后发现和另一个已有功能冲突，怎么排查和决策？

不需要完整写 spec，但要讲清楚**决策逻辑和每一步为什么这么选**。"
