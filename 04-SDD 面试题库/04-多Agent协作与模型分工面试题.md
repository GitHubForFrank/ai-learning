# 04 — 多 Agent 协作与模型分工面试题

> 覆盖多 Agent 拓扑、模型三档分工、人机职责分离、角色 agent 套件、跨模型协作

---

## 一、模型分工基础（⭐⭐）

### Q1：SDD 中模型按能力分几档？各自定位？
**参考答案：**

| 档位 | 代表（仅示例） | SDD 角色 | 单价/速度 |
|------|---------|----------|-----------|
| 重推理档 | Claude Opus / GPT-5 / Gemini Pro | "架构师+杠精" | 最贵/最慢 |
| 平衡档 | Claude Sonnet / GPT-5-mini / Gemini Flash | "主力工程师" | 中等 |
| 快速档 | Claude Haiku / GPT-5-nano | "实习生+文档秘书" | 最便宜/最快 |

**判别标准**：任务需同时持有 ≥5 个相互依赖的事实→重推理档；决策点 ≤3→平衡档；单步确定性→快速档。

### Q2：SDD 六阶段各用哪档模型？
**参考答案：**

| 阶段 | 推荐档位 | 原因 |
|------|---------|------|
| 需求分析 | 重推理档 | 识别歧义、刻意反问、补全边界 |
| Spec 生成 | 平衡档 | 模板填空、结构化生成 |
| Spec 评审 | 重推理档 | 对抗性挑刺、多视角 |
| 实现编码 | 平衡档 | 按 spec 翻译为代码 |
| 验证测试 | 平衡档（默）→重推理档（性能诊断） | AC↔用例映射 |
| 归档沉淀 | 快速档 | 摘要、CHANGELOG、索引 |

**速记口诀**：澄清和评审升档，写码和测试守档，归档降档。

### Q3：什么信号必须升档？什么信号应该降档？
**参考答案：**

**升档信号**：
- AI 连续 ≥3 次跑偏
- 任务跨 ≥3 个文件/模块/子系统
- 涉及并发/分布式/安全/合规
- spec 评审/架构决策节点

**降档信号**：
- 任务变成"按模板填空"
- spec 已锁死，纯翻译工作
- 重复批量操作
- 归档/摘要/索引类任务

**伪信号**："这次比较重要"≠需要升档（重要≠复杂）；"我不放心"≠升档（应补 review 流程）

## 二、多 Agent 协作拓扑（⭐⭐⭐）

### Q4：SDD 中有哪四种多模型协作拓扑？
**参考答案：**

| 拓扑 | 形态 | 何时用 | 代价 |
|------|------|--------|------|
| 单模型贯穿 | 同一模型跑全程 | mini-spec/个人脚本 | 贵+上下文污染 |
| 主从模式（推荐起点） | 重推理档只在关键节点出场 | 中型 feature | 交接要通过文件 |
| 角色专家组 | 每阶段一个独立 Agent | 跨栈大需求/合规 | 编排成本 |
| 选举/裁决 | N 个模型出方案，重推理档裁决 | 架构选型/安全决策 | ≥3 倍成本 |

### Q5：多 Agent 之间如何交接信息？为什么不能传 raw context？
**参考答案：**
- **正确方式**：通过冻结的 spec 文件/需求摘要/契约文件交接
- **禁止方式**：把对话历史一股脑丢给下一个 Agent

**原因**：
- raw context 携带生成 Agent 的偏见和错误
- 下一个 Agent 会继承前一个的"思维惯性"
- 上下文越滚越脏，需要的信息被噪声淹没
- 跨模型时 raw context 包含厂商特定 token

### Q6：为什么评审 Agent 必须独立会话且换模型？
**参考答案：**
- 同一模型刚生成的 spec，自审会有"自己写的总觉得对"的偏向
- 同一会话上下文有自己的生成痕迹，会被自己骗
- 跨厂商评审能识别单厂商训练偏差导致的盲区

**推荐**：3 个独立会话——杠精（重推理档）+ 安全专家（重推理档）+ 新人（平衡档）。

### Q7：5+1 个角色 Agent 套件是哪些？各自的红线？
**参考答案：**

| Agent | 上场时机 | 核心红线 |
|-------|---------|---------|
| 需求澄清官 | 新需求→spec 前 | 只问不答，每轮 ≤3 问 |
| Spec 作者 | 澄清后→spec draft | 缺信息用 [TBD]，不杜撰，不写代码 |
| 杠精评审员 | spec draft→approved | 必须刻薄，模糊词标 H |
| 实现工程师 | spec approved→写代码 | spec 未覆盖必反问 |
| 测试设计师 | 实现前/补测试 | 每条 AC 至少一用例 |
| Task 守护者 | 启动/提交/提 PR | 检测反例立即叫停 |

## 三、人机分工（⭐⭐⭐）

### Q8：SDD 的人机协作核心原则是什么？
**参考答案：**
**决策权永远在人，执行权可以下放给 AI；中间的灰色地带按"是否能客观验证"来判。**

| 类别 | 例子 | 谁做 |
|------|------|------|
| 决策权 | 选技术栈/是否上线/AC 算不算通过 | **人** |
| 执行权 | 翻译 spec→代码→测试→文档 | 可下放 AI |
| 决策输入 | 调研方案/列出 trade-off | AI 代劳，人审用 |

### Q9：人必须保留的 7 项不可代理职责？
**参考答案：**
1. 业务决策（scope/优先级/trade-off）
2. 验收（spec [APPROVED]/release）
3. Code Review（审 PR/批准合入）
4. 契约变更（API/DB/事件发起与审批）
5. 安全 review（权限/注入/数据泄漏最终判定）
6. 与人沟通（产品/客户/团队对话）
7. 责任承担（故障复盘/合规问询）

### Q10：AI 绝对不能做的 5 件事？
**参考答案：**
1. 替用户做需求决策（"我顺便加了 X，应该有用"）
2. 改 AC 让测试通过（SDD 最严重违规）
3. 在 spec 未覆盖处自由发挥（必须反问）
4. 自我审批自己的 spec 或代码
5. 改生产环境（哪怕代码正确）

## 四、跨模型协作（⭐⭐）

### Q11：跨厂商（Claude/OpenAI/Gemini）协作的最低约定？
**参考答案：**
1. spec/契约文件用 Markdown + YAML/JSON block（避免厂商专属格式）
2. 提示词避免依赖单厂商特殊 token（如 Claude 的 `<thinking>` 在其他模型行为不同）
3. 跨厂商裁决会话不要包含原模型的输出 context（否则偏向先发言者）
4. 交接透过冻结文件，不传 raw context

### Q12：AGENTS.md 的内容应该写什么？不应该写什么？
**参考答案：**

| 该写 | 不该写 |
|------|--------|
| 硬约束（流程红线、编码禁令） | 编码教程 |
| 流程红线（先 spec 再代码） | 长篇幅风格指南 |
| spec/ADR 路径约定 | 示例代码 |
| 角色 prompt 索引 | 解释性文字 |

**铁律**：AGENTS.md 只留硬约束+流程红线，教程外链；超过 500 行要清理。

## 五、深度场景（⭐⭐⭐）

### Q13：同一会话频繁切模型有什么问题？正确做法是什么？
**参考答案：**
- **问题**：每次切档新 context，丢失之前的对话历史；上下文越滚越脏
- **正确做法**：按"任务/PR"分会话，会话内固定档位；交接通过冻结 spec 文件

### Q14：多 Agent 系统中最常见的 6 种反模式？
**参考答案：**

| 反模式 | 症状 | 对治 |
|--------|------|------|
| 全场最贵 | Opus 跑 lint/写 CHANGELOG | 归档/格式类走快速档 |
| 全场最便宜 | Haiku 写 spec/做架构决策 | 澄清/评审/架构必须重推理档 |
| 同会话自审 | 生成 spec 的会话审自己 | 评审强制开新会话，最好换模型 |
| 多 Agent 各持 spec 副本 | 各自读 spec 后漂移 | 引用同一 spec 文件版本号 |
| 跨档不交接 | 对话历史一股脑丢过去 | 通过冻结的 spec/摘要文件交接 |
| 裁决偏向先发言者 | 读完 A 再读 B，偏 A | 并列对比，不带顺序偏好 |

### Q15：你设计的多 Agent 系统，怎么防止一个 Agent 的输出"污染"另一个？
**参考答案：**
1. Agent 之间不共享 raw context，只传结构化产出物
2. 每个 Agent 独立会话、独立选模型
3. 产出物有固定 schema（JSON/YAML/Markdown 模板）
4. 设置"信息防火墙"——Agent A 的思维过程不进 Agent B 的上下文
5. 关键节点（评审/架构决策）多 Agent 独立给出意见，合并后再决策
