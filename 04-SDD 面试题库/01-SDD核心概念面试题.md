# 01 — SDD 核心概念面试题

> 覆盖 SDD 定义、四大原则、与 Vibe Coding / PRD 的对比、spec 9 模板、适用场景与边界

---

## 一、SDD 基础概念（⭐）

### Q1：什么是 SDD？一句话定义
**参考答案：**
SDD（Specification-Driven Development，规范驱动开发）是一种以"规范文档先行、代码实现后置"为核心的开发范式——在写代码之前先用结构化文档把需求、接口、约束、验收标准显式固化，后续的编码、审查、测试、维护全部围绕这份规范展开。

> 一句话：**写代码之前先写"合同"，让人和 AI 照合同干活。**

### Q2：SDD 的四大核心原则是什么？
**参考答案：**

| 原则 | 含义 | 落地动作 |
|------|------|----------|
| 规范先行 | 先出 spec，再出代码 | spec 未评审通过不进入编码 |
| 文档即契约 | spec 是人+AI+测试的共同单一事实源 | AC 直接翻译为测试用例 |
| 可追溯 | 每个实现决策都能回溯到 spec | 代码注释引用 spec 章节号 |
| 可验证 | spec 必须能机械式检验是否满足 | AC 写成 Given-When-Then |

### Q3：SDD 和"写文档"有什么区别？
**参考答案：**
- 传统文档：代码写完再补、只给人看、一旦变更就失效
- SDD spec：代码开始前就必须存在的**契约**，同时给人+AI+测试消费，作为 Git 一等公民与代码一起演进

### Q4：SDD 主要解决什么问题？
**参考答案：**
SDD 解决 AI 辅助编程时代三个核心问题：
1. **需求漂移**：Vibe Coding 中每轮对话 AI 只看到局部，实现偏离原始意图
2. **不可审查**：没有中间产物，只能审最终代码
3. **不可复现**：同一需求交给 AI 两次，得到两份截然不同的实现

## 二、SDD vs 传统开发（⭐⭐）

### Q5：SDD 和传统 PRD 有什么不同？
**参考答案：**

| 维度 | 传统 PRD | SDD |
|------|----------|-----|
| 面向读者 | 业务方 | 人 + AI + 测试 |
| 粒度 | 粗（功能级） | 细（接口/字段/AC 级） |
| 可验证性 | 低 | 高（每条 AC 可直接写成测试） |
| 生命周期 | 写完即开始过时 | 与代码同 repo、同 PR、同演进 |
| AI 消费 | 难 | 原生可解析 |

### Q6：SDD 和 Vibe Coding 的核心差异是什么？
**参考答案：**

| 维度 | Vibe Coding | SDD |
|------|-------------|-----|
| 起点 | 一句话让 AI 写代码 | 生成结构化 spec.md |
| AI 角色 | 全程主导无约束 | 协助写 spec + 受 spec 约束 |
| 可审查性 | 只能审最终代码 | spec 审 + 代码审两层 |
| 质量上限 | 看 AI 当天状态 | 看 spec 质量 |
| 变更管理 | 极低成本但混乱 | 改 spec 触发代码重生成 |
| 适用规模 | 个人/玩具 | 个人到大型团队 |

### Q7：SDD 和 TDD / BDD / DDD 的关系？
**参考答案：**
- **TDD**：SDD 的执行方式之一——AC 用测试固化；但 SDD 还覆盖 TDD 不覆盖的非功能需求与业务背景
- **BDD**：SDD 验收标准推荐使用 Given-When-Then（BDD 风格），但 SDD 范围更广
- **DDD**：都讲"契约"，DDD 关注领域模型边界，SDD 关注规范文档如何驱动开发流程

## 三、spec 模板（⭐⭐）

### Q8：SDD 的 spec 包含哪 9 大模块？
**参考答案：**

```
1. 需求背景  ← WHY（为什么做）
2. 业务目标  ← WHAT outcome（主目标 + 非目标）
3. 功能范围  ← WHAT scope（MUST/SHOULD/MAY）
4. 非功能需求 ← HOW WELL（性能/安全/容量）
5. 接口定义  ← HOW API（入参/出参/错误码/幂等性）
6. 数据模型  ← HOW data（实体/字段/约束/索引）
7. 技术方案  ← HOW implementation（选型理由/时序/依赖）
8. 风险评估  ← WHAT IF（风险+可能性+缓解）
9. 验收标准  ← DONE?（Given-When-Then 或可断言条件）
```

### Q9：不是什么 spec 都要写满 9 模块，怎么判断简化版？
**参考答案：**
- **个人脚本（<100 行）**：10 行 README 顶部注释
- **小团队 feature（1-2 人周）**：mini-spec（3-5 节），至少保留"需求背景 + 功能范围 + 验收标准"
- **中型需求（1 月+）**：完整 9 模块
- **跨团队系统（半年+）**：主 spec + 子 spec 拆分

**铁律**：即使简化，模块顺序和标题保持不变，保证可扫读。

### Q10：如何让 spec "可验证"？举出 3 个具体技巧
**参考答案：**
1. **替换模糊词**："尽快"→"P99 < 300ms"，"多种"→枚举完整列表，"大量"→具体数量级
2. **反向提问法**："如果有人故意曲解这一句，能曲解几种版本？"
3. **可测性检查**：每条需求读完能浮现至少一条测试断言

## 四、适用边界（⭐⭐）

### Q11：什么场景下 SDD 收益最大？什么时候不要用？
**参考答案：**

**收益最大**：
- 多人协作、AI 辅助开发、长生命周期项目、对外接口/SDK、复杂业务逻辑（权限/计费/审批流）

**不要用**：
- 一次性脚本、探索性原型、UI 微调、极度紧急的线上 fix

**判断标准**：spec 的写作成本 < 它将节省的沟通与返工成本。

### Q12：SDD 等于"重流程、拖慢速度"吗？
**参考答案：**
不是。简单任务可用极简版 spec（10 行 Markdown），复杂度可伸缩。SDD 追求的是**前期集中投入换后期减少返工**，全生命周期通常是加速的。

## 五、深度理解（⭐⭐⭐）

### Q13：为什么说"AI 越强越需要 spec"？
**参考答案：**
AI 越强，生成代码速度越快，但**方向跑偏的速度也越快**。没有 spec 约束，AI 可能在错误方向上生成大量代码。spec 的价值在于：**用前期的约束投入，换后期不跑偏**。AI 的能力越强，这种约束的收益越大。

### Q14：SDD 的核心哲学是什么？和"意图即真理"的关系？
**参考答案：**
核心哲学：**决策权永远在人，执行权可以下放给 AI**。spec 是人的意图的结构化表达——"意图即真理"（Intent is the Source of Truth）。代码只是意图的一次实现，而非定义。这也是 SDD 和传统"代码即规范"的根本区别。

### Q15：AGENTS.md 和 CLAUDE.md 是什么关系？为什么推荐 AGENTS.md 作为主文件？
**参考答案：**

| 文件 | 谁自动读 | 工具中立度 |
|------|---------|-----------|
| `AGENTS.md` | Cursor/Windsurf/Aider/Continue/Cline 等 | 高 |
| `CLAUDE.md` | Claude Code | 仅 Claude Code |
| `.cursor/rules/` | Cursor | 仅 Cursor |

推荐以 `AGENTS.md` 为厂商中立主文件，其他厂商私有文件写一行 `See AGENTS.md` 级联。这样换工具不丢约束。

## 六、快问快答（10 秒挑战）

- **SDD 全称？** Specification-Driven Development
- **spec 的首要读者是谁？** 人 + AI + 测试用例（三者同等重要）
- **AC 是什么的缩写？** Acceptance Criteria（验收标准）
- **9 模板中哪个模块最容易被遗漏？** §8 风险评估
- **非目标（Non-goal）为什么要显式写？** 防止 scope creep（需求蠕变）
- **spec 和代码冲突时以谁为准？** 以 spec 为准，代码回退；若代码更合理则先改 spec
