## Q1. 如何在 Claude Code / Cursor / Windsurf 等 AI 工具中挂载 MCP Server？★★★

**答：** 虽不同工具配置文件路径不同，但 **`mcpServers` 字段的结构几乎统一**。

**通用配置模板：**

**Stdio 模式（本地）：**
```json
{
  "mcpServers": {
    "my-tools": {
      "command": "python",
      "args": ["-m", "my_mcp_server"],
      "env": {
        "API_KEY": "${MY_API_KEY}"
      },
      "cwd": "/path/to/project"
    }
  }
}
```

**Streamable HTTP 模式（远程）：**
```json
{
  "mcpServers": {
    "company-tools": {
      "type": "http",
      "url": "https://mcp.company.com/mcp",
      "headers": {
        "Authorization": "Bearer ${MCP_TOKEN}"
      }
    }
  }
}
```

**各工具配置文件位置：**

| 工具 | 用户级配置 | 项目级配置 |
|------|-----------|-----------|
| **Claude Code** | `~/.claude.json` | `.mcp.json` |
| **Cursor** | `~/.cursor/mcp.json` | `.cursor/mcp.json` |
| **Windsurf** | `~/.codeium/windsurf/mcp_config.json` | 项目根目录同名文件 |
| **Cline / Roo Code** | VS Code 扩展设置 | `.cline/mcp.json` |
| **Continue** | `~/.continue/config.yaml`（YAML 格式！） | — |
| **Claude Desktop** | `<user-config>/Claude/claude_desktop_config.json` | — |
| **Zed** | Settings → `context_servers` 块 | — |

## Q2. 环境变量在 MCP 配置中如何注入？★★★

**答：** 使用 `${VAR}` 占位符，在启动时由 Host 替换为实际环境变量值：

```json
{
  "mcpServers": {
    "database": {
      "command": "python",
      "args": ["-m", "pg_mcp_server"],
      "env": {
        "DATABASE_URL": "${DB_URL}",
        "API_KEY": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

**占位符注意事项：**
- 占位符名称必须与操作系统环境变量名完全一致（区分大小写）
- `${VAR:-default}` 语法支持默认值（部分 Host 支持）
- 如果环境变量不存在，Server 会收到空值——在 Server 端做存在性校验
- **绝对不要**在配置文件中直接写真实密钥（即使用户级配置）
- 本地开发：用 `.env` + `direnv` 或 `1Password CLI` 注入环境变量
- 团队共享：使用 OAuth（不共享 Token）

## Q3. MCP 配置的三级作用域是什么？优先级如何？★★☆

**答：**

```
优先级：Local > Project > User

第 1 级：User（用户级）    → ~/.claude.json 等全局配置
第 2 级：Project（项目级）  → .mcp.json（可提交到 Git，团队共享）
第 3 级：Local（本地覆盖）  → .mcp.local.json（不提交 Git，个人覆盖）
```

**合并规则：**
- 同名 Server：高优先级覆盖低优先级
- 不同名 Server：合并（取并集）
- `.mcp.local.json` 中的 `env` 覆盖 `.mcp.json` 中的同名 `env`

**使用场景：**
- **Project**：团队共享的数据库 Server、GitHub Server（提交到 Git）
- **Local**：个人开发用的本地数据库连接字符串（不提交到 Git）

## Q4. MCP Server 挂载后，AI 工具如何发现和使用 Tool？★★☆

**答：**

**发现阶段（启动时）：**
```
Host 启动
→ 读取配置文件
→ 启动每个 MCP Server
→ 执行 initialize 握手
→ 调用 tools/list（获取所有 Tool）
→ 调用 resources/list、prompts/list（获取所有能力和Prompt）
→ 将 Tool 列表注入 LLM 的 System Prompt
→ LLM 开始理解可用的能力
```

**运行时的 Tool 命名：**
```
原始 Tool 名: search_issues
挂载后全名: mcp__github__search_issues  (Claude Code 风格)
        或: github:search_issues         (Cursor 风格)
```

**运行时交互：**
- 用户输入指令 → LLM 判断需要调用哪个 Tool → Host 路由到正确 Server
- **Resource 引用**：`@github://owner/repo/issues/42`（类似 @mention）
- **Prompt 调用**：`/review_code language=python`（斜杠命令）

## Q5. AI 工具中的 MCP 权限控制有哪些模式？★★★

**答：**

**三种模式：**

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **Ask（询问）** | 每次调用 Tool 前弹出确认框 | 默认模式，高安全 |
| **Allowlist（白名单）** | 匹配的 Tool 自动执行，不询问 | 常用且信任的 Tool |
| **Denylist（黑名单）** | 匹配的 Tool 总是拒绝执行 | 敏感的或危险的 Tool |

**配置示例（Claude Code）：**
```json
{
  "permissions": {
    "allow": [
      "mcp__github__search_*",
      "mcp__github__get_*",
      "mcp__postgres__run_query"
    ],
    "deny": [
      "mcp__github__delete_*",
      "mcp__system__*"
    ]
  }
}
```

**规则：**
- Deny 优先于 Allow
- 支持通配符（`*`）
- 无匹配项 → 默认 Ask

## Q7. 如何选择一个合适的 MCP Server？（选择检查清单）★★☆

**答：**

```
□ 1. 来源可信吗？
   → 优先：modelcontextprotocol/servers 官方仓库
   → 其次：知名组织/公司的仓库

   → 是否有功能描述、安装命令、配置示例

□ 3. 维护活跃吗？
   → 最近 6 个月内有更新

```

## Q8. MCP 与 Agent Skills 在 Claude Code 中如何协同工作？★★★

**答：**

**Skill 引用 MCP Tool：**
```markdown
---
name: database-query
description: 安全地查询生产数据库
allowed-tools:
  - mcp__postgres__run_query
  - mcp__postgres__describe_table
---

# 数据库查询 Skill

## 规则
1. 只允许 SELECT 查询
2. 查询前先查看表结构（describe_table）
3. 限制返回行数（添加 LIMIT 100）
4. 不要在查询中包含敏感字段名
```

**Hook 拦截 MCP 调用：**
```json
{
  "hooks": {
    "preToolUse": [
      {
        "matcher": "mcp__postgres__run_query",
        "command": "sh ./hooks/validate-sql.sh"
      }
    ]
  }
}
```

**Subagent 的 MCP Tool 白名单：**
```
Subagent "data-analyst":
  allowed-tools:
    - mcp__postgres__run_query
    - mcp__postgres__describe_table
  # 不能调用 delete/drop 类 Tool
```

**协同模式：**
```
Skill（指导"怎么做"）
  ↓ 引用
MCP Tool（提供"能做什么"）
  ↓ 被拦截
Hook（安全校验，如 SQL 审查）
  ↓ 委托
Subagent（限定能力范围的子智能体）
```

## Q9. 如何在 VS Code / Cursor 中调试 MCP Server 连接问题？★★☆

**答：**

**常见问题和诊断：**

| 症状 | 可能原因 | 诊断方法 |
|------|---------|---------|
| Server 状态显示 "Error" | 启动失败 | 查看 Server 日志 |
| Tools 不出现 | capabilities 未声明 | 检查 `initialize` 响应 |
| Tool 调用卡住 | 内部阻塞 | 检查超时设置 |
| 权限弹窗太多 | 未配置白名单 | 添加 Allowlist |
| 环境变量不生效 | 占位符拼写错误 | 确认变量名一致性 |
| Windows 命令找不到 | 路径问题 | 使用绝对路径 |

**调试入口：**
- Claude Code: `~/.claude/logs/mcp-<name>.log` + `/mcp` 命令查看状态
- Cursor: Developer Tools → Console
- Cline: VS Code Output 面板 → "Cline" 频道
- Continue: `~/.continue/logs/`

**调试步骤（推荐顺序）：**
1. `/mcp` 查看连接状态
2. 查看 Server 日志文件
3. 用 MCP Inspector 隔离测试（排除 Host 因素）
4. 检查协议版本兼容性

## Q10. MCP Server 推荐的挂载数量是多少？为什么？★★☆

**答：**

**推荐数量：**
- Server 数量：**≤ 5 个**
- 总 Tool 数量：**≤ 50 个**

**原因：**
- 每个 Tool 的 name + description + inputSchema 都占用 LLM 上下文 Token
- 50 个 Tool 的定义可能消耗 5000+ Token（约为 Claude 上下文窗口的 2-3%）
- Tool 太多，LLM 选择准确率下降（"选择困难"）
- Resource 订阅也会持续消耗注意力

**优化策略：**
- 按项目/任务选择性地挂载 Server（不需要全部同时启用）
- 合并功能相近的 Server
- 不需要文件 Server 如果 Host 已有内置文件操作能力

## Q11. 如何为团队选择统一部署的 MCP Server 架构？★★☆

**答：**

```
团队共享的 MCP Server 部署模式：

模式 1：中心化 HTTP 部署
  所有团队成员 → 同一个 Streamable HTTP Server
  优点：集中管理、统一升级、共享连接池
  缺点：需要运维、认证必须配置

模式 2：每人本地 Stdio
  每人本地启动自己的 Server 实例
  优点：零运维、本地文件访问、隐私保护
  缺点：环境差异、版本不一致

模式 3：混合模式
  共享服务（数据库/API）→ HTTP 集中部署
  个人工具（文件/本地脚本）→ Stdio 本地运行
  通过 .mcp.json（Project 级）统一配置
```

**推荐**：模式 3 混合模式——数据库 Server 用 HTTP 共享，本地工具用 Stdio。

## Q12. ChatGPT / Custom GPT 是否原生支持 MCP？如何间接使用？★★☆

**答：** ChatGPT 和 Custom GPT **不原生支持 MCP**。

**间接方案：**

**方案 1：MCP → Function Calling 适配器**
```python
# 适配器将 MCP Tools 转为 OpenAI Function Calling 格式
mcp_tools = await session.list_tools()
openai_functions = [
    {
        "name": tool.name,
        "description": tool.description,
        "parameters": tool.inputSchema,
    }
    for tool in mcp_tools.tools

```

**方案 2：HTTP API Action**
- 将 MCP Server 包装为 HTTP API
- 在 Custom GPT 中配置为 Action
- 本质是绕过 MCP 协议，改用 REST

**方案 3：LangChain / AutoGen 等框架**
- 框架内置 MCP 适配器，可同时支持多种 LLM 后端
- Agent 层使用 MCP，模型层使用各自 API

## Q14. MCP 的 `@mention` Resource 引用是怎么工作的？★★☆

**答：**

在支持 MCP 的 AI 工具中，用户可以通过 `@` 引用 Resource：

```
用户输入："帮我分析 @github://mycompany/myproject/issues/42 这个 Issue"
```

**工作流程：**
1. Host 识别 `@github://...` 是一个 MCP Resource 引用
2. 查找匹配的 MCP Server（URI scheme `github://` 对应该 Server 注册的 scheme）
3. 调用 `resources/read` 获取 Resource 内容
4. 将内容注入 LLM 上下文，替换 `@` 引用

**优势：**
- 精确引用：不是"GitHub 上的第四个 Issue"，而是确定的 URI
- 减少 Tool 调用：不需要先 `search_issues` 再选一个
- 用户友好：类似 Slack/Discord 的 @mention 用户体验

## Q15. 如何通过 MCP Server 暴露内部企业 API？★★☆

**答：**

**架构模式——API 网关 MCP Server：**
```
              MCP 协议                  内部 HTTP/gRPC
AI Host ←──────────────→ MCP Server ←───────────────→ 内部服务
                             │
                    ┌────────┼────────┐
                    ↓        ↓        ↓
                用户服务   订单服务   库存服务
```

**实现要点：**
1. MCP Server 作为统一入口（协议适配器 + 安全网关）
2. 认证：MCP Client → Server（OAuth/SAML），Server → 内部 API（mTLS/内部 Token）
3. 不暴露内部 API 的原始接口（简化、聚合、脱敏后通过 Tool 暴露）
4. 审计日志：记录谁通过 MCP 调用了哪个内部服务

**示例 Tool：**
```python
@mcp.tool()
def get_order_status(order_id: str) -> str:
    """查询订单状态。当用户询问订单进度时使用。只读，无副作用。"""
    token = get_internal_service_token()
    response = httpx.get(
        f"https://order-api.internal/orders/{order_id}",
        headers={"Authorization": f"Bearer {token}"},
    )
    # 脱敏后返回
    return sanitize_order_data(response.json())
```

## Q17. 继续（Continue）插件的 MCP 配置有什么特殊之处？★★☆

**答：** Continue 使用 **YAML 格式**而非 JSON 格式：

```yaml
# ~/.continue/config.yaml
mcpServers:
  - name: github
    command: npx
    args:
      - "-y"
      - "@modelcontextprotocol/server-github"
    env:
      GITHUB_TOKEN: "${GITHUB_TOKEN}"
```

**注意事项：**
- 格式是 YAML（不是 JSON）
- Server 配置是**数组**形式（而非对象）
- 每个 Server 需要 `name` 字段（而非 JSON 配置中的 key）
- 路径是 `~/.continue/config.yaml`

## Q18. LangChain / LangGraph / CrewAI / AutoGen 如何集成 MCP？★★☆

**答：** 这些 Agent 框架通过 **MCP 适配器** 集成：

```python
# LangGraph 示例
from langchain_mcp import MCPToolkit

toolkit = MCPToolkit(
    server_params=StdioServerParameters(
        command="python", args=["-m", "my_server"]
    )
)

tools = await toolkit.get_tools()
# tools 可以直接用于 LangGraph Agent
agent = create_react_agent(llm, tools)
```

**通用模式：**
1. 框架提供 MCP Client 适配器
2. 适配器连接 MCP Server，获取 Tool 列表
3. 将 MCP Tool 转换为框架的原生 Tool 格式
4. Agent 使用这些 Tool 执行任务

**好处：** Agent 框架不需要为每种外部能力写适配器——只要对方实现了 MCP Server。

## Q19. 什么是 MCP 的 Project 级配置？为什么要用它？★★☆

**答：**

**Project 级配置**（`.mcp.json`放在项目根目录）：

```json
{
  "mcpServers": {
    "project-db": {
      "command": "python",
      "args": ["-m", "pg_mcp_server"],
      "env": {
        "DATABASE_URL": "${PROJECT_DB_URL}"
```

**为什么要用 Project 级配置：**
1. **团队共享**：提交到 Git，每个成员 clone 后自动获得相同的 MCP 配置
2. **上下文正确**：进入项目目录时自动加载项目相关的 Server
3. **可复现**：新人 onboarding 时不需要手动配置
4. **安全边界**：密钥使用 `${VAR}` 占位符，每个成员自己管理本地环境变量

## Q22. 如何监控 MCP Server 在 AI 工具中的使用情况？★★☆

**答：**

**1. 通过 `/mcp` 命令（Claude Code）：**
- 查看连接状态、协议版本、Tool/Resource/Prompt 数量
