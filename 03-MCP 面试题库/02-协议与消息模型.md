# 02 - 协议与消息模型

> 高频题 ★★★ | 进阶题 ★★☆ | 基础题 ★☆☆

---

## Q1. MCP 基于什么底层通信协议？为什么选择它？★★★

**答：** MCP 基于 **JSON-RPC 2.0** 协议。

**选择 JSON-RPC 2.0 的原因：**
- **语言无关**：纯 JSON 文本，任何语言都有成熟的 JSON 解析器
- **简单明确**：只有 3 种消息类型（Request、Response、Notification），易于理解和实现
- **传输无关**：可运行在 Stdio、HTTP、WebSocket 等任何双向传输层上
- **轻量级**：没有 SOAP 的复杂性，没有 gRPC 的二进制依赖
- **双向通信**：支持 Server → Client 的反向通知（Notification）
- **批量请求**：支持将多个请求打包为数组发送

## Q2. JSON-RPC 2.0 的三种消息类型是什么？各有什么特征？★★★

**答：**

**1. Request（请求）**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "search_issues", "arguments": { "query": "bug" } }
}
```
- 必须有 `id`（数字/字符串/NULL）
- 必须有 `method`
- 期望一个 Response

**2. Response（响应）——成功**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "content": [{ "type": "text", "text": "找到 3 个 Issue" }] }
}
```
- `id` 与对应 Request 匹配
- 包含 `result` 字段

**3. Response（响应）——错误**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": { "code": -32602, "message": "Invalid params" }
}
```
- `id` 与对应 Request 匹配
- 包含 `error` 对象（`code` + `message` + 可选 `data`）

**4. Notification（通知）**
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```
- **没有 `id` 字段**
- 不期待任何响应
- 用于单向事件推送

## Q3. 请描述 MCP 连接的完整生命周期（4 个阶段）。★★★

**答：**

```
阶段 1: 初始化握手 (Initialization)
  Client → Server: initialize {
    protocolVersion: "2025-03-26",
    capabilities: { roots: {...}, sampling: {...} },
    clientInfo: { name: "claude-code", version: "1.0.0" }
  }
  Server → Client: {
    protocolVersion: "2025-03-26",
    capabilities: { tools: {}, resources: { subscribe: true } },
    serverInfo: { name: "github-mcp-server", version: "1.2.0" },
    instructions: "Use this server to interact with GitHub..."
  }
  Client → Server: notifications/initialized  ← 标记握手完成

阶段 2: 运行期 (Operational)
  - tools/list, tools/call
  - resources/list, resources/read, resources/subscribe
  - prompts/list, prompts/get
  - ping (心跳)
  - notifications/progress, notifications/cancelled
  - sampling/createMessage (反向)

阶段 3: 关闭协商 (Shutdown)
  - Client 发送关闭信号
  - Server 完成正在执行的任务
  - 双方确认关闭

阶段 4: 传输断开 (Transport Disconnect)
  - Stdio: 子进程终止
  - HTTP: 关闭 TCP 连接，清理 Session
```

**关键点**：`notifications/initialized` 必须在 Server 返回 `initialize` 结果后由 Client 发送，在此之前 Server 不应处理任何运行期请求。

## Q4. 能力协商中 Client 和 Server 各自可以声明哪些能力？★★☆

**答：**

**Client 能力声明：**
```json
{
  "capabilities": {
    "roots": {
      "listChanged": true    // 支持根目录变更通知
    },
    "sampling": {}            // 允许 Server 请求 LLM 调用（高风险）
  }
}
```

**Server 能力声明：**
```json
{
  "capabilities": {
    "tools": {
      "listChanged": true    // Tool 列表变更时主动通知
    },
    "resources": {
      "subscribe": true,     // 支持资源变更订阅
      "listChanged": true    // 资源列表变更时主动通知
    },
    "prompts": {
      "listChanged": true    // Prompt 列表变更时主动通知
    },
    "logging": {}            // 支持日志级别设置
  }
}
```

**`listChanged` 的意义**：当 Server 的工具/资源/提示列表在运行时发生变化（如新增插件），可以通过 Notification 主动通知 Client 重新发现，无需重启连接。

## Q5. MCP 定义了哪些错误码？有什么特殊设计？★★★

**答：**

**JSON-RPC 2.0 标准错误码：**
| 错误码 | 说明 |
|--------|------|
| -32700 | Parse error（JSON 解析失败） |
| -32600 | Invalid Request（无效请求） |
| -32601 | Method not found（方法不存在） |
| -32602 | Invalid params（参数无效） |
| -32603 | Internal error（服务器内部错误） |

**MCP 扩展错误码：**
| 错误码 | 说明 |
|--------|------|
| -32002 | Resource not found（资源未找到） |
| -32001+ | Server 自定义错误范围 |

**关键设计：Tool 业务错误 vs 协议错误**

```
❌ 错误做法：SQL 超时返回 JSON-RPC error（-32603）
   → LLM 不知道发生了什么，无法自行恢复

✅ 正确做法：SQL 超时返回正常 Response + isError: true
   {
     "result": {
       "content": [{ "type": "text", "text": "查询超时，请减小查询范围重试" }],
       "isError": true
     }
   }
   → LLM 看到"工具执行了但失败了"，可以决定重试或换策略
```

## Q6. 进度通知（Progress Notification）是如何工作的？★★☆

**答：** 进度通知用于长耗时任务（如大文件处理、批量操作），避免 Client 因超时而误判 Server 无响应。

**机制：**
1. Client 在请求的 `_meta.progressToken` 中传入一个进度令牌
2. Server 在处理过程中周期性发送 `notifications/progress`：
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "token-123",
    "progress": 45,
    "total": 100,
    "message": "已处理 45/100 条记录..."
  }
}
```
3. Client 收到进度通知后更新 UI（如进度条）

**面试要点**：
- `progress` 和 `total` 都是可选的——Server 可以不提供总量（total=null 表示未知总量）
- `message` 可选，用于人类可读的进度描述
- 进度通知是单向的，不影响请求-响应流程本身

## Q7. 请求取消（Cancellation）是如何工作的？有什么限制？★★☆

**答：**

**取消流程：**
1. Client 发送 `notifications/cancelled`：
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": {
    "requestId": 42,
    "reason": "用户取消了操作"
  }
}
```
2. Server 应尝试中止正在执行的任务
3. 如果 Server 已接近完成，仍可返回正常 Response（此时 Client 应忽略该结果）

**关键限制**：
- 取消是**协作式**的（cooperative），不是强制性的
- 如果 Server 已提交副作用（如已发送邮件、已写入数据库），取消无法回滚
- Server 不必立即响应取消——它可以在方便的安全点中止
- 取消请求本身是 Notification，不期待响应

**面试话术**：取消是"尽力而为"的——类似于 HTTP 的请求取消，不能保证事务回滚。

## Q8. 消息批处理（Batching）在 MCP 中是如何支持的？★☆☆

**答：** JSON-RPC 2.0 原生支持将多个请求打包为一个 JSON 数组发送：

```json
[
  { "jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {} },
  { "jsonrpc": "2.0", "id": 2, "method": "resources/list", "params": {} },
  { "jsonrpc": "2.0", "method": "notifications/initialized" }
]
```

Server 返回一个数组，包含每个 Request 对应的 Response（Notification 不产生 Response）。Client 通过 `id` 匹配请求和响应。

**优势**：减少往返次数，提升启动时的能力发现效率。

## Q9. Server 的 `instructions` 字段有什么作用？★★☆

**答：** 在 `initialize` 的 `InitializeResult` 中，Server 可以返回一个 `instructions` 字符串：

```json
{
  "instructions": "Use this server to interact with GitHub repositories. For creating issues, use create_issue tool. For reading files, use get_file_contents resource."
}
```

**作用**：
- 这是一个**面向 LLM 的全局提示**，告诉 LLM 如何使用这个 Server
- 会被 Host 注入到 LLM 的 System Prompt 中
- 用于描述 Server 的整体用途、使用时机和注意事项
- 不替代单个 Tool 的 `description`，而是提供 Server 级别的上下文

**最佳实践**：简洁（2-4 句话），说明 Server 的定位和什么时候该用、什么时候不该用。

## Q10. 消息格式方面，Stdio 模式下有什么特别要求？★★☆

**答：** Stdio 传输使用 **NDJSON（Newline Delimited JSON）** 格式：

- 每条 JSON-RPC 消息占一行，以 `\n` 结尾
- 消息内部**不能有换行符**（否则会被误判为消息边界）
- stdin 读入，stdout 写出

```
stdin:  {"jsonrpc":"2.0","id":1,"method":"tools/list"}\n
stdout: {"jsonrpc":"2.0","id":1,"result":{...}}\n
```

**常见陷阱：**
1. **在 stdout 中 print 日志**：会污染协议通道，导致 Client 解析失败
2. **Windows `\r\n` 问题**：需确保使用 `\n` 而非 `\r\n`
3. **消息体中的意外换行**：JSON 序列化时必须确保紧凑格式（无额外空白）

## Q11. 如何调试 MCP 的 JSON-RPC 消息流？★★☆

**答：**

**方法 1：MCP Inspector（推荐）**
```bash
npx @modelcontextprotocol/inspector <command> <args>
```
提供可视化的消息流界面，可查看原始 JSON-RPC 消息、手动调用 Tool、测试 Resource 订阅。

**方法 2：环境变量启用调试日志**
```bash
# Python
MCP_LOG_LEVEL=DEBUG python server.py

# Node.js
DEBUG=mcp:* node server.js
```

**方法 3：tee 包装脚本（Stdio 模式）**
```bash
#!/bin/bash
# 将 stdout 同时写入协议通道和日志文件
python server.py 2>stderr.log | tee messages.log
```

**方法 4：HTTP 抓包（HTTP 模式）**
使用 Charles Proxy 或 mitmproxy 拦截 HTTP 请求/响应，查看 Session-Id、Authorization 头等。

## Q12. 如果在初始化握手阶段，Client 和 Server 的 protocolVersion 不兼容怎么办？★★☆

**答：**

1. Client 在 `initialize` 中声明其支持的 `protocolVersion`（如 `"2025-06-01"`）
2. Server 在 `InitializeResult` 中返回其实际支持的版本（如 `"2025-03-26"`）
3. 如果两者**完全不重叠**（Server 支持的版本早于/晚于 Client 能接受的范围），Server 应返回 JSON-RPC error，提示协议版本不兼容

**实际处理**：
- Host 通常内置了多个协议版本的兼容逻辑
- 如果 Host 无法兼容 Server 的版本，会显示 "协议版本不匹配" 错误并拒绝连接
- 解决方案：升级/降级 Server 或 Host 以匹配版本

## Q13. 什么情况下应该使用 `notifications/tools/list_changed`？★★☆

**答：** 当 Server 在运行期间动态新增或移除了 Tool 时（如热加载插件、条件性启用功能），应主动发送此通知：

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
```

Client 收到后应重新调用 `tools/list` 以获取最新的工具列表。

**适用场景**：
- 插件系统：用户安装/卸载插件后
- 权限变更：用户登录/登出导致可用工具集变化
- 数据源热切换：数据库连接变更导致查询工具参数变化

**同理**：`notifications/resources/list_changed` 和 `notifications/prompts/list_changed` 用于资源列表和提示模板列表的变更通知。

## Q14. `ping` 方法的作用是什么？如何使用？★★☆

**答：** `ping` 是一个简单的 Request-Response 方法，用于心跳检测和连接健康检查。

```json
// Request
{ "jsonrpc": "2.0", "id": 99, "method": "ping" }

// Response
{ "jsonrpc": "2.0", "id": 99, "result": {} }
```

**用途：**
- **心跳检测**：Client 每隔 30 秒发送一次 ping，确保 Server 仍在正常运行
- **连接健康判断**：如果 ping 超时或失败，触发重连逻辑
- **空闲保活**：防止某些网络中间件（如负载均衡器）因长时间无数据而断开连接

## Q15. MCP 如何处理大体积的返回结果？★☆☆

**答：** 多种策略：

1. **分页**：在 Tool/Resource 参数中加入 `limit` + `cursor`，返回 `nextCursor` 和 `hasMore`
2. **Resource 引用**：返回 `{ "type": "resource", "uri": "..." }` 而非完整数据，让 Client 按需拉取
3. **进度通知**：长耗时任务通过 `notifications/progress` 告知进度
4. **read_analysis_section 模式**：如 idea-plugin-analyzer 的做法——提供按字段读取大 JSON 的 Tool，避免 40MB 的响应塞满上下文

## Q16. JSON-RPC 的 `id` 字段可以是哪些类型？有什么注意事项？★☆☆

**答：** 根据 JSON-RPC 2.0 规范，`id` 可以是：
- **Number**（整数，不推荐浮点数）
- **String**
- **Null**（但带 null id 的请求被视为 Notification，不期待响应——MCP 中不常用）

**MCP 实践**：通常使用递增整数或 UUID 字符串作为 `id`，用于请求-响应匹配。

**注意**：Server 返回的 Response 中的 `id` 必须与对应 Request 的 `id` 完全一致（类型和值）。

## Q17. 描述 `resources/subscribe` 和 `resources/unsubscribe` 的工作机制。★★☆

**答：**

**订阅流程：**
```
1. Client → Server: resources/subscribe { uri: "file:///logs/app.log" }
2. Server → Client: {} (确认订阅成功)
3. 当 app.log 内容变化时...
4. Server → Client: notifications/resources/updated { uri: "file:///logs/app.log" }
5. Client → Server: resources/read { uri: "file:///logs/app.log" } (拉取新内容)
```

**取消订阅：**
```
Client → Server: resources/unsubscribe { uri: "file:///logs/app.log" }
```

**前提条件**：Server 必须在其能力声明中包含 `"subscribe": true`。

**使用场景**：日志监控、实时数据面板、文件变更检测。

## Q18. `logging/setLevel` 的作用是什么？★★☆

**答：** Client 可通过此方法动态调整 Server 的日志级别：

```json
// Request
{ "jsonrpc": "2.0", "id": 5, "method": "logging/setLevel", "params": { "level": "debug" } }
```

Server 随后通过 `notifications/message` 向 Client 推送日志：
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/message",
  "params": {
    "level": "debug",
    "logger": "my-server",
    "data": "执行 SQL: SELECT * FROM users WHERE id = 123"
  }
}
```

**用途**：运行时动态调试，无需重启 Server。

## Q19. MCP 的 `notifications/message` 和 stderr 日志有什么不同？★☆☆

**答：**

| 维度 | `notifications/message` | stderr 日志 |
|------|------------------------|-------------|
| **传输通道** | MCP 协议通道（stdout/HTTP） | 独立日志通道（stderr） |
| **目标接收方** | MCP Client（Host 应用） | 开发者/运维人员 |
| **格式化** | 结构化的 JSON-RPC Notification | 自由格式文本 |
| **可见性** | Host 可在 UI 中展示 | 通常写入日志文件，需要单独查看 |

两者互补：stderr 用于开发者调试，`notifications/message` 用于运行时可见的日志。

## Q20. 在实现 MCP Client 时，如何处理多 Server 的 Tool 名称冲突？★★☆

**答：**

**策略 1：命名空间前缀（最常用）**
```json
// 原始 Tool: search_issues
// 挂载后: mcp__github__search_issues（Claude Code 风格）
// 或: github:search_issues（Cursor 风格）
```

**策略 2：资源 URI 方案隔离**
- 每个 Server 使用不同的 URI scheme（如 `github://`、`postgres://`）
- 不会冲突，因为 URI scheme 是全局唯一的

**策略 3：Client 层路由表**
- Client 维护 Tool 名 → Server 的映射表
- 同名 Tool 由 Client 根据上下文选择（不推荐，易混淆）

**面试话术**：命名空间前缀是事实标准，由 Host 自动处理，Server 开发者不需要关心冲突问题。

## Q21. 什么是 Streamable HTTP 的 Session 管理机制？★★☆

**答：** Streamable HTTP 使用 `Mcp-Session-Id` HTTP Header 管理会话：

```
Client → Server: POST /mcp
  Headers: { "Content-Type": "application/json" }
  Body: { "jsonrpc": "2.0", "id": 1, "method": "initialize", ... }

Server → Client: 200 OK
  Headers: { "Mcp-Session-Id": "abc123" }
  Body: { "jsonrpc": "2.0", "id": 1, "result": { ... } }

后续请求:
Client → Server: POST /mcp
  Headers: { "Mcp-Session-Id": "abc123" }
  Body: ...
```

**关键点：**
- 首个请求（initialize）后 Server 返回 Session-Id
- 后续所有请求必须在 `Mcp-Session-Id` Header 中携带此 ID
- 使无状态的 HTTP 能够支持有状态的 MCP 会话
- 对于多实例部署，需要负载均衡器支持基于 Session-Id 的粘性路由（sticky session）

## Q22. 解释 MCP 中的 `_meta` 字段的用途。★★☆

**答：** `_meta` 是一个可选的元数据字段，可附加在 Request/Response/Notification 的参数对象中，用于携带协议扩展信息。

**常见用途：**
- `_meta.progressToken`：用于进度追踪
- 自定义追踪 ID（trace_id）：用于分布式追踪
- 分页游标：`_meta.nextCursor`、`_meta.hasMore`

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "batch_convert",
    "arguments": { "text": "hello world" },
    "_meta": {
      "progressToken": "token-42",
      "traceId": "trace-abc-123"
    }
  }
}
```
