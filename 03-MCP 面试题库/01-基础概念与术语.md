# 01 - 基础概念与术语

> 高频题 ★★★ | 进阶题 ★★☆ | 基础题 ★☆☆

---

## Q1. 什么是 MCP（Model Context Protocol）？谁提出的？设计目标是什么？★★★

**答：** MCP 是 Anthropic 于 2024 年 11 月开源的一种标准化通信协议，用于在 LLM（AI 模型）与外部工具/数据源之间建立标准化的通信桥梁。

**设计目标：**
- **解耦与模块化**：将 LLM 核心能力与外部工具分离，各自独立演进
- **安全性**：提供受控的沙箱环境，限制模型对敏感系统的访问
- **可扩展性**：无需重新训练或微调模型即可添加新能力（搜索、数据库查询、代码执行等）
- **标准化**：建立统一的通信标准，避免每个工具都要写定制集成代码

**核心价值**：类比为 "AI 世界的 USB 接口"——一次实现 MCP Server，跨所有兼容的 AI 应用（Client/Host）复用。

---

## Q2. 请描述 MCP 的三层架构（Host、Client、Server）及其关系。★★★

**答：**

```
┌─────────────────────────────────┐
│  Host（宿主应用）                  │
│  如 Claude Desktop、Cursor、VS Code │
│  ├─ Client A ────── Server 1     │
│  ├─ Client B ────── Server 2     │
│  └─ Client C ────── Server 3     │
└─────────────────────────────────┘
```

- **Host（宿主）**：AI 应用程序本身，管理 LLM 会话并协调 Client 与 Server 之间的通信
- **Client（客户端）**：Host 内部的协议通信组件，维持与单个 Server 的 1:1 连接，负责消息转发和生命周期管理。Server 之间的进程边界对 Host 是透明的
- **Server（服务端）**：实现具体能力（Tools/Resources/Prompts）的程序。可以是本地子进程或远程 HTTP 服务

**关键点**：一层 Host 可连接多个 Client，一个 Client 对应一个 Server。Host 通过 Client 发现和调用 Server 暴露的能力。

---

## Q3. MCP 解决的核心问题是什么？（N×M 问题）★★★

**答：** 在 MCP 出现之前，M 个 AI 应用 × N 个工具需要 M×N 次定制集成。每个工具（数据库、文件系统、API）都需要为每个 AI 应用单独写适配代码。

MCP 将复杂度降为 M+N：每个 AI 应用实现一个 MCP Client，每个工具实现一个 MCP Server，通过标准协议互操作。这种标准化使得生态可规模化增长。

```
之前：M 应用 × N 工具 = M×N 种集成
之后：M 应用 + N 工具 = M+N —— 只需 M 个 Client + N 个 Server
```

## Q4. MCP 与传统 Function Calling 的本质区别是什么？★★★

**答：**

| 维度 | MCP | 传统 Function Calling |
|------|-----|----------------------|
| **架构** | 解耦的 Client-Server 模型，模型与工具完全分离 | 紧耦合，函数硬编码在应用或 System Prompt 中 |
| **通信** | 标准化的 JSON-RPC 2.0，语言无关 | 供应商特定（OpenAI vs Anthropic 格式不同） |
| **动态性** | 运行时动态注册/发现工具，Server 可随时更新 | 静态函数列表，变更需重启应用 |
| **能力范围** | Tools + Resources + Prompts 三类原语 | 仅 Tools（函数调用） |
| **生态复用** | 一次实现 Server，跨所有 MCP Client 复用 | 通常不可跨应用复用 |
| **安全性** | 沙箱 + 审计 + 权限控制 | 依赖具体实现 |

**面试加分点**：MCP 是 Function Calling 的抽象和标准化层。MCP 不替代 Function Calling，而是提供一个开放标准来描述和管理工具，最终仍要通过 LLM 的 Function Calling 机制触发调用。

## Q5. MCP 与 REST API 有什么不同？★★☆

**答：**

| 维度 | MCP | REST API |
|------|-----|---------|
| **目标用户** | AI 模型（LLM） | 人类开发者 |
| **描述方式** | 面向 LLM 的语义描述（何时用、何时不用、有何副作用） | 面向人类的技术文档（请求/响应格式） |
| **发现机制** | 自动能力发现（`list_tools`、`list_resources`） | 需人工阅读文档或 OpenAPI Spec |
| **通信协议** | JSON-RPC 2.0（单向 + 双向通知） | HTTP（请求-响应） |
| **典型用途** | AI Agent 调用外部能力 | Web 服务间通信 |

**关键区别**：为 MCP 写 Tool description 时，必须站在 LLM 的角度——描述语义、时机、边界条件和副作用，而不是写技术参数说明。

---

## Q6. MCP 与 Agent Skills 是什么关系？★★☆

**答：** 两者是**互补关系**，而非替代关系。

| 维度 | MCP Server | Agent Skill |
|------|-----------|-------------|
| **本质** | 能力提供者（代码实现） | 指令模板（Markdown 文件） |
| **回答的问题** | "能做什么"（what to do it with） | "怎么做"（how to do it） |
| **执行方式** | LLM 调用 Tool → Server 执行 → 返回结果 | LLM 阅读 Skill 指令 → 按指令行动 |
| **典型内容** | 数据库查询工具、API 调用工具 | 工作流程指南、最佳实践、操作步骤 |
| **可组合性** | Skill 可引用 MCP Tool：`allowed-tools: [mcp__postgres__run_query]` | — |

**面试话术**：Skill 指导 LLM 如何正确使用 MCP Tools，一个 Skill 可以编排多个 MCP Server 的 Tools 协同工作。

## Q7. MCP 的三类核心原语（Primitives）是什么？各自的职责和触发方？★★★

**答：**

| 原语 | 职责 | 触发方 | 是否有副作用 | 典型例子 |
|------|------|--------|-------------|---------|
| **Tools** | 执行操作/动作 | LLM 决策 | 可能有 | 创建 Issue、发送邮件、运行 SQL |
| **Resources** | 暴露只读数据 | 用户/应用请求 | 无（只读） | 读取文件内容、查询数据库表结构 |
| **Prompts** | 可复用的对话模板 | 用户触发（斜杠命令/菜单） | 无（只读） | 代码审查模板、Issue 分类模板 |

**核心设计思想**：通过"谁发起"和"是否有副作用"两个维度，将外部能力清晰地划分为三类，避免 LLM 在资源读取和操作执行之间产生混淆。

---

## Q8. MCP 适用于哪些场景？不适合哪些场景？★★☆

**答：**

**适用场景：**
- AI 驱动的开发工具（IDE 集成数据库查询、Git 操作、API 调用）
- 企业内部 AI 平台（统一接入各种数据源和业务系统）
- 跨工具/跨平台 AI 应用（一次开发 Server，多工具复用）
- 需要沙箱安全隔离的工具调用场景

**不适用场景（反模式）：**
- **纯推理任务**：不需要外部工具或数据，LLM 自身即可完成
- **单供应商绑定应用**：如果只用一个闭源 AI 工具且不会迁移，MCP 的标准化优势不明显
- **极致低延迟场景**：MCP 的 JSON-RPC 协议开销可能不满足微秒级延迟要求
- **一次性脚本**：开发 MCP Server 的成本高于直接调用 API

## Q9. 请描述一次典型的 MCP 工作流程（从启动到工具调用完成）。★★★

**答：**

```
1. Host 启动
   → 读取配置文件（.mcp.json / claude_desktop_config.json）
   → 启动所有已配置的 MCP Server

2. 初始化握手
   → Client 发送 initialize（含 protocolVersion + capabilities + clientInfo）
   → Server 返回 InitializeResult（含 capabilities + serverInfo）
   → Client 发送 notifications/initialized

3. 能力发现
   → Client 调用 tools/list → 获取可用工具列表
   → Client 调用 resources/list → 获取可用资源列表
   → Client 调用 prompts/list → 获取可用提示模板列表

4. 用户输入
   → 用户向 Host 发出指令

5. LLM 规划
   → LLM 根据用户指令和可用工具列表，决定调用哪个工具
   → 生成 Function Calling 请求

6. 工具调用
   → Host 将 LLM 的 Tool Call 路由到对应的 MCP Server
   → Client 发送 tools/call（工具名 + 参数）
   → Server 执行逻辑，返回结果（content[] + isError）

7. 结果注入
   → 工具返回结果注入 LLM 上下文
   → LLM 根据结果生成最终回复给用户

8. 关闭
   → Host 关闭时发送优雅关闭信号
   → 断开传输层连接
```

## Q10. MCP 协议当前处于什么发展阶段？成熟度如何？★☆☆

**答：** MCP 自 2024 年 11 月开源后快速发展：

- **协议版本**：使用日期版本号（如 `2025-03-26`），持续迭代中
- **SDK 成熟度**：Python（`mcp`）和 TypeScript（`@modelcontextprotocol/sdk`）为 Anhtropic 官方维护，最成熟；Java（Spring AI）紧随其后；Go/Rust/C#/Swift 为社区维护
- **生态规模**：官方仓库 `modelcontextprotocol/servers` 收录大量参考实现，awesome-mcp 列表涵盖数百个社区 Server
- **行业采用**：Claude Desktop、Cursor、Windsurf、Cline、Continue、Zed 等主流 AI 工具均已支持

**面试要点**：MCP 正在快速标准化，但从 2025 到 2026 仍在快速演进中，API 可能仍存在 breaking change。

## Q11. 什么是 MCP 的"能力协商"（Capability Negotiation）？★★☆

**答：** 在初始化握手阶段，Client 和 Server 互相声明自己所支持的能力，确保后续通信中双方只使用共同支持的功能。

**Client 可声明的能力：**
- `roots`：暴露文件系统根目录供 Server 访问
- `sampling`：允许 Server 反向请求 Client 调用 LLM

**Server 可声明的能力：**
- `tools`（含 `listChanged: true`）：支持工具调用
- `resources`（含 `subscribe: true`、`listChanged: true`）：支持资源暴露和订阅
- `prompts`（含 `listChanged: true`）：支持提示模板
- `logging`：支持日志级别设置

**设计意义**：不是所有 Server 都需要实现全部能力。一个简单的只读 Server 可能只声明 `resources`，而一个自动化 Server 可能只声明 `tools`。能力协商避免了无意义的功能调用。

---

## Q12. MCP 协议的版本号机制是怎样的？为什么用日期格式？★★☆

**答：** MCP 使用日期格式的版本号，如 `2025-03-26`。

**原因：**
- 协议仍在快速演进中，传统 SemVer（语义化版本）难以表达向后不兼容的频繁变更
- 日期版本直观地反映了协议规范的"快照时间"
- 方便判断兼容性：Client 提议版本 → Server 回应其支持的版本 → 若无交集则为协议错误

**版本协商流程**：Client 在 `initialize` 中发送其支持的 `protocolVersion`，Server 在 `InitializeResult` 中返回它支持的版本。Host 根据双方版本决定后续行为。

---

## Q13. 什么是 MCP 的"反向能力"（Reverse Capabilities）？★☆☆

**答：** 常规 MCP 通信是 Client → Server 方向。但某些场景下，Server 需要反向请求 Client。两种反向能力：

1. **Sampling（采样）**：Server 请求 Client 代其调用 LLM
   - 风险：消耗 Client 的 LLM Token 配额
   - 必须由 Client 在能力协商中显式授权，默认禁用
   
2. **Roots（根目录）**：Server 请求 Client 暴露其文件系统根目录
   - 用于 Server 需要了解 Client 可访问的文件范围时
   - Client 通过回调函数提供根目录列表

**面试要点**：Sampling 是高风险能力——恶意 Server 可能滥用 LLM 配额或泄露上下文，因此生产环境中几乎总是禁用它。

## Q14. 请解释 MCP 区别于传统插件/扩展系统的核心优势。★★☆

**答：**

| 维度 | MCP | 传统插件系统 |
|------|-----|-------------|
| **接口标准** | 开放协议，与语言/平台无关 | 通常绑定特定语言或框架 API |
| **跨应用复用** | 一个 Server 可被任何 MCP 兼容 Host 使用 | 插件通常只能在特定应用中使用 |
| **AI 原生** | 面向 LLM 的描述和语义设计 | 面向人类开发者的 API 设计 |
| **动态发现** | 运行时自动发现能力，无需手动注册 | 通常需要预定义扩展点 |

**本质区别**：传统插件是"人写给人看"的接口，MCP 是"人写给人 + LLM 读"的协议。

## Q16. MCP Server 的命名规范有哪些要求？★★☆

**答：**

- **Server 命名**：小写字母 + 连字符，如 `github-mcp-server`、`postgres-mcp-server`
- **Tool 命名**：小写 + 下划线，动词前缀，如 `search_issues`、`run_query`，不超过 40 字符，不使用缩写和模糊词汇
- **Resource URI**：语义化 URI 方案，如 `github://owner/repo/issues/42`，不使用无意义的 ID
- **Prompt 命名**：同 Tool 规范，建议加意图前缀如 `analyze_`、`generate_`、`review_`
- **工具命名空间**：挂载后自动添加前缀，如 `mcp__github__search_issues`（Claude Code）或 `github:search_issues`（其他工具）

## Q17. MCP 协议中的 isError 与 JSON-RPC error 有什么区别？★★★

**答：**

这是一个关键设计差异：

- **JSON-RPC error**（协议级错误）：表示"调用本身失败了"——Tool 不存在、参数格式错误、方法名拼写错误。这类错误意味着请求未能被正确处理
- **`isError: true`**（业务级错误）：表示"调用成功了，但业务逻辑失败了"——如 SQL 超时、文件不存在、API 返回 500。这类"失败"是在工具正常执行过程中产生的

**为什么重要**：业务失败返回 `isError: true` 而不是 JSON-RPC error，可以让 LLM 理解"工具正常执行了，但结果不符合预期"，从而自行决定是否重试、修改参数或换用其他工具。JSON-RPC error 则直接中断流程，LLM 无法自行恢复。

## Q18. MCP 支持哪些内容类型（Content Types）作为返回值？★☆☆

**答：** MCP 工具和资源返回的 `content` 数组支持三种类型：

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| `text` | 纯文本 | 查询结果、错误消息、日志 |
| `image` | Base64 编码的图片 | 图表、截图、照片分析 |
| `resource` | URI 引用 | 嵌套引用另一个 MCP Resource |

**注意**：`resource` 类型允许 Server 返回一个"指针"而非完整数据，Client 需要根据 URI 再次请求具体内容。

## Q19. 什么是"面向 LLM 的描述"？与面向人类的 API 文档有什么不同？★★★

**答：**

**面向 LLM 的描述必须回答 4 个问题：**
1. 这个工具做什么？
2. 什么时候应该调用它？
3. 什么时候**不应该**使用它？
4. 它有什么副作用？

**示例对比：**

```
❌ 面向人类的描述（不好）：
"执行 SQL 查询并返回结果集"

✅ 面向 LLM 的描述（好）：
"对 PostgreSQL 数据库执行只读 SQL 查询。当用户询问数据库中的数据时使用此工具。
不要在需要修改数据时使用（使用 run_mutation 替代）。
此工具是只读的，无副作用。支持 SELECT 语句。
查询超时时间为 30 秒。"
```

**关键原则**：LLM 需要语义信息来做决策（"这工具是我该用的吗？"），而不只是技术参数说明。

## Q20. MCP Server 应该暴露多少 Tools？有没有数量限制？★★☆

**答：**

**最佳实践**：
- 单个 Server 建议 **≤ 20 个 Tools**
- 超过 20 个考虑按子域拆分成多个 Server
- 总挂载 Tools 建议 **≤ 50 个**（跨所有 Server），以控制上下文预算

**粒度原则**：单一职责——一个 Tool 做好一件事，不要设计"万能 Tool"（用 `action` 参数区分不同操作的反模式）。

**拆分信号**：当 Tool 的描述中出现 "如果 action=X 则..." 时，说明应该拆分成多个独立 Tool。

## Q21. 如何判断一个功能应该实现为 Tool 还是 Resource？★★★

**答：** 判断决策树：

```
需要 LLM 决定调用时机？ → Tool
有副作用（写操作）？ → Tool
参数组合复杂、依赖 LLM 推理？ → Tool

有稳定的 URI 可达？ → Resource
纯只读、无需 LLM 推理？ → Resource
需要订阅变更通知？ → Resource
用户/应用主动拉取？ → Resource

URI 不稳定、需要动态参数？ → Tool（返回数据）
一次性、不需复用？ → Tool（返回数据）
```

**反模式**：
- 用 Tool 做纯数据查询（应设为 Resource，支持缓存和订阅）
- 把不同职责的参数放进同一个 Tool（"万能 Tool"反模式）

## Q22. MCP Server 和 MCP Client 各有什么核心职责？★★☆

**答：**

**Server 职责：**
- 初始化握手，声明能力
- 响应能力发现请求（`tools/list`、`resources/list`、`prompts/list`）
- 执行具体操作（`tools/call`、`resources/read`、`prompts/get`）
- 推送变更通知（`notifications/resources/updated`）
- 上报日志和进度

**Client 职责：**
- 连接管理（启动/连接/握手/重连/心跳）
- 能力发现（扫描所有已连接 Server 的工具/资源列表）
- 消息路由（将 LLM 的 Tool Call 路由到正确的 Server）
- 反向请求处理（Sampling、Roots）
- 生命周期管理（超时、取消、优雅关闭）
- 权限与审计（白名单、日志记录）

## Q23. 什么是 Stderr 日志通道？为什么 MCP 中 Stdout 不能用于日志？★★☆

**答：** 在 Stdio 传输模式下，stdin/stdout 是 JSON-RPC 协议通道（NDJSON 格式，每行一条 JSON 消息）。如果在 stdout 中 `print()` 日志，会污染协议通道，导致 Client 解析 JSON-RPC 消息失败。

**正确做法**：
- **stdout**：仅输出 JSON-RPC 消息
- **stderr**：用于所有日志输出
- Python：`logging.basicConfig(stream=sys.stderr)`
- Node.js：`console.error()` 或 `process.stderr.write()`
- Go：`log.SetOutput(os.Stderr)`

## Q24. MCP 协议中的通知（Notification）与请求（Request）有什么区别？★☆☆

**答：**

| 维度 | Request | Notification |
|------|---------|-------------|
| **是否有 id** | 有 | 无 |
| **是否需要响应** | 是（必须返回 Response） | 否 |
| **用途** | 主动查询/调用（`tools/list`, `tools/call`） | 事件推送（`notifications/initialized`, `notifications/resources/updated`） |
| **JSON-RPC 2.0 定义** | 有 `id` 字段的请求对象 | 无 `id` 字段的请求对象 |

这是 JSON-RPC 2.0 的基础概念，MCP 完全遵循。
