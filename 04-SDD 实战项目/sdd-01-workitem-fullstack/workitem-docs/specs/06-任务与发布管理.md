## 1. 为什么要把"任务"和"spec 发版"分开

`05-跨仓协调.md` 已经规定 spec 仓通过 **git tag** 对外发版。但现实中有一个它不解决的问题:

> **一次 spec 发版包含的变更,不一定全部都要同时上 PROD**。
> Task001 和 Task002 都合入了 main、spec 也已打 tag,但业务方只想让 Task001 功能可见。

这就是分期上线(staggered rollout)。若没有专门纪律,团队会退化到:

- 分支长期 holdback(几周不 rebase 主干,变烂)
- 环境分叉 migration 目录(`migration/prod/` vs `migration/dev/`,真源碎裂)
- 手工删脚本 / 回滚 Flyway checksum(不可追溯)

本项目采用**行业主流方案**:

> **DDL 始终向前兼容(additive-only)+ 功能可见性由 feature flag 控制**。
> DDL 能随时上 PROD,功能何时可见由代码旁边的 flag 决定。

## 2. Task 编号体系

### 2.1 格式

```
Task<NNN>
```

- 三位数字,从 `001` 开始连续递增
- 不复用已分配号码(Task002 被废弃,下一个新任务仍是 Task003)
- 编号一经分配即与文字描述绑定,**不重新解读为别的任务**

### 2.2 分配与登记

- 编号由 `workitem-docs` 仓的 issue tracker 分配(GitHub Issues / Gitee Issues / 团队内部 tracker)
- 分配时机:有人开始写 spec 草稿 **之前**,先开 issue 锁定编号
- **轨迹持久化**:满足阈值(DDL / 跨仓 ≥2 / 引入 flag / Expand-Contract / 改 BR 或接口契约)的 Task **必须**在 `workitem-docs/tasks/` 建一份 md 文件,模板与状态机详见 **`07-多任务并行开发.md §6`**
- issue tracker 承载日常讨论,Task md 承载**最终决策与轨迹**;两者不重复

### 2.3 Task 的跨仓落点(强制对齐)

同一个 Task-ID 必须在**所有相关位置**出现,一致到字节:

| 出现位置 | 格式示例(占位符) |
|---|---|
| `workitem-docs` 的 issue 标题 | `Task001: <短描述>` |
| `workitem-docs` 的 PR 标题 | `Task001: spec update for <feature>` |
| `workitem-backend` / `workitem-frontend` 的 PR 标题 | `Task001: <impl 摘要>` |
| Flyway 脚本文件名 | `V<N>__task001_<snake_case>.sql` |
| Flyway 脚本头注释 | `-- [SDD-TASK: Task001]` |
| Java 代码追溯注释 | `// [SDD-TASK: Task001][SDD-SPEC: 02-功能规范.md §4 BR-06]` |
| TypeScript 代码追溯注释 | `// [SDD-TASK: Task001][SDD-SPEC: ...]` |
| Feature flag 名 | `feat_task001_<短语>`(见 §5.1) |
| commit 消息行首 | `Task001: <动作>` |

**目的**:从任何一个工件(脚本、代码行、flag、commit)都能反查到 Task-ID,再反查到 spec 章节和 issue 讨论。

### 2.4 Task 与 Epic / spec tag 的关系

| 概念 | 来源 | 定义 |
|---|---|---|
| **Task** | 本文 | **最小可部署单元**。一个 Task 的所有 PR 合入后,对应的 DDL / 代码 / flag 可一起上 PROD(功能是否可见由 flag 决定) |
| **Epic** | `05-跨仓协调.md §4` | **跨仓协调载体**。多仓 PR 的父问题 |
| **spec tag** | `05-跨仓协调.md §2` | spec 仓对外发版的版本号 |

关系:

- 简单功能:**1 Epic = 1 Task**(PR 编排和部署单元一致)
- 破坏性变更:**1 Epic = 2+ Task**(Expand 一个 Task,Contract 另一个 Task,详见 §4)
- 1 个 spec tag 可以包含多个 Task 的 spec 变更,但**每个 Task 独立部署**

## 3. 核心发布原则

这 4 条原则是本文档一切其他规则的根基。**违反任一条即被视为 SDD 纪律缺陷**。

### 3.1 Additive-only 默认

任何 DDL 改动**默认**必须向前兼容:

- ✅ CREATE TABLE
- ✅ ADD COLUMN(必须可空或有默认值)
- ✅ CREATE INDEX
- ✅ CREATE VIEW
- ❌ DROP TABLE / DROP COLUMN
- ❌ ALTER COLUMN 改 NOT NULL / 改类型缩范围 / 改列名
- ❌ 改主键 / 改外键约束

落在 ❌ 里的变更,**禁止单步完成**,必须走 §4 Expand-Contract。

### 3.2 破坏性变更必须 Expand-Contract

见 §4。

### 3.3 功能可见性在代码侧 feature flag,不在 DDL

- DDL 只负责"能力存在",不负责"功能可见"
- 代码用 feature flag 决定是否走新路径、是否暴露新 UI
- flag 默认 **关闭**,按 env 配置开启

### 3.4 脚本与代码强制带 Task 编号

见 §2.3。

## 4. Expand-Contract 模式

### 4.1 什么场景必须走 Expand-Contract

下列任一**破坏性 DDL** 必须拆成至少 2 个 Task:

- 加 NOT NULL 列
- 改列名 / 改列类型(缩范围)
- 删列 / 删表
- 改主键 / 改唯一键

### 4.2 三阶段流程

```
[Expand 阶段]                [过渡期]                  [Contract 阶段]
┌─────────────────┐         ┌─────────────────┐      ┌─────────────────┐
│ TaskNNNa:       │         │ 代码双写新旧     │      │ TaskNNNb:       │
│ 加新结构(可空)  │  ──>    │ / 数据回填       │ ──>  │ 清理旧结构       │
│ 代码开始兼容写   │         │ / 所有服务部署   │      │ / DDL 转为严格   │
└─────────────────┘         └─────────────────┘      └─────────────────┘
     部署 PROD                 部署 PROD                  部署 PROD
     DDL 上线                  代码上线                   DDL + 代码上线
     feature flag 关           逐步开启 flag               flag 保留或移除
```

**关键纪律**:

- Expand 和 Contract **必须是两个独立的 Task**(不同 Task-ID,不同 PR,不同脚本版本号)
- 过渡期**至少跨越一次部署**,让所有 replica / 实例都跑过兼容代码,避免某些实例读到半态
- Contract 阶段开始前,必须人工确认:无任何客户端在访问旧结构(通过日志 / 监控确认)

### 4.3 场景示例(占位符)

#### 场景:把 `<table>.<col>` 改为 NOT NULL

**TaskNNNa:Expand**
```
V<N>__taskNNNa_add_nullable_<col>.sql       # DDL: ADD COLUMN <col> NULLABLE
V<N+1>__taskNNNa_backfill_<col>.sql         # (可选) DML 回填历史数据
# 代码 PR: 写入路径开始填 <col>,读取路径兼容 null
```

**TaskNNNb:Contract**(**必须在 Expand 全部部署完成并验证之后**)
```
V<M>__taskNNNb_enforce_<col>_not_null.sql   # DDL: ALTER COLUMN <col> SET NOT NULL
# 代码 PR: 读取路径删除 null 分支
```

#### 场景:改列名 `<old>` → `<new>`

**TaskNNNa:Expand**
```
V<N>__taskNNNa_add_<new>_column.sql          # ADD COLUMN <new>
V<N+1>__taskNNNa_backfill_<new>_from_<old>.sql
# 代码 PR: 双写 <old> 和 <new>,读取仍走 <old>
```

**TaskNNNb:Migrate(可与 Contract 合并,视情况)**
```
# 代码 PR: 读取切到 <new>,写入仍双写
```

**TaskNNNc:Contract**
```
V<M>__taskNNNc_drop_<old>_column.sql         # DROP COLUMN <old>
# 代码 PR: 只读写 <new>
```

### 4.4 为什么不走"单步 RENAME"

MySQL 的 `ALTER TABLE ... CHANGE` / `RENAME COLUMN` 是原子 DDL,但:

- 部署窗口内存在"旧代码 + 新 DDL"的半态,旧代码找不到旧列名即崩
- 主从复制期间可能出现短时不一致
- 回滚成本极高

所以 Expand-Contract 的**本质**是用时间换安全:新旧结构共存一段时间,让所有消费者平滑迁移。

## 5. Feature Flag 纪律

### 5.1 命名

```
feat_task<NNN>_<snake_case_短语>
```

示例占位符:
- `feat_task001_user_avatar`
- `feat_task005_payment_v2`

**规则**:

- 前缀 `feat_` 表示 feature flag(其他类型 flag 不在本文件范围)
- 必须包含 `task<NNN>` —— 让 flag 可以反查 Task-ID
- `<NNN>` **严格三位十进制数字**(`001` ~ `999`),与 Task-ID 完全一致,**禁止**简写为 `task1` / `task01`
- 后缀业务短语简洁,不超过 30 字符
- flag 名**全小写 + 下划线**,便于在代码、配置、数据库存储中保持一致

### 5.2 生命周期

本项目 flag 的完整生命周期跨越 `08-环境晋级与选择性发布.md §2.1` 的五层阶梯:

```
引入(默认关) → SIT 开启 → UAT 按需 → PROD-mirror 验证 → PROD 灰度/全开 → 移除
   TaskNNNa      (随合入)   (晋级决定)   (晋级决定)        (晋级决定)        TaskXYZ
```

**规则**:

- Flag 引入时**必须默认关闭**(代码中 `application.properties` 默认 `false`),仅在 dev 环境显式打开
- 每次环境晋级开启 / 关闭 flag 必须走 release plan(见 `08 §6`)
- flag 移除**必须是一个独立 Task**(不能与业务改动混在一起)
- **最晚 2 个 spec 版本内移除已全开的 flag**,防止长期僵尸 flag 腐蚀代码

### 5.3 禁用清单

- ❌ flag 值依赖数据库表(性能问题 + 启动期循环依赖)
- ❌ 在 DDL 里判断 flag(DDL 无法判断)
- ❌ flag 作为权限 / 鉴权手段(feature flag ≠ ACL)
- ❌ flag 命名不带 Task-ID
- ❌ 删除代码但保留 flag(必须一起删)

## 6. DDL 场景处理矩阵

本矩阵是 §3 ~ §5 的**强制参考表**。任何 DDL 改动必须对应到矩阵中的一行。

| 场景 | Additive? | 拆 Task 数 | 脚本数 | 备注 |
|---|---|---|---|---|
| 新建表 | ✅ | 1 | 1 | 一步到位 |
| 加可空列 | ✅ | 1 | 1 | 一步到位 |
| 加带默认值的 NOT NULL 列(默认值业务可接受) | ✅ | 1 | 1 | 一步(MySQL 大表谨慎,考虑 online DDL) |
| 加索引 | ✅ | 1 | 1 | 一步(大表用 `ALGORITHM=INPLACE` 或 pt-online-schema-change) |
| 加视图 | ✅ | 1 | 1 | 一步 |
| **加 NOT NULL 列(无可接受默认值)** | ❌ | **≥2** | ≥2 | Expand 可空+回填 → Contract 改 NOT NULL |
| **改列名** | ❌ | **≥2** | ≥2 | 加新列+双写+切读 → 删旧列 |
| **改列类型(缩范围,如 VARCHAR(200)→VARCHAR(100))** | ❌ | **≥2** | ≥2 | 加新列+数据验证+切读 → 删旧列 |
| **改列类型(扩范围,如 VARCHAR(100)→VARCHAR(200))** | ✅ | 1 | 1 | 扩范围本身向前兼容 |
| **删列** | ❌ | **≥2** | ≥2 | 先代码停用 → 再 DROP |
| **删表** | ❌ | **≥2** | ≥2 | 先代码停用 → 再 DROP |
| **改主键** | ❌ | **≥2** | ≥3 | 加新主键 / 双写 / 切读 / 删旧 |
| **加唯一约束(已有数据可能违反)** | ❌ | **≥2** | ≥2 | 先代码保证唯一 + 清理历史 → 加约束 |

## 7. 分期上线的三种打法

| 打法 | 做法 | 评价 |
|---|---|---|
| **α. feature flag + 共同上 PROD** | A/B DDL 都是 additive,都合 main 都上 PROD;flag 只开 A 的功能 | ⭐⭐⭐ **本项目主推** |
| β. B 的 DDL 留分支 | B 的 V<N+1> 不合 main,长期 rebase | ⚠️ 分支易腐、V 号冲突频发;**仅在 DDL 无法 additive 化且 flag 无法控制时勉强可用** |
| γ. 环境分叉 migration 目录 | `migration/prod/` 和 `migration/dev/` 分目录 | ❌ **本项目禁用**,破坏真源唯一性 |

### 7.1 选择 α 的前置条件

- Task002 的所有 DDL 都是 additive(对照 §6 矩阵)
- Task002 的代码路径能被 feature flag 完全屏蔽
- flag 已在所有会走 Task002 路径的服务里初始化为 **关**

满足以上 3 条,**α 总是优于 β/γ**。

### 7.2 什么情况必须走 β

- Task002 必须走破坏性 DDL 且无法 Expand-Contract(极罕见)
- 业务方要求 Task002 **完全不进入**代码库(例如竞品保密)

即使如此,也应先尝试把 Task002 拆成更小的 Task,让其中可部分的部分走 α。

## 10. 本文档的维护规则

- Task 编号格式的任何变更(如从 3 位改 4 位)**必须走 MAJOR spec tag**
- 新的破坏性 DDL 场景发现 → 回来补 §6 矩阵,走 MINOR tag
- 新的 flag 命名约定扩展 → 补 §5.1,走 MINOR tag
- feature flag 管理工具选型(如引入 LaunchDarkly / Unleash / 自建)**不在本文档范围**,由工程侧决定;本文档只管纪律
