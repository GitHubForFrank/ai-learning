# 92 — 接口治理（API 版本化 / 错误码）

> **文档定位**:接口版本化策略与错误码分段、命名与演化纪律。
> **适用范围**:本项目所有对外 HTTP 接口（REST）+ 错误响应体。
> **SDD 阶段**:横切约束（随各阶段增量更新）

---

# Part A — API 版本化

## A.1 版本化策略：URI 版本化

```
/api/v<N>/<resource>
```

**理由**：语义显式，浏览器/curl/日志/监控直观可见；路由层原生支持；客户端缓存友好。

**不选的方案（本项目禁用）**：

| 方案 | 原因 |
|---|---|
| `Accept` 媒体类型版本 | 日志不直观，调试成本高 |
| `Api-Version` 自定义头 | URI 级缓存层无法区分 |
| Query 参数 `?version=2` | 破坏 RESTful 幂等性语义 |

## A.2 版本号格式

- **整数版本号**：`v1`、`v2`、`v3` ...
- **禁止**小数 / 语义化版本：`v1.1` / `v1.2.3`
- 版本号只升不降，从 1 连续递增，**禁止跳号**

## A.3 Breaking Change 强制对照表

| 变更 | Breaking? | 处理 |
|---|---|---|
| 新增接口 | ❌ | 当前版本追加 |
| 新增可空请求字段 | ❌ | 当前版本追加 |
| 新增响应字段 | ❌ | 当前版本追加 |
| 删除接口 | ✅ | 新版本 + deprecation 流程 |
| 删除请求字段 | ✅ | 同上 |
| 删除响应字段 | ✅ | 同上 |
| 改必填字段为可选 | ❌ | 当前版本追加 |
| 改可选字段为必填 | ✅ | 新版本 |
| 改字段类型（含范围收紧） | ✅ | 新版本 |
| 改接口 URL 路径 | ✅ | 新版本 |
| 改 HTTP method / 语义 | ✅ | 新版本 |
| 改错误码语义（同 code 换含义） | ✅ | 新版本 |
| 新增错误码（不改旧码） | ❌ | 当前版本追加 |
| 改响应 envelope 结构 | ✅ | 新版本 + MAJOR spec tag |

## A.4 跨版本共存期

| 阶段 | 旧版 URI | 新版 URI | 时长建议 |
|---|---|---|---|
| Announce | 活跃 | 上线 | ≥ 1 个 spec MINOR 版本 |
| Deprecation | 响应头附 `Deprecation: true` + `Sunset: <日期>` | 活跃 | ≥ 2 个 spec MINOR 版本 |
| Freeze | 仅存量客户端 | 主推 | ≥ 1 个 spec MINOR 版本 |
| Remove | 物理删除 | 唯一 | 视情况 |

**硬性**：

- Deprecation 阶段每个响应**必须**附 `Deprecation: true`（RFC 8594）和 `Sunset: <HTTP-date>`
- 日志必须可统计旧版本 URI 调用次数
- Remove 必须在 spec **MAJOR tag** 中执行

## A.5 Deprecation 流程

```
① spec PR 标 deprecated + 指向替代 → ② MINOR 发版 → ③ 响应头加 Deprecation + Sunset（代码 PR）
→ ④ 发版通告 → ⑤ 监控旧版调用量 → ⑥ Sunset 到期后 spec PR 删除旧版 → ⑦ MAJOR 发版 + 代码删除
```

每一步必须对应一个 Task + release plan。

# Part B — 错误码治理

## B.1 段位分配（硬性）

所有业务错误码遵循 **4 位整数**：

| 段位 | 含义 | 可分配范围 |
|---|---|---|
| `0` | 成功 | 仅此一个值 |
| `1xxx` | 客户端错误（参数/格式） | `1000~1999` |
| `2xxx` | 业务规则冲突/状态不允许 | `2000~2999` |
| `3xxx` | 鉴权/授权失败 | `3000~3999` |
| `4xxx` | 速率限制/资源限制 | `4000~4999` |
| `5xxx` | 外部依赖失败 | `5000~5999` |
| `9xxx` | 系统未知/未捕获 | `9000~9999` |

新增错误码按上表分段。

## B.2 错误码命名

格式：`<SEGMENT>_<SUBJECT>_<VERB>`（全大写+下划线）

示例：`PARAM_INVALID`、`WORKITEM_NOT_FOUND`、`WORKITEM_DONE_IMMUTABLE`、`AUTH_REQUIRED`

## B.3 message 规范

- `message` 面向**最终用户**，**禁止**暴露栈/内部类名/SQL/服务名
- `message` 可 i18n（后端返回 i18n key + 参数，前端渲染）
- 同一 `code` 在不同场景的 `message` **可以不同**（如参数校验失败时包含具体字段名）

## B.4 错误码生命周期

| 阶段 | 允许动作 |
|---|---|
| 新增 | spec PR 中增加到功能规范错误码表 |
| 活跃 | 代码引用 + 测试覆盖 |
| 弃用 | spec 标 `deprecated`，保持表中，代码返回时加日志 WARN |
| 移除 | 从 spec 删除 + 代码删除，仅在 spec MAJOR tag 中执行 |

## B.5 错误码与接口版本的关系

- 新增错误码**不要求**接口升版（backward-compatible）
- 改变错误码语义 = breaking change，**必须**升接口版本 + MAJOR spec tag

## B.6 HTTP 状态码与业务码

HTTP 状态码是传输层语义，业务 `code` 是业务语义，二者**并行**存在。新增错误码**必须**标注其 HTTP 状态码归属。

## B.7 响应 envelope 版本化

- 新增 envelope 字段 ≠ breaking，但**必须**在 spec 中标注
- 修改 envelope 字段（`code` / `message` / `data` 等）= breaking → MAJOR tag + 新接口版本

## B.8 追溯注释

- Controller / Router 文件首标明 `@api-version: v<N>`
- 错误码常量上方标明 `[SDD-SPEC: 02-功能规范.md §5]`
- Deprecation 拦截器标明 `[SDD-SPEC: 92-接口治理.md §A.5]`

## B.9 禁用清单

- ❌ 小数版本号（`v1.1`）
- ❌ 跳号（`v1` 直跳 `v3`）
- ❌ 当前版本内做 breaking change
- ❌ Deprecation 无响应头 / 无 Sunset 日期
- ❌ 同 `code` 不同环境返回不同语义
- ❌ `message` 暴露技术细节
- ❌ 删除错误码不走 MAJOR tag
