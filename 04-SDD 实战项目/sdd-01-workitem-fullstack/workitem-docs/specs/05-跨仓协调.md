# 05 - 跨仓协调

> **文档定位**:回答「spec 仓和 3 个代码仓怎么协同、PR 怎么排期、版本怎么对齐」
> **SDD 阶段**:附加阶段 — Cross-Repo Orchestration
> **上游依赖**:`02-功能规范.md`(给出模块拆分)、`04-验收标准.md`(给出三维矩阵)
> **本文解答**:多仓场景下 SDD 怎么不散架。

---

## 1. 为什么 spec 独立成仓

### 1.1 单仓时的痛点

把 spec 放在代码仓的子目录(如 `specs/`),随代码一起版本化。看似直观,实际几个隐性问题:

- spec 的 PR review 和代码 PR 混在一起,**讨论焦点经常漂移**
- 多人在不同分支并行改 spec 时,冲突频繁且难 merge
- 跨团队(前后端不同团队)时,任何一方修改 spec 都对另一方隐形
- 历史上"spec 什么版本"不清晰,只能靠 commit hash 反查

### 1.2 独立仓带来的好处

| 好处 | 体现 |
|---|---|
| **决策审计** | spec PR 独立 review,历史清晰可追溯 |
| **版本化消费** | 下游用 tag(`v1.2.0`)锚定,避免"浮动引用" |
| **跨团队对齐** | spec 变更对所有下游是**显式事件**,不会被代码 PR 埋没 |
| **契合 SDD 哲学** | spec 是"真源";真源独立存放是自然结果 |

### 1.3 独立仓带来的代价

| 代价 | 应对 |
|---|---|
| 下游需要主动跟进 spec 版本 | 本文 §3 的对齐义务 + README 顶部 `Implements: spec@vX.Y.Z` |
| 跨仓功能变更需编排多个 PR | 本文 §4 epic 协调 |
| 紧急修 bug 时流程变重 | 本文 §7 hotfix 捷径(有,但有纪律) |
| 新人理解成本上升 | 工作区 `README.md` 给出"哪里是哪里"速查 |

## 2. spec 仓的版本化

### 2.1 Semantic Versioning

spec 仓的 tag 格式 **严格** `vMAJOR.MINOR.PATCH`:

| 版本位 | 触发条件 | 示例 |
|---|---|---|
| `MAJOR` | 破坏性变更:删接口 / 改错误码语义 / 删 BR / 改 DB 列名 / 改枚举值字符串 | v1.x.x → v2.0.0 |
| `MINOR` | 向后兼容新增:加接口 / 加可空字段 / 加 BR / 加模块 / 加页面 | v1.1.0 → v1.2.0 |
| `PATCH` | 文档澄清 / 拼写 / 示例更正 / 不改变契约的措辞修正 | v1.2.0 → v1.2.1 |

> ⚠️ **"契约变了没"判断基准**:任何能让下游**测试用例期望改变**的变更,至少是 MINOR;任何能让**已通过的测试用例失败**的变更,是 MAJOR。

### 2.2 Tag 的打法

```
spec PR merge 到 main
  → 维护者在 main 上打 tag:git tag vX.Y.Z -m "..."
  → git push origin vX.Y.Z
  → 在 GitHub / Gitee 对应 tag 上写 Release Notes(列出改动的 BR / 模块 / 条款)
```

### 2.3 下游仓的锚定方式

下游 3 仓的 **README 顶部** 必须显式声明:

```markdown
**Implements**: spec@v1.2.0
```

- 下游仓代码**不依赖** spec 仓作为运行时依赖,不用 submodule
- 对齐是**文字声明**,依赖 PR review 和测试用例保障
- 升级 spec tag = 下游提 PR 改两处:README 顶部 + 对应实现代码

## 3. 下游仓对齐 spec 的义务

每个代码仓(`workitem-backend` / `workitem-frontend` / `workitem-db`)必须:

### 3.1 静态义务

1. README 顶部 `Implements: spec@vX.Y.Z` 始终与 main 实际实现一致
2. 目录结构符合 spec `../design/03-技术方案.md §2` 和 `specs/*`
3. 关键位置有 `[SDD-SPEC: xx.md §y.z]` 追溯注释,其中 `xx.md` 指 `workitem-docs/` 下的文件

### 3.2 动态义务(PR 纪律)

每个下游仓的 PR 描述**必须**包含三段:

```markdown
## Task
- Task001 (see workitem-docs issue #N)

## Implements
- spec version: v1.2.0 (no bump) / v1.2.0 → v1.3.0 (bump)

## Covers
- §3.4 API-04 update
- §4 BR-06 DONE immutable priority/dueDate
- §8 BE-02 service rule
- TC-U-05 (new), TC-IT-10 (updated)
```

- `Task` 给出 Task-ID,详见 `06-任务与发布管理.md §2`
- `Implements` 声明这次 PR 涉不涉及 spec tag 跳变
- `Covers` 列出覆盖的 spec 条款 / 模块 / 测试用例
- review 者据此反查 spec 原文,判断实现是否忠于契约

PR **标题**必须以 Task-ID 起头,格式:`Task<NNN>: <简述>`。

## 4. 跨仓功能的 Epic 协调

### 4.1 什么是"跨仓功能"

**只要一个功能的业务规则在三维矩阵中覆盖了 ≥ 2 个模块**,就是跨仓功能。典型场景:

- 新加一条 BR(如"任务支持标签"):spec + FE + BE + DB 四仓都要动
- 改错误码语义:spec + BE + FE 三仓都要动
- 加一列:spec + BE + DB 三仓,可能波及 FE

### 4.2 Epic 编号与 Task 的关系

每个跨仓功能立一个 **Epic ID**,格式 `EPIC-<NN>-<短名>`,例:`EPIC-01-add-tags`。

**Epic 与 Task 的映射关系**(Task 定义见 `06-任务与发布管理.md §2`):

| 场景 | 映射 | 说明 |
|---|---|---|
| 简单功能(additive-only) | **1 Epic = 1 Task** | 一次部署全部生效,PR 编排和部署单元一致 |
| 破坏性变更(走 Expand-Contract) | **1 Epic = 2+ Task** | Expand 一个 Task,Contract 另一个 Task,跨越多次部署 |
| 大型功能需分阶段交付 | **1 Epic = N Task** | 按业务 / 技术边界拆分,每个 Task 独立部署 |

Epic 的**协调物**可以是:

- (轻量)一个 GitHub / Gitee Issue,描述跨仓拆分和 PR 编排
- (重量)一个独立的项目管理工具条目

本项目**规划**使用轻量方案(Issue 贴在 `workitem-docs` 仓),实际工程按团队选择。

### 4.3 Epic 对应的 PR 编排(关键纪律)

**严格顺序**:

```
① workitem-docs PR     (spec 变更 + 打新 tag)
           ↓  tag 发布
② workitem-db PR       (DDL 跟进)
           ↓
③ workitem-backend PR  (BE 实现)
           ↓
④ workitem-frontend PR (FE 实现)
           ↓
⑤ 回到 workitem-docs   (若 04-验收标准.md 的三维矩阵需回填 TC-E2E-*,再提一个 PATCH PR)
```

**关键约束**:

- ② / ③ / ④ 的 PR 描述**必须**引用 ① 的 spec tag
- ③ 的集成测试跑通后,才能合 ④
- ④ 的 E2E 跑通后,Epic 才算关闭
- 若 ① 在 review 中被否决,② / ③ / ④ 一起作废,**不允许**"spec 没通过但代码先合"

### 4.4 为什么是这个顺序

- **spec 先行**:这是 SDD 本色,也防止"代码决定了 spec 措辞"的倒挂
- **DB 次之**:DDL 一旦在 staging 落地,BE 才能集测
- **BE 在 FE 前**:FE 的 E2E 需要真 BE 接口,BE 不就绪时 FE 只能 mock
- **E2E 回到 spec**:三维矩阵的 TC-E2E-* 补齐是验收闭环的最后一步

## 5. 破坏性变更(MAJOR)的处理

### 5.1 基本原则

MAJOR 变更意味着**下游仓必然需要代码改动**,不能平滑升级。本项目要求:

1. 先在 `workitem-docs` 提 RFC PR(议题讨论),**不合 main**,只作对齐
2. RFC 通过后,主 spec PR 提交,同时**预告下游仓 breaking change 清单**
3. 给下游团队**至少一个迭代周期**(本教学项目不做强制,生产按团队约定)
4. 下游仓在新 tag 发布后走 §4.3 顺序升级

### 5.2 不允许的做法

- ❌ "spec 偷偷改,下游仓自己发现" —— 必须走 RFC
- ❌ "下游代码先改,spec 事后追认" —— 这就是传统开发的反例
- ❌ 在 PATCH tag 里塞语义变更

## 6. 三维矩阵的维护责任归属

| 矩阵单元(BR × 模块 × TC)的变化 | 维护责任 |
|---|---|
| 新增 BR 的行 | `workitem-docs` 提 PR 时,同步在 `04-验收标准.md §2` 加行 |
| 某模块填入新 TC 编号 | 对应代码仓 PR 必须回到 `workitem-docs` 追加 TC 编号(可以是同一 Epic 下的最后一个 PR) |
| 废弃 TC | `workitem-docs` 提 PR 在矩阵中标删除线,**编号不复用** |

> ⚠️ 矩阵是"活文档",但只住在 `workitem-docs`。其他仓的测试代码里的 TC-* 注释是**引用**,不是真源。

---

## 7. Hotfix 捷径(有纪律的)

当线上出现严重 bug,上面 §4.3 的完整编排太慢,允许走 hotfix 流程:

### 7.1 前置条件

- 必须是**线上缺陷**,不是功能补救
- 修复方案必须符合现行 spec(即 spec 本身没错,是代码背离 spec)

### 7.2 流程

1. 在**对应代码仓**直接提 PR,PR 描述明确标 `Hotfix: spec unchanged`
2. 合入并发布
3. **事后 72 小时内**,在 `workitem-docs` 提 PATCH PR,更新 `04-验收标准.md` 新增一条 TC 覆盖该 bug 场景,打新 PATCH tag(如 v1.2.0 → v1.2.1)
4. 下游仓对应 tag 升级

### 7.3 纪律

- hotfix 的 PR 必须附事后 TC 补齐的 issue 链接;**72 小时不补齐视为违规**
- 如果修复方案要改动 spec 语义,**不走 hotfix**,走 §4 epic

## 8. 与 AI 协作的特别说明

spec 仓独立成仓的一个隐性好处是:它本身就是**最优质的 AI 上下文**。

- 让 AI 生成 / 修改代码时,优先把**整个 `workitem-docs` 仓**作为上下文投喂
- 代码 PR 的 prompt 应当引用 spec 版本:`"依据 spec@v1.2.0 §4 BR-06 实现 ..."`
- 当 AI 产出与 spec 冲突时,规则同人类:**改代码去对齐 spec**,除非先有 spec PR

---

## 9. 与本项目教学场景的简化说明

本项目把 4 仓**物理放在一个工作区目录下**,并没有真正 clone 4 个 git 仓库。因此:

- 本教学仓内的跨仓 PR 编排**无法真实演示**
- 但 spec tag 的打法、README 声明、PR 描述模板**完全适用**于真实 4 仓场景
- 学习者如果要完整演练,可以把 4 个目录拆成 4 个独立仓推到真实远端,然后按 §4.3 走一次完整 Epic

---

## 10. 本文档的维护

- 跨仓协作规则发现不合理 → 回来改本文档,走 MINOR tag
- 本文档只写**规则与纪律**,不写具体 spec 内容(那是 01~04 的职责)
- 任何与本文档冲突的下游约定,以本文档为准;如确需偏离,必须先在本文档加例外条款

---

**结语**:spec 独立成仓不是增加流程,而是**让原本隐式的决策边界显式化**。
当 spec tag、下游 README 声明、PR `Covers` 描述三者齐备时,任何一次跨仓变更都能反查到"为什么、谁拍的板、哪些实现要跟、哪些测试要补",这是真正让 SDD 在多团队多仓库下落地的核心机制。
