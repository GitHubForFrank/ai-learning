# 04 - 验收标准

> **文档定位**:回答「怎么证明三模块代码都做对了」
> **SDD 阶段**:Phase 4 — Acceptance
> **上游依赖**:`02-功能规范.md` 全部接口契约、BR-01 ~ BR-08、§8 模块拆分
> **核心原则**:
> 1. 每条 spec 规则都必须有**至少一条**自动化测试覆盖
> 2. 每条测试用例都能**反向追溯**到 spec 条款与模块编号
> 3. 跨模块业务规则必须有至少一条 **L4 E2E** 用例兜底
>
> ⚠️ 本文档的测试用例清单是**规划层产出**。具体用例方法名与实现在 P2/P3/P4 阶段补齐,本阶段先冻结"哪些用例必须存在"。

---

## 1. 验收层级

| 层级 | 验收方式 | 工具 | 对应文件(P2/P3/P4 产出) |
|---|---|---|---|
| **L1** 手工接口验收 | curl 命令 + 浏览器点击 | curl / Playwright headed | 本文档 §3 |
| **L2** 后端单元测试 | Service 层业务规则验证 | JUnit 5 + Mockito | `workitem-backend/src/test/java/.../application/service/` |
| **L3** 后端集成测试 | HTTP → Controller → Service → Repository → Mapper → MySQL 真库 | Spring Boot Test + MockMvc + **Testcontainers** | `workitem-backend/src/test/java/.../app/api/controller/` |
| **L3'** 前端组件测试 | 关键组件的行为与校验 | Vitest + Vue Test Utils | `workitem-frontend/src/**/*.spec.ts` |
| **L4** E2E 端到端 | 浏览器真实点击,贯穿三模块 | Playwright | `workitem-frontend/tests/e2e/*.spec.ts` |
| **L5** 回归测试 | 历史功能 + **DB 物理真相**不被破坏 | MockMvc + DB Probe + **`sdd-db-probe` MCP** | 见 **§5** 回归层 |

> ⚠️ 每个 BR 可能分布在多个层级,详见 §2 三维矩阵。

## 3. L1 手工验收用例

### 3.1 后端 curl 用例(HT-01 ~ HT-16)

> **前置**:
> - MySQL 容器已启动(验证:`docker ps | grep mysql`)
> - 后端已启动(验证:`curl http://localhost:10197/app/api/workitems` 返回 `code=0`)
> - 具体启动命令 P2 阶段在 README 补齐
> - 下列 curl 以 `BASE=http://localhost:10197/app` 为基准路径(与 `../design/03-技术方案.md §9.1` `server.port=10197` + `context-path=/app` 一致)

| ID | 场景 | 请求 | 期望 |
|---|---|---|---|
| HT-01 | 创建任务(正常) ✅ | `POST $BASE/api/workitems` body `{"title":"...","priority":"HIGH","dueDate":"2026-04-30"}` | HTTP 200;`code=0`;`data.status=TODO`;`data.priority=HIGH` |
| HT-02 | 创建(title 为空) ❌ | `POST $BASE/api/workitems` body `{"title":""}` | HTTP 400;`code=1001`;`message` 含 `title` |
| HT-03 | 创建(title 超 100) ❌ | `POST $BASE/api/workitems` body `{"title":"<101 chars>"}` | HTTP 400;`code=1001` |
| HT-04 | 创建(请求带 status=DONE,应被忽略) | `POST $BASE/api/workitems` body `{"title":"BR-01 test","status":"DONE"}` | `data.status=TODO`(BR-01 强制) |
| HT-05 | 创建(不传 priority,默认 MEDIUM) | `POST $BASE/api/workitems` body `{"title":"BR-02 test"}` | `data.priority=MEDIUM`(BR-02) |
| HT-06 | 分页查询(默认参数) ✅ | `GET $BASE/api/workitems` | HTTP 200;`data.page=1`;`data.size=10`;`data.list` 为数组 |
| HT-07 | 按状态筛选 | `GET $BASE/api/workitems?status=TODO&page=1&size=5` | `data.list` 中所有项 `status=TODO` |
| HT-08 | 分页越界 ❌ | `GET $BASE/api/workitems?page=0`<br>`GET $BASE/api/workitems?size=101`<br>`GET $BASE/api/workitems?status=INVALID` | 均 HTTP 400,`code=1001` |
| HT-09 | 排序验证 | 连续创建 3 个任务后 `GET $BASE/api/workitems` | `data.list[0]` 是最后创建的(`createdAt DESC`,BR-04) |
| HT-10 | 查询不存在任务 ❌ | `GET $BASE/api/workitems/999999` | HTTP 404;`code=1002`;`message` 含 `not found` 或"任务不存在" |
| HT-11 | 查询已删除任务 ❌ | `DELETE $BASE/api/workitems/1` 后 `GET $BASE/api/workitems/1` | HTTP 404;`code=1002`(BR-03) |
| HT-12 | 更新(空 body) ❌ | `PUT $BASE/api/workitems/1` body `{}` | HTTP 400;`code=1001`;`message` 含"至少"(BR-05) |
| HT-13 | DONE 任务改 priority ❌ | 先创建并 `PUT status=DONE`,再 `PUT priority=LOW` | 第二次 HTTP 409;`code=1003`(BR-06) |
| HT-14 | 正常更新 | `PUT $BASE/api/workitems/1` body `{"title":"新标题","status":"DOING"}` | HTTP 200;`data.title="新标题"`;`data.updatedAt` 已刷新(BR-08) |
| HT-15 | 逻辑删除 ✅ | `DELETE $BASE/api/workitems/1` | HTTP 200;`code=0`;`data=null`;数据库中 `deleted=1`(BR-07) |
| HT-16 | 重复删除 ❌ | `DELETE $BASE/api/workitems/1` 执行两次 | 第二次 HTTP 404;`code=1002`(幂等) |
| HT-17 | 登录正常 ✅ | `POST $BASE/api/login` body `{"username":"admin","password":"12346#@&"}` + header `traceparent: 00-...` | HTTP 200;`code=0`;`data.token` 非空;`data.expiresIn=3600` |
| HT-18 | 密码错误 ❌ | `POST $BASE/api/login` body `{"username":"admin","password":"wrong"}` | HTTP 401;`code=3001`(消息不区分用户名 vs 密码) |
| HT-19 | 失败 3 次后锁定 ❌ | 连续 4 次 HT-18 | 第 4 次 HTTP 423;`code=3002`;DB `app_user.locked_until` 在未来 15 分钟内 |
| HT-20 | 锁定期间正确密码也拒 | HT-19 后立即正确密码登录 | `code=3002`(锁定不放行,即使密码正确) |
| HT-21 | task API 缺 token ❌ | `GET $BASE/api/workitems` 不带 `Authorization` | HTTP 401;`code=3003` |
| HT-22 | task API 带有效 token ✅ | HT-17 取 token,`GET $BASE/api/workitems` header `Authorization: Bearer <token>` | HTTP 200;`code=0` |

> 部分用例有顺序依赖(如 HT-11 依赖 HT-01 创建的数据)。执行时先按编号顺序,或在每个用例前通过 `POST $BASE/api/workitems` 准备数据。

### 5.1 本地全量回归流程(规划)

1. 启动数据库容器
2. 运行后端 `mvn test`(含 L2 + L3,Testcontainers 会自动拉起 MySQL 容器用于测试)
3. 前端目录下运行 `npm run test:unit`(L3')
4. 前端 + 后端 + 数据库全部 `up` 后,运行 `npm run test:e2e`(L4)
5. 可选:手工跑一遍 §3 L1 用例

### 5.2 CI 流程(不在本项目范围)

本项目**不实现 CI 脚本**(参见 `01-需求分析.md §6`),但规划上三层自动化全部可在一个 job 内串起来。
