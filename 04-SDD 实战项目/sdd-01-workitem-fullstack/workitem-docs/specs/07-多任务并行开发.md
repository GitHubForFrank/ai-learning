# 07 - 多任务并行开发

> **文档定位**:回答「N 个 Task 同时进行时,怎么不打架、怎么留下完整轨迹」
> **SDD 阶段**:附加阶段 — Parallel Development Discipline
> **上游依赖**:`05-跨仓协调.md`(单 Epic 跨仓编排)、`06-任务与发布管理.md`(单 Task 发布纪律)
> **下游依赖**:本文规则被 4 仓 README 与 `specs/*` 引用
>
> **与 05 / 06 的分工**:
> - `05`:**一个**跨仓功能怎么编排 PR
> - `06`:**一个** Task 怎么编号、分期上线
> - `07`(本文):**N 个** Task 同时进行时,如何避免冲突、保留轨迹、并发部署

---

## 1. 适用场景

本文规则在以下情境强制生效:

- 同一 sprint / 同一周内有 ≥ 2 个 Task 在 4 仓中任一处提交 PR
- 任意 Task 的分支寿命预计 > 24 小时
- 任意 DDL 改动(无论单 Task 与否)
- 任意 BR-* / TC-* / FE-* / BE-* / DB-* 编号新增

只要触发其中之一,本文规则即生效;**全部走完才允许合入 main**。

---

## 2. 分支策略:Trunk-based + 短分支

### 2.1 核心原则

| 原则 | 含义 |
|---|---|
| **main 永远 deployable** | main 上的代码任一刻都能上 PROD,功能可见性由 feature flag 决定(参见 `06 §3.3`) |
| **从 main 拉短分支** | 每个 Task 一个分支,分支只解决该 Task 的问题 |
| **分支寿命 ≤ 5 工作日** | 超过即视为风险,需 review owner 介入或拆分 Task |
| **频繁 rebase**,不用 merge commit | 每天 rebase main,保持线性历史 |

### 2.2 分支命名

```
task<NNN>-<kebab-short-name>
```

例(占位符):`task001-add-user-tags`、`task003a-add-nullable-phone`(Expand 阶段)。

分支名**必须**与 Task-ID 严格对应,不允许:

- ❌ 一个分支承载多个 Task
- ❌ Task-ID 与分支名上的不一致
- ❌ 用 commit message 起头表达 Task-ID,而分支名隐藏

### 2.3 禁用清单

- ❌ **长寿分支(> 5 工作日)** —— 必须拆分或加速合入
- ❌ **大量 merge main 到分支** —— 用 rebase,避免 merge commit 污染历史
- ❌ **跨 Task 互相 cherry-pick** —— 真有依赖请走 §5 显式声明
- ❌ **release 分支 / hotfix 分支长期存在** —— hotfix 走 `05-跨仓协调.md §7`,完成即关
- ❌ **私有分支**(只在本机存在 > 2 天的工作)—— 必须 push 到远端,review 可见

## 3. 编号空间与撞号防控

### 3.1 会撞号的 5 类编号

| 编号 | 出处 | 撞号频率 |
|---|---|---|
| `Task<NNN>` | `06 §2` | 中(开新 issue 时锁) |
| `BR-<NN>` | `02-功能规范.md §4` | 中(spec PR 撞) |
| `TC-U-* / TC-IT-* / TC-FE-* / TC-E2E-*` | `04-验收标准.md §4` + `specs/90-工程约束.md Part B` | 中 |
| `FE-<NN> / BE-<NN> / DB-<NN>` | `02-功能规范.md §8` | 低(模块拆分动作不频繁) |
| `V<N>__*.sql` | `specs/91-编码规范.md §A.6` | **高**(DDL 一旦多人并行就撞) |

### 3.2 "先占位再写"

每类编号在**开始动手之前**必须先在 spec 仓**显式占住**编号:

| 编号 | 占位时机 | 占位机制 |
|---|---|---|
| `Task<NNN>` | 开 issue 时 | issue 标题 `Task<NNN>: <短描述>` + 在 `workitem-docs/tasks/Task<NNN>-*.md` 提一个**只含 frontmatter 的占位 PR**(参见 §6) |
| `BR-<NN>` | spec PR 草案打开时 | spec PR 在 `02-功能规范.md §4` 表格里**先添加一行**(描述列可暂为 `TBD`),并立即 push 到远端 |
| `TC-*` | 同 spec PR 草案 | 在 `04-验收标准.md §4` 添加用例占位行 |
| `FE-* / BE-* / DB-*` | 同 spec PR 草案 | 在 `02-功能规范.md §8` 模块表添加占位行 |
| `V<N>` | DB 仓 PR 创建时 | DB 仓本地 `git pull origin main` 看最新 V 号,再 +1;立即 push 分支占位 |

### 3.3 撞号已发生时的 resolve

- **后 push 的人 rebase**:重新分配下一个可用编号,改文件名 / 表格行 / 代码注释,一并 push
- **不允许**:用同号但加 `_v2` 后缀、留两个相同编号"看哪个先合"、口头协商不改实物
- 编号占位 commit 即使被 review 拒绝,**该编号仍消耗**(不复用);新一轮使用下一个编号

### 3.4 编号永不复用

无论占位是否最终落地,**已分配的编号永不复用**:

- Task002 被废弃 → 下一个新 Task 是 Task003
- BR-09 被删除 → 下一个新 BR 是 BR-10
- 这条规则是 SDD "决策审计可追溯"的基本要求

## 4. Flyway V 号撞号专项

V 号是**最容易撞号**的编号,因此单独成节。

### 4.1 撞号检测(DB 仓 CI 强制)

DB 仓 CI **必须**在每次 PR 上跑以下校验:

| 校验项 | 期望 |
|---|---|
| V 号唯一性 | `migration/V<N>__*.sql` 在 main + 当前 PR 中**总和无重复 N** |
| V 号连续性 | main 历史中 V 号从 1 起连续递增,无跳号 |
| Task-ID 存在性 | 每个脚本名中的 `task<NNN>` 段在 `workitem-docs/tasks/` 中对应 md 文件存在(可放宽为"对应 issue 存在") |
| Expand-Contract 配对完整性 | 若 frontmatter 标 `expand-contract-pair`,被指向的 Task md 必须存在(可为 active) |
| 已发布脚本未被修改 | main 历史中已合入的 `V<N>__*.sql` 在当前 PR 中**内容未变**(checksum 比对) |

任一失败 → PR 阻塞合入。

### 4.2 撞号 resolve 流程

```
PR-A (V5__task001_*.sql) 先合入 main
PR-B (V5__task002_*.sql) CI 校验失败:V5 已存在
   ↓
PR-B 作者操作:
  ① git pull origin main + git rebase
  ② 把 V5__task002_*.sql 改名为 V6__task002_*.sql
  ③ 检查 PR 描述、Task md、代码注释中所有 V5 引用,改为 V6
  ④ git push --force-with-lease(分支保护下安全)
  ⑤ CI 重跑,通过则进入 review
```

> ⚠️ 即使本地 PR-B 创建早于 PR-A,**先合入者赢得编号**。这是 git 物理事实,不是公平问题。

### 4.3 禁用做法

- ❌ 用日期戳 `V20260425120000__*.sql` 替代序号(Flyway 支持但顺序判读复杂,本项目禁用)
- ❌ 跳号(`V5` 后直接 `V7`)以"避免冲突"
- ❌ 修改已合入脚本的内容(无论是否影响 checksum)
- ❌ 撞号时通过删除对方的脚本来"resolve"

## 5. Task 间依赖管理

### 5.1 显式声明

每个 Task md 文件 frontmatter **必须**有 `depends-on` 字段,字段格式见 **§6.2**(此处不重复)。

**依赖的语义定义**:

依赖**仅指语义依赖** —— 本 Task 的实现需要另一个 Task 的 schema / 接口 / 类型 / flag 已存在。
以下情况**不算依赖**:

- 单纯"撞同一个 PR review queue"
- 两个 Task 改动撞同一个文件但逻辑独立(属于 §3 编号空间或 git 冲突,不是语义依赖)
- 两个 Task 都依赖的是 main 上已合入的历史 Task(那是"共同基础",不需要声明)

### 5.2 依赖图与拓扑顺序

- 所有 active Task 的依赖关系构成**有向无环图(DAG)**
- 没有依赖箭头连通的两个 Task **可以并行**
- 有依赖箭头的 Task,**被依赖方**必须先合入 main(并视情况打 spec tag)

### 5.3 串行 / 并行 决策

| 场景 | 处理 |
|---|---|
| 两个 Task 都是 additive、无共享文件 | **并行**,各自分支推进 |
| 两个 Task 都改同一份 spec md(如 §4 业务规则)| 串行(后者 rebase) |
| Task002 依赖 Task001 的新表 | **串行**:Task001 先合 + 打 tag,Task002 再开始 |
| Task002 依赖 Task001 的新接口,但 Task001 仅是 BE 改 | **可部分并行**:Task002 在 FE 用 mock 推进,等 Task001 合入再切真实接口 |
| Expand-Contract 配对(Task003a / Task003b) | **强制串行,跨多次部署**(参见 `06 §4`) |

### 5.4 循环依赖处理

发现两个 Task 互相依赖时:

1. **拆分**:把某一方再拆成 a / b 两个 Task,打破环
2. **合并**:确认实在无法拆,合并成一个 Task(但要警惕 Task 过大,寿命可能超 §2.1 限制)
3. **绝不允许**:让两个 Task 长期 holdback 等对方

## 6. Task 持久化文档:`tasks/Task<NNN>-*.md`

### 6.1 位置与命名

```
workitem-docs/
└── tasks/
    ├── README.md                            # 目录说明
    ├── Task001-add-user-tags.md             # 一 Task 一文件
    ├── Task002-fix-priority-default.md
    ├── Task003a-add-nullable-phone.md       # Expand 阶段
    └── Task003b-enforce-phone-not-null.md   # Contract 阶段
```

- **单目录**(不分 active / archive 子目录),状态由 frontmatter 控制
- 文件名:`Task<NNN>-<kebab-short>.md`
  - `<NNN>` 三位数字
  - `<kebab-short>` 小写 + 短横线,3~6 个英文单词
- **代码追溯注释**仍只引用 Task-ID(`[SDD-TASK: Task001]`),不带文件名后缀,便于跨工件 grep

### 6.2 frontmatter 字段(必填)

```yaml
---
id: Task001                       # 与文件名前段一致
title: <短描述>                    # 一行,< 60 字符
status: active                    # active | closed
owner: <name>                     # 单一 owner
depends-on: []                    # 显式依赖,见 §5.1
spec-version-target: v1.2.0       # 预期合入后 spec 仓的 tag
expand-contract-pair: null        # 若是 Expand-Contract 中一员,填配对 Task-ID
created: YYYY-MM-DD               # 开 issue 当日
closed: null                      # 关闭时填 YYYY-MM-DD
---
```

### 6.3 正文必填段

```markdown
## Why
<1~3 行业务动机,不重复 spec 里的 BR 描述,只说"为什么现在做这个 Task">

## Scope
- 仓: <workitem-docs / workitem-backend / workitem-frontend / workitem-db,挑相关的列出>
- 影响编号: <BR-09 (新) / FE-07 / DB-04 等,与 02-功能规范 §8 对齐>
- DDL: <V<N>__taskNNN_*.sql 列表;若无 DDL 写 "none">
- Feature flag: <feat_taskNNN_<短语>;若无 flag 写 "none">

## Out of Scope
<显式不做的事;承载的是"边界澄清",不是"未来工作">

## Regression
- 必跑用例(合入前):<TC-R-API-* / TC-R-DB-* 列表,至少列出本 Task 影响的 BR 对应用例>
- 新增用例:<若本 Task 新增了 TC-R-* 编号,列出;无则写 "none">
- 有 DDL 时必配:<TC-R-MIG-<NN>;否则写 "N/A">
- MCP 查询审计:<若本 Task 调用过 sdd-db-probe MCP,列出 query_id 列表;否则写 "N/A">

## PR Trail (合入时回填)
- workitem-docs PR:    <link>
- workitem-db PR:      <link>
- workitem-backend PR: <link>
- workitem-frontend PR:<link>

## Closure (PROD 上线后回填)
- spec tag bumped to: vX.Y.Z
- PROD 上线日期: YYYY-MM-DD
- Lessons: <可选,1~3 行>
```

> ⚠️ 总长度建议**控制在 60 行内**。Task md 不是 RFC,是**轨迹记录**。
> 大型设计讨论留在 issue / PR review 评论中,Task md 只沉淀"这件事是什么、最终怎么做的"。

### 6.4 阈值(并非所有 Task 都强制 md)

| Task 类型 | 是否需 md | 理由 |
|---|---|---|
| 涉及 DDL | ✅ 必须 | DDL 持久化决策,需轨迹 |
| 跨仓 ≥ 2 仓 | ✅ 必须 | PR Trail 需汇总 |
| 引入 / 移除 feature flag | ✅ 必须 | flag 生命周期需追踪 |
| Expand-Contract 配对 | ✅ 必须(双方各一个 md,互相 link) | 跨多次部署需严格追溯 |
| 新增 BR-* / 改 BR 语义 | ✅ 必须 | 决策审计 |
| 新增 / 改 接口契约 | ✅ 必须 | 同上 |
| spec PATCH(typo / 措辞) | ❌ 免 | 走普通 PR 即可 |
| 单仓内部重构(不动契约) | ❌ 免 | 走普通 PR 即可 |
| 升级依赖版本(无契约影响) | ❌ 免 | 走普通 PR |

### 6.5 状态机

```
issue 开启
   ↓
创建 Task md,frontmatter:status: active
   ↓
进入开发(分支创建,跨仓 PR 推进)
   ↓
所有相关 PR 合入 main + spec tag(若需要)发布
   ↓
PROD 上线后,回填 Closure 段,frontmatter:status: closed,closed: YYYY-MM-DD
   ↓
md 文件留在原位,不删不移动(保留历史轨迹)
```

**禁用做法**:

- ❌ 删除已 closed 的 Task md
- ❌ 把 closed 的 md 移动到子目录(破坏链接)
- ❌ 长期 active 的 md 不更新 status(超过 spec-version-target 两个版本仍 active 必须强制关闭或拆分)

### 6.6 跨仓 PR 引用规范

每个跨仓 PR 描述里(模板见 `05 §3.2`)的 `## Task` 段必须**链接到 Task md 文件**:

```markdown
## Task
- Task001 — see [tasks/Task001-add-user-tags.md](../workitem-docs/tasks/Task001-add-user-tags.md)
```

> ⚠️ 链接是相对工作区路径(教学项目);真实多仓场景下用 spec 仓的 web URL(如 `https://github.com/<org>/workitem-docs/blob/main/tasks/Task001-...md`)。

## 7. 环境分层(通用视角)

> 本节给出**通用环境概念**(用于并行纪律讨论)。
> 具体 SIT / UAT / PROD 晋级规则 + flag 矩阵 + release plan 详见 **`08-环境晋级与选择性发布.md`**。

### 7.1 dev(个人开发)

- **每个开发者本机自备 MySQL 8.0**(本机已装 / 内网共享开发库 / 任何可达实例均可),互不干扰
- Flyway 跑本机 main + 本地分支的所有 migration
- 想"重置"环境:`DROP DATABASE zmz_sdd_demo; CREATE DATABASE zmz_sdd_demo ...`,Flyway 自动重跑

### 7.2 staging / SIT(共享集成)

> ⚠️ **术语说明**:本项目中 "staging"、"SIT"、"共享集成环境"**完全等价**,指向同一个环境实例。
> 下文通用语境沿用 "staging";具体晋级规则与环境阶梯(含 UAT / PROD-mirror / PROD)以 "SIT" 命名,详见 `08 §2`。

- 一个共享 MySQL 实例,跟随 `workitem-db` 的 main 自动 migrate
- 部署节奏:每个仓 main 合入 → 自动部署到 SIT
- 任何破坏 SIT 的 PR(如 V 号撞号未 resolve)**必须立即回滚**
- 详见 `08 §2` SIT 层职责

### 7.3 preview(每 PR 临时,可选)

- 每个 PR 拉起一组临时 MySQL + BE + FE 容器
- 用于:跨仓 Task 的早期联调、QA 实地验证
- 本项目教学场景**不强制实现**,仅作规划

### 7.4 UAT / PROD-mirror / PROD

- UAT / PROD-mirror / PROD 的晋级流程、flag 子集、release plan 规则**全部**在 `08` 中定义
- 本文不重复,仅强调:
  - 受控发布:只接收带 spec tag 的版本(`05 §2`)+ 通过所有 CI
  - 每次发布必须有对应 release plan(`08 §6`),不是随意 release note

## 8. 跨仓 PR review SLA 与排队

### 8.1 优先级

| 顺位 | 仓 | 理由 |
|---|---|---|
| 1 | `workitem-docs` | 上游真源,所有下游等它 |
| 2 | `workitem-db` | 下游 BE 等 schema |
| 3 | `workitem-backend` / `workitem-frontend` | 可并行 |

### 8.2 SLA(本项目教学约定)

| PR 类型 | 首次响应 SLA | 完成 review SLA |
|---|---|---|
| `workitem-docs` | 24 小时 | 48 小时 |
| `workitem-db` | 24 小时 | 48 小时 |
| `workitem-backend` / `workitem-frontend` | 48 小时 | 5 工作日 |
| Hotfix(任意仓) | 4 小时 | 24 小时 |

### 8.3 超时机制

- 超首次响应 SLA → 自动 ping review owner
- 超完成 review SLA → upgrade 到 review owner 的 backup
- 任何 PR 待 review > 5 工作日且无任何评论 → 视为流程缺陷,需复盘

## 9. 多 Task 并发部署节奏

### 9.1 默认:都可同时部署

只要满足以下,所有 main 上的 Task 默认**可同时上 PROD**:

- 所有 DDL 都是 additive(对照 `06 §6` 矩阵)
- 所有功能受 feature flag 控制(默认 off)
- 通过 staging 验证

### 9.2 部署节奏可选

| 节奏 | 何时用 |
|---|---|
| 一锅端(所有 main 上 Task 一次部署) | 默认 / staging 全绿时 |
| 灰度(分批开 flag) | 需要观测的功能 |
| 单 Task 独立部署 | 紧急 hotfix(`05 §7`) |

### 9.3 回滚边界

- **代码回滚**:revert PR + 重新部署(快速)
- **DDL 回滚**:**禁止 SQL 层 ROLLBACK**;走 Expand-Contract 反向操作(`06 §4`)
- **feature flag 关闭**:首选回滚手段(秒级,无 DDL 风险)

> ⚠️ Task A 与 Task B 一起部署后,只回滚 A 不回滚 B,**只能通过关闭 A 的 feature flag**实现。
> 若 A 的 DDL 影响了 B 已开启的 flag(罕见),整个部署一起回(关 A flag + 关 B flag)。

## 11. 常见冲突案例与处理

| 案例 | 处理 |
|---|---|
| 两人 V 号同时定到 V5 | 后合入者 rebase 改 V6,见 §4.2 |
| 两人同时新增 BR-09 | 后合入者 rebase 改 BR-10,顺带改自己的 spec / TC / 代码注释 |
| 两人同时改 `02-功能规范.md §4 BR-06`(语义冲突) | review 阻塞,要求两人在 issue 里先达成 spec 一致再各自 rebase |
| Task002 急需 Task001 的新接口,但 Task001 仍 review 中 | Task002 在 FE 用 mock 推进;Task001 合后 Task002 切真实接口并 rebase |
| staging Flyway 因撞号失败 | 立即 revert 失败的 PR,本地 reset 容器,**不允许**手工改 staging 数据库 |
| 一个 Task md 长期 active(spec-version-target 已发布两个版本) | 强制关闭(拆分剩余工作为新 Task)或重新评估 owner |
| 多个 Task 都引入了 feature flag,但都没有清理已全开的 flag | 季度专项 Task 集中清理(每个 flag 删除是独立 Task,见 `06 §5.2`) |

## 13. 维护规则

- 本文规则的任何变更走 MINOR spec tag(除非删规则,那是 MAJOR)
- 编号空间扩展(如新增模块编号 `INFRA-*`)→ §3.1 加行
- CI 强校验项扩展 → §10 加项
- 新的常见冲突案例 → §11 加行(避免冲突再次发生)
- 本文不承载**单 Task 内部决策**(那是 Task md 的职责)
- 本文不承载**单 Task 发布纪律**(那是 06 的职责)
