# Part A — 数据库约束

## A.1 本项目 DB 范围边界

✅ 覆盖：基础表（CREATE TABLE）、索引（CREATE INDEX）、视图（CREATE VIEW）

❌ 不覆盖：存储过程、触发器、函数、事件、分区表

若未来引入，需先在需求分析中移除对应"不做"项，再在本文件增补规则。

## A.2 表命名约束

| 维度 | 规则 | 反例 |
|---|---|---|
| 字符集 | 小写字母 + 下划线 + 数字 | `OrderLine` |
| 单复数 | **单数** | `tasks`、`users_list` |
| 风格 | snake_case | `purchaseOrder` |
| 前缀 | **不加** `t_` / `tb_` / `tbl_` | `t_order` |
| 长度 | ≤ 40 字符 | - |
| 保留字 | 避免 SQL 保留字；`user` 用 `app_user` 替代 | - |

## A.3 列命名约束

### 通用列（强制）

每张业务表**必须**包含：

| 列名 | 类型建议 | 说明 |
|---|---|---|
| `id` | `BIGINT AUTO_INCREMENT PRIMARY KEY` | 主键 |
| `created_at` | `DATETIME NOT NULL` | 创建时间 |
| `updated_at` | `DATETIME NOT NULL` | 最后更新时间 |
| `deleted` | `TINYINT NOT NULL DEFAULT 0` | 逻辑删除，`0`=活跃，`1`=已删除 |

### 业务列规则

| 维度 | 规则 |
|---|---|
| 风格 | snake_case，小写+下划线 |
| 布尔字段 | 以 `is_` 或 `has_` 开头（`deleted` 除外） |
| 时间字段 | 以 `_at` 结尾（时刻）或 `_date` 结尾（仅日期） |
| 外键 | `<ref_table>_id` |
| 枚举字段 | 存 `VARCHAR(16)` 枚举字符串，不存数字 |
| 金额字段 | `DECIMAL(N, M)`，**禁止** `FLOAT` / `DOUBLE` |

类型选择：主键 `BIGINT AUTO_INCREMENT`，短字符串 `VARCHAR(N)`，长文本 `TEXT`，编码 `utf8mb4`，引擎 `InnoDB`。

## A.4 索引命名约束

| 类型 | 前缀 | 格式 | 示例 |
|---|---|---|---|
| 普通索引 | `idx_` | `idx_<table>_<cols>` | `idx_workitem_status_deleted` |
| 唯一索引 | `uk_` | `uk_<table>_<cols>` | `uk_app_user_username` |
| 全文索引 | `ft_` | `ft_<table>_<cols>` | - |

- 多列索引按字段在定义中的顺序连接
- 长度 ≤ 64 字符
- 索引添加必须有理由（伴随慢查询记录或查询模式说明）

## A.5 视图命名约束

- 前缀：`v_`，格式：`v_<table>_<purpose>` 或 `v_<purpose>`
- 仅用于只读场景

## A.6 Flyway 迁移脚本约束

### 归属仓

迁移脚本**唯一归属 DB 仓**（`workitem-db/migration/`），**不放**后端代码仓。

### 文件命名

```
V<N>__task<NNN>_<snake_case_desc>.sql
```

- `V`（大写）+ 版本号 + **双下划线** + `task<NNN>_` + 描述
- 版本号从 1 连续递增，**禁止**跳号或复用
- 描述动词起头：`init_<table>_table`、`add_<col>_to_<table>`

### 已发布脚本不可变性

任何已合入 DB 仓 main 分支的 `V<N>__*.sql` **禁止修改内容**。如需修正，必须新增 `V<N+k>__*.sql`。Flyway 会校验 checksum。

### 脚本内容约束

- 一个脚本只做一件事（建表 / 加列 / 加索引 / 改列）
- 多独立变更**必须**拆成多个脚本
- 脚本头部**必须**加追溯注释

### 本项目不使用的 Flyway 特性

- ❌ Repeatable migrations（`R__*.sql`）
- ❌ Callbacks
- ❌ `baselineOnMigrate=true`

## A.7 DDL 原子性与回滚

- 每份 Flyway 脚本是单向前进的 DDL
- 本项目不提供回滚脚本；生产回滚以"新脚本反做"形式

## A.8 追溯注释

所有 Flyway 脚本、视图等**必须**加双标签注释：

```sql
-- [SDD-TASK: Task<NNN>]
-- [SDD-SPEC: <spec>.md §<section>] <简述>
```

必须加注释的位置：每份 `V<N>__*.sql` 文件顶部、每个 CREATE VIEW / CREATE INDEX 上方、Expand-Contract 成对脚本相互标注。

## A.9 逻辑删除实现路径

通过 **MyBatis Plus 全局逻辑删除**机制：

- Entity 的 `deleted` 字段标 `@TableLogic`
- 全局配置：`logic-delete-field=deleted`，`logic-delete-value=1`，`logic-not-delete-value=0`
- 自定义 XML SQL 必须手工加 `AND deleted=0`

**硬性禁止**：
- ❌ 应用层重复写 `deleted=0`（对 `BaseMapper` 方法）
- ❌ 关闭 MP 逻辑删除以"优化 SQL"

## A.10 DB 禁用清单

- ❌ `SELECT *`
- ❌ 硬编码 schema 前缀
- ❌ 生产数据 `INSERT INTO` 放 Flyway 脚本
- ❌ 裸字符串拼接 SQL
- ❌ 存储过程 / 触发器 / 函数

# Part B — 前端约束

## B.1 目录结构

```
src/
├── main.ts                 # 入口（初始化+挂载）
├── App.vue                 # 根组件（<router-view /> + 全局容器）
├── router/index.ts         # 路由定义，所有 view 懒加载
├── api/<entity>.ts         # 按实体分文件，汇聚所有 API 函数
├── views/<Entity><Action>View.vue  # 页面级组件
├── components/<Common>.vue        # 可复用组件
├── types/<entity>.ts       # 与后端 DTO 对齐的 TS 类型
├── utils/
│   ├── http.ts             # HTTP 实例 + 统一拦截器
│   └── <util>.ts           # 其他纯函数工具
└── assets/                 # 静态资源
```

**禁止**：views/ 放工具函数、components/ 放路由页面、api/ 里写 UI 代码、组件里裸 `axios`/`fetch`。

## B.2 命名约束

| 对象 | 规则 | 示例 |
|---|---|---|
| 视图组件 | PascalCase + 功能后缀 + `View.vue` | `TaskListView.vue`、`TaskEditView.vue` |
| 通用组件 | PascalCase，不带 `View` | `ConfirmDialog.vue` |
| 路由 path | 小写 kebab-case | `/tasks`、`/tasks/new`、`/tasks/:id/edit` |
| API 函数 | `<verb><Entity>` camelCase | `createTask`、`listTaskPage`、`softDeleteTask` |
| TS 类型 | PascalCase | `TaskVO`、`TaskCreateReq`、`PageVO<T>`、`Result<T>` |
| 枚举值 | 字符串 literal 联合类型（不用 `enum`） | `type Status = 'TODO' \| 'DOING' \| 'DONE'` |

## B.3 组件职责边界

- **views/**：UI 渲染、表单 state、用户事件、跳转；不关心 HTTP 细节
- **components/**：只接受 props + emit events；不直接调 `api/`，不依赖 `router`
- **api/**：纯函数，返回 `Promise<T>`（T 是业务数据，Result 壳由拦截器脱掉）；不展示 UI
- **utils/**：纯函数，无副作用（除 `http.ts` 拦截器）

## B.4 表单校验

1. 前端校验必须与后端 spec 一致（长度、必填、枚举）。发现偏差回去改 spec
2. 校验失败不发送请求
3. 校验失败高亮对应字段 + 显示错误文案
4. 校验规则抽取为独立 `validators.ts`，不散落在模板里
5. 后端约束之外可做 UX 软约束，但必须标明为软约束

## B.5 HTTP 统一错误处理

所有 HTTP 交互**必须**经过 `utils/http.ts` 的 axios 实例：

- **响应拦截器分支**（按优先级）：
  1. 网络错误 / HTTP 5xx → toast "网络异常" + reject
  2. HTTP 4xx → toast(res.message) + reject
  3. HTTP 200 + `code === 0` → resolve(res.data)
  4. HTTP 200 + `code !== 0` → toast(res.message) + reject

- 组件层 `.catch()` 只处理跳转或 state 回滚，**不写** `alert` / `console.error`

## B.6 路由与样式约束

- 所有路由集中在 `router/index.ts`，view 组件必须懒加载
- 路由 `name` 与 view 文件名主干一致
- 优先 `<style scoped>`，全局样式仅限 CSS reset / 字体 / 变量
- 不引入 UI 组件库、CSS 预处理器
- 颜色/间距写成 CSS 变量（`:root { --color-primary: ...; }`）

## B.7 与后端 DTO 对齐

- `types/<entity>.ts` 必须一一对应后端 VO / Req / PageVO / Result
- 字段名、可空性、枚举值**必须**完全一致
- 不对外暴露的字段（如 `deleted`）**禁止**出现在前端类型中

## B.8 前端禁用清单

- ❌ `any` 类型（除临时过渡并标注 TODO）
- ❌ `console.log` 留存提交代码
- ❌ 组件里裸 `fetch` / `axios`
- ❌ 内联事件处理器中写业务逻辑
- ❌ 组件外全局可变状态
- ❌ 跨路由 URL 硬编码

# Part C — 国际化约束

## C.1 支持语言

- 默认语言：`en-US`（作为 fallback 与 i18n key 源语言）
- Fallback 链：请求语言 → 简化语言 → `en-US`
- 教学场景不强制多语，仅规定扩展规则

## C.2 文案 key 命名

格式：`<domain>.<feature>.<element>.<variant?>`

- 全小写 + 点分隔
- 示例：`task.list.title`、`common.button.save`、`task.form.error.title_required`

**禁止**：key 含实际文案（中文 key）、key 中拼接动态数据（用 `{param}` 占位符）

## C.3 资源文件组织

```
src/i18n/
├── en-US.json    # 默认/源语言（source of truth）
├── zh-CN.json
└── index.ts      # 加载器
```

`en-US.json` 是 source of truth；其他语言 key 集合必须是其子集或等集。

## C.4 日期时间与时区

| 层 | 表示 |
|---|---|
| 存储层（DB） | **始终** UTC |
| API 传输（JSON） | ISO-8601 UTC（带 `Z`） |
| 展示层（前端） | 按用户本地时区 + locale 格式化 |
| 日志 | ISO-8601 UTC |

**硬性禁止**：后端返回"本地时间字符串"（无时区）、前端把展示格式送回后端、DB 列用 session time_zone 默认（必须显式 `+00:00`）。

## C.5 业务消息 i18n

后端响应体必须提供 `message_key` + `message_params`：

```json
{
  "code": 1002,
  "message_key": "error.task.not_found",
  "message_params": { "id": 42 },
  "message": "Task not found: id=42"
}
```

- 前端用 `message_key` 查本地化资源；找不到 fallback 到 `message`
- 日志始终用英文 key

## C.6 国际化禁用清单

- ❌ 非英文作为 key
- ❌ 界面文案硬编码在组件里
- ❌ 后端硬编码中文返回
- ❌ 日志非 ASCII 业务文案
- ❌ DB 存本地时间字符串

# Part D — 依赖版本与升级纪律

## D.1 依赖分级

| 级别 | 范围 | 升级策略 |
|---|---|---|
| L1 环境级 | JDK / Node.js / MySQL / Docker base image | 走 MAJOR spec tag + 完整回归 |
| L2 核心框架 | Spring Boot / Vue / Vite / MyBatis Plus / Flyway | 走独立 Task + 回归测试 |
| L3 业务依赖 | axios、vue-router、SpringDoc、Jackson 等 | 跟随 L2 或按需 |
| L4 开发工具 | Maven / npm / TypeScript / 测试框架 | 内部操作，不影响契约 |
| L5 传递依赖 | 由直接依赖引入 | 不主动锁 |

## D.2 版本锁定策略

- L1/L2/L3：**锁精确版本**（`pom.xml` / `package.json` 用定值，**不用** `^` / `~`）
- L4：锁 minor，允许 patch 自动更新
- ❌ `RELEASE` / `LATEST`（Maven 已废弃）
- ❌ 提交代码不提交锁文件更新

## D.3 锁文件纪律

| 仓 | 锁文件 | 入 git |
|---|---|---|
| workitem-backend | `pom.xml` | ✅ |
| workitem-frontend | `package.json` + `package-lock.json` | ✅ |

锁文件变更**不得**与业务改动同 PR 混合（小规模例外）。

## D.4 Major 升级流程

```
RFC（spec 仓 issue）→ Review ≥ 1 周 → 独立 Task → 升级指南 → 正常晋级 → 稳定 2 周关闭
```

必须全部自动化测试通过 + 评估下游兼容性 + 准备回滚预案。

## D.5 安全漏洞响应（CVE）

| 严重性 | 响应 SLA |
|---|---|
| Critical（CVSS ≥ 9.0） | 24 小时内 |
| High（7.0~8.9） | 3 工作日内 |
| Medium（4.0~6.9） | 7 工作日内 |
| Low（< 4.0） | 季度巡检 |

## D.6 新增依赖审核

必须评估：必要性、许可证兼容、维护活跃度、安全记录、传递依赖膨胀、bundle size（前端）。

## D.7 依赖禁用清单

- ❌ 非精确版本（`^` / `~` / `LATEST` / `RELEASE`）
- ❌ 锁文件未入 git
- ❌ 绕过锁文件手工改版本
- ❌ CVE 超 SLA 未响应
- ❌ 环境级依赖各开发者版本不一致

# Part E — 异步入口约束

## E.1 异步入口分类

| 类型 | 触发方式 | 本项目是否默认启用 |
|---|---|---|
| **Scheduled Job** | cron 表达式 / 固定周期 | ✅ Spring `@Scheduled` |
| **Event Listener** | 应用内事件 | ✅ |
| **MQ Consumer** | 消息队列 | ❌ 本项目不引入 |
| **Batch Job** | 大批量数据处理 | ❌ 按需走 |

## E.2 命名规范

- **Scheduled Job**：类名 `<Verb><Subject>Job`（PascalCase），如 `CleanupExpiredTaskJob`
- cron 元数据必标：`name`（kebab-case）、`workitem_id`、`cron`、`timezone`（默认 UTC）、`timeout`、`enabled`
- **Event Listener**：类名 `<Event>Listener`
- **Batch Job**：步骤 `<phase>-<subject>`

## E.3 幂等性（硬性）

**所有**异步入口**必须**幂等。多次执行结果一致。**禁止**依赖"事件只送一次"的假设。

## E.4 失败处理与重试

| 参数 | 默认 |
|---|---|
| 最大重试次数 | 3 |
| 退避策略 | 指数退避（1s / 5s / 30s） |
| 死信 | 超重试入 DLQ / 人工介入 |

每次失败日志必记：入口名、输入（幂等 key，PII 脱敏）、错误类型/stack、重试次数。

## E.5 分布式追踪

- Scheduled Job / Event / 消息消费**必须**生成新 `trace_id`
- HTTP 手动触发 job 时继承请求的 `trace_id`

## E.6 与 Feature Flag 协作

| 场景 | 规则 |
|---|---|
| 新 job 首次引入 | 必须带 flag（`feat_task<NNN>_<短语>`），默认 OFF |
| flag OFF | **不触发**；入口判断 flag 后直接 return |
| flag 中途关闭 | 允许当前执行跑完；下次触发判断 |

**硬性**：flag 判断放最外层入口，不在业务逻辑之后。

## E.7 可观测性

每个异步入口必须暴露：`job_execution_total`（counter）、`job_duration_seconds`（histogram）、`job_last_success_timestamp`（gauge）。

每次执行至少 2 条结构化日志（开始 INFO + 结束 INFO/ERROR + 耗时）。

执行 > 5 分钟须支持 checkpoint + 心跳日志。

## E.8 异步禁用清单

- ❌ 非幂等异步入口上线 PROD
- ❌ Scheduled Job 跨实例同时执行（须有分布式锁）
- ❌ 无 flag 的新 job 直接上 PROD
- ❌ 无超时的长任务
- ❌ 失败静默（无日志 / 无告警）
- ❌ `@Scheduled(fixedRate=...)` 高频触发（每秒多次）
