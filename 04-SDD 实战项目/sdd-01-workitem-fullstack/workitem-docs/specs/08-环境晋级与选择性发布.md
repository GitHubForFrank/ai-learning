# 08 - 环境晋级与选择性发布

> **文档定位**:回答「同一份代码 + 同一份 spec,如何让 SIT 跑 10 个功能、UAT 只对外 5 个、PROD 只放 3 个,而不破坏 SDD 真源唯一性」
> **SDD 阶段**:附加阶段 — Environment Promotion & Selective Release
> **上游依赖**:`06-任务与发布管理.md`(Task 编号、additive-only、feature flag 命名)、`07-多任务并行开发.md`(并行纪律、Task md)
> **下游依赖**:本文规则被 4 仓 README、`../design/03-技术方案.md` 配置项、`specs/*` 引用
>
> **与 05 / 06 / 07 的分工**:
> - `05`:**一个 Epic 跨仓 PR**怎么编排
> - `06`:**一个 Task** 怎么编号与分期上线
> - `07`:**N 个 Task 并行**怎么不打架
> - `08`(本文):**N 个 Task × N 个环境**,每个环境只暴露其中一部分时,怎么不破坏 SDD

---

## 1. 适用场景

本文规则在以下情境强制生效:

- 项目存在 ≥ 2 个晋级环境(典型:dev → SIT → UAT → PROD)
- 任一环境开放的功能集是上游环境的**真子集**(如 PROD = UAT 的 3/5)
- 任一晋级动作改变 feature flag / 配置 / 部署版本

只要触发其中之一,本文规则即生效;每次晋级**必须**留下可审计的 release plan。

---

## 2. 环境分层

### 2.1 标准阶梯

| 层 | 简称 | 职责 | 数据 | flag 默认 |
|---|---|---|---|---|
| 开发 | dev | 个人开发与本地联调 | 各自容器,可重置 | 全开(本地实验) |
| 系统集成测试 | **SIT** | 跨仓集成测试、QA 全量验证 | 共享,持续刷新 | **当前 main 全部 Task 的 flag 全开** |
| 用户验收测试 | **UAT** | 业务方 / 用户用真实数据验收 | 类生产数据,受控 | **本轮晋级被批准的 Task flag 开** |
| 准生产(可选) | **PROD-mirror** | 灰度前最后一道闸门,验证 PROD 的 flag 配置和数据 | PROD 数据子集 / 脱敏 | **与 PROD 计划一致** |
| 生产 | **PROD** | 终态 | 真实业务数据 | **本次发布批准的最小子集 flag 开** |

> ⚠️ `07 §7` 用过 "staging" 一词,在本项目语义里**等价于 SIT**(`07` 文档保留通用表述,本文落地为 SIT)。

### 2.2 关键不变量

- 所有环境**跑同一个 build artifact**(同一 git SHA / 同一镜像)
- 所有环境**跑同一组 Flyway 脚本**(因 DDL additive,无副作用)
- 所有环境的 spec 真源**指向同一个 `workitem-docs` tag**
- **唯一**变化的是:**环境配置**(以 feature flag 为主,详见 §4)

任何破坏以上不变量的做法 → 本项目禁用,见 §10。

---

## 3. 五条核心原则

| 原则 | 含义 | 违反后果 |
|---|---|---|
| **3.1 同 build 跨环境** | SIT/UAT/PROD 跑同一个镜像,**不允许**为某环境单独打包 | 失去 build 不可变性,事故定位 / 回滚链路断裂 |
| **3.2 DDL 全环境执行** | Flyway 在所有环境都跑全部脚本(因 additive,功能可见性 ≠ schema 存在性) | 各环境 schema 漂移 / Flyway checksum 失配 |
| **3.3 功能可见性 = flag 配置** | 同一 Task 的功能在不同环境的"开关"由各自配置决定,**与代码无关** | 代码分支化 / 环境分支化 / spec 分裂 |
| **3.4 每次晋级是显式决策** | SIT→UAT、UAT→PROD 都需 release plan 文档,记录批准人、Task 列表、flag 变更 | 晋级无审计 / "为什么 PROD 没上 Task007" 没人答得上 |
| **3.5 回滚靠关 flag,不回滚 DDL/build** | 出问题的功能 → 关该 Task 的 flag(秒级);**禁止**降级 build / 回滚已合入的 DDL | DDL 回滚是雪崩起点 |

## 4. Feature flag 配置外部化

### 4.1 配置存储选型

| 选项 | 机制 | 评价 |
|---|---|---|
| **(a) 环境变量驱动** | `application.properties` 内 flag 默认全 OFF;启动时通过环境变量(或 configmap 挂载文件)按环境覆盖 | ⭐⭐⭐ **本项目教学落地选这个** |
| (b) 独立 Flag 服务(Unleash / LaunchDarkly) | 集中管理 + 灰度 / AB 试验能力 | 生产级首选,本项目仅在 spec 中规划提及,不实施 |
| (c) DB 表 + 热加载 | flag 值存数据库,启动期 + 周期拉取 | 启动期循环依赖、配置审计困难,**本项目禁用** |

> 选 (a) 的关键约束:
> - `application.properties` 里所有 flag 默认值**必须**是 `false`(安全默认)
> - 各环境的实际 flag 值由部署侧的环境变量 / configmap 设置,**不进 workitem-backend 仓**
> - **flag 名字与默认值在 workitem-backend 仓**(因为代码要读),**flag 实际开关在配置侧**

### 4.2 不止 feature flag —— 所有"环境差异"都要外部化

下列配置**全部**走 §4.1 的外部化路径,不允许 hardcode 进代码:

| 配置项 | 为什么必须外部化 |
|---|---|
| Feature flag 当前值 | 本文核心 |
| 数据库连接 URL / 凭据 | 各环境数据库不同 |
| CORS 允许来源 | dev:`localhost:5173`;PROD:正式域名 |
| 前端 API base URL | 各环境后端域名不同 |
| 日志级别 | dev 详细 / PROD 收敛 |
| 第三方服务 endpoint(若有) | 环境隔离 |

### 4.3 配置变更纪律

- 任何环境配置变更**必须**走 PR(配置仓 / configmap repo / Flag 服务的变更也要审计)
- PR 描述格式参考 release plan 模板(§6)
- 紧急变更:走 hotfix 流程(`05 §7`),事后 72h 内补 release plan

## 5. Feature Flag Matrix 文档

### 5.1 位置与定位

```
workitem-docs/
└── 03-技术方案.md §12           # Feature flag 矩阵,描述所有 flag 在所有环境的"当前态"
```

- 这是 **当前快照**(snapshot of now),不是历史
- 历史变更轨迹由 `releases/` 下的 release plan 承载(§6)
- 所有环境的运维 / 部署都以本文件为**目标态**

### 5.2 模板必填段(占位符)

```markdown
# Feature Flags — 当前态

> 本表是各环境的 flag 目标态,部署侧据此配置实际环境。
> 任何变更必须配套 `releases/<env>-<date>-<seq>.md` release plan。

## 1. 当前 spec 真源版本

- workitem-docs tag: vX.Y.Z
- 最近更新日期: YYYY-MM-DD

## 2. flag 矩阵

| Flag 名 | Task | 引入版本 | dev | SIT | UAT | PROD-mirror | PROD | 备注 |
|---|---|---|---|---|---|---|---|---|
| feat_task001_<短语> | Task001 | v1.0.0 | ON | ON | ON | OFF | OFF | UAT 验证中 |
| feat_task003_<短语> | Task003 | v1.1.0 | ON | ON | OFF | OFF | OFF | 仅 SIT 测试中 |
| ... | | | | | | | | |

## 3. 待移除 flag(已全开稳定 ≥ 2 个 spec 版本)

| Flag 名 | 全开起始版本 | 计划移除 Task | 计划移除时间 |
|---|---|---|---|
| ... | | | |
```

### 5.3 维护规则

- 每次新增 / 移除 flag → 本文件 PR + 一个对应 Task md(`07 §6`)
- 每次环境晋级改动 flag → 本文件 PR 同步当前态 + 提交对应 release plan(§6)
- 任何 PROD 列变化 → 同行的 PROD-mirror 列必须**先**变化并验证通过
- 列顺序固定为 `dev → SIT → UAT → PROD-mirror → PROD`,**禁止**调换(便于跨 PR diff 对比)

## 6. Release Plan 文档

### 6.1 位置与命名

```
workitem-docs/
└── releases/
    ├── README.md                          # 目录说明
    ├── sit-2026-04-25-01.md               # SIT 晋级记录
    ├── uat-2026-04-30-01.md               # UAT 晋级记录
    ├── prod-2026-05-10-01.md              # PROD 晋级记录
    └── ...
```

- 命名:`<env>-<YYYY-MM-DD>-<seq>.md`
  - `<env>`:`sit` / `uat` / `prod-mirror` / `prod`(全小写)
  - `<YYYY-MM-DD>`:计划晋级日期(不是创建日期)
  - `<seq>`:同日同环境的序号,从 `01` 起,两位数
- 一次晋级 = 一份 release plan

### 6.2 frontmatter 字段(必填)

```yaml
---
env: prod                                 # sit | uat | prod-mirror | prod
date: 2026-05-10                          # 计划晋级日期
seq: 01                                   # 当日序号
status: planned                           # planned | approved | executed | rolled-back
spec-tag: v1.5.0                          # 本次晋级使用的 spec 版本
build-sha: <git-sha-or-image-tag>         # 同一 build,跨环境流转
epic: EPIC-<NN>-<短名>                     # 可选:若本次晋级服务于一个或多个 Epic(见 05 §4);多个用列表
prepared-by: <name>
approved-by: <name>                       # planned 时为空,approved 后填
executed-at: null                         # 执行时填 ISO8601 时间戳
---
```

### 6.3 正文必填段

```markdown
## 1. 本次晋级目标
<1~3 行业务说明:为什么现在做、覆盖哪些业务点>

## 2. 涉及 Task
- Task<NNN> — 链 [tasks/Task<NNN>-*.md](../tasks/Task<NNN>-*.md)
- Task<NNN> — ...

## 3. Flag 变更(diff)
| Flag | 上一态(本环境) | 本次变更后 | 关联 Task |
|---|---|---|---|
| feat_task001_<短语> | OFF | ON | Task001 |
| ...

## 4. 上游环境验证状态
- (PROD 晋级时)PROD-mirror 验证通过日期 / QA 签字
- (UAT 晋级时)SIT 通过的关键测试用例编号
- 关键测试用例编号必须包括 BR-* 三维矩阵覆盖(参见 04-验收标准.md §2)

## 5. DDL / 数据影响
- Flyway 脚本范围: V<a> ~ V<b>
- 是否涉及数据回填 / 后台任务 / 索引重建
- 回填 / 重建预计耗时 / 影响窗口

## 6. 回滚预案
- 优先方案:关哪些 flag(秒级)
- 兜底方案:若 flag 回滚不足,是否需要 hotfix Task(预先评估)
- 不允许:DDL 回滚 / build 回滚

## 7. 风险与已知未决
<列出已知风险、未解疑问、已接受的妥协>

## 8. 执行后回填(executed 状态时填)
- 实际执行时间
- 实际生效的 flag 配置(应与 §3 一致;不一致是事故)
- Lessons
```

### 6.4 release plan 与 flag matrix 的关系

| 维度 | flag matrix(`../design/03-技术方案.md §12`) | release plan(`releases/*.md`) |
|---|---|---|
| 时态 | **当前态** | **历次变更** |
| 数量 | 单文件 | 每次晋级一份 |
| 变更 | 每次晋级**同步**更新 | 每次晋级**新增**一份(已有不改) |
| 审计粒度 | 谁批了什么状态(coarse) | 每次晋级的细节(fine) |

**强制对齐**:任何 release plan 的 §3 flag 变更行,合入后**必须**同步反映到 `../design/03-技术方案.md §12` 矩阵的对应单元格。CI 可校验该一致性(详见 §10)。

## 7. 晋级流程

### 7.1 标准晋级路径

```
spec 合入 main + 打 tag
       ↓
build 出镜像(任一环境用同一个)
       ↓
[A] 部署 SIT → flag 默认全开(SIT 是 main 的全量映射)
       ↓
QA 验证 / 业务方初验
       ↓
[B] 决定哪些 Task 进 UAT
       ↓
开 UAT release plan PR(planned) → review → approved
       ↓
ops 按 plan 调整 UAT flag 配置 + 部署同一 build
       ↓
[C] UAT 业务方验收(可能反复)
       ↓
[D] 决定哪些 Task 进 PROD-mirror(可选)→ 验证
       ↓
[E] 决定哪些 Task 进 PROD
       ↓
开 PROD release plan PR(planned) → review → approved
       ↓
ops 在 PROD 调整 flag + 部署同一 build → 监控 → 标 executed
```

### 7.2 晋级前置硬性条件(所有 gate 共用)

晋级到任一上游环境前,**必须**先通过以下回归门槛,任一未满足即**阻塞晋级**:

| # | 条件 | 依据 |
|---|---|---|
| 1 | 目标 release plan 涉及的 Task 在当前环境(上游)已全部 `closed` 或明确 in-flight 但有 release plan 批准 | `07 §6.5` + `08 §6.2` |
| 2 | 全量 **TC-R-API-\*** 在上游环境绿(Testcontainers MySQL 跑) | `regression-conventions §7.2` |
| 3 | 全量 **TC-R-DB-\*** 在上游环境绿(含 BR-07 的 `deleted=1` 物理断言) | 同上 |
| 4 | 若本次晋级涉及 DDL,对应 **TC-R-MIG-\*** 在上游环境绿 | 同上 |
| 5 | 全量 **TC-R-E2E-\*** 在上游环境绿(含 `sdd-db-probe` MCP 后台断言) | 同上 |
| 6 | `sdd-db-probe` 审计日志无异常告警(最近 24h)| `regression-conventions §4.4` |

> ⚠️ 回归门槛**不可由晋级决策者 override**;若回归失败但需紧急上线,走 `05 §7` hotfix 流程(独立 Task + 独立审计)。

### 7.3 各 gate 的审批权

| Gate | 决策角色 | 决策依据 |
|---|---|---|
| SIT 晋级 | 自动(任何 main 合入触发) | CI 全绿 |
| SIT → UAT | 产品 / QA Lead | SIT 验证通过 + UAT 业务诉求 |
| UAT → PROD-mirror | 产品 / 技术 Lead | UAT 验收记录 |
| PROD-mirror → PROD | 技术 Lead / Ops Lead | mirror 监控指标合格 |

> 教学项目可简化:owner 既是 Lead 也是 reviewer,但**审批动作必须留痕**(release plan PR review 通过即留痕)。

### 7.4 反向流程:从 PROD 撤功能

PROD 已开启的功能临时撤掉(如发现问题但还未达 hotfix 程度):

1. 开 `prod-<date>-<seq>.md` release plan,标 `status: planned` + 注明 "rollback by flag"
2. plan §3 列出 ON → OFF 的变更
3. 加急 review + approved
4. ops 关 PROD flag(秒级) + 同步更新 `../design/03-技术方案.md §12`
5. plan 标 `status: rolled-back`,补充 `Lessons`
6. **不**回滚 build / DDL

## 8. 跨环境冲突案例与处理

| 案例 | 处理 |
|---|---|
| PROD 准备开 Task005,但 Task005 依赖 Task002 schema,Task002 在 PROD 一直关 | ✅ 无问题:Task002 schema 在 PROD 的 DB 已 additive 上线,即使 flag 关也不影响 Task005 用 |
| Task007 仅引入 SIT,但 SIT 的 Flyway 自动跑包含 Task007 的 V<N> | ✅ 无问题:DB schema 在 SIT 已存在,flag 默认全开生效 |
| 想把 Task009 的 DDL 留到 PROD 才执行 | ❌ **禁用**:违反 §3.2;若需延后,改为 Expand 版 DDL 先上 + Contract 版 DDL 后上(`06 §4`) |
| UAT 出现 PROD 没有的"多余表" | ✅ 这是正常的:UAT 已用 V<N> 但 PROD 部署在更早 build,等 PROD 跟进 build,DDL 自然到位 |
| 需要在 PROD 做"灰度":同一 Task 只对 1% 用户开 | 本项目教学场景**不演示**;生产场景需选 §4.1 的 (b) Flag 服务,在本文档之外补 spec |
| PROD 紧急关 Task005 后,Task005 历史数据残留表里 | 不立即处理,作为脏数据 / 历史数据保留;后续 Task 走 Contract 流程清理(`06 §4`) |
| 同一日多次晋级 PROD | 用 seq 区分(`prod-2026-05-10-01.md`、`prod-2026-05-10-02.md`),互不覆盖 |

## 9. Hotfix 与部分上线的交互

### 9.1 PROD 已开 Task A,A 出 bug,需 hotfix

1. **首选**:关 A 的 flag(`prod-<date>-<seq>.md` rollback plan)+ 同步 `../design/03-技术方案.md §12`
2. 若 flag 关闭后业务可接受:**完成**,后续按 §9.3 处理
3. 若必须修代码:走 hotfix 流程(`05 §7`),开新 Task<NNN+1> 走标准 SIT→UAT→PROD,但 SLA 缩短

### 9.2 PROD 已开 A,要求紧急再开 Task C(原计划下次发布)

- 不允许"硬塞":必须开新 release plan PR
- 但 SLA 可缩短(参考 hotfix)
- C 必须已通过 SIT + UAT(若 UAT 未过,**不允许**进 PROD)

### 9.3 hotfix 完成后的清理

- hotfix Task 的 md 文件标 `closed`
- `../design/03-技术方案.md §12` 同步当前 PROD 实际 flag 态
- 该问题 Task A 的 md 在 Closure 段补 Lessons

## 10. 与"SDD 不破坏"的关键校验(CI 与人工)

### 10.1 CI 自动校验项

- [ ] 所有 release plan 的 frontmatter 格式正确
- [ ] release plan 的 §3 flag 变更与 `../design/03-技术方案.md §12` 对应环境列**最终一致**(应用 plan 后)
- [ ] release plan 引用的 Task md 在 `tasks/` 中存在
- [ ] release plan 的 `spec-tag` 在 spec 仓 tag 列表中存在
- [ ] release plan 的 `build-sha` 在下游代码仓 main 历史中存在(需跨仓查询能力)
- [ ] PROD 列在 `../design/03-技术方案.md §12` 的任何变化必须有 `prod-*.md` release plan 关联
- [ ] release plan `status` 字段的变更符合状态机:
  - 允许:`planned → approved`、`approved → executed`、`planned → rolled-back`、`executed → rolled-back`
  - **禁止**:任何回退(`executed → planned`、`approved → planned`、`rolled-back → *`)
  - **禁止**:跳级(`planned → executed` 绕过 approved)
- [ ] release plan 若 `epic` 字段有值,引用的 `EPIC-<NN>-<短名>` 在 `05` 或 issue tracker 中存在

### 10.2 人工审计项(每月一次)

- [ ] 没有"幽灵 flag"(代码引用了但 matrix 没有,或反之)
- [ ] 没有 active > 90 天且未变化的 flag(僵尸 flag,应安排移除)
- [ ] 各环境实际 flag 配置 vs `../design/03-技术方案.md §12` 对账
- [ ] 各环境实际 build SHA vs 最近 release plan 对账

## 11. 禁用清单(本项目硬性禁止)

- ❌ **环境特定的代码分支**(`if (env == 'PROD')`)
- ❌ **环境特定的 Flyway 目录**(`migration/prod/`)
- ❌ **跳跃式 build**(SIT 跑 build A,UAT 跑 build B,但 A != B)
- ❌ **绕过 release plan 直接改 PROD 配置**(运维拍脑袋开 flag)
- ❌ **回滚 DDL** / **回滚 build** 作为常规手段(只在事故级且无 flag 兜底时考虑)
- ❌ **PROD 未经 PROD-mirror 验证就开新 flag**(PROD-mirror 可选则 SIT/UAT 必须充分)
- ❌ **release plan 事后补**(plan 必须先 approved 才执行)
