## 1. 项目背景

### 1.1 背景

本项目是一个**独立的** SDD 方法论多仓参考实现。核心目标**不是业务扩展**,而是回答真实工程中的问题:

> 当一个功能同时涉及**前端页面、后端接口、数据库表**,并且这三者各自独立成 git 仓库时,
> SDD 的"规范 → 代码 → 验收 → 环境晋级"链条怎么串?

### 1.2 业务载体选择

选 **Task(任务管理)** 作为业务域的理由:

- 业务模型单一(单表 CRUD),避免业务复杂度盖过 SDD 流程本身
- 涵盖 REST API 最常见的 5 种操作(增 / 删 / 改 / 单查 / 列表查)
- 可扩展空间大(后续可加标签 / 评论 / 用户等),便于二次演练

### 1.3 本项目的方法论贡献

| 贡献 | 体现在 spec 的哪里 | 解决什么问题 |
|---|---|---|
| 多仓架构 | `../design/03-技术方案.md §2` + 4 仓 README | 真实工程中 spec / BE / FE / DB 各独立仓时的协作 |
| 约束 spec 分层 | `specs/` 9 份独立约束 | 跨功能的通用规则从功能 spec 中剥离、可复用 |

| 多任务并行 | `07-多任务并行开发.md` | N 个 Task 同时推进时的编号空间、冲突、轨迹 md |
| 环境晋级 | `08-环境晋级与选择性发布.md` | SIT / UAT / PROD 不同子集的同 build 晋级 |

## 3. 用户故事(User Stories)

> 格式:As a **{角色}**, I want **{能力}**, So that **{价值}**.
> **重要**:USF-* 为**前端可见行为**,USA-* 为**纯 API 级行为**(无前端入口),两类明确分开,以便 §8 模块拆分时区分"要不要做页面"。

| ID | 用户故事 | 前端入口 | 优先级 |
|---|---|---|---|
| USF-01 | As a **用户**, I want **在"新建任务"页填写标题/描述/优先级/截止日期并保存**, So that **我可以把待办事项记下来** | ✅ 新建页 | P0 |
| USF-02 | As a **用户**, I want **在"任务列表"页按状态筛选、分页浏览任务**, So that **在任务多时也能快速定位** | ✅ 列表页 | P0 |
| USF-03 | As a **用户**, I want **在列表项上点击进入"任务详情"页查看完整信息**, So that **我能看到创建/更新时间等完整信息** | ✅ 详情页 | P0 |
| USF-04 | As a **用户**, I want **在"编辑任务"页修改标题/描述/状态/优先级/截止日期**, So that **任务信息能随进度更新** | ✅ 编辑页 | P0 |
| USF-05 | As a **用户**, I want **在列表或详情页一键删除任务(软删除)**, So that **列表保持整洁但数据可追溯** | ✅ 列表/详情页按钮 | P0 |
| USF-06 | As a **用户**, I want **操作失败时看到明确的错误提示(而不是空白页)**, So that **我知道发生了什么** | ✅ 全局 toast | P0 |
| USF-07 | As a **用户**, I want **登录后才能访问任务管理功能**, So that **不同用户的任务隔离 + 访问受控** | ✅ 登录页 + 受保护跳转 | P0 |
| USA-01 | As a **API 调用方**, I want **所有接口返回统一的 Result 结构**, So that **前端或第三方调用方能统一处理** | ❌ 仅后端 | P0 |
| USA-02 | As a **运维 / 安全审计**, I want **每次登录(成功 / 失败 / 锁定)都被记录(账号 / IP / 时间 / traceparent)**, So that **可追溯** | ❌ 仅后端 + 日志 | P0 |

## 4. 功能清单(P0 范围)

> 本项目覆盖 5 个核心功能(F-01 ~ F-05)+ 1 个跨模块功能(F-06),功能编号与业务规则编号在后续 spec 中保持稳定。

| 功能 ID | 功能名称 | 对应 User Story | 备注 |
|---|---|---|---|
| F-01 | 创建任务 | USF-01 | 前端新建页 + 后端 POST 接口 |
| F-02 | 分页 + 按状态筛选查询 | USF-02 | 前端列表页 + 后端 GET 分页接口 |
| F-03 | 按 ID 查询详情 | USF-03 | 前端详情页 + 后端 GET 详情接口 |
| F-04 | 更新任务 | USF-04 | 前端编辑页 + 后端 PUT 接口 |
| F-05 | 逻辑删除任务 | USF-05 | 前端删除按钮 + 后端 DELETE 接口(MP 逻辑删除) |
| F-06 | 统一错误提示 | USF-06 | 前端全局拦截 + toast |
| F-07 | 登录与登录态保护 | USF-07, USA-02 | 登录页 + JWT + 失败 3 次锁定 + 登录日志 + traceparent 透传 |

> 详细接口契约见 `02-功能规范.md §3`;详细模块拆分见 `02-功能规范.md §8`。

## 5. 非功能需求(NFR)

| 类别 | 要求 | 验收方式 |
|---|---|---|
| **性能** | 单机开发环境下,单接口 P95 < 150ms(覆盖 MySQL 往返开销) | 手工 curl 计时 / E2E 耗时记录 |
| **可用性** | 开发者本机 MySQL 就绪后,`mvn spring-boot:run` + `npm run dev` 两条命令即可跑通 | P2/P3 阶段在 README 补 verify 命令 |
| **安全** | 后端参数 Bean Validation;前端表单同步校验(与 spec §3 保持一致) | TC-FE-* / TC-IT-* 覆盖 |
| **可观测性** | 后端统一日志格式;前端控制台不打 warning/error | 代码审查 + 浏览器 devtools |
| **可维护性** | 前端/后端/DB 三处关键位置均有 `[SDD-SPEC: xx §y]` 追溯注释 | 代码审查 |
| **文档** | SpringDoc 生成 OpenAPI;前端 README 列出所有页面路径 | 手工访问 |
| **数据库可演进性** | 所有 DDL 通过 Flyway 迁移脚本,发布后脚本不可变(详见 `specs/91-编码规范.md §A.6`) | Flyway 校验 checksum |
| **可复现性** | MySQL 版本统一为 8.0,Flyway 迁移脚本版本化(开发者自备本机 MySQL) | spec 中 §1.3 明确版本要求 + Flyway checksum 校验 |

## 6. 范围边界(明确"不做什么")

> SDD 核心实践:**范围外**比**范围内**更重要。

**业务层面不做**:

- ✅ **最小登录(由 Task001 引入,见 §3 USF-07 / §4 F-07)**:账号密码 + JWT + 失败 3 次锁定 15 分钟 + 登录日志
- ❌ 任务归属、团队协作、分配给他人(虽然有用户但 task 业务层面仍单租户)
- ❌ 注册 / 重置密码 / 多用户管理后台(默认管理员 `admin` 由 Flyway 种子,生产场景另议)
- ❌ 角色 / 权限(RBAC):本项目的"鉴权"仅为登录态校验,无角色概念
- ❌ Refresh token(Task001 仅 access token,1 小时过期)
- ❌ 标签、子任务、任务依赖
- ❌ 附件上传、富文本描述
- ❌ 通知、提醒、定时任务
- ❌ 物理删除(仅 `deleted=1`,由 MP `@TableLogic` 实现)

**工程层面不做**:

- ❌ **前端 UI 框架**(Element Plus / Ant Design Vue):只用 Vue 3 原生 + 极简手写 CSS,避免组件库 API 盖过 SDD 方法论
- ❌ **前端状态管理库**(Pinia / Vuex):demo 页本地 state 足够
- ❌ **前端 SSR / Nuxt**:纯 SPA
- ❌ **生产部署**(Nginx / K8s / CI/CD):仅本地开发跑通
- ❌ **多环境配置**(dev/staging/prod):只有开发环境
- ❌ **数据库存储过程 / 触发器**(详见 `specs/91-编码规范.md`,范围仅到表/列/索引/视图)
- ❌ **前端单元测试覆盖率门槛**(仅做关键组件的基本用例,详见 `specs/90-工程约束.md`)

**将来可能做**:

- ⏸ 批量删除、任务搜索、按时间排序
- ⏸ 接入 Pinia + Element Plus
- ⏸ 引入 tag 表 + 多对多关系(作为"需求变更闭环"演练)

## 7. 关键术语表

| 术语 | 定义 |
|---|---|
| **前端模块 / FE** | `workitem-frontend` 仓的 Vue 3 + Vite + TS 项目 |
| **后端模块 / BE** | `workitem-backend` 仓的 Spring Boot 项目 |
| **数据库模块 / DB** | MySQL 实例 + Flyway 迁移脚本(唯一归属 `workitem-db/migration/`,不在后端仓) |
| **规范仓 / spec 仓** | `workitem-docs` 仓,业务契约的唯一真源,通过 git tag 版本化对外发布 |
| **约束 spec** | `workitem-docs/specs/` 下的 3 份文档,规定**通用命名/目录/追溯规则**,独立于具体功能 |
| **功能 spec** | `workitem-docs/01-需求分析.md` ~ `04-验收标准.md`,描述具体业务 |
| **跨仓协调 spec** | `workitem-docs/05-跨仓协调.md`,多仓 SDD 的版本 tag、epic、PR 编排纪律 |
| **三维矩阵** | `04-验收标准.md §2` 提出的追溯矩阵:BR(业务规则) × 模块(FE/BE/DB) × TC(测试用例) |
| **模块编号** | FE-01 / BE-01 / DB-01 形式,与 BR-01 正交,用于模块拆分和测试追溯 |
| **迁移脚本** | Flyway 管理的 DDL 脚本,命名为 `V{N}__{snake_case}.sql`,合入 DB 仓 main 后不可修改 |
| **traceparent** | W3C Trace Context 标准请求头,前端生成,后端透传到日志 / login_log,跨服务追溯 |
| **JWT(access token)** | 登录后由后端签发,前端存 localStorage,后续请求 `Authorization: Bearer <token>`;1 小时过期 |
| **登录日志(login_log)** | 每次登录尝试(成功/失败/锁定)的审计记录,字段含 username / ip / user_agent / traceparent |

## 8. 验收视角

本文档的验收不是"代码跑通",而是:

1. ✅ 所有 USF-* 在 `02-功能规范.md §8 模块拆分` 中都有**前端**入口(至少一个 FE-* 编号)
2. ✅ 所有 USA-* 都有**后端**落点(至少一个 BE-* 编号)
3. ✅ 所有 P0 功能在 `04-验收标准.md §2 三维矩阵` 中,**每一条 BR 至少有一条自动化用例**(TC-U / TC-IT / TC-FE / TC-E2E 任一)
4. ✅ 所有"范围外"项在最终代码中**确实没有出现**(反向校验,详见 `04-验收标准.md §6.4`)
5. ✅ `workitem-docs/specs/` 三份约束文档**只写规则和占位符**,不嵌入 `task`、`TaskListView.vue` 等具体名字
6. ✅ 4 个 git 仓(`workitem-docs` / `workitem-backend` / `workitem-frontend` / `workitem-db`)的 README 顶部都声明了 `Implements: spec@vX.Y.Z`(详见 `05-跨仓协调.md`)
