## 1. 通用约定

### 1.1 统一响应结构 `Result<T>`

所有后端接口**必须**返回以下结构:

```jsonc
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "timestamp": 1713945600000
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | int | 业务码。`0` 成功;非 0 失败。见 §5 错误码表 |
| `message` | string | 人类可读的提示信息(**前端直接用作 toast 文案**) |
| `data` | object / null | 业务数据。失败时为 `null` |
| `timestamp` | long | 服务端响应时间戳(ms) |

### 1.2 HTTP 状态码约定

| 场景 | HTTP Status | 业务 `code` |
|---|---|---|
| 成功 | 200 | 0 |
| 参数校验失败 | 400 | 1001 |
| 资源不存在 | 404 | 1002 |
| 业务规则冲突 | 409 | 1003 |
| 鉴权失败(凭据无效 / 缺 token / token 过期) | 401 | 3001 / 3003 |
| 资源被锁(账号锁定中) | 423 | 3002 |
| 服务端未知错误 | 500 | 9999 |

> ⚠️ 即使 HTTP 是 4xx/5xx,响应体仍是 `Result<T>` 结构(F-06 的前提)。

### 1.3 通用请求约定

- `Content-Type`: `application/json;charset=UTF-8`
- 日期时间:ISO-8601 `yyyy-MM-dd'T'HH:mm:ss`
- 日期:`yyyy-MM-dd`
- ID 字段:后端 `Long` ↔ 前端 `number`

### 1.4 CORS

- 开发环境后端允许来源 `http://localhost:5173`(Vite 默认端口)
- 不使用 `*` 通配
- 鉴权后必须允许 `Authorization` 与 `traceparent` 出现在 `Access-Control-Allow-Headers`

### 1.5 通用请求头(全部接口必含)

| Header | 必填 | 说明 |
|---|---|---|
| `Content-Type: application/json;charset=UTF-8` | ✅ | 写接口 |
| `traceparent` | SHOULD | W3C Trace Context 格式,前端 SHOULD 生成;缺失时后端 Filter 兜底生成(BR-13)。无论来源,落到日志 / `login_log` 必须存在 |
| `Authorization: Bearer <jwt>` | ✅(除 `/api/login`)| 登录后所有 task API 必填(BR-15) |

### 1.6 鉴权约定

- 登录后由后端签发 **JWT**(HS256,1 小时过期);前端存 localStorage
- task 域所有接口(API-01 ~ API-05)**必须**带 `Authorization: Bearer <jwt>`,否则返回 `code=3003 AUTH_REQUIRED`
- JWT 由后端 `JwtFilter`(BE-08)校验;校验失败 / 过期统一返回 `3003`
- 详细技术方案见 `../design/03-技术方案.md §1.1` + `specs/90-工程约束.md §A.10`

## 2. 数据字典

### 2.1 Task 字段

| 字段名 | 类型 | 长度/范围 | 必填 | 约束 | 说明 |
|---|---|---|---|---|---|
| `id` | Long | - | 系统生成 | 主键,自增 | 任务 ID |
| `title` | String | 1 ~ 100 字符 | ✅ | 非空、trim 后非空 | 任务标题 |
| `description` | String | 0 ~ 500 字符 | ❌ | 可为 null | 任务描述 |
| `status` | Enum | - | 创建时默认 `TODO` | 见 §2.2 | 任务状态 |
| `priority` | Enum | - | 创建时默认 `MEDIUM` | 见 §2.3 | 优先级 |
| `dueDate` | LocalDate | - | ❌ | 可为 null,允许早于当日 | 截止日期 |
| `createdAt` | LocalDateTime | - | 系统生成 | 创建时间 | - |
| `updatedAt` | LocalDateTime | - | 系统生成 | 最后更新时间 | - |
| `deleted` | int | 0/1 | 系统生成 | 0=未删除,1=已删除 | **不对外暴露**(BE 响应与 FE 均不含) |

### 2.2 状态枚举 `TaskStatus`

| 值 | 含义 | 前端列表页 Badge 颜色(建议) |
|---|---|---|
| `TODO` | 待办(新建时默认) | 灰 |
| `DOING` | 进行中 | 蓝 |
| `DONE` | 已完成 | 绿 |

### 2.3 优先级枚举 `TaskPriority`

| 值 | 含义 |
|---|---|
| `LOW` | 低 |
| `MEDIUM` | 中(新建时默认) |
| `HIGH` | 高 |

### 2.4 `app_user` 字段(Task001 引入)

> ⚠️ 表名 `app_user` 而非 `user`,避免 MySQL 保留字(参见 `specs/91-编码规范.md §A.2`)。
>
> **命名映射**:本表"字段名"列采 Java 驼峰(如 `passwordHash`),SQL 列对应蛇形(`password_hash`);自动映射由 MyBatis Plus `map-underscore-to-camel-case=true` 完成(`../design/03-技术方案.md §1.1`)。BR 描述与 Java 代码统一用驼峰,DDL 与 Mapper XML 用蛇形。

| 字段名(Java)| 类型 | 长度/范围 | 必填 | 约束 | 说明 | PII |
|---|---|---|---|---|---|---|
| `id` | Long | - | 系统生成 | 主键,自增 | 用户 ID | NO |
| `username` | String | 3 ~ 50 字符 | ✅ | 唯一,trim 后非空,字母 / 数字 / 下划线 | 登录名 | NO |
| `passwordHash` | String | BCrypt 输出 ~60 字符 | ✅ | BCrypt strength=12 | 密码哈希,**不对外暴露** | NO(哈希值非 PII) |
| `status` | Enum | - | 默认 `ACTIVE` | `ACTIVE` / `DISABLED` | 账号启用状态 | NO |
| `failedLoginCount` | int | 0 ~ 99 | 默认 0 | 失败计数,登录成功重置 | 登录失败次数 | NO |
| `lockedUntil` | LocalDateTime | - | ❌ 默认 null | null 表示未锁定 | 锁定到期时间 | NO |
| `createdAt` / `updatedAt` / `deleted` | 同 §2.1 | - | - | - | - | NO |

`passwordHash` **永不**对外暴露:Controller / VO / 日志 / 任何响应**绝不**包含此字段。

### 2.5 `LoginType` 枚举

| 值 | 含义 |
|---|---|
| `SUCCESS` | 登录成功 |
| `FAIL_INVALID` | 用户名或密码错误 |
| `FAIL_LOCKED` | 账号已锁定时尝试登录 |

### 2.6 `login_log` 字段(Task001 引入)

> 命名映射同 §2.4:Java 驼峰 ↔ SQL 蛇形,MP 自动转换。

| 字段名(Java)| 类型 | 必填 | 约束 | 说明 | PII |
|---|---|---|---|---|---|
| `id` | Long | 系统 | 主键自增 | - | NO |
| `appUserId` | Long | ❌(可空)| 外键到 `app_user.id`(逻辑外键,不强约束)| 用户名不存在的 fail 此字段 null | NO |
| `usernameAttempted` | String(50)| ✅ | trim 后非空 | 即使用户名不存在也记录 | NO |
| `loginType` | Enum | ✅ | 见 §2.5 | - | NO |
| `loginIp` | String(64)| ✅ | IPv4 / IPv6 | 客户端 IP(经反代解析)| **YES**,日志输出需脱敏 |
| `userAgent` | String(512)| ❌ | 可空 | 浏览器 UA | 可视为 PII |
| `traceparent` | String(128)| ❌ | W3C 格式 | 关联本次请求的 trace | NO |
| `createdAt` | LocalDateTime | 系统 | - | 写入时间 | NO |

## 3. 接口清单

| # | 接口 ID | Method | URL | 功能 ID | 鉴权 | 简述 |
|---|---|---|---|---|---|---|
| 1 | API-01 | POST | `/api/workitems` | F-01 | ✅ JWT | 创建任务 |
| 2 | API-02 | GET | `/api/workitems` | F-02 | ✅ JWT | 分页查询任务 |
| 3 | API-03 | GET | `/api/workitems/{id}` | F-03 | ✅ JWT | 查询任务详情 |
| 4 | API-04 | PUT | `/api/workitems/{id}` | F-04 | ✅ JWT | 更新任务 |
| 5 | API-05 | DELETE | `/api/workitems/{id}` | F-05 | ✅ JWT | 逻辑删除任务 |
| 6 | API-06 | POST | `/api/login` | F-07 | ❌ 公开 | 登录(返回 JWT) |

### 3.1 API-01 创建任务

**URL**:`POST /api/workitems`

**请求体示例**:

```jsonc
{
  "title": "写 SDD 文档",
  "description": "完成 04 章节的 4 份规范",
  "priority": "HIGH",
  "dueDate": "2026-04-30"
}
```

**请求字段校验**:

| 字段 | 规则 | 违反时错误码 |
|---|---|---|
| `title` | 必填,trim 后长度 1~100 | 1001 |
| `description` | 可空,长度 ≤ 500 | 1001 |
| `priority` | 可空(默认 `MEDIUM`),必须是枚举值 | 1001 |
| `dueDate` | 可空,格式 `yyyy-MM-dd` | 1001 |

**业务规则**:

- 创建时 `status` 强制为 `TODO`,忽略请求中传入的 `status`(BR-01)
- 创建时 `priority` 若未传,默认 `MEDIUM`(BR-02)
- `createdAt` / `updatedAt` 由系统设置为当前时间(BR-08)

**成功响应**(HTTP 200):

```jsonc
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "title": "写 SDD 文档",
    "description": "完成 04 章节的 4 份规范",
    "status": "TODO",
    "priority": "HIGH",
    "dueDate": "2026-04-30",
    "createdAt": "2026-04-25T10:00:00",
    "updatedAt": "2026-04-25T10:00:00"
  },
  "timestamp": 1713945600000
}
```

### 3.2 API-02 分页查询任务

**URL**:`GET /api/workitems?status=TODO&page=1&size=10`

**Query 参数**:

| 参数 | 类型 | 必填 | 默认 | 约束 |
|---|---|---|---|---|
| `status` | String | ❌ | 不筛选 | 必须是 `TODO` / `DOING` / `DONE` 之一,否则 1001 |
| `page` | int | ❌ | 1 | ≥ 1,否则 1001 |
| `size` | int | ❌ | 10 | 1 ≤ size ≤ 100,否则 1001 |

**业务规则**:

- 仅返回 `deleted = 0` 的任务(BR-03,由 MP 逻辑删除全局配置实现,见 `../design/03-技术方案.md §5.3`)
- 按 `createdAt DESC` 排序(BR-04)

**成功响应**:

```jsonc
{
  "code": 0,
  "message": "success",
  "data": {
    "page": 1,
    "size": 10,
    "total": 42,
    "totalPages": 5,
    "list": [
      { "id": 42, "title": "...", "status": "TODO", "...": "..." }
    ]
  },
  "timestamp": 1713945600000
}
```

**`PageVO<T>` 字段**:

| 字段 | 类型 | 说明 |
|---|---|---|
| `page` | int | 当前页码(从 1 开始) |
| `size` | int | 每页大小 |
| `total` | long | 总记录数(已过滤 `deleted=0`) |
| `totalPages` | int | `ceil(total / size)` |
| `list` | array | 当前页数据 |

### 3.3 API-03 查询任务详情

**URL**:`GET /api/workitems/{id}`

**Path 参数**:`id`(Long,> 0)

**业务规则**:

- 若 `id` 不存在或对应记录 `deleted=1` → 返回 `code=1002 WORKITEM_NOT_FOUND`(BR-03)

**成功响应**:结构同 API-01 的 `data` 字段。

### 3.4 API-04 更新任务

**URL**:`PUT /api/workitems/{id}`

**Path 参数**:`id`(Long,> 0)

**请求体**(所有字段均可选,但**至少传一个**):

```jsonc
{
  "title": "新标题",
  "description": "新描述",
  "status": "DOING",
  "priority": "LOW",
  "dueDate": "2026-05-01"
}
```

**请求字段校验**:

| 字段 | 规则 | 违反时错误码 |
|---|---|---|
| `title` | 若传,trim 后长度 1~100 | 1001 |
| `description` | 若传,长度 ≤ 500 | 1001 |
| `status` | 若传,必须是枚举值 | 1001 |
| `priority` | 若传,必须是枚举值 | 1001 |
| `dueDate` | 若传,格式 `yyyy-MM-dd` | 1001 |

**业务规则**:

- 更新请求**必须**至少传一个字段,否则 1001(BR-05)
- 任务不存在或已删除 → `code=1002 WORKITEM_NOT_FOUND`
- **已完成的任务(`status=DONE`)不允许修改 `priority` 和 `dueDate`** → `code=1003 WORKITEM_DONE_IMMUTABLE`(BR-06);但**允许**将 `status` 从 `DONE` 改回其他状态(撤销完成)
- 更新后 `updatedAt` 设为当前时间(BR-08)
- 未传的字段保持原值(PATCH 语义,非全量替换)

**成功响应**:返回更新后的完整任务。

### 3.5 API-05 逻辑删除任务

**URL**:`DELETE /api/workitems/{id}`

**Path 参数**:`id`(Long,> 0)

**业务规则**:

- 任务不存在或**已经**被删除 → `code=1002 WORKITEM_NOT_FOUND`(幂等性:重复删除也返回 404)
- 成功时将 `deleted` 置为 `1`(BR-07,由 MP `@TableLogic` + `deleteById` 自动转为 UPDATE)
- 被删除的任务在所有查询接口中不可见

**成功响应**:

```jsonc
{
  "code": 0,
  "message": "success",
  "data": null,
  "timestamp": 1713945600000
}
```

### 3.6 API-06 登录

**URL**:`POST /api/login`(无需 JWT)

**请求体**:

```jsonc
{
  "username": "admin",
  "password": "12346#@&"
}
```

**请求字段校验**:

| 字段 | 规则 | 违反时错误码 |
|---|---|---|
| `username` | 必填,trim 后长度 3~50 | 1001 |
| `password` | 必填,长度 ≥ 1(不暴露密码强度上限到前端) | 1001 |

**业务规则**:

- 用户不存在 / 密码错 → `code=3001 AUTH_INVALID_CREDENTIAL`,**写 login_log**(`loginType=FAIL_INVALID`,BR-14)
- 账号锁定中(`lockedUntil > now()`)→ `code=3002 AUTH_LOCKED`,**写 login_log**(`FAIL_LOCKED`)
- 密码错且失败次数 ≥ 3(本次错误后)→ 设置 `lockedUntil = now() + 15min`,清空 `failedLoginCount`(BR-10)
- 登录成功 → 重置 `failedLoginCount=0` + `lockedUntil=null`,**写 login_log**(`SUCCESS`),签发 JWT

**成功响应**(HTTP 200):

```jsonc
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "<jwt-string>",
    "expiresIn": 3600,
    "username": "admin"
  },
  "timestamp": 1713945600000
}
```

**失败响应示例**:

```jsonc
{ "code": 3001, "message": "Invalid username or password", "data": null, "timestamp": ... }
{ "code": 3002, "message": "Account locked. Try again after <ISO-8601>", "data": null, "timestamp": ... }
```

> ⚠️ `3001` 的 message **不能区分**"用户名不存在"vs"密码错",防止用户名枚举攻击(`security-conventions §13`)。

### 3.7 API-07 获取当前用户

**URL**:`GET /api/me`(必须 JWT)

**业务规则**:

- 从 JWT `sub` 取 `appUserId`,查 `app_user`,返回非敏感字段
- JWT 缺失 / 过期 / 篡改 → `code=3003 AUTH_REQUIRED`(BR-15)

**成功响应**:

```jsonc
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "username": "admin",
    "status": "ACTIVE"
  },
  "timestamp": 1713945600000
}
```

## 4. 业务规则汇总

> 所有业务规则**必须**在 Service 层实现,Controller 仅做参数接收。

| 规则 ID | 规则描述 | 对应接口 |
|---|---|---|
| BR-01 | 创建时 `status` 强制为 `TODO` | API-01 |
| BR-02 | 创建时 `priority` 未传则默认 `MEDIUM` | API-01 |
| BR-03 | 所有查询过滤 `deleted=0`(由 MP 逻辑删除全局配置实现,参见 `../design/03-技术方案.md §5.3`) | API-02, API-03 |
| BR-04 | 列表按 `createdAt DESC` 排序 | API-02 |
| BR-05 | 更新时至少传 1 个字段 | API-04 |
| BR-06 | `status=DONE` 的任务不允许修改 `priority` / `dueDate` | API-04 |
| BR-07 | 删除是逻辑删除(`deleted=1`,由 MP `@TableLogic` + `deleteById` 实现) | API-05 |
| BR-08 | 任何修改操作都要刷新 `updatedAt`(由 MP `MetaObjectHandler` 自动填充,参见 `../design/03-技术方案.md §5.4`) | API-01, API-04, API-05 |
| **BR-09** | 密码**必须**以 BCrypt 哈希存储(strength ≥ 12),`passwordHash` 永不出现在响应体 / VO / 日志 | API-06,Flyway V4 种子 |
| **BR-10** | 登录失败次数 ≥ 3 时,设置 `lockedUntil = now() + 15min`,后续清空 `failedLoginCount` 计数(避免叠加) | API-06 |
| **BR-11** | 锁定期间(`lockedUntil > now()`)登录尝试 → `code=3002 AUTH_LOCKED`,**不**进入密码校验 | API-06 |
| **BR-12** | 登录成功 → `failedLoginCount=0` + `lockedUntil=null` | API-06 |
| **BR-13** | 所有 HTTP 入口请求**逻辑上必须**含 `traceparent` 请求头(W3C 格式)。客户端 SHOULD 提供;若缺失,后端 `TraceparentFilter`(BE-09)**MUST 兜底生成**新值并打 INFO 级"补全"日志。Filter 解析或生成后写入 MDC,使所有日志 / `login_log` 字段携带同源 `trace_id`。详见 `monitoring-conventions §5.2` | 所有 API |
| **BR-14** | 登录尝试(成功 / 失败 / 锁定)**必须**写一条 `login_log`,字段含 `username` / `loginType` / `loginIp` / `userAgent` / `traceparent` / `createdAt` | API-06 |
| **BR-15** | API-01 ~ API-05 与 API-07 **必须**带有效 JWT,缺失 / 过期 / 签名无效 → `code=3003 AUTH_REQUIRED`;由 `JwtFilter` 在 Controller 之前拦截 | API-01 ~ 05, API-07 |

## 5. 错误码表

> ⚠️ **错误码治理规则见** `specs/92-接口治理.md Part B`(段位分配、命名、生命周期)。
> 本表仅列**当前已分配**的错误码。新增错误码必须遵守段位规则(`1xxx/2xxx/3xxx/4xxx/5xxx/9xxx`)。

| `code` | 常量名 | HTTP Status | message 示例(兜底英文) | 触发场景 | 备注 |
|---|---|---|---|---|---|
| `0` | `SUCCESS` | 200 | `success` | 成功 | - |
| `1001` | `PARAM_INVALID` | 400 | `Parameter invalid: title is blank` | Bean Validation / 枚举非法 | 正常 `1xxx` 段 |
| `1002` | `WORKITEM_NOT_FOUND` | 404 | `Task not found: id=42` | id 不存在或已被逻辑删除 | **历史遗留**:语义属 `2xxx`,已冻结不改 |
| `1003` | `WORKITEM_DONE_IMMUTABLE` | 409 | `Done task cannot change priority or dueDate` | BR-06 | **历史遗留**:语义属 `2xxx`,已冻结不改 |
| **`3001`** | `AUTH_INVALID_CREDENTIAL` | 401 | `Invalid username or password` | 用户名不存在或密码错(BR-09 语义合并不区分)| Task001 引入 |
| **`3002`** | `AUTH_LOCKED` | 423 | `Account locked. Try again after <ISO-8601>` | BR-11 锁定期间登录 | Task001 引入 |
| **`3003`** | `AUTH_REQUIRED` | 401 | `Authentication required` | BR-15 缺失 / 过期 / 无效 JWT | Task001 引入 |
| `9999` | `INTERNAL_ERROR` | 500 | `Internal server error` | 未捕获异常 | 正常 `9xxx` 段 |

> 新增错误码**必须**遵守:

> - 命名见 `92-接口治理.md Part B`
> - message i18n 纪律见 `91-编码规范.md §C.5`

## 7. 前端页面契约

> 本节规定**前端页面应该长什么样**。不提供视觉稿,仅规定:路由、核心行为、表单校验与后端对齐方式。

### 7.1 路由清单

| 路由 | 名称 | 对应 User Story | 对应接口 |
|---|---|---|---|
| `/` | 首页(若已登录重定向到列表;未登录重定向到登录页) | - | - |
| `/login` | 登录页(Task001 引入) | USF-07 | API-06 |
| `/tasks` | 任务列表页 | USF-02, USF-05 | API-02, API-05 |
| `/tasks/new` | 新建任务页 | USF-01 | API-01 |
| `/tasks/:id` | 任务详情页 | USF-03, USF-05 | API-03, API-05 |
| `/tasks/:id/edit` | 编辑任务页 | USF-04 | API-03(加载)+ API-04(保存) |

> 所有 `/tasks/*` 路由**必须**经路由守卫检查 JWT,无 token 或 401 响应时跳转 `/login`。

### 7.2 页面核心行为

#### 7.2.1 列表页 `/tasks`

- 加载时调用 API-02(默认 `page=1&size=10`,无 `status` 筛选)
- 提供 `status` 下拉筛选(四选一:全部 / TODO / DOING / DONE),变更即重新请求
- 提供分页器,翻页重新请求
- 列表项字段:`title`、`status`(Badge)、`priority`、`dueDate`、操作按钮(详情 / 编辑 / 删除)
- 删除按钮**必须二次确认**再调用 API-05
- 空列表显示空状态占位,不显示空表格

#### 7.2.2 新建页 `/tasks/new`

- 表单字段:`title`(必填)、`description`、`priority`、`dueDate`
- **前端校验规则必须与 spec §2.1 完全一致**(长度/必填/枚举),失败时同步高亮错误,不发送请求
- 通过后调用 API-01,成功则跳转 `/tasks`(列表),失败则在原页展示 toast(走 F-06)

#### 7.2.3 详情页 `/tasks/:id`

- 挂载时调用 API-03;id 不存在(1002)时显示"任务不存在"占位,不显示表单
- 展示所有字段(不含 `deleted`);提供"编辑"和"删除"按钮

#### 7.2.4 编辑页 `/tasks/:id/edit`

- 挂载时调用 API-03 加载原值填表
- 表单校验同新建页,额外:**若原状态为 DONE**,`priority` 与 `dueDate` 输入框必须 disable(前端同步 BR-06,但**后端仍以 BR-06 为准**,不因前端禁用就省掉后端校验)
- 保存时调用 API-04;成功跳回 `/tasks/:id`,失败展示 toast

### 7.3 F-06 统一错误提示

- 前端统一 HTTP 拦截器:对任何响应(含 `code=0`)都检查 `code`
- `code≠0` → 触发全局 toast,内容为 `message`(直接取用后端文案)
- 不在组件里手写 `if (res.code !== 0)` 判断

### 7.4 与后端 spec 的对齐原则

前端**必须**与后端 spec 一致:

1. 校验规则(长度、必填、枚举值)
2. 枚举文案(TODO/DOING/DONE 的中文映射可选,但枚举值严格一致)
3. 日期格式(`yyyy-MM-dd`)
4. 错误码语义(1001/1002/1003 的处理分支一致)

> ⚠️ 这条对齐是**双向的**:后端 spec 变,前端必须跟改;前端发现不合理,必须回去改 spec 而不是私自放宽校验。

## 8. 模块拆分(sdd-01 核心增量)🎯

> **一个业务规则(BR)可能落在多个模块(FE / BE / DB)**,
> 所以追溯不能只有 BR → TC 一维,必须扩展为 **BR × 模块 × TC 三维**(见 `04-验收标准.md §2`)。
>
> 本节给每个模块分配**独立编号**(FE-* / BE-* / DB-*),它们与 BR-* 正交。

### 8.4 跨模块业务规则落点举例

为了让"三维矩阵"更直观,举两个 BR 的跨模块落点:

**BR-06**:`status=DONE` 的任务不允许修改 `priority` / `dueDate`

| 模块 | 落点 | 说明 |
|---|---|---|
| FE-04 | 编辑页在加载到 `status=DONE` 时 disable 两个字段(UX 软约束) | 仅提升体验,不作为安全防线 |
| BE-02 | Service 层抛 `BizException(WORKITEM_DONE_IMMUTABLE)` | **安全防线**(真源) |
| BE-04 | GlobalExceptionHandler 转 `code=1003` + HTTP 409 | 统一响应 |
| FE-05 | 拦截器收到 `code=1003` 展示 toast(message 来自后端) | 提示 |

BR-06 对应的测试分布:
- TC-U-05(BE Service 单测)+ TC-IT-* (BE 集测)+ TC-FE-* (FE 组件测)+ TC-E2E-* (端到端)

**BR-03**:所有查询过滤 `deleted=0`

| 模块 | 落点 | 说明 |
|---|---|---|
| BE-03 | Mapper XML 的 WHERE 子句固定带 `deleted=0` | **安全防线** |
| DB-02 | 索引包含 `deleted` 列,保障性能 | 性能支撑 |
| DB-03(可选) | 视图 `v_task_active` 封装过滤,减少 Mapper 重复 | 约定式保护 |
| FE-01 | 列表页永远不发 `deleted` 参数,也不展示已删除项 | 消费侧 |

### 8.5 模块拆分的使用方式

在 `04-验收标准.md §2` 的三维矩阵中,每一条 BR 都会横向列出涉及的模块编号,纵向列出测试用例编号。
这样:

- **从 BR 出发**:可以查出这条规则在哪些模块有落点、分别用哪些 TC 验证
- **从模块出发**:可以查出这个模块承担了哪些 BR
- **从 TC 出发**:可以反查测试对应的模块和 BR
