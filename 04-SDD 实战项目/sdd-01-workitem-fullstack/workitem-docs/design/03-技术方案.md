## 1. 技术栈

### 1.1 后端技术栈

| 层次 | 选型 | 版本 | 选型理由 |
|---|---|---|---|
| 语言 | Java | **21** | 利用 pattern matching / records / virtual threads 等新特性;Spring Boot 4 推荐基线 |
| 框架 | Spring Boot | **4.0.5** | 当前主干稳定版;与 MP starter4 对齐 |
| Web | Spring Web MVC | 随 SB | 最经典的 REST 实现 |
| 持久层 | **MyBatis Plus** | 3.5.16 | 在 MyBatis 上提供 BaseMapper 通用 CRUD、条件构造器、全局逻辑删除;减少 XML 样板 |
| MP Starter | `mybatis-plus-spring-boot4-starter` | 3.5.16 | 与 Spring Boot 4 专用 |
| SQL 解析 | `mybatis-plus-jsqlparser` | 3.5.16 | 支持 MP 分页插件等 SQL 解析能力 |
| 连接池 | **HikariCP** | 随 SB | 高性能,官方推荐 |
| 数据库 | **MySQL** | 8.0(服务端)/ connector 9.5.0 | 生产级真库 |
| 数据库迁移 | **Flyway** | 10.x(随 SB) | 发布后脚本不可变 + 版本号校验,契合 SDD "决策可追溯" |
| 参数校验 | Jakarta Validation | 3.x | 声明式校验 |
| 代码简化 | **Lombok** | 随 SB | `@Data`、`@Slf4j`、`@Builder` 等,压缩样板 |
| 日志 | **Logback**(`spring-boot-starter-logging`)+ 外挂 `logback-config.xml` | 随 SB | 支持 springProfile 分环境路由 |
| API 文档 | SpringDoc OpenAPI | 最新兼容 SB 4 | 自动生成 OpenAPI 3 |
| **鉴权框架** | **Spring Security**(随 SB 4)| 随 SB | FilterChain 集成 JWT,生产实践 |
| **JWT 库** | **Nimbus JOSE JWT** | 随 Spring Security | Spring Security 已内置依赖,签发 / 校验 HS256 |
| **密码哈希** | **BCryptPasswordEncoder** | 随 Spring Security | 工作因子 ≥ 12,符合 `../specs/90-工程约束.md Part A` |
| 构建 | Maven | 3.8+ | - |
| 测试 | JUnit 5 + Mockito + MockMvc + **Testcontainers** | 随 SB | 集测用 Testcontainers 启动 MySQL,与 Flyway 一致 |

**关键集成点**(与技术选型强绑定):

- MP 全局逻辑删除:通过 `mybatis-plus.global-config.db-config.logic-delete-field=deleted` 让所有 `BaseMapper` 的 `select*` / `update*` 自动过滤 `deleted=0`、`delete*` 自动转为 `UPDATE SET deleted=1`(参见 `specs/91-编码规范.md Part A` 逻辑删除小节)
- MP 驼峰映射:`mybatis-plus.configuration.map-underscore-to-camel-case=true`
- HikariCP 池命名与监控项暴露
- Logback 日志按 `sit / uat / prod` springProfile 分路由(见 §9.3)

**不引入**:

- ❌ Spring Data JPA(与 MyBatis Plus 二选一)
- ❌ MapStruct(DTO 转换少量,Translator 手写即可)
- ❌ 任何 JDBC 直连样板(统一走 MP)

## 4. 后端分层(DDD 四层 + 核心配置)

### 4.1 层职责表

| 分层 | 包路径(相对 `com.zmz.sdd.fullstack.app`) | 对应模块编号 | 职责 |
|---|---|---|---|
| **API 接口层** | `api.controller` | BE-01 | HTTP 接入 / 参数绑定 / `@Valid`;**不碰**业务规则 |
| **应用服务层** | `application.service` + `application.service.impl` | BE-02 | 业务规则(BR-01~08)/ 事务边界 / 编排多个 repository |
| **领域层** | `domain.model` + `domain.repository` | 部分 BE-02 | 领域模型与仓储**接口**;不依赖框架 |
| **基础设施层** | `infrastructure.dao.entity` + `infrastructure.dao.mapper` + `infrastructure.repository.impl` + `infrastructure.repository.translator` | BE-03 | Entity(@TableName)/ Mapper(BaseMapper)/ 仓储实现 / Model↔Entity 转换 |
| **核心横切** | `core.config` + `core.common` | BE-04 + BE-05 | Result / ErrorCode / BizException / 全局异常 / MP 配置 / CORS |

### 4.2 分层调用方向

```
api.controller
    ↓
application.service (接口)
    ↓
application.service.impl(实现,依赖 domain.repository)
    ↓
domain.repository (接口)
    ↓
infrastructure.repository.impl(实现,用 translator 转 Entity)
    ↓
infrastructure.dao.mapper(BaseMapper)
    ↓
DB
```

**硬性禁止**:

- ❌ Controller 直接依赖 Mapper(绕过 Service + Repository)
- ❌ Service 直接依赖 Mapper(必须走 domain.repository 接口)
- ❌ Entity 泄漏到 Service 之上(Service / Controller 层仅见 domain.model 与 DTO)
- ❌ domain 层 import MyBatis Plus 或 Spring 注解(保持领域纯净)

### 4.3 启动类

`TaskFullstackApplication.java` 位于 `com.zmz.sdd.fullstack` 根包,仅承载 `@SpringBootApplication` 与 `@MapperScan("com.zmz.sdd.fullstack.app.infrastructure.dao.mapper")`。

## 5. 数据库设计

### 5.1 表结构 `task`

由 **Flyway** 管理(位于 `workitem-db/migration/`):

```sql
-- 位于 workitem-db/migration/V1__task001_init_task_table.sql  (独立 DB 仓,不在后端仓内)
-- [SDD-TASK: Task001]
-- [SDD-SPEC: 02-功能规范.md §8 DB-01]

CREATE TABLE `workitem` (
    `id`          BIGINT        NOT NULL AUTO_INCREMENT,
    `title`       VARCHAR(100)  NOT NULL,
    `description` VARCHAR(500)  DEFAULT NULL,
    `status`      VARCHAR(16)   NOT NULL DEFAULT 'TODO',
    `priority`    VARCHAR(16)   NOT NULL DEFAULT 'MEDIUM',
    `due_date`    DATE          DEFAULT NULL,
    `created_at`  DATETIME      NOT NULL,
    `updated_at`  DATETIME      NOT NULL,
    `deleted`     TINYINT       NOT NULL DEFAULT 0,
    PRIMARY KEY (`id`),
    KEY `idx_workitem_status_deleted` (`status`, `deleted`),
    KEY `idx_workitem_deleted_created` (`deleted`, `created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

> ⚠️ 表 / 列 / 索引命名必须符合 `specs/91-编码规范.md Part A`;DDL 变更必须**新增** `V<N+1>__*.sql`,**禁止**修改已发布脚本。

-- workitem-db/migration/V4__task001_seed_default_admin.sql
-- [SDD-TASK: Task001]
-- [SDD-SPEC: 02-功能规范.md §8 DB-06] 默认管理员账号 admin / 12346#@&
-- ⚠️ 默认密码哈希值由 P2 阶段用 BCrypt(strength=12) 生成,本 spec 仅占位
INSERT INTO `app_user` (`username`, `password_hash`, `status`, `failed_login_count`, `created_at`, `updated_at`, `deleted`)

> ⚠️ V4 的 `<BCRYPT_HASH_OF_..._PLACEHOLDER>` 在 P2 阶段用 BCrypt 生成具体哈希值后替换;**默认密码 `12346#@&` 仅教学用**,生产场景必须强制首次登录改密码 + 走 secrets manager(`../specs/90-工程约束.md Part A`)。

### 5.2 字段映射(DB ↔ Java ↔ TS)

| DB 列 | Java(Entity)| Java(Domain Model)| TypeScript | 类型链 |
|---|---|---|---|---|
| `id` | `Long id` | `Long id` | `id: number` | BIGINT ↔ Long ↔ number |
| `title` | `String title` | `String title` | `title: string` | - |
| `description` | `String description` | `String description` | `description: string \| null` | - |
| `status` | `String status` | `TaskStatus status` | `'TODO' \| 'DOING' \| 'DONE'` | translator 转枚举 |
| `priority` | `String priority` | `TaskPriority priority` | `'LOW' \| 'MEDIUM' \| 'HIGH'` | - |
| `due_date` | `LocalDate dueDate` | `LocalDate dueDate` | `dueDate: string \| null` | DATE ↔ LocalDate ↔ `yyyy-MM-dd` |
| `created_at` | `LocalDateTime createdAt` | `LocalDateTime createdAt` | `createdAt: string` | DATETIME ↔ LocalDateTime ↔ ISO-8601 |
| `updated_at` | `LocalDateTime updatedAt` | `LocalDateTime updatedAt` | `updatedAt: string` | - |
| `deleted` | `Integer deleted`(仅 Entity,标 `@TableLogic`) | ❌ 不出现 | ❌ 不出现 | MP 逻辑删除字段,上层不可见 |

- MP 配置 `map-underscore-to-camel-case=true`,DB 下划线 ↔ Java 驼峰自动映射
- `deleted` 字段**只在 Entity 可见**,Entity 通过 `@TableLogic` 注解告知 MP 自动处理;Service 层与以上**完全感知不到**该字段
- Model ↔ Entity 由 `infrastructure.repository.translator` 负责

### 5.3 MyBatis Plus 逻辑删除(全局配置)

**不在 SQL / Mapper 层显式写 `WHERE deleted = 0`**,而是通过 MP 全局配置让所有 `BaseMapper` 自动处理:

```properties
mybatis-plus.global-config.db-config.logic-delete-field = deleted
mybatis-plus.global-config.db-config.logic-delete-value = 1
mybatis-plus.global-config.db-config.logic-not-delete-value = 0
```

**效果**:

| 方法 | MP 自动生成 SQL |
|---|---|
| `mapper.selectById(id)` | `SELECT ... WHERE id=? AND deleted=0` |
| `mapper.selectList(wrapper)` | `SELECT ... WHERE <wrapper conditions> AND deleted=0` |
| `mapper.deleteById(id)` | `UPDATE task SET deleted=1, updated_at=NOW() WHERE id=?` |
| `mapper.updateById(entity)` | `UPDATE ... WHERE id=? AND deleted=0` |

> ⚠️ 若 Mapper 里有**自定义 XML SQL**,**必须手工加** `deleted=0`(MP 的自动逻辑删除仅对继承 `BaseMapper` 的通用方法生效)。详见 `specs/91-编码规范.md Part A` 逻辑删除节。

### 5.4 自动填充 `created_at` / `updated_at`

通过 MP 的 `MetaObjectHandler` 机制:

- Entity 字段标 `@TableField(fill = FieldFill.INSERT)` / `@TableField(fill = FieldFill.INSERT_UPDATE)`
- 实现 `MetaObjectHandler` 在 `insertFill` / `updateFill` 中写入 `LocalDateTime.now()`
- 配置类 `MetaObjectHandlerConfig` 注册它
- Service / Controller 层**不关心**这两个字段的赋值

### 5.5 Flyway 约定(本项目侧)

**核心原则**:迁移脚本**唯一归属 `workitem-db/migration/`**(独立仓),**不放** `workitem-backend` 内。
详细命名约束见 `specs/91-编码规范.md §A.6`,此处仅给**版本规划 + 跨仓加载方式**。

#### 5.5.1 脚本版本规划

> 脚本命名**必须**带 Task-ID 前缀(`V<N>__task<NNN>_<desc>.sql`),详见 `specs/91-编码规范.md §A.6` 与 `../specs/06-任务与发布管理.md §2`。

| 版本 | 脚本文件名(位于 `workitem-db/migration/`) | 对应 Task | 内容 | 引入时机 |
|---|---|---|---|---|
| V1 | `V1__task001_init_task_table.sql` | Task001 | 建 `task` 表 + 2 个索引 | P2 阶段 |
| V2 | `V2__task001_init_app_user_table.sql` | Task001 | 建 `app_user` 表 + 唯一索引 | P2 阶段 |
| V3 | `V3__task001_init_login_log_table.sql` | Task001 | 建 `login_log` 表 + 2 个索引 | P2 阶段 |
| V4 | `V4__task001_seed_default_admin.sql` | Task001 | 种子默认管理员 admin | P2 阶段 |
| V5+ | (未规划) | 后续 Task | 未来变更按序追加 | 需求变更时 |

#### 5.5.2 workitem-backend 如何加载外部脚本

| 场景 | 配置方式 | 对应环境 |
|---|---|---|
| A. 本地开发(工作区布局) | `spring.flyway.locations=filesystem:../workitem-db/migration` | 本地跑 `mvn spring-boot:run` |
| B. 镜像构建 | 构建期把 `workitem-db/migration/*.sql` 拷贝进 jar 的 `classpath:db/migration/` | CI/CD / 生产 |
| C. Testcontainers 集测 | 测试类路径同方案 A,`@TestPropertySource` 指向相对 `workitem-db/` | `mvn test` |

本项目 P2 阶段只落 **方案 A**;方案 B/C 留作规划。

#### 5.5.3 为什么 DDL 放独立仓

- 独立仓与 SDD "决策可追溯" 天然契合:DDL 变更独立 PR、独立 review
- 独立仓让"谁能动 DDL"的权限可单独授予(DBA / 平台团队把关)
- 任何已合入 `workitem-db` main 的脚本即视为"已发布",checksum 锁定

## 7. 异常与错误处理策略

### 7.1 后端

| 异常类型 | 处理器动作 |
|---|---|
| `MethodArgumentNotValidException` / `ConstraintViolationException` | `code=1001`, HTTP 400 |
| `HttpMessageNotReadableException`(JSON 解析 / 枚举非法) | `code=1001`, HTTP 400 |
| `BizException(WORKITEM_NOT_FOUND)` | `code=1002`, HTTP 404 |
| `BizException(WORKITEM_DONE_IMMUTABLE)` | `code=1003`, HTTP 409 |
| MyBatis Plus 数据访问异常(`DataAccessException` 子类) | 视场景分流:约束冲突 → `code=1001`;其他 → `code=9999` |
| `Exception`(兜底) | `code=9999`, HTTP 500,日志打印堆栈 |

所有异常统一通过 `core.config.GlobalExceptionHandler`(`@RestControllerAdvice`)转为 `Result` 返回。

### 7.2 前端(F-06)

```
axios 响应拦截
   │
   ├── 网络错误 / HTTP 5xx ──→ toast("网络异常,请稍后重试") + reject
   │
   ├── HTTP 4xx(响应体仍是 Result)──→ toast(res.message) + reject(携带 code)
   │
   └── HTTP 200
         ├── code === 0  → resolve(res.data)
         └── code !== 0  → toast(res.message) + reject(携带 code)
```

> ⚠️ 组件只处理**业务回调**(成功跳转 / 失败留在原页),不负责**展示错误文案**。
