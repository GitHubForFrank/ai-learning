## Q10.1 什么是 Multi-Agent 系统？与单 Agent 相比有哪些优势和挑战？

**参考答案**：
**Multi-Agent 系统**是由多个 AI Agent 协作完成复杂任务的架构模式。

| 维度 | 单 Agent | Multi-Agent |
|------|---------|------------|
| 任务处理 | 串行、单一视角 | 并行、多视角互补 |
| 可靠性 | 单点故障 | 交叉验证、鲁棒性强 |
| 复杂度 | 简单可控 | 通信开销大、协调复杂 |
| 适合场景 | 明确单一任务 | 复杂多步骤、需多方视角 |

**核心挑战**：
1. **通信协议设计**：Agent 间如何传递信息（广播/Hub-Spoke/点对点/A2A 协议）
2. **任务分配**：谁做什么，何时切换
3. **冲突解决**：多个 Agent 给出矛盾结论时如何裁决
4. **幻觉放大**：一个 Agent 的幻觉可能被下游 Agent 放大

**难度**：⭐⭐⭐ | **考察点**：Multi-Agent 理解

## Q10.2 Multi-Agent 常见的协作模式有哪些？

**参考答案**：

### 主从协作（Manager-Worker）
中心 Manager 拆解任务、分配给 Worker、汇总结果。适合结构化任务。

### 辩论博弈模式
多个 Agent 对同一问题给出方案 → 互相质疑 → 综合裁决。适合高风险决策场景。

### 层级委派模式
上层 Agent 做战略决策 → 中层拆解 → 底层执行。适合大型复杂项目。

### 点对点协商模式
无中心调度，Agent 之间直接通信协商。适合开放性问题。

**选择原则**：
- 任务边界清晰 → Manager-Worker
- 需要多视角验证 → 辩论模式
- 超大复杂任务 → 层级委派

**难度**：⭐⭐⭐ | **考察点**：协作模式设计

## Q10.3 如何设计 Agent 的记忆系统？（三层记忆架构）

**参考答案**：

| 记忆层级 | 存储位置 | 生命周期 | 典型实现 |
|---------|---------|---------|---------|
| 工作记忆 | 上下文窗口 | 当前会话 | LLM Context |
| 短期记忆 | 会话摘要/滑动窗口 | 多次对话 | 上下文压缩摘要 |
| 长期记忆 | 向量数据库/文件系统 | 永久 | ChromaDB/FAISS + CLAUDE.md |

**工作记忆**：当前对话的完整上下文，受 Token 限制
**短期记忆**：会话间隔摘要 + 关键信息保留，通过滑动窗口压缩机制
**长期记忆**：项目知识、用户偏好、经验积累，写入文件或向量库持久化

**短期记忆转长期记忆的判断策略**：
- 被反复引用的信息 → 升级为长期记忆
- 与项目核心相关的决策 → 写入 CLAUDE.md
- 临时性的过程信息 → 保持在短期，过期丢弃

**难度**：⭐⭐⭐ | **考察点**：记忆系统设计

## Q10.4 上下文管理与压缩机制是怎样的？

**参考答案**：

**核心问题**：LLM 上下文窗口有限，长对话需要压缩。

**常见策略**：
1. **滑动窗口**：保留最近 N 轮对话，旧内容丢弃或摘要
2. **分层摘要**：对历史对话做逐层摘要（对话级 → 任务级 → 会话级）
3. **选择性保留**：按重要性评分保留关键信息（代码变更、决策、错误信息）
4. **渐进式披露**：Skill 放外部注册表，按需加载，不占用上下文

**触发压缩的时机**：
- Token 使用达阈值（如 80%）
- 上下文压缩事件触发
- 手动触发（/compact）

**为什么 CLAUDE.md 不等于上下文**：CLAUDE.md 是持久记忆文件，始终注入；上下文是当前会话的临时状态。

**难度**：⭐⭐⭐ | **考察点**：上下文工程

## Q10.5 Agent 异常处理的三层机制是什么？

**参考答案**：

```
第一层：工具层硬隔离
  - 每个工具调用独立 try-catch
  - 错误不跨工具传播
  - 工具超时自动终止

第二层：推理层熔断
  - 连续 N 次工具调用失败 → 提示 Agent 更换策略
  - 循环检测：相同参数调用同一工具超过 3 次 → 中断并提示

第三层：规划层自修正
  - 任务无法推进时重新规划
  - 回溯到上一检查点
  - 降级策略：主方案失败则执行方案 B
```

**关键约束**：
- Max Iteration 限制防止死循环
- 每个循环必须有"步骤推进"的验证，否则视为漂移
- 错误信息必须回传给 Observation → Reflection → 重试

**难度**：⭐⭐⭐ | **考察点**：工程稳健性

## Q10.6 什么是渐进式披露（Progressive Disclosure）？

**参考答案**：
渐进式披露是指**不把所有 Skill/Tool 的定义全部注入上下文，而是按需加载**的策略。

**实现方式**：
- Skill 注册表存外部，只注入 description 摘要
- 当 Agent 匹配到某 Skill 时，才加载其完整指令
- Tool 数量过多时，用语义检索只拉回相关 Top-K 个 Tool

**为什么需要**：
- 100+ Tool 全部注入 → 大量消耗上下文
- 无关 Tool 的 description 会干扰 Agent 的判断
- Token 预算有限，每节省一个字节都可能提升性能

**难度**：⭐⭐ | **考察点**：上下文优化

## Q10.7 工具调用幻觉如何治理？

**参考答案**：
**工具调用幻觉**：Agent 错误地判断该调用哪个工具，或传入错误的参数。

**治理策略**：
1. **Schema 校验**：工具必须有严格的 inputSchema（类型、约束、必填项）
2. **Few-Shot 示例**：在 Tool description 中提供正确的调用示例
3. **负样本注入**：在 description 中明确说明"不要用于 X 场景"
4. **错误回传**：工具调用失败时，详细的错误信息回传给 Agent，让它自我纠正
5. **调用前确认**：高风险操作先返回"即将做什么，请确认"

**关键原则**：错误信息不要只说"失败了"，而要说"哪个参数不符合什么规则，建议怎么修正"。

**难度**：⭐⭐⭐ | **考察点**：幻觉防控

## Q10.8 如何将 Claude Code 集成到 CI/CD 流水线？

**参考答案**：

**核心方式**：无头模式（Headless Mode）`claude -p "prompt"`

**典型场景**：
1. **自动化 Code Review**：PR 创建时自动运行 review Skill
2. **提交信息检查**：检查 commit message 是否符合规范
3. **文档一致性检查**：验证 README 与代码是否同步
4. **安全扫描**：依赖检查、密钥泄露检查

**安全注意事项**：
- API Key 通过 CI Secret 注入，不写配置文件
- 限制 CI 模式下可用的工具（禁止 Bash 写操作等）
- 设置超时和 token 预算上限

**难度**：⭐⭐ | **考察点**：CI/CD 集成

## Q10.9 Sandbox 选择和安全规范有哪些考虑？

**参考答案**：

**Sandbox 的作用**：限制 Agent 的操作范围，防止误操作影响宿主机。

**常见 Sandbox 方案**：
| 方案 | 隔离级别 | 适用场景 |
|------|---------|---------|
| Docker 容器 | 进程级隔离 | 通用开发/测试 |
| 虚拟机 | 完全隔离 | 高风险操作 |
| 工作树（Worktree） | 文件级隔离 | Git 操作隔离 |

**安全规范**：
1. **权限最小化**：Skill 的 `allowed-tools` 只给必需的
2. **网络隔离**：敏感 Skill 不允许外网访问
3. **文件写保护**：只允许在指定目录写入
4. **操作确认**：危险操作（删除、强制推送）需用户确认
5. **审计日志**：关键操作记录到日志

**难度**：⭐⭐ | **考察点**：安全意识
