## Q6.1 常驻服务（Always-On）vs 按需调用（On-Demand）如何选择？

**参考答案**：

| 维度 | 常驻服务 | 按需调用 |
|------|---------|---------|
| 响应延迟 | 低（毫秒级） | 高（冷启动秒级到分钟级） |
| 资源占用 | 持续占用 | 不调用时零占用 |
| 成本 | 较高（7×24） | 较低（按量计费） |
| 适合场景 | 高频调用、低延迟 | 低频调用、耗时任务 |
| 适合技术 | Spring Boot, FastAPI, Express | Lambda, Cloud Functions |

**决策建议**：
- **选择常驻服务**：调用频率高（>10 次/分钟）、有严格低延迟要求（<500ms）、需要维护内存缓存（如加载 NLP 模型）
- **选择按需调用**：调用频率极低、任务耗时长、希望降低成本、每次调用完全独立

**推荐默认方案**：大多数 Agent Skill 建议从**常驻服务**起步——Agent 调用 Skill 的场景通常对响应时间敏感，而且常驻服务的开发和调试体验更好。

**难度**：⭐⭐⭐ | **考察点**：部署架构决策

## Q6.2 开发 Agent Skill 时如何选择编程语言？Python / Java / Node.js / Go 各有什么优劣？

**参考答案**：

| 维度 | Python | Java | Node.js | Go |
|------|--------|------|---------|-----|
| AI/ML 生态 | ★★★★★ | ★★☆☆☆ | ★★★☆☆ | ★★☆☆☆ |
| 启动速度 | ★★★☆☆ | ★★☆☆☆ | ★★★★☆ | ★★★★★ |
| 内存占用 | ★★★☆☆ | ★★☆☆☆ | ★★★☆☆ | ★★★★★ |
| 并发能力 | ★★★☆☆ | ★★★★★ | ★★★★☆ | ★★★★★ |
| 开发效率 | ★★★★★ | ★★★☆☆ | ★★★★☆ | ★★★☆☆ |
| 类型安全 | ★★★☆☆ | ★★★★★ | ★★☆☆☆ | ★★★★★ |

**决策树**：
```
Skill 需要直接调用 LLM / 加载模型 / 使用向量库？
  ├─ 是 → Python（生态优势压倒一切）
  └─ 否 → 看团队技术栈
           ├─ 团队主力是 Java → Java（Spring Boot）
           ├─ 团队主力是 JS/TS → Node.js + TypeScript
           └─ 对资源占用极度敏感 → Go
```

**推荐默认组合**：
- AI 相关 Skill：**Python + FastAPI**（SDK 最全，社区最活跃）
- 纯业务逻辑/系统集成 Skill：**Java + Spring Boot**（适合已有 Java 体系团队）

**避免的误区**：
- 不要因为"Java 性能好"就用 Java 写 AI Skill——调用 LLM API 的瓶颈在网络 IO 和模型推理，不在语言
- 不要因为"Python 慢"就否定 Python——Skill 响应时间 99% 取决于外部服务延迟

**难度**：⭐⭐⭐ | **考察点**：技术选型决策

## Q6.3 源代码和 Skill 定义放在同一个仓库还是分开？（单仓库 vs 双仓库）

**参考答案**：

### 单仓库（推荐常驻服务）
```
weather-skill/
├── skill.json          ← Skill 元数据
├── README.md
├── src/                ← 服务源代码
└── Dockerfile
```
**优点**：版本天然绑定、发布流程简单、PR 上下文完整
**适用**：Skill 由单个团队独立维护

### 双仓库
```
weather-skill-spec/     ← 仓库 1：描述（轻量）
weather-skill-service/  ← 仓库 2：实现（代码密集）
```
**优点**：描述可先行稳定、实现仓库可设置更严格权限
**缺点**：版本同步是最大痛点——需跨仓库 CI/CD 协调，容易漂移

**推荐**：
| 部署模式 | 推荐 | 理由 |
|---------|------|------|
| 常驻服务，单团队维护 | 单仓库 | 版本绑定，发布简单 |
| 常驻服务，接口与实现分属不同团队 | 双仓库 | 职责边界清晰 |
| 按需调用 | 单仓库 | Skill 本身轻量，无拆分必要 |

**难度**：⭐⭐⭐ | **考察点**：仓库架构决策

## Q6.4 HTTP REST vs MCP vs Function Calling 如何选择？

**参考答案**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| HTTP REST | 简单、通用、生态完善 | 需自己定义接口规范 | 多框架共用、内部系统 |
| MCP | 标准化、原生兼容各主流框架 | 生态相对新 | Claude/主流 Agent 框架 |
| Function Calling | 兼容 OpenAI 生态 | 非通用标准 | OpenAI/Azure OpenAI 生态 |

**最佳实践**：设计 Skill 时，业务逻辑与协议层解耦。核心逻辑写在 Service 层，Controller 层可以同时暴露 REST 和 MCP 接口。

**难度**：⭐⭐⭐ | **考察点**：协议选型

## Q6.5 如何决定 Skill 的拆分粒度？单一 Skill vs 聚合 Skill？

**参考答案**：

**判断标准**：
```
1. 这两个功能是否共享同一批底层依赖/配置？
   → 是：可以放一起；否：考虑拆分

2. 这两个功能是否有不同的扩展策略？
   → 是（如 CPU 密集 vs IO 密集）：应该拆分；否：可以放一起

3. 这两个功能的使用场景是否高度相关？
   → 是（如"查询天气"和"天气预警"）：放在一起
   → 否（如"发送邮件"和"查询天气"）：拆分
```

**推荐边界划分**：
- 按**业务域**拆分，而非按技术层拆
- 一个 Skill = 一个业务域的完整能力集合
- 示例：`weather-skill`（含查询、预报、历史）——不是拆成三个独立 Skill

**粒度问题的权衡**：
- 过细：编排复杂、网络往返多、维护成本分散
- 过粗：臃肿难维护、部分更新需全量部署、资源需求不均

**难度**：⭐⭐ | **考察点**：粒度设计

## Q6.6 Skill 服务应该是有状态还是无状态？如何管理状态？

**参考答案**：

**推荐：设计为无状态（Stateless）**
- 可以水平扩展（多实例部署）
- 实例重启不影响业务连续性
- 更容易进行灰度发布和滚动更新

**如果需要状态，应外置存储**：

| 状态类型 | 推荐存储方案 |
|---------|------------|
| 会话上下文 | Redis（TTL 管理） |
| 任务执行状态 | Redis 或数据库 |
| 用户偏好/配置 | 数据库 |
| 短期缓存 | Redis 或内存缓存（允许丢失） |
| 文件/附件 | 对象存储（S3/MinIO） |

**例外**：模型文件、词向量等只读大型资源，在进程启动时加载到内存是合理的。

**难度**：⭐⭐ | **考察点**：状态管理设计

## Q6.7 调用链的超时和重试如何设计？

**参考答案**：

**超时设置原则**：每一层超时 < 上一层超时
```
Agent 平台超时（60s）
  └─ Skill 服务处理超时（50s）
       └─ 外部服务调用超时（30s）
```

**重试策略**：
- **可以重试**：网络抖动导致的超时（5xx）、外部服务临时不可用
- **不应重试**：参数校验失败（4xx）、幂等性无法保证的写操作（如"发送邮件"、"扣款"）

**实现示例**（Python）：
```python
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=10),
    retry=retry_if_exception_type(httpx.TimeoutException),
)
async def call_with_retry(self, input_text: str) -> str:
    return await self._client.process(input_text)
```

**难度**：⭐⭐ | **考察点**：容错设计

## Q6.8 本地工具函数 vs 独立 Skill 服务——何时需要独立部署？

**参考答案**：

**保持为本地工具函数，当**：
- 逻辑简单（不超过 100 行代码）
- 不需要独立的资源配置（无数据库、无重型依赖）
- 功能与 Agent 高度耦合，不可能被其他 Agent 复用

**抽成独立 Skill 服务，当**：
- 功能需被多个 Agent 复用
- 功能有独立的外部依赖（数据库、外部 API）
- 功能本身有独立的扩展和资源需求
- 涉及复杂业务逻辑，需独立测试和版本管理

**判断流程**：
```
这个功能会被多个 Agent 用到吗？
  ├─ 是 → 独立 Skill 服务
  └─ 否 → 有外部依赖或需独立扩展吗？
           ├─ 是 → 独立 Skill 服务
           └─ 否 → 本地工具函数
```

**难度**：⭐⭐ | **考察点**：服务边界判断
