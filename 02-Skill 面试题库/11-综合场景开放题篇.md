## Q11.1 设计题：为"代码审查"设计一个完整的 Agent Skill

**题目**：
请为团队设计一个代码审查 Skill，要求覆盖：
- Skill 的定义文件（frontmatter + 执行指令）
- 触发方式
- 审查维度和输出格式
- 异常场景处理
- 与 MCP 工具的协同方案

**参考答案要点**：

**Skill 定义**：
```yaml
---
name: review
description: |
  对代码进行 Code Review，检查质量、安全性和性能问题。
  触发条件：用户说"审查"、"review"、"检查代码"并给出代码或文件路径。
  不触发：纯格式化需求（/format）、性能专项（/perf-analyze）。
allowed-tools: [Read, Grep, Glob]
---

审查模式：$ARGUMENTS（默认 standard，可选 strict/security）

审查维度：
1. 正确性：边界条件、空值处理、并发安全
2. 可维护性：单一职责、命名清晰、嵌套层级
3. 安全性：SQL注入、XSS、敏感信息泄露、权限校验
4. 性能：N+1查询、内存泄漏、缓存使用

输出格式：
| 级别 | 位置 | 问题描述 | 改进建议 |
|------|------|---------|---------|

结论：[通过 / 需修改后通过 / 不通过]
```

**与 MCP 协同**：可集成 `mcp__github__pull_request` 获取 PR 变更，自动定位变更文件。

**异常处理**：文件不存在 → 提示用户确认路径；读取超时 → 分批审查；空文件 → 跳过。

**难度**：⭐⭐⭐⭐ | **考察点**：综合设计能力

## Q11.2 设计题：设计一个跨平台的 Skill 迁移方案

**题目**：
你有一套 Claude Code 平台上运行的 Skill（commit、review、api-doc），现在需要迁移到 Dify 平台供非开发团队使用。请设计迁移方案。

**参考答案要点**：

**Step 1：分析差异**
| 维度 | Claude Code | Dify |
|------|------------|------|
| Skill 形态 | Markdown 文件 | 可视化 + YAML |
| 工具调用 | Bash/Read/Write | 自定义 Tool（Python 函数） |
| 参数传递 | $ARGUMENTS | 变量插值 |
| 触发方式 | 斜杠命令 | 对话/API |

**Step 2：核心迁移**
1. 提取核心 Prompt 指令（复用）
2. 将 Bash 调用改写为 Dify Tool（Python 函数）
3. 参数从 `$ARGUMENTS` 改为 Dify 的结构化 parameters
4. 错误处理适配 Dify Tool 的响应格式

**Step 3：架构调整**
```
Claude Code 方式：           Dify 方式：
用户 /commit ──→ Skill ──→   用户对话 ──→ Workflow ──→ LLM 节点 ──→ Tool 节点
bash git diff           （可视化编排，含条件分支、循环）
```

**Step 4：验证**
- 用同一批测试场景在 Dify 重跑，对比输出
- 检查是否有行为差异（工具能力、格式、异常处理）

**难度**：⭐⭐⭐⭐ | **考察点**：跨平台架构能力

## Q11.3 设计题：从零设计一个"天气查询"Skill 服务

**题目**：
设计一个天气查询 Skill，从需求分析到部署上线。需要考虑：
- 后端技术栈选型及理由
- 部署模式决策
- 系统架构（分层）
- 协议选择
- 健康检查和监控

**参考答案要点**：

**技术选型**：
- Python + FastAPI（AI 场景首选，httpx 异步调外部天气 API）
- 部署模式：常驻服务（天气查询对响应时间敏感）

**系统架构**：
```
api/controller/WeatherController  → 接收请求、参数校验
service/WeatherService            → 业务逻辑：调外部 API、数据格式化
infrastructure/WeatherApiClient   → 封装外部天气 API（超时、重试、缓存）
schemas/request & response        → Pydantic 模型定义
core/config & exceptions          → 配置管理、统一异常处理
```

**协议**：HTTP REST + MCP 双接口（核心逻辑解耦，Controller 层暴露两套）

**健康检查**：`GET /api/v1/health` → `{"status": "UP", "service": "weather-skill"}`

**监控**：
- Metrics：请求量、响应时间分位数（P50/P95/P99）、外部 API 错误率
- 结构化日志：trace_id + request_id + 耗时

**超时设计**：Agent 60s → Skill 50s → 外部 API 30s

**难度**：⭐⭐⭐⭐ | **考察点**：全链路设计能力

## Q11.4 追问：你的 Agent 调了三个工具就死循环了，异常处理在哪写的？

**题目**：
面试官追问：你设计的 Agent 在调用工具时陷入死循环——不断用相同参数调同一个工具。你的异常处理机制在哪里？怎么防止？

**参考答案要点**：

**三层防护机制**：

**第一层：工具调用层**
```python
# 循环检测：记录每次工具调用的 (tool_name, params_hash)
_call_history = {}
MAX_SAME_CALL = 3

def check_loop(tool_name, params):
    key = f"{tool_name}:{hash(str(params))}"
    _call_history[key] = _call_history.get(key, 0) + 1
    if _call_history[key] > MAX_SAME_CALL:
        raise LoopDetectedError(f"检测到循环调用: {tool_name}")
```

**第二层：推理层熔断**
- 连续 N 次工具调用返回错误 → 中断并提示 Agent 更换策略
- 总迭代次数 > MAX_ITERATIONS → 强制终止并输出当前结果

**第三层：规划层自修正**
- 任务无法推进时 → 触发重规划（退回到上一个检查点）
- 降级策略：Plan A 失败 → 尝试 Plan B（更简单但可能不完美的方案）

**回答关键**：不只说"加了 try-catch"，要说清楚每一层做什么、怎么检测、怎么恢复。

**难度**：⭐⭐⭐⭐ | **考察点**：工程细节追问

## Q11.5 场景题：100+ MCP Tool 的管理问题

**题目**：
你的项目中集成了 100+ 个 MCP Tool，上下文窗口快爆了，怎么解决？

**参考答案要点**：

1. **语义路由（Tool Retrieval）**：将所有 Tool 的 description 做向量化 → Agent 需要时做语义检索 → 只拉回 Top-K（如 5-10 个）相关 Tool 的定义，而不是全部 100+

2. **工具分层**：高频 Tool 常驻上下文，低频 Tool 按需检索

3. **渐进式披露**：先只给 Tool 名称 + 一句话描述，Agent 明确需要时才加载完整 inputSchema

4. **Skill 封装**：将相关 Tool 组合成 Skill（如 `db-analyze` Skill 封装 3 个数据库 Tool），只暴露 Skill 入口，减少 Tool 数量

5. **按项目管理**：用 `disabledMcpServers` 在不需要的项目中禁用特定 MCP Server

**难度**：⭐⭐⭐ | **考察点**：规模化问题解决

## Q11.6 场景题：如何处理团队中 AI 工具使用的规范化？

**题目**：
你的团队有 20 个开发者都在用 Claude Code，但质量参差不齐——有人提交信息不规范，有人从不做 code review。如何用 Skill 系统规范团队行为？

**参考答案要点**：

**三层治理体系**：

**第一层：CLAUDE.md 全局约束**
```markdown
# 团队所有成员必须遵守
- 所有提交必须符合 Conventional Commits
- 每次提交前自动运行 review Skill
- 禁止直接提交到 main 分支
```

**第二层：项目 Skill 统一管理**
```
.claude/skills/
├── commit.md      ← 强制使用 /commit，不允许手动 git commit -m
├── review.md      ← PR 前必须 /review
└── test-gen.md    ← 新功能必须带测试
```
纳入 Git，团队共享，所有人拉取即生效。

**第三层：Hook 自动化门禁**
```jsonc
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "python .claude/hooks/check_dangerous_cmd.py"
      }]
    }],
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "python .claude/hooks/validate_commit_msg.py"
      }]
    }]
  }
}
```

**效果**：规范从"人记住"变成"工具强制执行"。

**难度**：⭐⭐⭐ | **考察点**：团队治理方案

## Q11.7 面试官反问：你如何看待 Skill 和传统微服务的区别？

**参考答案要点**：

| 维度 | Agent Skill | 传统微服务 |
|------|------------|----------|
| 调用方 | AI Agent（LLM） | 其他微服务或前端 |
| 接口定义 | natural language description + JSON Schema | 严格的 API 契约（OpenAPI/gRPC） |
| 容错方式 | Agent 可根据错误描述自我纠正 | 调用方按错误码处理 |
| 发现机制 | description 语义匹配 | 服务注册中心（Consul/Nacos） |
| 版本管理 | URL 版本 + 兼容期 | API 版本 + 灰度发布 |
| 核心差异 | Skill 面向 LLM 的"阅读理解" | 微服务面向程序的"精确调用" |

**核心洞察**：Skill 的 description 就像给 LLM 的"API 文档"——必须让模型在语义层面理解"什么时候调用、怎么传参、返回什么"，而不是精确到字段级的二进制协议。

**难度**：⭐⭐⭐⭐ | **考察点**：概念深度和类比思维
