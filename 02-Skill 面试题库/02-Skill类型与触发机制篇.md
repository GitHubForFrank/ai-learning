# Skill 类型与触发机制篇 · 面试题库

> 涵盖 Skill 的分类体系、触发方式、参数传递、作用域与 Hook 机制

## Q2.1 Skill 有哪几种分类方式？各自包含什么类型？

**参考答案**：

### 按触发方式分类

| 类型 | 触发者 | 特点 | 典型形式 |
|------|--------|------|---------|
| 用户可调用 Skill | 用户显式触发 | 斜杠命令或界面按钮 | `/commit`、`/review` |
| 代理自调用 Skill | Agent 语义判断自动调用 | 降低学习成本 | "帮我提交代码" → 自动匹配 commit Skill |
| 事件驱动 Skill | 系统事件触发 | 与 Hook 机制结合 | PreToolUse 安全检查、PostToolUse 格式化 |

### 按功能形态分类

| 类型 | 能力来源 | 适合场景 |
|------|---------|---------|
| 纯指令型 Skill | 仅 Prompt 指令 | 代码审查、写作指导等不依赖外部工具的任务 |
| 工具增强型 Skill | Prompt + 外部工具 | 需要读写文件、调 API 的场景 |
| 参数化 Skill | 接收运行时参数 | 需要动态目标对象（如生成特定文件的测试） |
| 组合 Skill | 编排多个子任务 | 复杂工作流（设计→开发→测试→文档） |

**难度**：⭐⭐ | **考察点**：分类体系的理解

## Q2.2 用户可调用 Skill 和代理自调用 Skill 的关键区别是什么？如何让 Agent 准确自动匹配 Skill？

**参考答案**：

| 维度 | 用户可调用 | 代理自调用 |
|------|----------|----------|
| 控制权 | 用户决定何时触发 | Agent 根据上下文决定 |
| 学习成本 | 需要记住命令 | 自然语言即可 |
| 确定性 | 行为可预期 | 可能误匹配 |

**让 Agent 准确自动匹配的关键**：`description` 字段是 Skill 的"索引"。

```yaml
# 不好的 description（无法帮助 Agent 准确匹配）
description: 代码审查工具

# 好的 description（明确触发条件、使用场景、不触发的条件）
description: |
  当用户需要对代码进行 Code Review、质量检查、安全审查时使用。
  适用于：完成编码后提交前的质量把关、PR 审查。
  不适用于：纯格式化需求（使用 /format）、性能专项分析（使用 /perf-analyze）。
```

**核心原则**：description 应描述**何时触发、何时不触发**，而非仅描述功能。

**难度**：⭐⭐⭐ | **考察点**：触发机制设计能力

## Q2.3 事件驱动 Skill 的 Hook 机制是怎样的？Claude Code 有哪些 Hook 事件？

**参考答案**：
Hook 机制允许在特定事件发生时自动触发 Skill 或脚本。

**Claude Code 可用 Hook 事件**：

| 事件 | 触发时机 | 典型用途 |
|------|---------|---------|
| PreToolUse | 工具调用前 | 安全检查、确认危险操作 |
| PostToolUse | 工具调用后 | 格式化代码、更新日志 |
| Notification | Agent 发送通知时 | 桌面通知、消息推送 |
| Stop | 会话结束时 | 生成摘要、清理临时文件 |
| SubagentStop | 子 Agent 结束时 | 汇总子任务结果 |

**配置示例**：
```jsonc
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "echo '即将执行 Bash 命令'"
        }]
      }
    ]
  }
}
```

**难度**：⭐⭐⭐ | **考察点**：Hook 机制的理解

## Q2.4 参数传递的机制有哪些？$ARGUMENTS 是如何工作的？

**参考答案**：
参数传递主要有两种机制：

### 命令行参数（$ARGUMENTS）
斜杠命令后的所有内容作为 `$ARGUMENTS` 传入。

```markdown
# Skill 定义
---
name: explain
description: 详细解释指定的代码概念
---
请详细解释以下内容：**$ARGUMENTS**
```

```
用户输入：/explain Promise.all
实际执行：请详细解释以下内容：Promise.all
```

### 上下文变量（Claude Code 专属）
| 变量 | 说明 |
|------|------|
| $CLAUDE_FILE_PATHS | 当前相关文件路径 |
| $CLAUDE_TOOL_NAME | 当前触发的工具名 |
| $CLAUDE_TOOL_INPUT | 工具调用的输入参数（JSON） |
| $CLAUDE_TOOL_RESPONSE | 工具调用的输出结果（JSON） |

**建议**：Skill 中应处理参数为空的情况，可用条件判断给出提示。

**难度**：⭐⭐ | **考察点**：参数机制

## Q2.5 Skill 的作用域是如何分级的？冲突时如何处理？

**参考答案**：

| 作用域 | 路径 | 适用场景 |
|--------|------|---------|
| 全局 | `~/.claude/skills/` | 个人通用 Skill，所有项目可用 |
| 项目 | `.claude/skills/` | 项目专属 Skill，仅在该项目中可用 |
| 内联 | `CLAUDE.md` 中直接定义 | 简单的一次性 Skill |

**优先级**：项目 Skill > 全局 Skill（项目级覆盖全局同名 Skill）

**版本管理建议**：项目 Skill（`.claude/skills/`）应纳入 Git 版本控制；个人本地状态（`settings.local.json`、会话缓存等）忽略。

**难度**：⭐⭐ | **考察点**：作用域管理

## Q2.6 如何选择合适的 Skill 类型？（决策树）

**参考答案**：
```
你的需求是什么？
│
├── 固定操作，每次触发相同流程
│   └── 用户可调用 Skill（斜杠命令）
│
├── 根据意图自动匹配，降低用户学习成本
│   └── 代理自调用 Skill（优化 description 字段）
│
├── 需要在特定操作前/后自动执行
│   └── 事件驱动 Skill（Hook 配置）
│
├── 需要调用外部 API 或读写文件
│   └── 工具增强型 Skill（配置 allowed-tools）
│
└── 多步骤复杂工作流
    └── 组合 Skill（分步骤编排）
```

**难度**：⭐⭐ | **考察点**：决策能力
