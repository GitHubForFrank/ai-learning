## Q7.1 Skill Prompt 设计的六大核心原则是什么？

**参考答案**：

### 原则一：角色先行（Role First）
在 Prompt 开头明确 Agent 的角色定位，建立执行上下文。

```markdown
# 不好的写法
分析这段代码的性能问题。

# 好的写法
你是一位专注于 JVM 性能优化的 Java 高级工程师，
拥有10年以上的系统调优经验。请分析以下代码的性能问题。
```

**角色声明要素**：专业领域、经验层次、关注视角。

### 原则二：指令原子化（Atomic Instructions）
将复杂任务拆解为清晰的原子步骤，避免歧义。

### 原则三：约束输出格式（Output Constraints）
明确定义输出的结构、格式和风格，确保结果可预期。

### 原则四：边界声明（Boundary Declaration）
明确告知 Agent 哪些事情不做，防止过度执行。

```markdown
✅ 允许：优化算法复杂度、减少内存分配
❌ 禁止：修改函数签名、引入新外部依赖、改变业务逻辑
```

### 原则五：提供示例（Few-Shot Examples）
对于格式复杂或理解难度高的任务，提供具体的输入/输出示例。

### 原则六：描述字段的战略重要性
`description` 字段是 Skill 的"索引"，直接影响 Agent 的自动匹配能力。

**难度**：⭐⭐⭐ | **考察点**：Prompt 设计方法论

## Q7.2 列举常见的 Prompt 反模式。

**参考答案**：

| 反模式 | 问题 | 修正 |
|--------|------|------|
| 指令模糊性 | "帮我写好一点" | 明确目标受众、语言风格、字数限制 |
| 过度宽泛的范围 | "全面分析这个项目" | 明确关注点：检查依赖安全性/性能/架构 |
| 缺乏错误处理指导 | "读取配置文件并解析" | 说明文件不存在怎么办、格式错误如何处理 |
| 角色和任务不匹配 | "你是初级程序员，设计高并发架构" | 角色定位与任务要求必须对齐 |
| 缺乏禁止声明 | 导致执行了额外操作 | 添加"禁止做以下操作"条款 |

**难度**：⭐⭐ | **考察点**：Prompt 质量意识

## Q7.3 如何编写高质量的 Skill description 字段？

**参考答案**：

`description` 字段是 Skill 的"索引"，直接影响 Agent 的自动匹配能力。

```yaml
# 弱 description（无法帮助 Agent 准确匹配意图）
description: 代码审查工具

# 强 description（明确触发条件、使用场景、能力边界）
description: |
  当用户需要对代码进行 Code Review、质量检查、安全审查时使用。
  适用于：完成编码后提交前的质量把关、PR 审查、重构前的现状评估。
  支持语言：Java、Python、TypeScript、Go。
  会输出：问题清单（按严重程度分级）+ 改进建议 + 修复代码示例。
  不适用于：纯格式化需求（使用 /format）、性能专项分析（使用 /perf-analyze）。
```

**核心要点**：
- 描述**何时触发**，而非仅描述功能
- 包含使用场景、输入类型
- 明确"不适用于"的场景（反触发条件）

**难度**：⭐⭐ | **考察点**：description 编写能力

## Q7.4 如何调试和优化一个 Skill 的 Prompt？

**参考答案**：

### 黄金测试三角
对每个 Skill 进行三类测试：
1. **正常场景测试**：标准输入是否产生期望输出
2. **边界场景测试**：极端输入（空值、超大值、特殊字符）
3. **对抗场景测试**：恶意或错误输入是否被正确处理

### 问题定位速查表

| 症状 | 可能原因 | 修复方向 |
|------|---------|---------|
| 输出格式不稳定 | 格式约束不够明确 | 增加"严格按照以下格式" + 示例 |
| 遗漏关键步骤 | 步骤描述不完整 | 拆分为有序步骤列表 |
| 执行了额外操作 | 缺少禁止声明 | 添加"禁止做以下操作"条款 |
| 无法处理边界情况 | 未考虑异常路径 | 添加异常场景处理说明 |
| Agent 误解意图 | description 不准确 | 重写 description，增加场景描述 |

### 迭代优化流程
```
写初稿 → 测试执行 → 发现问题 → 定位原因 → 针对性修改 → 回归测试
    ↑                                                           │
    └───────────────────────────────────────────────────────────┘
```

**A/B 对比测试**：对关键 Skill 保存多个版本对比，测试相同场景，选择最优版本。

**难度**：⭐⭐⭐ | **考察点**：Prompt 优化方法论

## Q7.5 写出三类常用 Skill Prompt 模板——分析类、生成类、检查类。

**参考答案**：

**分析类模板**：
```markdown
你是一位[专业角色]，专注于[专业领域]。
请对以下[分析对象]进行全面分析：
**分析维度：** 1. [维度1] 2. [维度2] 3. [维度3]
**输出格式：** ## 分析摘要 / ## 详细分析 / ## 关键结论
**约束：** 基于事实分析，每个结论必须有对应证据
```

**生成类模板**：
```markdown
你是一位[专业角色]。
根据以下输入生成[目标产物]：$ARGUMENTS
**生成要求：** - [要求1] - [要求2]
**输出格式：** [具体格式定义]
**示例：** 输入：[示例] 输出：[示例]
**禁止：** - [禁止行为1] - [禁止行为2]
```

**检查类模板**：
```markdown
你是一位[专业角色]，负责[检查职责]。
对以下[检查对象]逐项评估：
| 检查项 | 标准 | 优先级 |
|--------|------|--------|
| [检查项1] | [具体标准] | P0/P1/P2 |
**输出格式：** 问题描述 + 严重程度 + 位置 + 修复建议
**通过标准：** 无 Critical 级别问题
```

**难度**：⭐⭐⭐ | **考察点**：模板化思维能力

## Q7.6 如何在 Prompt 中平衡"自由度"和"约束力"？

**参考答案**：
核心在于**精确的边界设定**：

1. **流程约束**：拆为有序步骤，确保必要动作不遗漏
2. **格式约束**：定义输出模板，减少随意发挥
3. **行为边界**：明确"允许做"和"禁止做"的清单
4. **保留自由度**：在分析和判断环节留出空间——只约束"必须考虑什么维度"，不约束"结论必须是什么"
5. **示例引导**：通过 Few-Shot 示例隐式传达期望，而非逐字规定

**反例**：约束过死 → Agent 变成填表工具，失去推理价值
**反例**：全无约束 → Agent 自由发挥，输出不稳定

**难度**：⭐⭐⭐ | **考察点**：Prompt 设计平衡感
