## Q9.1 描述一个完整的 Agent Skill 项目开发流水线。

**参考答案**：

```
⓪ 开工前缀（每次对话必带 rule）
① 需求草稿（人）   → ② 大模型扩写 PLAN → ③ 迭代打磨 PLAN
   → ④ 启动实现   → ⑤ 跑样本 + 生成 FINDINGS
   → ⑥ 按 FINDINGS 反向优化（记 PLAN → 改代码 → 补测试 → 升版本 → 同步三文档）
   → ⑦ 专项扩展（新开关 / 知识库扩充 / 质量修复 / 补 README）
   → ⑧ 交付前清单
```

**人机分工**：
| 人做的事 | 大模型做的事 |
|---------|------------|
| 想清楚"解决什么问题" | 把需求扩写成 PLAN |
| 把提示词复制给大模型 | 写代码、建目录、跑测试 |
| 提供样本、审核产出 | 生成 FINDINGS、回归修复 |
| 决定是否升 MAJOR 版本 | 执行升版、同步改 README/SKILL.md |

**难度**：⭐⭐⭐ | **考察点**：工程化开发流程

## Q9.2 Skill 核心目录和开发过程资料如何隔离？为什么？

**参考答案**：

```
skill-XX-<name>/
├── <skill-name>/        # 📦 Skill 核心目录（交付单元）—— 可被 Claude 加载
│   ├── .meta/VERSION
│   ├── SKILL.md
│   ├── README.md
│   ├── scripts/
│   └── tools/
├── development/          # 📝 开发过程资料 —— 不参与交付
│   ├── plans/PLAN_vNN.md
│   └── findings/FINDINGS_<时间戳>.md
└── tools-src/            # 🛠️ 自研 artifact 源码 —— 不参与交付
    └── <artifact-name>/
```

**硬约束**：
- 开发过程文档（plan、findings、调研笔记）只能放 `development/`
- 核心目录里不得出现 `development/`、`plans/`、`findings/` 同名子目录
- `tools/` 只放可运行的 artifact，不放源码（源码去 `tools-src/`）

**为什么**：保证交付单元干净——用户 `git clone` 后 skill 即可运行，不混入开发草稿和源码构建依赖。

**难度**：⭐⭐ | **考察点**：工程化意识

## Q9.3 什么是"三者同步"铁律？违反会导致什么？

**参考答案**：

**三者 = Plan / README.md / SKILL.md**

**铁律**：功能描述、目录结构、参数契约必须一致。改一个就要联动核查另外两个。

**违反的常见后果**：
- 用户看 README 以为有某个功能，但 SKILL.md 没写 → Skill 不生效
- PLAN 已经改了架构，README 目录树还是旧的 → 死链、误导
- SKILL.md 的 description 写了一种触发词，README 写的是另一种 → 用户打错命令

**验证方法**：交付前逐项对比——PLAN 的功能列表 vs README 的功能列表 vs SKILL.md 的工作流描述。

**难度**：⭐⭐ | **考察点**：质量意识

## Q9.4 Skill 的版本号如何管理？语义化版本的三个段位分别代表什么？

**参考答案**：

采用语义化版本 `MAJOR.MINOR.PATCH`：

| 段位 | 触发条件 | 示例 |
|------|---------|------|
| MAJOR | 输入输出契约不兼容变更、目录结构不兼容变更 | 删除了必填参数 |
| MINOR | 新增功能、扩展知识库（向后兼容） | 新增一个模式开关 |
| PATCH | Bug 修复、文案修正、性能优化 | 修复空输入崩溃 |

**硬约束**：
- 每次改动合并前，按最高一档升位
- `.meta/VERSION` 和 `tools-src/pom.xml <version>` 必须保持一致
- **不得忘记升 VERSION**

**版本号唯一事实源**：`.meta/VERSION` 文件（README 不硬编码版本号，用指针"见 `.meta/VERSION`"）

**难度**：⭐⭐ | **考察点**：版本管理

## Q9.5 Java-capability skill 的核心设计原则是什么？

**参考答案**：

**核心原则：Skill 是运行单元，不是构建单元。**

```
# 反模式（错误做法）—— 把 .java 源码塞进 Skill 交付目录
<skill-name>/
└── tools/
    └── src/main/java/...  ← ❌ 用户需要 JDK + Maven 才能跑

# 正确做法 —— 预构建 jar + 源码分离
skill-XX-<name>/
├── <skill-name>/
│   └── tools/case-converter.jar       ← ✅ 预构建 fat jar（用户只需 JRE）
└── tools-src/case-converter/          ← 源码 + 构建配置（开发者才需）
    ├── pom.xml
    └── src/main/java/...
```

**设计要点**：
1. `tools/` 只放预构建 artifact（jar / exe / wasm）
2. 源码与构建配置统一放 `tools-src/<artifact-name>/`
3. `tools-src/` 每个 artifact 一个独立构建项目（独立 pom.xml / Cargo.toml）
4. jar 版本 = skill `.meta/VERSION` 保持同步

**命名约定**：Skill 名用"**对象 + 动作**"描述功能：
- ✅ `case-converter`、`log-summarizer`
- ❌ `java-hello-world`（技术栈 + demo 词，不描述功能）

**难度**：⭐⭐⭐ | **考察点**：Java skill 架构设计

## Q9.6 FINDINGS 文件的标准结构是什么？严重度如何分级？

**参考答案**：

**FINDINGS 标准结构**：
1. **头部元信息**：测试时间、Skill 版本、样本来源、运行环境
2. **样本清单与结果矩阵**：每组样本的通过/失败结论
3. **覆盖率/量化指标矩阵**（可选）：识别率、准确率、耗时等
4. **发现列表**：每条包含——现象、触发样本、根因假设、建议修复、影响范围

**严重度分级**：
- **P1**：功能阻断 / 数据错误（必须修）
- **P2**：体验差（下个版本修）
- **P3**：小幅打磨（有空再修）

**归档规则**：
- 路径：`development/findings/FINDINGS_<YYYYMMDD_HHMMSS>.md`
- 一次测试一个文件，不覆盖历史
- 若确认为 bug → 同步回 plan 的"已知陷阱"

**难度**：⭐⭐ | **考察点**：测试与质量流程

## Q9.7 Bug 修复的标准 5 步流程是什么？

**参考答案**：

```
① 在最新 PLAN 的"已知陷阱"章节记录现象与根因
② 修复代码
③ 补充 / 更新测试用例
④ 升 PATCH 版本号（若涉及新功能升 MINOR，契约不兼容升 MAJOR）
⑤ 若修复改动了目录或契约 → 同步更新 README.md、SKILL.md
```

**关键要求**：
- 修复后必须用**同一批样本**回归验证（换样本等于作弊）
- 确认上一轮 P1/P2 条目确实消失
- 版本号按语义化规则升位

**难度**：⭐⭐ | **考察点**：Bug 修复流程

## Q9.8 交付前的检查清单包括哪些项目？

**参考答案**：

| # | 条目 | 验证方法 |
|---|------|---------|
| 1 | 最新 plan 已更新，且与代码现状一致 | `ls development/plans/ | tail -1` 查看 |
| 2 | README.md 目录结构、功能列表与实际一致 | 对比 `ls` 与 README 目录树 |
| 3 | SKILL.md 的 description、工作流与 README 一致 | 对比两文件 |
| 4 | `.meta/VERSION` 已按规则升位 | `git diff .meta/VERSION` |
| 5 | 若有 bug 修复 → plan "已知陷阱"已补充 | 打开最新 plan 检查 |
| 6 | 本轮 FINDINGS 已归档 | `ls development/findings/` |
| 7 | 旧 plan、旧 findings 未被删除或覆盖 | `git diff --name-status` |
| 8 | sources 无历史轨迹注释 | grep 检查 |

**附加检查**：
- samples/ 下保留至少 1 个能完整跑通的示例
- 开发过程文档没有污染核心目录
- 所有大输出有 token 上限/降级策略

**难度**：⭐⭐ | **考察点**：质量门禁

## Q9.10 典型开发陷阱有哪些？

**参考答案**：

| 陷阱 | 症状 | 对策 |
|------|------|------|
| 基于旧 PLAN 开工 | 改的代码与 PLAN 对不上 | 开工前强制读最新 PLAN |
| 核心目录放开发日志 | 发布后混入草稿 | 过程文档只放 development/ |
| 覆盖旧 PLAN/FINDINGS | 历史无法对比 | 文件名带版本号或时间戳 |
| 只改代码不改 PLAN/README | 三文档漂移 | 交付清单每次必跑 |
| 忘记升 .meta/VERSION | 下游不知道版本变了 | 修改收尾明确提示升位 |
| SKILL.md ≠ README.md | 触发词/参数不一致 | 专项核对 |
| 回归偷偷换样本 | 修复无法验证 | 样本集固定，每次确认 |

**难度**：⭐⭐⭐ | **考察点**：经验沉淀
