# 04 — Agent 提示词工程

---

## ★☆☆ 基础概念

### Q1: 写给子 Agent 的 prompt 和写给人的 prompt 有什么不同？

| 维度 | 写给人 | 写给子 Agent |
|------|--------|-------------|
| 背景信息 | 可以省略，人能自己问 | 必须写全，它问不了 |
| 输出格式 | 可以口述 | 必须结构化，主 Agent 会解析 |
| 边界说明 | 可以模糊 | 必须明确，否则会跑偏 |
| 示例样本 | 可选 | 强烈推荐，特别是格式复杂时 |

### Q2: 子 Agent 提示词设计的六要素是什么？

```
1. 角色定义      → 你是谁，负责什么
2. 任务描述      → 这次具体要做什么
3. 输入说明      → 给你的数据是什么格式
4. 约束条件      → 不能做什么，边界在哪
5. 输出格式      → 返回什么，格式是什么
6. 示例（可选）  → 给一个输入输出的例子
```

### Q3: 为什么子 Agent 的提示词格外关键？

子 Agent 只能看到主 Agent 传给它的那段 prompt，不知道整体任务背景。prompt 模糊会导致：
- 自己脑补意图 → 方向跑偏
- 返回格式不对 → 主代理解析失败
- 做多或做少 → 影响后续所有 Agent

写给子 Agent 的 prompt 必须**自包含**——包含完成任务所需的全部信息。

## ★★☆ 应用场景

### Q4: 子 Agent 提示词设计的四个核心原则？

| 原则 | 说明 |
|------|------|
| 单一职责 | 一个子 Agent 只做一件事 |
| 输出格式强约束 | 明确告诉它返回什么结构 |
| 边界清晰 | 明确说不做什么 |
| 上下文最小化 | 只给它完成任务必要的信息 |

### Q5: 对比一个差的和一个好的子 Agent prompt

**差的 prompt：**
```
帮我分析这段代码有没有问题
```

**好的 prompt：**
```
你是一个代码审查专家，只负责检查安全漏洞。

输入：一段 Python 函数代码
任务：识别其中的 SQL 注入、XSS、敏感信息泄露风险
约束：不要优化性能，不要修改代码，只做安全分析
输出格式：
  - 问题列表（每条包含：位置、风险类型、严重程度）
  - 如无问题，返回"无安全风险"
```

差距：好的 prompt 有**角色定义、明确任务、清晰约束、结构化输出格式**。

### Q6: Agent 提示词规范应包含哪些部分？

基于实战项目的提示词规范，每个 Agent 文件应包含：

1. **智能体概述**：一句话说明核心定位
2. **角色定义**：核心定位、专业领域、工作风格、目标导向
3. **核心职责**：具体负责什么（3-7 条）
4. **行为准则**：工作中的约束原则
5. **工作流程**：标准执行步骤
6. **输出标准**：交付物的质量标准
7. **职权边界与禁用指令**：明确不做什么、红线在哪
8. **提示词示例**：核心提示词 + 辅助提示词 + Few-shot 示例

### Q7: 什么是 Few-shot 示例？为什么对 Agent 很重要？

Few-shot 示例是在 prompt 中提供输入→期望输出的样例，让模型通过模仿学习理解任务格式和要求。

对 Agent 的重要性：
- 复杂输出格式难以用自然语言精确描述，样例最直观
- 能显著提高输出格式的符合率
- 让模型理解质量标准（"好的输出长什么样"）

### Q8: 如何设计 Agent 的"职权边界"？

职权边界（Guardrails）告诉 Agent 什么不能做：

设计原则：
1. **负面约束优先**：先说不做什么，再说做什么
2. **场景具体化**：不是泛泛的"注意安全"，而是"不直接修改生产数据库"
3. **风险分级**：区分禁止（阻断）、警告（确认后可行）、建议（可自主决定）
4. **兜底策略**：遇到不确定的情况时默认怎么做

示例："输入上下文应按相关性排序与裁剪，仅保留任务必需信息；避免向 AI 系统注入与当前任务无关的历史噪音。"

## ★★★ 架构设计

### Q9: 什么是 SDD（规范驱动开发）？和 Prompt 工程的关系？

SDD（Specification-Driven Development）是将模糊需求转化为精确的 AI 可执行规范：

- 用结构化规格文档替代口头描述
- 包含数据模型、API 契约、验收标准、Given-When-Then 场景
- 让 Agent 从"猜测需求"变为"执行规格"

一份可执行的规格至少包含：数据模型、API 契约、验收标准、行为场景、校验规则、权限矩阵。

### Q10: 如何做 Prompt 的版本管理和 A/B 测试？

1. **版本管理**：提示词模板纳入 Git，变更记录 diff、影响面与回滚策略
2. **A/B 测试**：同一场景使用两套 prompt，对比关键指标（准确率、格式符合率、token 消耗）
3. **评估体系**：离线标注数据集 + 自动评测 + 人工抽检
4. **灰度上线**：新 prompt 先在 10% 流量验证，达标后全量

### Q11: 如何设计"通用基线 + 特化"的 Agent 提示词体系？

基于实战项目的实践：

```
通用基线文件（70% 共享内容）
  ├─ 角色定位
  ├─ 核心职责
  ├─ 行为准则
  ├─ 工作流程（通用）
  ├─ 输出标准（通用）
  └─ 职权边界（通用部分）

特化文件（≤30% 差异内容）
  └─ 职权边界（语言/平台相关的具体规则）
```

例如：代码工匠（通用）定义了统一的角色、流程、输出标准，而代码工匠（Java/Python/VUE）各自补充语言特定的安全模式和工具链。
