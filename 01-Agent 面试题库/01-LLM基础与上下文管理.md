# 01 — LLM 基础与上下文管理

---

## ★☆☆ 基础概念

### Q1: 大模型的"记忆"功能是怎么实现的？

大模型本身是**无状态的**，每次调用只能看到当前输入，没有原生记忆能力。"记忆"是工程层面的技巧：把需要记住的内容在下次调用时塞进 prompt 里。

常见实现方式：
- 拼接对话历史（每次把完整历史一起发送）
- 摘要压缩（超长后先总结再替换）
- 外部存储 + 检索/RAG（历史存数据库，按需检索塞入）
- 文件持久化（关键信息写入文件，每次读入 context）

### Q2: 什么是上下文窗口（Context Window）？

上下文就是模型每次调用时能"看到"的全部内容的总长度，用 token 数量衡量。

一次调用中计入上下文的内容：
```
系统提示 + 对话历史（所有轮次） + 当前提问 + 模型回答
= 总量不能超过上下文限制
```

Token 数量参考：
- 1K token ≈ 750 个英文单词 / ~500 汉字
- 128K token ≈ 一本中等厚度的书
- 1M token ≈ 75 万英文单词，相当于整个代码仓库

### Q3: 多轮对话中，模型的回答会传递到下一轮吗？

**会。** 多轮对话的本质是每次都把完整历史拼进去：

```
第1轮发送：[问题1]
第2轮发送：[问题1] + [回答1] + [问题2]
第3轮发送：[问题1] + [回答1] + [问题2] + [回答2] + [问题3]
```

问题：对话越长，每次发送的 token 越多，超过上下文限制后平台会截断早期内容或自动压缩，导致模型"忘掉"最开始的内容。

### Q4: Token 和字数的换算关系是什么？

- 英文：1 token ≈ 0.75 个单词，所以 1000 tokens ≈ 750 个英文单词
- 中文：1 token ≈ 0.5 个汉字，所以 1000 tokens ≈ 500 个汉字
- 代码：不同语言差异大，通常 1 token ≈ 3-4 个字符

实际消耗取决于具体模型的分词器（Tokenizer）。

### Q5: 什么是 Token 预算（Token Budget）？

Token 预算是指在上下文窗口内为不同用途分配 token 配额的策略。常见分配：

| 预算项 | 典型占比 |
|--------|---------|
| 系统提示词 | 10-20% |
| 对话历史 | 30-50% |
| 检索上下文/RAG | 20-30% |
| 当前提问 | 5-10% |
| 模型回答 | 10-20% |

## ★★☆ 应用场景

### Q6: 控制上下文的三个抓手是什么？

| 抓手 | 应用场景 |
|------|---------|
| 主动摘要 | 长对话快要触顶时人工触发，替换历史 |
| 外部持久化 | 关键事实写入文件/数据库，下次按需读入 |
| 分层上下文 | 全局/任务/局部分层注入，避免全量广播 |

### Q7: 对话过长导致模型"遗忘"时怎么处理？

1. **主动摘要法**：用 prompt 让模型生成对话摘要，然后开启新对话：
   ```
   请把我们本次对话的关键信息总结成一段简洁的上下文摘要，
   包括：已确认的需求、做过的决策、当前进展、待解决的问题。
   我将用这段摘要开启新对话继续工作。
   ```

2. **外部持久化法**：将关键信息写入文件，新对话时读入

3. **分层上下文法**：将信息分为全局层（项目背景）、任务层（当前任务）、局部层（执行细节），按需注入

### Q8: RAG（检索增强生成）的核心流程是什么？

```
用户提问
  → 查询理解/改写
  → 向量检索（从向量库召回相关文档片段）
  → 重排序（对召回结果精排）
  → 上下文压缩（选取 Top-K 片段）
  → 拼接 prompt（系统提示 + 检索上下文 + 用户问题）
  → LLM 生成回答
```

关键环节：文档分块策略、Embedding 模型选型、检索管道设计（粗筛→精排→压缩）。

### Q9: 向量数据库在 Agent 系统中扮演什么角色？

向量数据库是 Agent 的外部记忆载体：
- 存储知识的向量 Embedding
- 支持语义相似度检索
- 实现"按需记忆"——不是全量加载，而是按相关性召回

常用选型对比维度：HNSW vs IVF-PQ 索引、召回率、延迟（P95）、规模。

### Q10: 什么情况下用摘要压缩而非全量保留对话历史？

- 对话历史已超过上下文窗口的 50%
- 早期对话信息对当前任务不再关键
- 需要控制 token 成本

但摘要会丢失细节，需要权衡。经验法则：事实性信息（代码、决策记录）应外部持久化，推断性信息（讨论过程）可摘要压缩。

## ★★★ 架构设计

### Q11: 设计一个支持长对话的上下文管理系统

要点：
1. **分层架构**：全局上下文（项目规范，所有 Agent 可见）+ 任务上下文（当前任务详情）+ 局部上下文（执行细节，单 Agent 内部）
2. **自动摘要触发**：当对话 token 超过窗口 70% 时自动压缩历史
3. **外部持久化**：关键决策写入文件/数据库，下次会话按需加载
4. **按需检索**：历史信息存向量库，需要时语义检索而非全量加载
5. **上下文隔离**：多租户场景下确保信息隔离

### Q12: 如何评估一个 RAG 系统的质量？

离线评估指标：
- 召回率（Recall@K）：相关文档是否被检索到
- 命中率（Hit Rate）：Top-K 中是否包含正确答案
- MRR（Mean Reciprocal Rank）：正确答案的平均排名
- 答案一致性：生成答案与检索片段的语义对齐度

上线后监控：
- 日采样人工标注
- 漂移监控（数据分布变化）
- 回滚阈值：一致性低于 90% 不放量

### Q13: 混合检索（Hybrid Search）相比纯向量检索有什么优势？

混合检索 = 关键词检索（BM25）+ 向量语义检索

优势：
- 精确匹配场景（产品编号、错误码）关键词检索更准
- 语义理解场景（同义词、概括性问题）向量检索更优
- 两者结合互补，提升整体召回率

典型管道：BM25 粗筛 → 向量召回 Top-K → 重排序 → 输出。
