# 03 — 多 Agent 协作与编排

---

## ★☆☆ 基础概念

### Q1: 什么是多 Agent 协作？

多 Agent 协作是指由多个 Agent 共同完成一个复杂任务，每个 Agent 各司其职，通过信息传递进行协同。

核心原则：**信息按需传递，不是全量广播。**

类比：项目经理（主 Agent）把任务拆解后分给不同专家（子 Agent），专家完成各自部分后，项目经理汇总结果。

### Q2: 多 Agent 协作有哪几种常见模式？

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 编排者+执行者 | 主 Agent 分解任务，派发子 Agent | 有明确分工的复杂任务 |
| 串行流水线 | Agent A → Agent B → Agent C | 有严格先后依赖 |
| 并行执行 | 多个 Agent 同时工作 | 子任务互相独立 |
| 共享外部存储 | Agent 通过文件/数据库交换信息 | 跨会话或异步协作 |
| 结构化消息传递 | Agent 间传 JSON 摘要而非原始对话 | 需精确控制信息流 |

### Q3: 什么是 Plan-and-Execute 模式？

Plan-and-Execute（先规划再执行）是多 Agent 系统中最成熟的编排模式：

```
Planning   → 主 Agent 分析任务，输出任务清单
Dispatch   → 按清单创建子 Agent（串行或并行）
Execute    → 每个子 Agent 执行自己的子任务
Aggregate  → 主 Agent 汇总所有结果，输出最终答案
```

### Q4: 主 Agent 和子 Agent 的角色分工是什么？

| 层级 | 角色 | 职责 |
|------|------|------|
| 用户 | 需求方 | 下达高层任务 |
| 编排层 | 调度器 | 规划、调度、上下文管理 |
| 主 Agent | 决策层 | 决定派谁做什么 |
| 子 Agent | 执行层 | 各司其职，完成即销毁 |
| 工具 | 执行力 | 读写文件、运行命令、检索网络 |

## ★★☆ 应用场景

### Q5: 多 Agent 协作时上下文怎么处理？

每个 Agent 都是独立的 LLM 调用，各自有独立的上下文窗口。处理方法：

1. **编排者传递摘要**：主 Agent 只把必要上下文传给子 Agent，收到结果后做摘要
2. **共享外部存储**：Agent A 写入文件 → Agent B 读取需要的部分 → 处理 → 写回
3. **结构化消息传递**：Agent 之间只传结构化摘要（JSON），不传原始对话
4. **上下文分层**：全局/任务/局部三层，按角色授权

### Q6: 什么时候用串行，什么时候用并行？

| 条件 | 策略 |
|------|------|
| 任务之间有明确先后依赖 | 串行创建 |
| 任务互相独立，可并行 | 并行创建，最后统一汇总 |
| 任务数量/深度未知 | 按需创建，主 Agent 边做边派 |
| 跨会话共享结果 | 共享外部存储 |

经验法则：一眼能说清步骤的任务不用 Plan，需要想两秒的就值得 Plan。

### Q7: 没有 Plan 时容易出现什么问题？

- 中途改变方向
- 重复创建功能相似的子 Agent
- 子任务之间依赖关系混乱
- 出问题时无法定位是"哪一步"跑偏的

### Q8: 什么时候值得先规划再执行（Plan）？

**值得 Plan 的场景：**
- 任务步骤超过 3 步，且步骤之间有依赖
- 需要跨多个文件/模块的改动
- 目标比较抽象（如"重构这个模块"），需要先拆解
- 团队协作，需要对齐执行思路

**不需要 Plan 的场景：**
- 单步任务（查一个 API、改一个字符串）
- 探索性问答
- 已经很熟悉的重复操作

### Q9: 上下文分层的三级模型是什么？

| 层级 | 内容 | 谁能看到 |
|------|------|---------|
| 全局上下文 | 项目背景、规范、技术栈 | 所有 Agent |
| 任务上下文 | 当前子任务详情、输入输出契约 | 负责该任务的 Agent |
| 局部上下文 | 执行细节、中间推理 | 单个 Agent 内部 |

### Q10: 子 Agent 的生命周期是怎样的？

```
创建 → 初始化 → 执行 → 返回结果 → 销毁
```

1. **创建**：主 Agent 调用 Agent 工具，传入任务 prompt
2. **初始化**：上下文填入系统提示 + 任务 prompt
3. **执行**：子 Agent 工作，可调用工具，可再创建下级 Agent
4. **返回结果**：只返回最终输出，不返回完整执行过程
5. **销毁**：上下文完全清除，不保留任何状态

## ★★★ 架构设计

### Q11: 设计一个支持动态调度的多 Agent 编排系统

核心要素：
1. **任务分解器**：将复杂任务自动拆解为 DAG（有向无环图）
2. **依赖分析**：识别子任务间的依赖关系，决定串/并行策略
3. **动态调度**：运行时根据中间结果动态调整后续任务
4. **上下文管理**：为每个 Agent 构造最小必要上下文
5. **结果聚合器**：汇总并去重各子 Agent 结果
6. **故障恢复**：子 Agent 失败时重试或降级

### Q12: AI 编程工具中 Agent 编排的实现对比

| 工具 | 派发子 Agent 能力 | 隔离机制 |
|------|-------------------|---------|
| Claude Code | Agent 工具（subagent_type 参数） | git worktree |
| Cursor | Cascade + Background Agent | 沙箱分支 |
| Cline/Roo Code | Plan Mode + Act Mode 双模式 | 当前工作区 |
| Aider | --architect 模式（Architect + Editor） | 当前工作区 |

通用结论：能力是连续谱——从"工具内置编排"到"自研框架完全可编程"。

### Q13: 什么是编排层（Orchestration Layer）？它做什么？

编排层把"一个高层任务"翻译成"一组 Agent 的协作流水线"：

```
用户输入任务
  ↓
编排层
  ├─ 规划：调用 Plan Agent 拆解任务
  ├─ 调度：管理子 Agent 创建/销毁/并发
  ├─ 工具调用：读文件、写代码、执行命令、调外部 API
  ├─ 上下文管理：压缩、分层传递、跨阶段交接
  └─ 结果汇总：把多个子 Agent 的输出拼回给用户
```

一句话：编排层把"一个任务"翻译成"一组 Agent 的协作流水线"，具体由谁实现（CLI/IDE/框架）不影响这个模式本身。
