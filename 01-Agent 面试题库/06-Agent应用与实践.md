# 06 — Agent 应用与实践

---

## ★☆☆ 基础概念

### Q1: Agent 在企业中有哪些典型应用场景？

基于实战项目分类：

| 类别 | 典型 Agent | 职责 |
|------|-----------|------|
| 管理决策 | 全局统筹助手、决策参谋官、战略中枢 | 统筹全局、辅助决策 |
| 研发技术 | 代码工匠、架构师、代码审核、测试专家 | 软件开发全流程 |
| 运营增长 | 内容创作官、增长引擎、市场洞察者 | 内容营销与用户增长 |
| 客户支持 | 贴心服务助手、售后解忧官 | 客户咨询与售后服务 |
| 行政人事 | 人事小管家、招聘筛选专员 | 考勤、招聘、行政 |
| 财务法务 | 财务核算能手、合规审查官 | 财务核算与法务咨询 |
| 数据智能 | 数据工程师、机器学习工程师 | 数据管道与 AI 应用 |
| 安全合规 | 安全审计专家 | 安全审计与风险评估 |
| 设计创意 | UI 设计专家 | 界面设计与评审 |

### Q2: "代码工匠" Agent 的核心职责是什么？

代码工匠是专业代码实现专家，核心职责：

1. **代码实现**：按需求输出可读、可维护、可测试的代码
2. **规范遵循**：遵循目标语言的官方风格指南与项目编码规约
3. **错误处理**：对可预见失败路径显式处理，资源具备清理保证
4. **测试自检**：提交前完成单测与关键路径自测
5. **可读注释**：对非显然的"为什么"加注释，不复述代码已表达的"是什么"

### Q3: "代码审核" Agent 的五个审查维度是什么？

1. **正确性**：边界条件、错误处理、并发安全
2. **可读性**：命名、结构、模块边界
3. **规范**：对照官方风格指南与项目规约
4. **性能**：热点路径反模式、资源使用风险
5. **安全**：注入、反序列化、SSRF、越权、敏感日志

问题分级：阻断 > 严重 > 一般 > 建议。

## ★★☆ 应用场景

### Q4: 如何用 Agent 实现"测试专家"？

测试专家 Agent 的核心能力：

1. **测试策略制定**：风险驱动，高风险模块优先覆盖
2. **分层覆盖**：单元 → 接口 → 集成 → 端到端，各层职责清晰
3. **自动化测试开发**：编写可独立运行、可参数化的测试用例
4. **性能测试**：阶梯加压 + 资源基线 + 瓶颈定位
5. **安全测试**：覆盖 OWASP Top 10 + SAST/DAST 工具链
6. **缺陷管理**：把失败用例回填回归套件，形成质量防护网

### Q5: "上下文管理专家" Agent 做什么？

1. **上下文工程**：动态上下文组装，Token 预算管理
2. **向量库建设**：文档分块、Embedding、混合检索
3. **知识图谱**：实体链接、语义推理、跨源整合
4. **智能记忆**：工作/情景/语义/程序四层记忆架构
5. **RAG 优化**：粗筛→精排→压缩三阶段管道
6. **多 Agent 协调**：上下文传递协议、状态管理

### Q6: "架构师" Agent 的输出标准是什么？

1. 清晰的架构设计文档和架构图（C4 视图）
2. 详细的技术选型理由和模块划分原则
3. 各组件间的交互关系和数据流向（含超时预算）
4. 扩展策略和安全考虑说明
5. 关键路径的容量估算与性能预算

### Q7: "安全审计专家" Agent 的工作流程？

1. **范围确认与授权**：明确审计范围、时间窗口、测试手段
2. **信息收集**：资产清点
3. **技术审计**：代码/架构/配置审计
4. **渗透测试**：对已发现弱点做受控验证
5. **风险评估**：量化风险（CVSS），分级报告（P0/P1/P2/P3）
6. **报告输出与跟踪**：交付报告并跟踪修复闭环

### Q8: Agent 在工作流自动化（Coze/N8N）中扮演什么角色？

工作流制作 Agent 负责：
1. **流程设计**：拆分节点、分支与子流程，确定串/并行策略
2. **系统集成**：通过 HTTP/SDK/原生节点接入上下游系统
3. **触发与调度**：选择合适的触发器（事件/Webhook/定时/手动）
4. **凭据与安全**：敏感数据最小可见与必要脱敏
5. **可观测与运维**：执行日志、错误处理、重试与告警

### Q9: 如何用 Spec 驱动设计 Agent 实现从需求到代码？

SDD 流程：
```
业务需求
  → Spec Agent 编写规格文档（数据模型 + API 契约 + 验收标准 + GWT 场景）
  → Code Agent 按规格实现代码
  → Test Agent 按验收标准验证
  → Review Agent 审核代码与规格的一致性
```

可执行规格的要素：数据模型、API 契约、验收标准、行为场景、校验规则、权限矩阵。

## ★★★ 架构设计

### Q10: 设计一个完整的研发团队 Agent 套件

```
需求阶段：Spec Agent（规格化需求）
  ↓
设计阶段：架构师 Agent（系统设计）
  ↓
实现阶段：代码工匠 Agent（多语言）×N
  ↓
审核阶段：代码审核 Agent（多语言）×N
  ↓
测试阶段：测试专家 Agent
  ↓
部署阶段：DevOps/Jenkins Agent
  ↓
运维阶段：可观测性 Agent + 事故响应 Agent
```

每个 Agent 独立上下文，通过结构化消息传递结果。

### Q11: 如何评估一个 Agent 系统的业务价值？

1. **效率指标**：完成任务的时间 vs 人工时间
2. **质量指标**：代码缺陷率、测试覆盖率、线上事故率
3. **成本指标**：API 调用费用、人均 Agent 使用次数
4. **满意度**：开发者和使用者的 NPS/满意度评分
5. **ROI**：节省的人工成本 vs Agent 的总成本

### Q12: 多 Agent 系统中如何处理"知识传递"？

1. **结构化摘要传递**：每个 Agent 输出标准格式的结果摘要
2. **共享知识库**：利用向量库进行语义检索
3. **上下文模板**：为不同角色预定义上下文模板
4. **版本化**：知识变更可追溯、可回滚

关键原则：信息按需传递，不是全量广播。
