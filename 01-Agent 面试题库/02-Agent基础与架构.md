# 02 — Agent 基础与架构

---

## ★☆☆ 基础概念

### Q1: 什么是 AI Agent？它和普通 LLM 调用有什么区别？

AI Agent 是能够**自主感知环境、制定计划、调用工具、执行行动**的智能体。

| 维度 | 普通 LLM 调用 | Agent |
|------|-------------|-------|
| 交互模式 | 单轮问答 | 多轮推理+行动循环 |
| 工具使用 | 无 | 可调用工具（搜索、代码执行、API等） |
| 自主性 | 被动响应 | 主动规划与执行 |
| 上下文 | 单次调用 | 多步上下文管理 |
| 输出 | 文本 | 文本 + 行动结果 |

### Q2: Agent 的核心组件有哪些？

```
Agent = LLM + 规划 + 记忆 + 工具使用
```

1. **LLM（大脑）**：推理与决策核心
2. **规划模块**：任务分解、步骤排序、依赖管理
3. **记忆模块**：短期记忆（上下文窗口）、长期记忆（向量库/外部存储）
4. **工具使用**：调用外部工具扩展能力（搜索、代码执行、API调用）

### Q3: 什么是子 Agent（Subagent）？

子 Agent 是由主 Agent 按需创建的一次性工作单元：

**生命周期：**
```
创建 → 初始化（填入任务 prompt）→ 执行（可调用工具）→ 返回结果 → 销毁
```

**关键特征：**
- **无状态**：销毁后不保留任何记忆
- **上下文隔离**：不共享主 Agent 的上下文
- **结果单向传递**：只返回结果摘要，不返回执行过程
- **可嵌套**：子 Agent 内部可再创建子 Agent

### Q4: Agent 和 RAG 是什么关系？

- RAG 是 Agent 的记忆/检索子系统之一
- Agent 可能包含 RAG 组件来获取外部知识
- 但不是所有 Agent 都需要 RAG，也不是所有 RAG 系统都是 Agent
- 简单区分：RAG = 检索 + 生成，Agent = 规划 + 工具 + 执行循环

### Q5: 什么是 ReAct 模式？

ReAct = Reasoning（推理）+ Acting（行动），是 Agent 的基本推理框架：

```
思考 → 行动 → 观察 → 思考 → 行动 → 观察 → ... → 最终答案
```

模型在每一步：
1. **Thought**：分析当前状态，决定下一步
2. **Action**：调用工具或执行操作
3. **Observation**：获取工具返回结果
4. 循环直到任务完成

## ★★☆ 应用场景

### Q6: 子 Agent 为什么能控制 token 膨胀？

主 Agent 的上下文**不会因为子 Agent 的执行过程而膨胀**：

```
主 Agent 上下文
  ├─ 用户任务（保留）
  ├─ 子 Agent A 结果摘要（只保留结果，不保留过程）
  ├─ 子 Agent B 结果摘要
  └─ 最终答案
```

子 Agent 内部的读文件、递归搜索、反复试错等开销，全部被它自己的生命周期吸收，用完即焚。

### Q7: Agent 的工具调用（Function Calling）是如何实现的？

1. **工具定义**：用 JSON Schema 描述工具名称、参数、功能
2. **模型决策**：LLM 根据用户意图决定是否调用工具、调用哪个、传什么参数
3. **执行**：系统执行实际的函数调用
4. **结果返回**：将执行结果作为新的上下文注入对话
5. **继续推理**：LLM 根据结果决定下一步

### Q8: Agent 的记忆系统如何分层设计？

| 层级 | 内容 | 时效 | 存储方式 |
|------|------|------|---------|
| 工作记忆 | 当前任务的执行上下文 | 单次任务 | 上下文窗口 |
| 情景记忆 | 历史对话/任务的过程记录 | 短期 | 对话历史/摘要 |
| 语义记忆 | 知识、事实、规范 | 长期 | 向量库/知识图谱 |
| 程序记忆 | 工作流、SOP、模板 | 永久 | 文件系统/数据库 |

### Q9: Agent 如何处理不确定性？

1. **置信度评估**：对输出进行信心评分
2. **多路验证**：关键问题调用多个工具交叉验证
3. **人工介入**：低置信度时请求人工确认
4. **溯源引用**：输出附带证据来源，用户可自行验证
5. **安全兜底**：不确定时倾向于保守回答，标注"待验证"

### Q10: 描述 Agent 执行一次任务的完整流程

```
1. 接收任务
2. 理解与分解：将高层任务分解为可执行的子任务
3. 规划：确定步骤顺序与依赖关系
4. 执行循环：
   - 当前步骤需要哪些信息？（调用读文件/搜索工具）
   - 当前步骤需要哪些操作？（调用写代码/命令执行工具）
   - 结果是否符合预期？（验证 → 修正）
5. 汇总：将所有步骤结果整合为最终输出
6. 交付
```

## ★★★ 架构设计

### Q11: 设计一个企业级 Agent 平台的架构

核心层次：
```
┌─────────────────────────────────┐
│         用户/API 接入层          │
├─────────────────────────────────┤
│         编排层（Orchestrator）    │
│  规划 → 调度 → 上下文管理 → 汇总  │
├─────────────────────────────────┤
│          Agent 执行层            │
│  ┌──────┐ ┌──────┐ ┌──────┐    │
│  │Agent A│ │Agent B│ │Agent C│    │
│  └──────┘ └──────┘ └──────┘    │
├─────────────────────────────────┤
│          工具层                  │
│  文件IO │ 搜索 │ 代码执行 │ API  │
├─────────────────────────────────┤
│          基础设施层              │
│  向量库 │ 消息队列 │ 日志 │ 监控  │
└─────────────────────────────────┘
```

### Q12: Agent 的可观测性如何设计？

三大支柱：
1. **日志**：Agent 推理链路完整记录（检索片段、工具调用参数、提示模板版本、中间步骤）
2. **指标**：延迟（P50/P95/P99）、成功率、token 消耗、工具调用频次
3. **链路追踪**：为每次 Agent 调用分配 trace_id，支持跨 Agent 关联

关键 SLO 示例：端到端延迟 P99 < 5s，可用性 99.9%。

### Q13: 什么是 Agent 的"上下文预算管理"？

在有限的上下文窗口中合理分配 Token：

- 系统提示：固定开销，应精简
- 检索上下文：限制 Top-K 片段的总 token（如 ≤ 4K）
- 对话历史：设定最大保留轮数或 token 上限
- 工具结果：长结果先摘要再注入

目标是：在正确的时机为 Agent 提供正确的信息，让有限的 Token 窗口内有更多"有效信息密度"。
