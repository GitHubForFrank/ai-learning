# 三类能力原语详解：Tools / Resources / Prompts

> 定位：深入理解 MCP 的能力模型与选型

---

## 1. 为什么是三类

MCP 通过三种不同的"暴露语义"把外部能力分层：

| 语义维度 | Tools | Resources | Prompts |
|---------|-------|-----------|---------|
| **发起方** | LLM 模型 | 用户/客户端 | 用户 |
| **交互模式** | 调用（Invoke） | 读取（Read） | 渲染（Render） |
| **是否可写** | 可能 | 只读 | 只读 |
| **控制者** | model-controlled | application-controlled | user-controlled |
| **类比** | 函数 | 文件 | 模板 |

三者互补而非替代——**同一业务能力可以同时以三种形式暴露**，让不同场景选最合适的入口。

---

## 2. Tools（工具）

### 2.1 定义

Tools 是 LLM 可以**主动决定调用**的函数。Server 声明函数签名，Client 把声明转换成 Function Calling 规格传给 LLM，LLM 推理出调用时机与参数。

### 2.2 生命周期

```
Server 启动
   ├─ tools/list      ← Client 发现可用 Tool 列表
   └─ tools/call      ← LLM 决定调用时触发
         ├─ name
         ├─ arguments（符合 inputSchema）
         └─ 返回：content[] + isError?
```

### 2.3 定义结构

```jsonc
{
  "name": "search_issues",
  "description": "搜索 GitHub 仓库的 issues，支持按状态、标签、作者过滤。适用于：排查历史 bug、统计工作量、检索讨论。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "repo":   { "type": "string", "description": "格式：owner/name" },
      "state":  { "type": "string", "enum": ["open", "closed", "all"], "default": "open" },
      "labels": { "type": "array", "items": { "type": "string" } },
      "limit":  { "type": "integer", "minimum": 1, "maximum": 100, "default": 20 }
    },
    "required": ["repo"]
  }
}
```

### 2.4 返回值结构

```jsonc
{
  "content": [
    { "type": "text", "text": "共检索到 12 条..." },
    { "type": "image", "data": "base64...", "mimeType": "image/png" },
    { "type": "resource", "resource": { "uri": "issue://owner/name/42" } }
  ],
  "isError": false
}
```

支持的 content 类型：
- `text` — 纯文本/JSON 字符串
- `image` — base64 编码图片
- `resource` — 引用一个 Resource URI（让 Client 可按需读取）

### 2.5 设计要点

| 要点 | 规则 |
|------|------|
| **描述面向 LLM** | 写"什么时候用"而非只写"接口做什么" |
| **参数校验严格** | 用 JSON Schema 的 enum/minimum/pattern 约束 |
| **幂等优先** | 能只读的绝不写；写操作明确写在 description |
| **副作用提示** | 破坏性操作返回"需要确认"的文本，不直接执行 |
| **错误可恢复** | 参数错误返回 isError=true + 提示，便于 LLM 纠正重试 |

### 2.6 何时用 Tools

- 需要执行动作（查询、写入、调用第三方 API）
- LLM 需要自主决策是否调用
- 结果需要进入 LLM 上下文继续推理

## 3. Resources（资源）

### 3.1 定义

Resources 是 Server 暴露的**只读数据**，通过 URI 寻址。类似于给 AI 的"虚拟文件系统"——可被枚举、可被读取，但读不读由 Client 或用户决定。

### 3.2 URI 模型

Resource URI 采用自定义 scheme，Server 自行定义语义：

```
file:///home/user/notes/todo.md
github://owner/repo/issues/42
postgres://schema/table/row?id=123
config://app/settings.json
log://service/2026-04-21
```

### 3.3 生命周期

```
Server 启动
   ├─ resources/list       ← 枚举可用资源
   ├─ resources/read       ← 读取单个资源内容
   └─ resources/subscribe  ← 订阅资源变更（可选）
         └─ notifications/resources/updated  ← Server 推送变更
```

### 3.4 定义结构

```jsonc
{
  "uri": "postgres://prod/users/schema",
  "name": "users 表结构",
  "description": "生产库 users 表的列定义与索引",
  "mimeType": "application/json"
}
```

读取返回：

```jsonc
{
  "contents": [
    {
      "uri": "postgres://prod/users/schema",
      "mimeType": "application/json",
      "text": "{\"columns\": [...], \"indexes\": [...]}"
    }
  ]
}
```

### 3.5 Resources 与 Tools 的边界

容易混淆的一个问题：`get_user_schema` 这种 Tool，为什么不做成 Resource？

| 判断依据 | 选 Tool | 选 Resource |
|---------|---------|-------------|
| **是否有参数决策** | 有（LLM 要推理） | 无或固定（URI 即键） |
| **谁来决定触发** | LLM 自主 | 用户/应用显式 |
| **是否有副作用** | 可能有 | 绝不允许有 |
| **结果是否稳定** | 可变 | URI 稳定时内容相对稳定 |

**经验法则**：只读且能用稳定 URI 表达的数据 → Resource；需要 LLM 推理参数或有副作用 → Tool。

### 3.6 动态 Resources（Resource Templates）

支持 URI 模板，由 Client 填参：

```jsonc
{
  "uriTemplate": "github://{owner}/{repo}/issues/{number}",
  "name": "GitHub Issue",
  "description": "读取指定仓库的单个 Issue"
}
```

## 4. Prompts（提示词模板）

### 4.1 定义

Prompts 是 Server 提供的**可复用对话模板**，用户显式触发（通常通过斜杠命令或菜单），把模板渲染成消息序列后发给 LLM。

### 4.2 与本地 Skill 的区别

| 维度 | Claude Code Skill | MCP Prompts |
|------|-------------------|-------------|
| **存储位置** | 本地 `.claude/skills/` | 远程 Server |
| **更新方式** | 编辑文件重启 | Server 端随时更新 |
| **分发** | 每台机器各自拷贝 | 一次部署全员生效 |
| **鉴权** | 文件权限 | Server 可加鉴权 |

MCP Prompts 适合**集中治理的组织级模板**。

### 4.3 定义结构

```jsonc
{
  "name": "review_pr",
  "description": "按团队规范审查 PR",
  "arguments": [
    { "name": "pr_url", "description": "PR 链接", "required": true },
    { "name": "focus", "description": "审查重点：security|perf|style", "required": false }
  ]
}
```

### 4.4 渲染结果

```jsonc
{
  "description": "按团队规范审查 PR",
  "messages": [
    {
      "role": "user",
      "content": {
        "type": "text",
        "text": "请审查以下 PR：https://... \n重点关注：security\n输出格式..."
      }
    }
  ]
}
```

Prompt 可以返回**多轮消息**（user/assistant 交替），用于少样本示例。

## 5. 三类原语组合设计

一个成熟的 MCP Server 往往同时暴露三类：

**示例：GitHub MCP Server**

```
Tools（可执行动作）：
  ├─ create_issue           创建 Issue
  ├─ merge_pull_request     合并 PR
  └─ search_code            代码搜索

Resources（只读数据）：
  ├─ github://{owner}/{repo}/readme        README
  ├─ github://{owner}/{repo}/issues/{n}    单个 Issue
  └─ github://{owner}/{repo}/stats         仓库统计

Prompts（模板）：
  ├─ triage_issues          按规则分诊 Issue
  └─ release_notes          从 commits 生成 Release Notes
```

## 6. 选型决策树

```
这个能力我想暴露给谁？
│
├─ LLM 自己判断何时使用
│   └──► Tools
│
├─ 用户/应用显式请求读取数据
│   │
│   ├─ 有稳定的 URI 标识
│   │   └──► Resources
│   │
│   └─ URI 不稳定或需推理参数
│       └──► Tools（返回数据）
│
└─ 用户想一键套用某个工作流模板
    └──► Prompts
```

## 7. 反模式

### 7.1 用 Tool 做只读数据拉取

```
❌ Tool: get_config_file(path)
✅ Resource: file://{path} + Tool: list_config_files()
```

理由：Resource 让 Host UI 可以直接预览、可缓存、可共享给其他对话，Tool 结果是一次性的。

### 7.2 把所有能力塞进一个 Tool

```
❌ Tool: database_operation(action, params)   ← action 为 query/insert/delete...
✅ 拆成 run_query / insert_rows / delete_rows / explain_query
```

理由：一个 Tool 的语义越窄，LLM 越容易正确调用。

### 7.3 Prompts 里硬编码业务参数

```
❌ Prompt 模板里写死 "查询 prod 数据库"
✅ 用 arguments 传入目标环境，让模板通用
```

---

## 8. 小结

- **Tools** — LLM 的"动词"，用来做事
- **Resources** — LLM 的"名词"，用来看数据
- **Prompts** — 用户的"套路"，用来一键触发模板化工作

三者分层清晰、语义正交，组合起来可以覆盖从数据探索到动作执行到工作流编排的完整光谱。
