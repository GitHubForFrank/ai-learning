## 1. 子 Agent 是固定的还是按需创建的

主流做法是 **按需创建**：每次需要时动态启动，任务完成后销毁，不常驻。

```
主对话
  → 调用某种「派生 Agent」的能力（传入任务描述）
  → 新建子 Agent 实例（独立上下文）
  → 子 Agent 执行任务
  → 返回结果给主对话
  → 子 Agent 销毁
```

为什么不常驻：
- 子 Agent 的上下文是任务专属的，长期保留会污染后续任务
- 模型计费按 token，常驻会持续消耗
- 失败重启成本低（任务可重新派发）

某些自研系统会做"长期专家 Agent"（如客服坐席），但**编程协作场景**几乎都是按需。

## 2. 子 Agent 的常用类型

跨工具的通用语义可以归为三类：

| 角色 | 典型职责 | 输出形态 |
|------|---------|---------|
| **Explore / 探索** | 跨目录搜索、理解代码结构、定位实现位置 | 文件清单 + 摘要 |
| **Plan / 规划** | 拆解复杂任务、输出实施计划 | 步骤化任务清单 |
| **Execute / 执行** | 编辑文件、运行命令、做多步操作 | 实际代码改动与运行结果 |

派发时通常可指定的参数：

| 参数 | 含义 |
|------|------|
| 任务描述（prompt） | 子 Agent 要做什么 |
| 角色类型 | Explore / Plan / Execute 之一 |
| 是否后台执行 | 主对话能否继续别的事 |
| 工作区隔离 | 是否在独立沙箱（如 git worktree、容器）跑 |
| 工具白名单 | 子 Agent 允许调用哪些工具 |

不同工具暴露这些参数的方式不同（CLI 选项、JSON 字段、API 参数），但语义大同小异。

## 3. 编排层做的事

无论是 CLI、IDE 插件还是自研 Agent 框架，**编排层**承担同一件事：把「一个高层任务」翻译成「一组 Agent 的协作流水线」。

```
用户输入任务
  ↓
编排层
  ├─ 规划：调用 Plan Agent 拆解任务
  ├─ 调度：管理子 Agent 创建 / 销毁 / 并发
  ├─ 工具调用：读文件、写代码、执行命令、调外部 API
  ├─ 上下文管理：压缩、分层传递、跨阶段交接
  └─ 结果汇总：把多个子 Agent 的输出拼回给用户
```

用户感知不到这些，因为编排层把「人工多 Agent 协作流程」自动化了。

## 4. 一次典型任务背后发生了什么

以"在这个项目里新增一个登录接口"为例：

```
用户输入
  ↓
① Plan Agent：产出任务清单
  - 找到现有接口分层
  - 定位鉴权模块
  - 新增接口 + 单测 + 文档

② Explore Agent ×2（并行）：
  - Agent A 扫描 controller 层目录
  - Agent B 扫描 security 相关模块

③ Execute Agent：
  - 根据定位结果，创建 Controller / Service / DTO
  - 运行测试、查看结果
  - 根据失败信息修复

④ 主 Agent：汇总所有子 Agent 的摘要结果
  ↓
输出给用户
```

此流程**与具体工具无关**，只要工具支持"派子 Agent + 工具调用 + 结果回传"就能跑。

## 5. 跨工具实现对照

| 工具 / 平台 | 派发子 Agent 的能力 | 隔离机制 | 备注 |
|------|---------|---------|------|
| Claude Code | `Agent` 工具（`subagent_type` 参数） | git worktree（可选） | 内置 Plan / Explore / general-purpose 等类型 |
| Cursor | Cascade（Composer）+ Background Agent | 沙箱分支 | Plan 由 Cascade 内部完成，不暴露独立"Plan Agent" |
| Windsurf | Cascade（与 Cursor 同名异源） | 类似 Cursor | 同上 |
| Cline / Roo Code（VS Code） | Plan Mode + Act Mode 双模式切换 | 当前工作区 | "切模式"等价于切 Agent 角色 |
| Aider | `--architect` 模式（Architect + Editor 两个 LLM 角色） | 当前工作区 | 角色固定为两个，不可派任意子 Agent |
| Continue | 自定义 `agent` / `chat` / `edit` 模式 | 当前工作区 | 通过 config.yaml 配置不同模型分工 |
| ChatGPT / Custom GPT | 单会话内通过 prompt 切角色；或多会话人工编排 | 无 | 子 Agent 由人手工开新会话模拟 |
| Gemini / Gem | 同上 | 无 | 同上 |
| 自研 Agent 框架（LangGraph / AutoGen / CrewAI / LlamaIndex Agents） | 显式定义 Agent 节点 + 边 | 容器 / 进程 / 协程 | 可完全自定义；编排逻辑全在代码里 |
| 直连 LLM API | 由你的应用代码自行编排 | 自定 | 自己实现 Plan-Execute 循环 |

**通用结论**：能力是连续谱，从"工具内置编排"到"全靠手工开多会话"再到"自研框架完全可编程"。SDD 章节里的"5 角色 agent 套件"就是把这套编排**降级到纯文本提示词**，让所有工具都能跑（详见 `04-SDD 学习笔记/14-SDD-工具无关的Agent套件.md`）。

## 6. 小结：编排层的本质

| 层级 | 角色 |
|------|------|
| **你** | 下达高层任务（"加一个登录接口"） |
| **编排层** | 规划、调度、上下文管理（Claude Code CLI / Cursor Cascade / 自研框架等） |
| **主 Agent** | 决策层：决定派谁做什么 |
| **子 Agent** | 执行层：各司其职，完成即销毁 |
| **工具** | 执行力：读写文件、运行命令、检索网络 |

一句话：**编排层把"一个任务"翻译成"一组 Agent 的协作流水线"**——具体由谁实现这个翻译过程，不影响这个模式本身。
