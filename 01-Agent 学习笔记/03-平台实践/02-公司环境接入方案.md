## 1. 总体决策流程

接入公司自建大模型环境是「能力对齐」问题，不是「选某款工具」问题。先判断对方提供什么，再选适配它的工具：

```
公司大模型平台
  ├─ 有没有暴露 HTTP API？
  │   ├─ 有 → 看 §2 ~ §4
  │   └─ 没有 → 只能用网页端 / IDE 插件，自动化能力受限
  ├─ API 协议是什么？
  │   ├─ OpenAI 兼容（/v1/chat/completions） → 兼容性最好，几乎所有工具都能接
  │   ├─ Anthropic 兼容（/v1/messages） → Claude Code / Aider 等开箱可接
  │   ├─ 私有协议 → 需要写适配层（OpenAI-Proxy 网关）
  │   └─ 仅 SDK（无 HTTP） → 需要包一层 HTTP 服务
  └─ 鉴权方式？
      ├─ 永久 API Key → 直接配置
      ├─ Web Token / 短期凭证（小时级过期） → §5 / §6
      └─ 无鉴权（内网信任） → 直接可用
```

## 2. 网页端 ≠ 有 API

```
网页端（人工操作，浏览器访问）
  ≠
API 接口（程序调用，返回结构化数据）
```

很多公司 DS 平台只做了网页问答，没暴露 API。要主动问技术负责人：

> "DS 有没有 HTTP 接口可以程序调用？兼不兼容 OpenAI 格式？"

如果有接口地址，能做的事：

| 方式 | 说明 |
|------|------|
| 接入 AI 编程工具 | 改工具的 BaseURL 指向内网地址（详见 §4） |
| 自己写编排脚本 | Python / Node 直连 API，自实现多 Agent 逻辑 |
| 接入开源 Agent 框架 | LangChain、LangGraph、AutoGen、CrewAI 等 |

## 3. 通过 IDE 插件反推 API 地址

公司若提供了自建的 IDE 插件（IDEA / VS Code 形态），插件背后一定调用了某个 API：

1. 用 jadx / IDEA 反编译插件 `.jar` / `.vsix` 文件
2. 搜索关键词：`http`、`/v1/chat`、`completions`、`baseUrl`、`endpoint`、`messages`
3. 找到内网接口地址与协议形式

> 反编译前确认公司信息安全政策；找插件开发同事直接问通常成本更低。

## 4. 按 API 协议匹配工具

接到内网 API 后，把它喂给具体工具的方式各不相同。下面是常见适配方式：

### 4.1 OpenAI 兼容协议

绝大多数工具都内建支持，只要改环境变量或配置文件：

| 工具 | 接入方式 |
|------|---------|
| Cursor | 设置 → Models → 添加 OpenAI 兼容端点 + API Key |
| Continue | `~/.continue/config.yaml` 写自定义 model 块 |
| Cline / Roo Code | 设置面板选 "OpenAI Compatible" + Base URL |
| Aider | `--openai-api-base <URL>` + `OPENAI_API_KEY` |
| Windsurf | 类似 Cursor，模型设置中添加 |
| ChatGPT 客户端 | 大部分第三方客户端（如 ChatBox、Cherry Studio）支持自定义 BaseURL |
| 直连脚本 | OpenAI / Anthropic SDK 都允许传 `base_url` 参数 |

### 4.2 Anthropic 兼容协议

| 工具 | 接入方式 |
|------|---------|
| Claude Code | `ANTHROPIC_BASE_URL` + `ANTHROPIC_AUTH_TOKEN` 环境变量 |
| Aider | `--anthropic-api-base <URL>` |
| 直连脚本 | Anthropic SDK 的 `base_url` 参数 |

### 4.3 私有协议 / 非标准接口

走 **OpenAI-Proxy 网关** 模式：起一层本地或内网代理，把 OpenAI 请求翻译成公司协议，再返回 OpenAI 格式。这样下游工具完全不用动。

```
[Cursor / Continue / Aider]
        ↓ OpenAI 格式请求
[本地代理服务（FastAPI / Express）]
        ↓ 翻译成公司协议
[公司大模型 API]
```

开源项目可以参考：`one-api`、`openai-forward`、`LiteLLM` 等"多协议聚合代理"。

### 4.4 仅 SDK 无 HTTP

包一层 HTTP 服务（FastAPI 暴露 OpenAI 兼容端点，内部调公司 SDK），再走 §4.1。

## 5. Token 类型与刷新策略

| | 永久 API Key | Web Token / 短期凭证 |
|--|---------|-----------|
| 有效期 | 长期 | 几小时 |
| 获取方式 | 配置一次 | 每次登录后从 cookie / 网络请求里提取 |
| 使用体验 | 顺畅 | 过期就报错，要重新提取 |
| 推荐度 | 优先 | 兜底 |

**短期 Token 的成本**：一天可能要手动刷新 3~5 次，会打断工作流。所以优先级永远是：

> **永久 API Key > 内网 OpenAI 兼容接口 > 自动化刷新的 Web Token > 手动刷 Web Token**

## 6. 自动化 Token 刷新（MCP 兜底）

如果公司只发 Web Token，可以用 **MCP chrome-devtools / Playwright Server** 自动化整个流程（任何支持 MCP 的工具都能挂载，不限 Claude Code）：

```
检测到 Token 过期
  → MCP 打开浏览器
  → 自动登录公司网页
  → 抓取网络请求里的新 Token
  → 写入工具配置文件 / 环境变量
  → 继续工作，无感知
```

也可以脱离 MCP，直接写一个 cron 脚本：定时跑无头浏览器刷新 Token、写到 `~/.config/<tool>/credentials`。

前提条件：

- 公司大模型 API 在本机网络可达
- 浏览器自动化不违反公司安全策略（Cookie / SSO 自动化通常要确认）

## 7. 现实方案总结

| 场景 | 工具组合 | 能做什么 |
|------|---------|---------|
| 公司只有 IDE 插件 | 公司内插件 | 代码补全、单轮问答；自动化能力受限 |
| 公司只有网页端 | 网页 + prompt 模板 | 手动多轮编排，上下文靠粘贴 |
| 公司有 OpenAI 兼容 API | Cursor / Cline / Continue / Aider 任选 | 完整编程 Agent 能力，按团队习惯选工具 |
| 公司有 Anthropic 兼容 API | Claude Code / Aider | 同上 |
| 公司有私有协议 API | 任意工具 + OpenAI-Proxy 网关 | 一次适配，所有工具复用 |
| 个人学习 / 提效（家庭网络） | 任选官方 API + 工具组合 | 完整 Agent 能力 |

核心判断：**先把"协议"对齐成工具能听懂的形态，再自由选工具。** 不要因为某个工具流行就强行接，否则容易被协议差异卡住。
