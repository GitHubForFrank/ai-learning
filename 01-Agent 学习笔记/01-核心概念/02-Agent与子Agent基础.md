## 1. 子 Agent 和子代理是同一个东西吗

是的，只是叫法不同：

| 叫法 | 说明 |
|------|------|
| 子 Agent | 中文常用叫法 |
| 子代理 | 中文另一种叫法 |
| Subagent | 英文叫法 |
| Worker Agent | 强调执行角色时的叫法 |

对应关系：
```
Orchestrator = 主代理 / 编排者
  └─ Subagent  = 子代理 / 子 Agent
```

## 2. 子 Agent 的生命周期是怎样的

子 Agent 是一次性的，完整流程：

```
创建 → 初始化 → 执行 → 返回结果 → 销毁
```

**详细阶段：**

**① 创建**
主代理调用 Agent 工具，传入任务 prompt 和参数，系统实例化一个新的 LLM 会话。

**② 初始化**
子 Agent 的上下文被填入：
```
系统提示（如果有）
+ 主代理传入的任务 prompt
= 子 Agent 的初始上下文
```

**③ 执行**
子 Agent 开始工作，可以：
- 调用工具（读文件、搜索、写代码等）
- 产生多轮内部推理
- 甚至再创建下一级子 Agent

**④ 返回结果**
任务完成后把结果返回给主代理，主代理只收到**最终输出**，不是子 Agent 的完整执行过程。

**⑤ 销毁**
子 Agent 的上下文完全清除，不保留任何状态。

---

## 3. 关键特征

| 特征 | 说明 |
|------|------|
| 无状态 | 销毁后什么都不记得 |
| 上下文隔离 | 不共享主代理的上下文 |
| 结果单向传递 | 只有结果回传，过程不回传 |
| 可嵌套 | 子 Agent 内部还可以创建子 Agent |

---

## 4. 小结：子 Agent 为什么能控制 token 膨胀

主代理的上下文**不会因为子 Agent 的执行过程而膨胀**，这是多 Agent 架构控制 token 的核心手段之一：

```
主代理上下文
  ├─ 用户任务（保留）
  ├─ 子 Agent A 结果摘要（只保留结果，不保留执行过程）
  ├─ 子 Agent B 结果摘要
  └─ 最终答案
```

子 Agent 内部那些读文件、递归搜索、反复试错的开销，全部被它自己的生命周期所吸收，用完即焚。
