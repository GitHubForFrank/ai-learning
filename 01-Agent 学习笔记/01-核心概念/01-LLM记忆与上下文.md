## 1. 大模型的记忆功能是怎么实现的

大模型本身是**无状态的**，每次调用只能看到当前输入，没有原生记忆能力。

"记忆"是工程层面的技巧，本质是：**把需要"记住"的内容，在下次调用时塞进 prompt 里。**

常见实现方式：

| 方式 | 说明 | 缺点 |
|------|------|------|
| 拼接对话历史 | 每次把完整历史一起发送 | 对话越长 token 越多 |
| 摘要压缩 | 超长后先总结再替换 | 可能丢失细节 |
| 外部存储 + 检索（RAG） | 历史存数据库，按需检索塞入 | 实现复杂 |
| 文件持久化 | 把关键信息写入文件，每次读入 context | 需要工程支持 |

## 2. 上下文（Context Window）是什么意思

**上下文就是模型每次调用时能"看到"的全部内容的总长度**，用 token 数量衡量。

一次调用中，以下内容都计入上下文：

```
系统提示
+ 对话历史（所有轮次）
+ 当前提问
+ 模型回答
= 总量不能超过上下文限制
```

**Token 数量参考：**

| 数量 | 大约等于 |
|------|---------|
| 1K token | ~750 个英文单词 / ~500 汉字 |
| 128K token | 一本中等厚度的书 |
| 1M token | ~75 万英文单词，相当于整个代码仓库 |

**DeepSeek-Coder 各版本上下文：**

| 模型 | 上下文 |
|------|--------|
| DeepSeek-Coder 1.0 | 16K |
| DeepSeek-Coder-V2 | 128K |
| DeepSeek-V3 | 64K |
| DeepSeek-R1 | 64K |

## 3. 多轮对话中模型回答会传递到下一轮吗

**会。** 多轮对话的本质是每次都把完整历史拼进去：

```
第1轮发送：[你的问题1]

第2轮发送：[你的问题1] + [模型回答1] + [你的问题2]

第3轮发送：[你的问题1] + [模型回答1] + [你的问题2] + [模型回答2] + [你的问题3]
```

**问题：** 对话越长，每次发送的 token 越多，超过上下文限制后平台会截断早期内容或自动压缩，导致模型"忘掉"最开始说的事。

**应对方法（普通用户）：** 在对话变长时，用以下 prompt 生成摘要，然后新建对话继续：

```
请把我们本次对话的关键信息总结成一段简洁的上下文摘要，
包括：已确认的需求、做过的决策、当前进展、待解决的问题。
我将用这段摘要开启新对话继续工作。
```

## 4. 小结：控制上下文的三个抓手

| 抓手 | 应用场景 |
|------|---------|
| **主动摘要** | 长对话快要触顶时人工触发，替换历史 |
| **外部持久化** | 关键事实写入文件/数据库，下次按需读入 |
| **分层上下文** | 全局信息 / 任务信息 / 局部信息 分层注入，避免全量广播 |
