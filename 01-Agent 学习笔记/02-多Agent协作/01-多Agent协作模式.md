## 1. 多 Agent 协作时上下文是怎么处理的

每个 Agent 都是独立的 LLM 调用，各自有独立的上下文窗口。核心原则：**信息按需传递，不是全量广播。**

### 1.1 编排者 + 执行者模式

```
Orchestrator（主代理）
  ├─ 分解任务
  ├─ 只把必要的上下文传给子 Agent
  └─ 收到结果后做摘要，再传给下一个子 Agent
```

### 1.2 共享外部存储

```
Agent A 写入结果 → 外部存储（文件/数据库）
Agent B 读取需要的部分 → 处理 → 写回
```

### 1.3 结构化消息传递

Agent 之间只传结构化摘要，不传原始对话：
```jsonc
{
  "task": "分析用户需求",
  "result": "用户需要一个登录功能",
  "next_action": "生成代码"
}
```

### 1.4 上下文分层

| 层级 | 内容 | 谁能看到 |
|------|------|---------|
| 全局上下文 | 项目背景、规范 | 所有 Agent |
| 任务上下文 | 当前子任务详情 | 负责该任务的 Agent |
| 局部上下文 | 执行细节 | 单个 Agent 内部 |

## 2. 主 Agent 一次对话会产生一批子 Agent 吗

不一定是"一批"，主代理根据任务需要**动态决定**创建几个、什么时候创建。

### 2.1 串行创建（有依赖关系时）

```
主代理
  → 创建子Agent A（完成后销毁）
  → 创建子Agent B（完成后销毁）
  → 创建子Agent C（完成后销毁）
```

### 2.2 并行创建（任务互相独立时，效率更高）

```
主代理
  → 同时创建子Agent A
  → 同时创建子Agent B
  → 同时创建子Agent C
  → 等全部返回后汇总
```

### 2.3 按需创建（最常见）

主代理边推理边决定，可能只创建 1 个，也可能创建 10 个，完全取决于任务复杂度。

> 就像项目经理不是一开始就把所有人叫来开会，而是需要谁就找谁，做完就散。

## 3. 小结：如何选择协作模式

| 场景 | 推荐模式 |
|------|---------|
| 任务之间有明确先后依赖 | 串行创建 + 编排者传递摘要 |
| 任务互相独立，可并行 | 并行创建，最后统一汇总 |
| 任务数量/深度未知 | 按需创建，主代理边做边派 |
| 跨会话共享结果 | 共享外部存储（文件 / 向量库） |
| Agent 之间频繁通信 | 结构化消息 + 上下文分层 |

核心判断：**上下文能不能瘦下来**。能瘦的信息用结构化消息，不能瘦的放外部存储，能复用的抽成共享层。
