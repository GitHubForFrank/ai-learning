## 1. 主 Agent 结合 Plan 是不是更好地使用子 Agent

对，这是目前多 Agent 系统里最成熟的模式，叫 **Plan-and-Execute**。

核心思想：**先规划，再执行**。主代理不再"走一步看一步"，而是先产出一个全局任务清单，再按清单分派子 Agent。

---

## 2. 没有 Plan 时的问题

- 中途改变方向
- 重复创建功能相似的子 Agent
- 子任务之间依赖关系混乱
- 出问题时无法定位是"哪一步"跑偏的

---

## 3. Plan-and-Execute 四步流程

```
第1步：Planning
  主代理分析任务
  → 输出任务清单（做什么、顺序、依赖关系）

第2步：Dispatch
  按清单创建子 Agent
  → 串行 or 并行取决于依赖关系

第3步：Execute
  每个子 Agent 执行自己的子任务

第4步：Aggregate
  主代理汇总所有结果 → 输出最终答案
```

---

## 4. 有无 Plan 的对比

| 没有 Plan | 有 Plan |
|---------|---------|
| 走一步看一步 | 全局可控 |
| 容易跑偏 | 方向锁定 |
| 子 Agent 职责模糊 | 每个子 Agent 任务精准 |
| 难以排查问题 | 哪步出错一目了然 |

各家 AI 编程工具对此模式的实现命名不一：Claude Code 有 `Plan` 子 Agent、Cursor / Windsurf 的 Cascade 内建规划阶段、Cline / Roo Code 区分 Plan / Act 模式、Aider 用 `--architect` 双角色。不论命名，"先规划后执行"在复杂任务上都能显著降低跑偏。

---

## 5. 小结：什么时候值得先规划再执行

**值得 Plan 的场景：**

- 任务步骤超过 3 步，且步骤之间有依赖
- 需要跨多个文件/模块的改动
- 目标比较抽象（如"重构这个模块"），需要先拆解
- 团队协作，需要对齐执行思路

**不需要 Plan 的场景：**

- 单步任务（查一个 API、改一个字符串）
- 探索性问答（先搞清楚问题再决定要不要做）
- 已经很熟悉的重复操作

经验法则：**一眼能说清步骤的任务不用 Plan，需要想两秒的就值得 Plan。**
