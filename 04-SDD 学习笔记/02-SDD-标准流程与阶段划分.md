# SDD 标准流程与阶段划分

> 版本：1.0 | 定位：流程方法论

---

## 1. 总览：SDD 六阶段流程

SDD 把一次"开发任务"拆成 **6 个可检查点**，每个阶段都有明确的产出物和准入准出条件：

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 需求分析 │→ │ 规范定义 │→ │ 评审确认 │→ │ 实现编码 │→ │ 验证测试 │→ │ 归档沉淀 │
└──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
  一句话          draft          approved          code           green           merged
  需求            spec            spec             & tests         tests           + archived
```

三个关键特征：

- 每个阶段**失败可以回退**，越早回退成本越低
- 每个阶段都有"**机器可检测**"的准出条件，不是拍脑袋
- AI 可以在任意阶段介入，但**必须显式切换角色**

---

## 2. 阶段一：需求分析

### 2.1 输入
- 业务方一句话需求
- Issue / 工单
- 用户反馈片段

### 2.2 核心动作
- 澄清 "**谁、在什么场景、要解决什么问题、用什么衡量成功**"
- 一个问题问到底：**为什么做（WHY）→ 做什么（WHAT）→ 怎么做（HOW 先不展开）**
- 和 AI 对话时，让它**反问**，而不是直接给方案

### 2.3 产出物
`需求摘要`（5–10 行），包含：

- **场景**：用户在什么情境下触发
- **目标**：做完后用户获得什么
- **非目标**：刻意不做什么（关键！）
- **成功指标**：如何判断完成

### 2.4 检查点
- [ ] 非目标是否被显式写出？
- [ ] 成功指标是可观测 / 可度量的吗？
- [ ] 能用一句话向外行解释清楚吗？

---

## 3. 阶段二：规范定义

### 3.1 核心动作

基于需求摘要，产出完整 spec（详见第 03 章的 9 大模块）。

推荐书写顺序：

1. 先写 **功能范围 + 非功能需求**（最容易达成共识）
2. 再写 **接口定义 + 数据模型**（进入设计层）
3. 最后写 **技术方案 + 风险评估 + 验收标准**（固化决策）

### 3.2 产出物
- `spec.md`（例如 `docs/specs/<feature>.md`），纳入 Git 管理
- 在 PR 描述或 Issue 里贴链接

### 3.3 检查点
- [ ] 所有接口都有明确的输入 / 输出
- [ ] "可能"、"大概"、"之类"等模糊词已被替换为具体描述
- [ ] 验收标准可以直接翻译为测试用例
- [ ] 风险章节非空（即使是"暂无已知风险"也要显式写出）

---

## 4. 阶段三：评审确认

### 4.1 核心动作

spec 进入审查，而**不是**跳过直接写代码。

- **自审**：对照一份标准 checklist 过一遍
- **他人审**：至少一位同事（或另一个 AI 角色）独立审
- **AI 审**：让 AI 扮演「杠精评审员」角色找漏洞

常用提问：

- "这份 spec 有没有哪里可能被误解？"
- "如果我是新来的工程师，哪里会让我卡住？"
- "哪些场景没覆盖？"

### 4.2 产出物
- 评审意见清单（Review Comments）
- 更新后的 `spec.md v0.2`

### 4.3 检查点
- [ ] 所有评审意见都已**显式**处理（接受 / 拒绝 + 理由）
- [ ] spec 版本号已迭代
- [ ] 本阶段至少发现 **≥ 1 个问题**（若一个都没有，多半是审得不认真）

---

## 5. 阶段四：实现编码

### 5.1 核心动作

严格按 spec 生成代码。关键是**让代码与 spec 双向可追溯**：

- 实现函数 / 类在 doc comment 中引用 spec 章节号，如 `// spec §5.2`
- 每个 commit 对应 spec 中的一段变更
- AI 编码时必须**显式告知它 spec 文件路径**

### 5.2 产出物
- 代码 + 单元测试
- 变更 diff

### 5.3 检查点
- [ ] 所有 spec 里的接口都已实现
- [ ] 所有 spec 里的验收标准都有对应测试
- [ ] 存在**不在 spec 里**但新增的代码吗？（若有，问：必要吗？是否回补 spec？）

---

## 6. 阶段五：验证测试

### 6.1 核心动作
- **自动化验证**：单元测试、集成测试，覆盖验收标准
- **手工验证**：按 spec 的"用户场景"一条条走
- **边界验证**：对照非功能章节（性能、安全、容量）测试

### 6.2 产出物
- 测试报告
- 若发现偏差，记录为 bug 并决定：**修代码 / 修 spec / 两者都修**

### 6.3 检查点
- [ ] 所有验收标准都有绿色测试
- [ ] 测试覆盖率达到 spec 要求（若有）
- [ ] 边界 / 异常路径已测

---

## 7. 阶段六：归档沉淀

### 7.1 核心动作
- 把最终 spec、实现决策、遗留问题一并归档到仓库
- 在 `CHANGELOG.md` 或 release note 中引用 spec 版本
- 如果 spec 体现了通用模式，提炼进团队 **spec 库**

### 7.2 产出物
- 归档后的 `spec.md`（版本冻结）
- 必要时生成 `ADR`（Architecture Decision Record）
- 更新过的项目级约束文件（`AGENTS.md` / `CLAUDE.md`）/ `README.md` 入口索引

### 7.3 检查点
- [ ] 在仓库中能快速搜到这份 spec
- [ ] 半年后被问"为什么这么做"，spec 能回答吗？

---

## 8. 在 AI 编程工具中落地每个阶段

下表的操作是**工具无关**的——任何具备"读写文件 + 长上下文 + 工具调用"三项能力的 AI 编程工具（Claude Code / Cursor / Aider / Continue / 直连 API 等）都能照做。Claude Code 用户在"实际操作"列直接对号入座；其他工具的对应做法见 [04 章 §6 在不同工具里加载 SDD 上下文](./04-SDD-工具落地与工作流整合.md)。

| 阶段 | 实际操作 |
|------|------------------------------|
| 需求分析 | 纯对话驱动 AI 反问，生成需求摘要 |
| 规范定义 | 让 AI 写出 `docs/specs/<feature>.md`，在项目级约束文件（`AGENTS.md` / `CLAUDE.md`）中登记索引 |
| 评审确认 | 用"切换角色"技巧：让 AI 扮演评审者重新审视 spec |
| 实现编码 | 提示词里**显式加上 "请严格遵循 docs/specs/xxx.md"** |
| 验证测试 | 让 AI 按 spec §验收标准一条条生成测试 |
| 归档沉淀 | 更新约束文件中的 spec 索引，commit message 引用 spec 版本号 |

> 第 04 章会把每个阶段的**提示词模板**详细展开。

## 9. 阶段间的检查点矩阵

| 阶段 | 进入条件 | 退出条件 | 失败处理 |
|------|----------|----------|----------|
| 需求分析 | 有原始需求输入 | 需求摘要 ≤ 10 行、非目标已写 | 补充澄清对话 |
| 规范定义 | 需求摘要已确认 | spec 9 模块齐全 | 回到分析补信息 |
| 评审确认 | spec draft 存在 | 所有评审意见已处置 | 重写 spec |
| 实现编码 | spec 已 approved | 代码通过自测 | 发现 spec 漏洞 → 回到规范定义 |
| 验证测试 | 代码已提交 | 验收标准全绿 | 根据偏差类型分流 |
| 归档沉淀 | 测试已通过 | spec 入库、索引已更新 | 补齐归档材料 |

## 10. 可以跳过某阶段吗？

SDD 不是教条。下面是**可以合理精简**的场景：

| 场景 | 可跳过 / 精简的阶段 |
|------|---------------------|
| 独自写玩具项目 | 评审确认（但自审不能跳） |
| Bug 紧急修复 | 规范定义写成 5 行：病因 + 修复方案 + 验收 |
| 小型 UI 调整 | 事后补 mini-spec |
| 原型探索 | 仅保留"需求分析 + 归档"两头 |

> 判断标准：**"是否存在信息丢失的风险"**。只要工作可能被其他人（或未来的自己、或 AI）误解，这一阶段就不该跳。

## 11. 示例串讲：一次完整的 SDD 流程

以"给 CLI 工具加 `--json` 输出开关"为例：

```
[阶段 1 需求分析]
   摘要：CI 脚本需要机读数据，当前只有人读表格；仅加开关，不改默认行为。
   非目标：暂不加 XML/YAML；不改列名。

[阶段 2 规范定义]
   spec.md 更新：
   - §3 功能范围：新增 --json flag
   - §5 接口：json schema 示例
   - §9 验收：3 条用例

[阶段 3 评审确认]
   评审意见：json 数值字段类型未指定 → 补"所有计数字段为 integer"

[阶段 4 实现编码]
   新增 3 个函数，每个注释 // spec §3-5

[阶段 5 验证测试]
   跑 3 条用例，1 条失败（空结果时输出 [] 还是 {}？）
   → 回到 §5 补齐规定 → 重测通过

[阶段 6 归档]
   CHANGELOG 记录：feat(cli): add --json flag (spec v1.2)
```

整个过程如果走 Vibe Coding，很可能在"空结果格式"这里埋坑，等 CI 脚本挂了才发现。

## 12. 思考与练习

1. **回溯题**：找一个你最近完成的小功能，倒推它如果走完 SDD 六阶段应该长什么样。哪一阶段你实际上跳过了？跳过带来了什么后果？
2. **流程设计题**：你们团队现有的开发流程（例如从 Jira 到 PR）如何映射到 SDD 六阶段？哪些阶段已经有制度保障，哪些是空白？
3. **权衡题**：如果"评审确认"阶段只能让 AI 来审（没有同事协作），你会怎么设计提示词，让评审不流于形式？

   > 参考思路：切换多种角色分别审（杠精 / 安全专家 / 新人）；显式要求 AI 产出 "N 条问题 + 优先级"。
4. **场景题**：线上紧急 bug 需要 30 分钟内修复，六阶段该如何压缩？哪一步绝对不能省？

   > 参考思路：压缩到 "mini-spec（5 行）→ 修复 → 验证 → 归档"，评审可异步补；**验证绝不能省**。
