# SDD 规范文档模板详解

> 版本：1.0 | 定位：模板与写作规范

---

## 1. 为什么需要统一的 spec 模板

同样一个"登录功能"的 spec，不同人写出来可能是：

- 甲：500 字散文
- 乙：30 条 JIRA 子任务
- 丙：PlantUML 时序图

三种都不能被机器消费，也无法跨项目复用。统一模板的价值：

- **对人**：新人一眼知道该看哪一节
- **对 AI**：可稳定解析、可按章节引用
- **对工具**：可自动生成测试骨架、可自动校验完整性

---

## 2. 通用 spec 9 大模块

```
┌──────────────────────────────────────────┐
│ 1. 需求背景       ← WHY                  │
│ 2. 业务目标       ← WHAT (outcome)       │
│ 3. 功能范围       ← WHAT (scope)         │
│ 4. 非功能需求     ← HOW WELL             │
│ 5. 接口定义       ← HOW (API)            │
│ 6. 数据模型       ← HOW (data)           │
│ 7. 技术方案       ← HOW (implementation) │
│ 8. 风险评估       ← WHAT IF              │
│ 9. 验收标准       ← DONE?                │
└──────────────────────────────────────────┘
```

> 不是每个 spec 都必须写满 9 个模块。小任务可以退化为 **"1 + 3 + 9"**（需求背景 + 功能范围 + 验收标准），但**模块顺序与标题保持不变**，保证可扫读。

---

## 3. 模块一：需求背景

### 3.1 编写要点
- 讲清楚 "**为什么现在要做**"：触发原因 / 痛点 / 上游依赖
- **不写**具体解决方案
- 3–5 句话足够

### 3.2 示例

> 当前用户登录仅支持邮箱密码，运营反馈接入"手机号一键登录"后新用户转化率可提升约 18%（X 渠道实测数据）。为匹配 Q2 增长目标，本期补齐手机号登录能力。

### 3.3 避坑
- ❌ 跳过该模块直接写功能 → 半年后无人记得为何要做
- ❌ 把"产品经理说要做"当背景 → 依赖一个随人员流动消失的 single source of truth

---

## 4. 模块二：业务目标

### 4.1 编写要点
- **可度量**（数值、百分比、时间）
- 区分 **主目标 (Primary)** 和 **次目标 (Secondary)**
- 显式写出 **非目标 (Non-goals)**

### 4.2 示例

> - **主目标**：上线手机号登录通道，手机号登录占比在 1 个月内达到 20%+
> - **次目标**：为后续"第三方授权登录"预留扩展位
> - **非目标**：本期不做跨设备账号合并；不做海外号码段

### 4.3 避坑
- ❌ "提升用户体验"这类不可度量的目标
- ❌ 写了主目标但不写非目标 → 最容易导致 scope creep

---

## 5. 模块三：功能范围

### 5.1 编写要点
- **用户故事 (User Story)** 或 **功能清单** 任选其一
- 每条都要能对应到后面"验收标准"的某一条
- 用 **MUST / SHOULD / NICE-TO-HAVE** 标注优先级（参考 RFC 2119）

### 5.2 示例

```
[MUST]   F1 用户可输入手机号并获取验证码
[MUST]   F2 用户输入正确验证码后完成登录
[MUST]   F3 验证码 60 秒内不可重复请求
[SHOULD] F4 连续 5 次失败后冻结该号码 10 分钟
[NICE]   F5 登录后提示"绑定邮箱以便恢复账号"
```

### 5.3 避坑
- ❌ 范围写得含糊（"支持常见登录方式"）
- ❌ 把实现细节混进来（"使用阿里云短信 SDK" 应归到 §7 技术方案）

---

## 6. 模块四：非功能需求

### 6.1 编写要点

覆盖以下 5 类（缺项时显式写 N/A）：

| 类别 | 示例条目 |
|------|----------|
| 性能 | P99 响应时间 < 300ms |
| 容量 | 支持 100 QPS，峰值 500 QPS |
| 安全 | 验证码 HMAC 签名，不落明文日志 |
| 可用性 | 短信通道降级策略 |
| 兼容性 | 兼容 iOS 13+ / Android 8+ |

### 6.2 避坑
- ❌ 遗漏"安全 / 日志脱敏"类条目 → 审计时补救成本极高
- ❌ "尽可能快"这种无数字的条件

---

## 7. 模块五：接口定义

### 7.1 编写要点
- 每个接口至少包含：**方法、路径、入参、出参、错误码、幂等性**
- 推荐用 **OpenAPI 片段** 或 **Given-When-Then** 表达
- 对外接口附 **示例请求 / 响应**

### 7.2 示例

```yaml
POST /api/auth/send-code
Request:
  phone: string (E.164 格式)
Response 200:
  code: "OK"
  cooldownSec: 60
Response 429:
  code: "TOO_FREQUENT"
  retryAfter: int
幂等性：同一号码 60s 内多次请求返回同一 cooldown
```

### 7.3 避坑
- ❌ 只写成功路径，不写错误码
- ❌ 不说明幂等性

---

## 8. 模块六：数据模型

### 8.1 编写要点
- 关键实体 + 字段 + 约束（NOT NULL / 唯一键 / 外键）
- **重点说明索引策略**：哪些字段必须建索引、为什么
- 涉及历史数据需说明迁移策略

### 8.2 示例

```
Table: user_auth_phone
  id            BIGINT PK
  user_id       BIGINT FK → user.id  (unique)
  phone_e164    VARCHAR(20)          (unique, index)
  verified_at   DATETIME
  created_at    DATETIME

索引：phone_e164 唯一索引（登录时 lookup）
迁移：首次创建表，无存量数据
```

---

## 9. 模块七：技术方案

### 9.1 编写要点
- **选型理由**：为什么是 A 不是 B（1–2 句即可）
- **关键时序 / 状态流转**
- 外部依赖清单

### 9.2 示例

```
选型：短信服务使用 XX 云（已接入、到达率 > 99%）

时序：
  Client → /send-code → AuthSvc → SmsProvider → 用户
  Client → /verify    → AuthSvc → Redis(验证码校验) → 签发 Token

外部依赖：XX 云短信 API、Redis 集群、JWT 私钥管理服务
```

### 9.3 避坑
- ❌ 把选型当结论塞给读者而不给 1 句理由
- ❌ 省略外部依赖清单 → 上线时才发现需要申请权限

---

## 10. 模块八：风险评估

### 10.1 编写要点

每条风险必须包含三要素：**风险描述 + 可能性 + 缓解措施**。

### 10.2 示例

| 风险 | 可能性 | 缓解 |
|------|--------|------|
| 短信通道被恶意刷爆 | 中 | 每号码每天限 5 次 + 图形验证码 |
| 手机号碰撞（换号用户） | 低 | 命中活跃账号时走合并流程 |
| 短信服务商宕机 | 低 | 预留语音验证码备份通道 |

### 10.3 避坑
- ❌ 写"暂无风险"了事 → 99% 是没思考
- ❌ 只写风险不写缓解

---

## 11. 模块九：验收标准

### 11.1 编写要点
- 用 **Given-When-Then** 或编号清单
- **每条都能直接写成自动化测试**
- 功能性验收 与 非功能性验收 分开标注

### 11.2 示例

```
[AC-1]      Given 新手机号, When 请求验证码, Then 返回 200 且收到短信
[AC-2]      Given 同一号码 60s 内重复请求, Then 返回 429
[AC-3]      Given 正确验证码, When verify, Then 返回 JWT 且用户表新增一行
[AC-4]      Given 连续 5 次错误, Then 号码进入冻结队列 10 分钟
[AC-NFR-1]  /send-code P99 < 300ms（压测 100 QPS）
```

### 11.3 避坑
- ❌ "功能应正常工作"这类无法验证的描述
- ❌ 验收标准数量少于功能项 → 必有漏测

## 12. 如何让 spec 足够「明确、可验证」

3 个实操技巧：

### 12.1 替换模糊词

| 模糊表达 | 替换为 |
|----------|--------|
| 尽快 | 具体 SLA（例如 P99 < 300ms） |
| 支持多种…… | 枚举完整列表 |
| 通常 / 一般 | 明确条件 |
| 大量 / 少量 | 具体数量级 |
| 用户友好 | 可度量的 UX 指标 |

### 12.2 反向提问法

写完后自问：**"如果有人故意曲解这一句，能曲解出几种版本？"**
每曲解一次就补一条边界定义。

### 12.3 可测性检查

每条需求读完，心里要能浮现出**至少一条测试断言**。
若想不出 → 改写，直到想得出为止。

## 13. 进阶：通用约束 spec（conventions）—— 9 支柱

主 spec 9 模板解决"**单个 feature** 的规范"，但中长期项目里总有一类条目反复出现在每份 spec 里：命名规则、目录约定、错误码段位、测试编号、安全合规底线……这些**跨 feature 的通用约束**塞进每份主 spec 等于复制粘贴。

实战项目 `04-SDD 实战项目/sdd-02-task-fullstack/task-spec/conventions/` 把它们抽出来沉淀为**约束 spec**，独立成文件，与主 spec 平级。下表的"9 支柱"是经过实战检验的最小完整集，可作为模板直接照搬：

| 支柱 | 文件名建议 | 职责（≤1 句） |
|---|---|---|
| 前端 | `fe-conventions.md` | 目录结构 / 组件命名 / HTTP 客户端 / 追溯注释格式 |
| DB | `db-conventions.md` | 表 / 列 / 索引 / 迁移工具命名 / 已发布脚本不可变 |
| 测试 | `test-conventions.md` | TC 编号体系（TC-U / TC-IT / TC-FE / TC-E2E）/ 用例命名 |
| 安全 | `security-conventions.md` | 输入校验 / XSS / SQL 注入 / CORS / 速率限制 / secrets / PII / 审计日志 |
| 监控 | `monitoring-conventions.md` | Metrics / Logs / Traces / SLO / 告警 / 上线必备项 |
| API 版本化 | `api-versioning.md` | `/api/v<N>` 路径策略 / deprecation 流程 / 错误码段位 |
| 异步任务 | `async-job-conventions.md` | 定时任务 / 事件监听的幂等 / 重试 / 超时 / flag 协作 |
| 国际化 | `i18n-conventions.md` | 文案 key 命名 / 时区 / BR message 国际化 / 前后端分工 |
| 依赖策略 | `dependency-policy.md` | 锁版本规则 / 大版本升级流程 / CVE 响应 SLA |

### 13.1 主 spec 与约束 spec 的分工

| 维度 | 主 spec（`docs/specs/NNNN.md`） | 约束 spec（`conventions/*.md`） |
|---|---|---|
| 作用域 | 单个 feature / 变更 | 跨所有 feature 的通用规则 |
| 生命周期 | 随 feature 起止 | 长期，季度级修订 |
| 变更频率 | 高（每个 feature 一份） | 低 |
| 谁审 | feature owner + 评审员 | 架构师 / 团队全体 |
| 引用方向 | **主 spec 引用约束 spec**，反之不许 |  |

铁律：**主 spec 不重写约束 spec 的内容，只引用**。例：

```markdown
## 5. 接口定义
本接口路径遵循 conventions/api-versioning.md §2，错误码段位见 §4。
```

### 13.2 什么时候上 9 支柱

不是每个项目都需要 9 份。判据：

- 团队 ≥3 人 / 项目周期 ≥6 个月 → 至少上 fe / db / test / security 四份（高频踩坑）
- 涉及对外开放 API → 加 api-versioning
- 有线上 SLA → 加 monitoring
- 国际业务 → 加 i18n
- 个人项目或 1–2 月小项目 → 0 份，主 spec 内嵌一段"约定"足矣

### 13.3 约束 spec 的边界

约束 spec **写抽象规则与契约，不嵌入具体业务**：

- ✅ "DB 表名一律 snake_case，业务前缀 + 实体名"
- ❌ "用户表叫 `user`，订单表叫 `order`"（这是具体业务，归主 spec / 数据字典）

把具体业务写进约束 spec 会让它变成"数据字典 v2"，每个新 feature 都要回头改它，背离"低频变更"的初衷。

> 完整 9 份 conventions 实参见实战项目 `04-SDD 实战项目/sdd-02-task-fullstack/task-spec/conventions/`。

## 14. 反例对照

### 13.1 糟糕的 spec

> "系统要做一个好用的登录功能，支持多种方式，要考虑安全。响应要快。"

问题：

- 没有背景（为什么做？）
- 没有范围（"多种"是几种？哪几种？）
- 没有非功能数字（"快"是多快？）
- 没有验收（怎么知道做完了？）

### 13.2 合格的 spec

见第 3–11 节每一节的示例。对比可知：**结构化、可度量、有边界**。

## 15. 示例串讲：一份最小可用 spec

以"给团队博客加阅读计数器"为例，体积仅约 30 行，但 9 模块齐全：

```markdown
# 博客阅读计数 — Spec v0.1

## 1. 需求背景
作者希望看到每篇文章的热度，当前无数据。

## 2. 业务目标
- 主：每篇文章页展示阅读数
- 非目标：本期不做独立 UV/PV 区分

## 3. 功能范围
[MUST] F1 页面加载时展示阅读次数
[MUST] F2 每次真实用户访问 +1（同一 IP 5 分钟内去重）

## 4. 非功能需求
- 性能：计数读接口 < 50ms
- 安全：不暴露任何用户信息

## 5. 接口定义
GET  /api/posts/{id}/views → { count: int }
POST /api/posts/{id}/view  → 204

## 6. 数据模型
Redis Hash: post:views → field=postId, value=count

## 7. 技术方案
Redis INCR + 5 分钟 IP 级去重键

## 8. 风险评估
| 风险 | 可能性 | 缓解 |
| 恶意刷量 | 中 | IP 级限流；反爬本期不做 |

## 9. 验收标准
[AC-1] 打开文章页 → 显示数字
[AC-2] 同 IP 5 分钟内刷新 → 计数不变
[AC-3] 新 IP 访问 → 计数 +1
```

## 16. 思考与练习

1. **改写题**：下面这段需求有哪些"不可验证"的表述？请改写成可验证版本。

   > "订单系统要稳定可靠，性能要好，不能让用户等太久，数据不能丢。"
2. **完成题**：从上面的最小 spec 出发，若要加"文章热度排行榜"功能，9 大模块各需补充哪些条目？
3. **排序题**：以下 5 条需求按 MUST / SHOULD / NICE 排序：
   - (a) 页面加载时展示阅读数
   - (b) 支持按月份筛选统计
   - (c) 同 IP 去重
   - (d) 文章作者可导出 CSV
   - (e) 显示增长趋势图

   > 参考：(a)(c) MUST；(b) SHOULD；(d)(e) NICE
4. **开放题**：在你当前项目中，§8 风险评估最容易被忽略。设计一个"风险脑暴 prompt"，让 AI 协助你产出 3–5 条候选风险。
