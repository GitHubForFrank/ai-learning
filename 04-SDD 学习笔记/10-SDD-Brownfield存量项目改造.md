# Brownfield：存量项目的 SDD 化改造（全栈视角）

> 版本：1.0 | 定位：高频问题专题 — "我项目都老了能上 SDD 吗？"

比起 greenfield（从零开始）上 SDD，**Brownfield（存量改造）才是大多数团队的真实处境**。本章只回答一件事：**一个从未跑过 SDD 的全栈项目，如何把 SDD 逐步挂上去而不翻车**。

---

## 1. Brownfield 的四个特殊难题

1. **代码即唯一事实源**——现行行为由代码定义，但代码本身不回答"为什么"
2. **历史知识已流失**——原作者离职、commit message 只写 "fix"、隐式契约散在各处
3. **团队有既有习惯**——"以前没 spec 也能发版"的路径依赖
4. **文档债务风险更高**——补的 spec 若不维护，反而会把人和 AI 同时带偏

**关键认知**：Brownfield SDD 的目标**不是"给全部存量补 spec"**，而是**把 SDD 挂到合适的切入点上，让新增 / 高价值变更都走 SDD**。追求"全覆盖"是这条路上最大的陷阱。

---

## 2. 三种切入模式（先选一种起步）

| 模式 | 做什么 | 成本 | 适合场景 |
|------|--------|------|----------|
| **A. 只约束新功能** | 存量不动，所有新 feature 走 SDD | 低 | 团队刚起步、项目仍健康 |
| **B. 契约层先固化** | 导出 OpenAPI / DB schema / 事件 schema 作为契约，实现不改 | 中 | 跨团队协作、对外 API 多 |
| **C. 借重构做 spec 考古** | 结合已计划的重构 / 迁移，倒推出 spec | 高 | 本来就要动的模块 |

**铁律**：第一天别贪大。推荐顺序 **A → 2 周后加 B → 3 个月后遇到契机再做 C**。同时推 B + C 是最常见的翻车姿势。

## 3. 全栈现状盘点清单（Week 1–2）

改造前必须知道"项目现在长什么样"。按层做一次快速盘点；产出物是 `docs/audit/` 下的盘点报告——**它本身还不是 spec，是后续所有 spec 考古的地图**。

### 3.1 前端层

- [ ] 路由清单（SPA 路由 / 多入口 / SSR 页面）
- [ ] 关键页面、组件、布局容器
- [ ] 状态管理方案和核心 store / context 结构
- [ ] 依赖的后端接口清单（按页面归组）
- [ ] 构建 / 部署方式、关键环境变量

### 3.2 后端层

- [ ] 接口清单（反向导出 OpenAPI，交由各框架内置能力完成）
- [ ] Domain Model / 核心业务实体
- [ ] 外部依赖清单（DB / 缓存 / MQ / 第三方 API）
- [ ] 事务边界与幂等策略（通常散在各 Service，要梳理）
- [ ] 定时任务 / 异步消费者 / 批处理清单

### 3.3 数据层

- [ ] 表清单 + 关键索引（从 `INFORMATION_SCHEMA` 或等价视图导出）
- [ ] 存储过程、触发器、物化视图（若有）
- [ ] 迁移脚本历史与当前基线版本
- [ ] 数据量级 & 增长速度（决定后续压测 fixture 规模）

### 3.4 基础设施层

- [ ] CI / CD 流水线（触发条件、检查项、部署策略）
- [ ] 环境拓扑（环境数量、灰度、多机房）
- [ ] 监控 / 告警接入点、日志聚合链路
- [ ] 密钥 / 配置管理方案

盘点不要追求精美，**3–5 天出初版，先跑起来**；遗漏的细节会在后续 spec 考古中被迫补齐。

---

## 4. 契约层优先：Brownfield SDD 最高 ROI 的动作

盘点完后，**第一件要做的不是写 spec，而是固化契约层**。理由在 07 章 §4 已论证：契约层是跨栈协作的命门，一旦成文，前后端 / DB 自然被迫对齐。

三类契约的冻结顺序：

1. **API 契约（最优先）**
   - 用后端框架自带的反向导出能力产出 OpenAPI
   - 人工补全缺失的错误码、示例、字段描述——导出只能给出结构，语义要靠人
   - 放入 `docs/contracts/openapi.yaml`，作为一等公民 review
   - 从此：**接口要变，先改 `openapi.yaml`，再改代码**
2. **DB 契约**
   - 导出当前 schema 作为 baseline，归档进仓库
   - 后续所有表变更必须走迁移脚本 + review，不允许手工改生产库
   - 破坏性变更（删列、改类型、改索引）走独立 PR
3. **事件 / 消息契约**（若有 MQ / Event Bus）
   - 列出所有 topic / queue、生产者、消费者、payload 字段语义
   - 没 schema 的先加一个最宽松的，再逐步收紧

契约层固化完成 ≈ 拿到 SDD 的 50% 收益。剩下 50% 才是 spec 本身。对很多团队，只做到这一步就已经够用。

## 5. Spec 考古：从代码反推 spec

对高价值存量模块，可以做"考古式 spec"——从代码倒推 spec。做之前先降低期望：它**只能还原 WHAT，很难还原 WHY**。

### 5.1 5 步考古法

1. **抽入口 / API 清单**——所有 public 函数、路由、消费者
2. **读测试**——测试覆盖的就是隐性 AC，这是考古的一级金矿
3. **翻 PR 与访谈**——`git blame` → PR 描述 → 原作者对话；记录"为什么当时这么做"
4. **对照线上行为**——监控 / 日志里的真实调用模式，有些隐藏约束（如只处理某类租户）只有线上看得到
5. **产出"现行行为 spec"**——先还原现状，不要在这一步优化；任何"当前这样不合理"的观察标记 TODO 即可，不要边考古边改

**铁律**：考古 spec 必须在顶部标明 `状态:[REVERSE-ENGINEERED v1.0]`，和 `[APPROVED]` 的 greenfield spec 区分开。前者是"行为倒推了决定"，权威性不同，后续遇到冲突时优先级也不同。

### 5.2 不值得考古的信号

出现以下任一，停止考古、保留为"代码即规范"即可：

- 模块 3 个月内会被重写或下线
- 代码量 < 500 行且语义明显
- 原作者还在团队里，能随时答疑（人本身就是 spec）
- 预估考古成本 > 重写成本

## 6. 全栈各层的 SDD 化要点

改造不是"统一模板套下去"，不同层的 spec 粒度、AC 重点、契约约束都不同。

### 6.1 前端层

- **spec 粒度**：按"用户旅程"拆，不按"组件"拆。一个 spec 对应一个完整场景（如"手机号登录"），跨多个页面也没关系
- **AC 重点**：页面状态机（loading / empty / error / success）、键盘 / 可访问性、移动端适配、离线降级
- **契约约束**：依赖的 API 必须引用 `openapi.yaml`，不得在前端本地定义一套字段
- **项目级约束文件（`AGENTS.md` / `CLAUDE.md` 等）必写**：组件拆分阈值、禁用的全局状态模式、样式方案约定、新增依赖前必须问

### 6.2 后端层

- **spec 粒度**：按"业务能力"拆（登录 / 下单 / 结算），不按"接口数"拆
- **AC 重点**：幂等性、事务边界、失败重试策略、限流、审计日志、权限校验位置
- **契约约束**：接口签名以 `openapi.yaml` 为准；Domain Event 必须有 schema；DB 访问必须经 Repository 层
- **项目级约束文件（`AGENTS.md` / `CLAUDE.md` 等）必写**：分层架构约定、异常处理策略、日志规范、禁止硬编码密钥 / 环境

### 6.3 数据层

- **spec 形态**：每次 schema 变更都是一份 mini-spec，而不是一份大 spec 管整张表
- **AC 重点**：迁移可回滚、锁表时间上限、数据量影响、索引影响评估、是否破坏性
- **契约约束**：迁移脚本 + 回滚脚本**必须成对**；破坏性变更需专门审批流
- **项目级约束文件（`AGENTS.md` / `CLAUDE.md` 等）必写**：禁止直连生产库、禁用的 SQL 模式（如无 WHERE 的 DELETE）、迁移工具与命名约定

### 6.4 基础设施层

- **spec 形态**：主要形态不是 spec 而是 Runbook / Playbook；SDD 在这层的产物是**部署契约**（环境变量清单、依赖服务清单、回滚步骤）
- **AC 重点**：部署可回滚、SLI / SLO、容量规划、告警接入、凭据轮换
- **项目级约束文件（`AGENTS.md` / `CLAUDE.md` 等）必写**：禁止代码内写死环境信息、密钥获取方式、发布流程与审批门槛

## 7. 六个月路线图（参考模板）

| 时间 | 动作 | 里程碑 |
|------|------|--------|
| Week 1–2 | 全栈盘点 + 契约层导出 | `docs/audit/`、`docs/contracts/` 成型 |
| Week 3–4 | 下一个新 feature 按完整 SDD 走一次 | 团队首份标准 spec + 模板 `docs/specs/TEMPLATE.md` |
| Month 2 | PR 模板强制勾选"是否涉及 spec / 契约变更" | CI 开始校验契约文件格式 |
| Month 3 | 挑一个高价值存量模块做 spec 考古（试点） | 首份 `[REVERSE-ENGINEERED]` spec |
| Month 4 | `AGENTS.md` / `CLAUDE.md` 沉淀团队编码约束 | AI 编程工具（Claude Code / Cursor 等）开始主动拦截不规范代码 |
| Month 5 | 契约变更走独立 PR，与实现 PR 解耦 | 契约 PR 成为常态 |
| Month 6 | 复盘：哪些 spec 救场、哪些是文档债，决定下阶段范围 | 形成第二版落地策略 |

节奏是参考值，重点是"**每个月至少一个可见产物**"，避免改造感只存在于 wiki 里。

## 8. 常见陷阱与对治

| 陷阱 | 症状 | 对治 |
|------|------|------|
| **考古贪全** | 立项要给所有存量模块补 spec | 只考古高价值模块（改动频繁 / 新人易坑 / 合规要求） |
| **马后炮 spec** | 代码写完再补 spec，纯粹走过场 | 把 spec 作为 PR **准入**门槛，而非收尾动作 |
| **双标膨胀** | 新功能走 SDD、存量随意，双轨变多轨 | 明确双轨边界（哪些模块算新、哪些算存量），季度性评估升档 |
| **文档债务反弹** | 半年前补的 spec 早已过期，新人被误导 | 过期 spec 立即标 `[STALE]`；严重不一致直接归档而非留着 |
| **契约层懒政** | 导出一次 OpenAPI 后就不更新 | 把契约文件更新纳入 PR 检查清单；**OpenAPI 不改 = 接口不能改** |
| **强推失败** | 顶层下令、基层抵触、最终流于形式 | 试点 → 数据 → 推广；避免直接行政命令铺开 |
| **绕过契约走"临时字段"** | 前端加了个 schema 外的字段，后端悄悄支持 | 契约违约一律在 PR review 挡下；例外必须有 ADR |

## 9. 决策图：存量项目值不值得上 SDD

```
项目剩余生命周期 ≥ 12 个月? ─── 否 ──→ 只做契约层 + 新 feature SDD
          │是
          ▼
对外 API / 多团队协作? ─── 是 ──→ 契约层必做;spec 按模块优先级补
          │否
          ▼
近 3 个月将发生重构 / 迁移? ─── 是 ──→ 借重构机会做 spec 考古
          │否
          ▼
团队 ≥ 3 人且有新人流入? ─── 否 ──→ mini-spec + 新功能 SDD 即可
          │是
          ▼
   启动完整 Brownfield SDD 路线图（§7）
```

## 10. 诚实结论

- **Brownfield SDD 是"驯兽"不是"建造"**：你不能推翻现实，只能在现实之上加约束。接受这一点，不要追求完美覆盖
- **契约层 ROI 远高于 spec 本身**：如果只能做一件事，先固化 API / DB / 事件契约
- **考古 spec 的价值在"拦截下一次误改"**：不是留给后人瞻仰，是保护未来的变更不跑偏
- **不是所有模块都值得 SDD 化**：短命模块、原作者还在、代码足够短的模块，维持现状即可
- **SDD 是过程不是终点**：6 个月路线图结束 ≠ 完工，而是进入日常维护节奏

**最大的风险不是做得不完美，而是半途而废后留下一堆过期 spec**——那才是真的比没上 SDD 还糟。Brownfield SDD 宁缺毋滥。

## 11. 思考与练习

1. **盘点题**：挑一个你维护的存量模块，按 §3 做一次 30 分钟快速盘点，产出前端 / 后端 / DB / 基础设施四层各 5 条的盘点清单。哪一层信息最难凑齐？反映出什么团队现状问题？
2. **契约题**：你现在的项目如果要**只做契约层不写 spec**，能导出一份可用的 OpenAPI / DB schema / 事件 schema 吗？缺哪块？缺的那块意味着什么风险？
3. **考古题**：用 §5.1 的 5 步法对一个 500–2000 行的存量模块做一次小型考古。哪一步信息最不全？这条信息缺失会在哪类未来变更中咬人？
