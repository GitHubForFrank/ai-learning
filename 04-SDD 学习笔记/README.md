# 04-SDD 学习笔记

> 一套面向 **1–3 年 Java 后端开发者** 的 **SDD（Specification-Driven Development，规范驱动开发）** 系统化笔记。
> 从概念到落地，从模板到案例，帮你在 AI 辅助开发时代建立 **"规范先行"** 的工作方式。

> **🛠️ 工具中立声明**
> SDD 是方法论，不是某款 AI 工具的专属玩法。本笔记适用于 **Claude Code / Cursor / Windsurf / Cline / Aider / Continue / ChatGPT / Gemini / 国产模型** 以及 **Spec Kit / Kiro 等 SDD 专用工具**。
> 文中代码示例多以 Claude Code 演示（因其 `CLAUDE.md` 自动加载机制最直观），但每条做法都给出了在其他工具里的对应方式（见各章"跨工具实操"小节）。
> **项目级约束文件优先用 `AGENTS.md`（厂商中立）**，`CLAUDE.md` 仅作为 Claude Code 的私有补充（一行 `See AGENTS.md` 级联即可）。

## 为什么要学 SDD

当 LLM 让"写代码"变得越来越便宜，"**把要做的事情说清楚**"反而成了最贵的技能：

- **Vibe Coding** 起步快、半途乱 → 需要约束
- **传统 PRD** 太重、AI 读不懂 → 需要轻量可机读的格式
- **人 + AI 协作** 需要一份共同契约 → spec 应运而生

SDD 给出的答案：**先写规范（spec），再写代码，让人和 AI 共守一份契约。**

## 术语速查

| 术语 | 含义 | 首次出现 |
|------|------|----------|
| **SDD** | Specification-Driven Development，规范驱动开发 | 01 §1 |
| **Vibe Coding** | 无规范、凭感觉让 AI 写代码的风格 | 01 §2.1 |
| **PRD** | Product Requirements Document，产品需求文档；业务方写给研发的传统需求说明书，通常较重、面向人读 | 01 §2.2 |
| **spec** | 规范文档，SDD 的核心产出物；相比 PRD 更细、面向人 + AI + 测试共同消费 | 01 §1 |
| **spec.md** | 物理文件形式的 spec | 04 §2 |
| **AGENTS.md / CLAUDE.md** | 项目级 AI 约束文件（厂商中立 / Claude Code 专用） | 04 §3、14 §2 |
| **AC (Acceptance Criteria)** | 验收标准条目 | 03 §11 |
| **AC-NFR / AC-E2E** | 非功能 AC / 端到端 AC | 09 §1 |
| **ADR (Architecture Decision Record)** | 架构决策记录 | 04 §1 |
| **TBD** | To Be Determined，spec 里的占位符 | 04 §5.1 |
| **MUST / SHOULD / MAY** | RFC 2119 定义的优先级关键词 | 03 §5 |
| **Given-When-Then** | BDD 风格的验收写法 | 03 §11 |
| **非目标 (Non-goal)** | 刻意不做的事，用于防止 scope creep | 02 §2.3 |
| **Brownfield / Greenfield** | 存量改造 / 从零起步 | 10 §1 |
| **三维追溯矩阵** | BR × 模块 × TC 的覆盖关系表，活在 spec 仓 | 09 §1.2 |
| **L1~L4 验收层级** | 单元 / 集成 / 前端 / 端到端，4 层垂直拆分 | 09 §1.1 |
| **Task** | 最小可部署单元（≠ Epic ≠ spec tag） | 15 §2.1 |
| **Expand-Contract** | 破坏性 DDL 拆两步：先加新结构兼容写、再清理旧结构 | 15 §4 |
| **feature flag** | 用配置而非代码控制功能可见性，与环境晋级解耦 | 15 §5 |
| **SIT / UAT / PROD-mirror / PROD** | 标准环境晋级阶梯 | 15 §7.1 |
| **Conventions（9 支柱）** | 跨 feature 的通用约束 spec（fe/db/test/security 等） | 03 §13 |

## 学完后你应该能

- [ ] 用一张表解释 SDD / 传统 PRD / Vibe Coding 的差异
- [ ] 独立起草一份符合 9 模板的 spec，结构完整且可验证
- [ ] 在任意 AI 编程工具（Claude Code / Cursor / Aider / ChatGPT 等）中通过对话驱动 **spec → 代码 → 测试** 全链路
- [ ] 识别并改写不合格的 spec（模糊词、缺验收、无风险评估）
- [ ] 根据项目规模选择 SDD 的复杂度档位
- [ ] 在团队内渐进落地 SDD，不招致反感
- [ ] 面对一个**无 SDD 的存量项目**，能从全栈角度盘点现状、固化契约层、制定 6 个月改造路线图

> 祝你写下的每一行 spec，都能在未来某天帮你省下一次深夜加班。
