# 常见误区与渐进落地

> 版本：1.0 | 定位：避坑清单 + 团队推广节奏

读完前面几章，团队真正想推 SDD 时通常卡在两件事：**自己写出的 spec 不合格**、**同事不愿意配合**。这一章把两类陷阱合在一起讲。

---

## 1. 常见误区

### 1.1 过度设计

症状：

- spec 里写进 "如未来需要可扩展为 X"
- 技术方案章节堆砌 5 种备选、缺少选型结论
- 接口给了 10 个参数，8 个 "未来可能会用"

为何有害：

- 拉长评审时间
- 代码实现时必然半途而废
- 给未来的人误导：以为"扩展位"被设计过了，其实只写了个接口壳

**对治**：

- spec 只记录**本期承诺交付**的内容
- 真有未来构想 → 单独写在 §8 风险评估或外挂 ADR

---

### 1.2 Spec 写得太模糊

5 个信号：

1. 读完不知道**边界**在哪
2. 验收标准数量少于功能项
3. 出现 "等"、"等等"、"诸如此类"
4. 接口只写正常路径、无错误码
5. 非功能需求全是定性词（"快"、"稳"、"好"）

**对治**：套用 03 章 §12 的"替换模糊词"和"反向提问法"过一遍。

---

### 1.3 写了不遵守

表现：spec 是 spec，代码是代码，两者渐行渐远。

根因矩阵：

| 根因 | 对治 |
|------|------|
| spec 粒度太粗，每次实现都"被迫自由发挥" | 补充到接口 + 验收级别 |
| 工具不强制引用 spec | 上 PR 模板、CI 校验 |
| 团队文化仍是"改代码快就行" | 选一个小样板项目做示范、复盘差异 |
| AI 自作主张加功能 | 提示词里加"凡 spec 未覆盖的决策必须反问" |

---

### 1.4 只有 AI 看 spec

表现：只有 AI 在严格读 spec，人类开发者懒得看。

**对治**：

- 把 spec 变成 PR 描述的**必填字段**，reviewer 必须点开看
- 每周 5 分钟 "spec review" 例会快速过最近变更
- 给写得好的 spec 公开点赞，形成正向反馈

---

## 2. 渐进式落地建议

给想引入 SDD 但团队还没准备好的读者：

1. **本周**：给自己的下一个任务写一份 mini-spec，别人看不看都写
2. **下周**：拉一位同事做 peer review，只审 spec 不审代码
3. **2 周后**：在某个试点模块把 spec 作为合入门槛
4. **1 个月后**：写入项目级约束文件（`AGENTS.md` / `CLAUDE.md` 等），让 AI 来督促全员
5. **3 个月后**：复盘哪些 feature 有 spec 质量高、哪些没 spec 踩了坑，数据说话

> 本章讲的是"节奏怎么排"；存量项目的特殊路径见 10 章 Brownfield。

---

## 3. 思考与练习

1. **诊断题**：本章 4 类误区中，你们团队最容易踩的是哪一个？为什么？
2. **推广题**：你负责的模块如果强制每个 PR 都要附 spec，估计会被同事骂吗？为什么？怎么分档让大家能接受？
3. **节奏题**：第 2 节的 5 步节奏，你团队卡在第几步？卡点是工具问题、文化问题、还是某个具体的人？
