# SDD 核心概念与价值

> 版本：1.0 | 定位：基础入门

---

## 1. 什么是 SDD

**SDD（Specification-Driven Development，规范驱动开发）** 是一种以 **"规范文档先行、代码实现后置"** 为核心的开发范式：在动手写代码之前，先用结构化文档把**需求、接口、约束、验收标准**显式固化下来，后续的编码、审查、测试、维护全部围绕这份规范展开。

> 一句话定义：**写代码之前先写"合同"，让所有人（包括 AI）照合同干活。**

### 1.1 和"写文档"的区别

不是所有写文档的过程都叫 SDD。传统文档经常是：

- 代码写完之后才补的"说明书"
- 只给人看、无法驱动开发流程
- 一旦代码变更就失效、再无人维护

SDD 的 spec 则是：

- 代码开始前就必须存在的**契约**
- 同时给**人 + AI + 测试用例**消费
- 作为 Git 一等公民，和代码一起演进

对一位 Java 开发者：可以把 spec 理解为**接口层的 OpenAPI 规范 + 业务层的契约测试 + 架构层的 ADR 合并起来的"超文档"**。

## 2. SDD 要解决的问题

### 2.1 Vibe Coding 的三宗罪

在 LLM 普及后，很多开发者进入一种"**凭感觉编程（Vibe Coding）**"的状态：

```
用户："帮我写一个用户登录接口"
    ↓
AI 直接生成一坨代码
    ↓
用户："哦好像不是我要的，登录要支持手机号"
    ↓
AI 再改
    ↓
用户："还要有失败锁定"
    ↓
AI 再改
    ↓
...（反复打补丁，三天后代码已无法阅读）
```

核心问题：

- **需求漂移**：每一轮对话 AI 都只看到局部，最终实现和最初设想已偏离
- **不可审查**：没有中间产物，PR review 只能看最终代码，无法回溯"为什么这么设计"
- **不可复现**：同一个需求交给 AI 两次，得到两份截然不同的实现

### 2.2 传统开发的另一极

硬写一份 20 页 **PRD（Product Requirements Document，产品需求文档）** 然后发给工程师开发。问题：

- 文档主要给人读，**AI 无法高质量消费**
- 写得越详细，过时得越快
- 工程师不会真的逐条对照，落地质量全凭自觉

SDD 介于二者之间：**比 Vibe 严谨，比传统 PRD 轻量，且能让 AI 直接读。**

## 3. SDD 四大核心原则

| 原则 | 含义 | 对应的动作 |
|------|------|------------|
| **规范先行** | 先出 spec，再出代码。spec 未评审通过不进入编码 | 显式告诉 AI "先输出 spec，现在别写代码" |
| **文档即契约** | spec 是代码、测试、AI 行为共同遵守的单一事实源 | 把验收标准直接翻译为单元测试 |
| **可追溯** | 每个实现决策都能回溯到 spec 的某一条 | PR 描述 / 代码注释引用 spec 章节号 |
| **可验证** | spec 必须能被"机械式"检验是否满足 | 验收标准写成 Given-When-Then |

> 注意：这四条原则都不是"新东西"。DDD 讲契约、TDD 讲可验证、REST 讲接口先行——SDD 做的是把它们整合进**一个**文档和**一套**工作流。

---

## 4. SDD vs 传统开发 vs Vibe Coding

| 维度 | 传统瀑布/PRD | Vibe Coding | **SDD** |
|------|--------------|-------------|---------|
| **起点** | 写 20 页 Word 文档 | 直接一句话让 AI 写 | 生成轻量结构化 `spec.md` |
| **AI 介入** | 几乎不参与早期设计 | 全程主导，无约束 | 协助写 spec + 受 spec 约束 |
| **文档与代码关系** | 文档容易滞后 | 几乎没文档 | spec 与代码同目录、同 PR |
| **可审查性** | 文档可审，代码审难 | 只能审最终代码 | spec 审 + 代码审 两层 |
| **变更成本** | 高（改 PRD 走流程） | 极低但混乱 | 改 spec 可触发代码重生成 |
| **质量上限** | 看团队流程 | 看 AI 当天状态 | 看 spec 质量 |
| **学习曲线** | 中 | 低（但上限也低） | 中（需要学写 spec） |
| **适合规模** | 大型长周期 | 玩具级 demo | 从个人到团队都适用 |

## 5. 适用场景

### 5.1 SDD 收益最大的场景

- **多人协作**：spec 作为共同语言，减少口头沟通成本
- **AI 辅助开发**：给 LLM 明确边界，降低幻觉、减少返工
- **长生命周期项目**：半年后回来改代码，spec 就是团队的长期记忆
- **对外接口 / SDK**：契约必须清晰，否则破坏下游
- **复杂业务逻辑**：如权限、计费、审批流等强状态机场景

### 5.2 什么时候不要用 SDD

- **一次性脚本**：写完就丢的小工具，写 spec 不如直接 vibe
- **探索性原型**：连需求都不明确，强行写 spec 反而固化错的方向
- **UI 微调**：改个按钮颜色不需要 spec
- **极度紧急的线上 fix**：先止血，再补文档

> 判断标准：**spec 的写作成本 < 它将节省的沟通与返工成本**，才值得投入。

---

## 6. SDD 带来的收益

| 收益 | 说明 |
|------|------|
| **降低 AI 幻觉** | 明确边界让 LLM 不再自由发挥 |
| **审查可分层** | 先审 spec（轻量）再审代码（重度），尽早发现设计问题 |
| **知识沉淀** | spec 是比代码注释更高层的团队资产 |
| **降低维护成本** | 新人读 spec 比读代码上手快 10 倍 |
| **测试有据** | 验收标准直接成为测试用例，避免漏测 |
| **可回放** | 同一份 spec 可以在不同技术栈 / 不同 AI 下重新生成实现 |

---

## 7. 常见误解

| 误解 | 澄清 |
|------|------|
| "SDD = 重流程，会拖慢速度" | 简单任务可用极简版 spec，10 行 Markdown 即可 |
| "AI 能写代码就不需要 spec" | 恰恰相反，AI 越强越需要 spec 控制方向 |
| "spec 就是 PRD" | PRD 面向业务方；spec 面向实现侧，粒度更细、更可验证 |
| "先写完整代码再补 spec 也一样" | 补出来的 spec 只是说明书，失去"驱动"的意义 |
| "spec 要一次写到完美" | spec 可以迭代，但每次迭代必须显式升版本号 |

## 8. 示例串讲：从一句话需求到一份 spec

**输入：** "给团队做个命令行工具，能统计每个人最近一周的 Git 提交数。"

❌ **Vibe Coding 的反应**：AI 直接写出一段 `git log | awk` 的脚本。

问题：没人定义"最近一周"是按自然周还是滚动 7 天；没人定义是否排除 merge commit；没人定义输出格式。

✅ **SDD 的反应**：先生成极简 spec 如下：

```markdown
# Git 提交统计 CLI — Spec v0.1

## 1. 需求背景
团队 leader 每周例会前希望一眼看清提交活跃度，当前无工具。

## 2. 业务目标
- 主：一条命令展示指定时间段内每个作者的提交统计
- 非目标：不做跨仓库聚合、不做趋势图

## 3. 功能范围
[MUST] F1 统计最近 N 天（默认 7）每个作者的提交数
[MUST] F2 默认排除 merge commit
[MUST] F3 输出表格：作者 | 提交数 | 变更行数
[SHOULD] F4 支持 --repo 指定仓库路径

## 4. 非功能
- 执行时间 < 2s（1 万提交规模内）
- 仅读仓库，不写入任何文件

## 9. 验收标准
[AC-1] 在本仓库运行 → 至少输出 1 行
[AC-2] --days=1 → 只含 24 小时内提交
[AC-3] 默认不含 merge，加 --include-merges 才含
```

**差异一目了然**：spec 把所有"默认值、边界、验收点"都讲清楚了，接下来 AI 写代码不会跑偏、人看代码也心里有数。

## 9. 思考与练习

1. **对比题**：回忆你最近一次和 AI 协作写代码的经历。如果改用 SDD 模式，会多哪些动作？少哪些？哪些地方会更顺？
2. **判断题**：以下哪些任务适合 SDD？
   - (a) 临时帮运营导一份 Excel
   - (b) 重写公司支付对接 SDK
   - (c) 在博客站加个分享按钮
   - (d) 设计一个内部工单系统

   > 参考：(b)、(d) 强烈适合；(a)、(c) 通常不必。
3. **应用题**：找出你负责模块中最近一个 PR，倒推一份 mini-spec（约 15 行）。写完后回答：这份 spec 能让另一位工程师（或 AI）**不看代码**就复现出等价实现吗？
4. **开放题**：SDD 的原则"可验证"和 TDD 的"测试先行"看起来相似，它们的关系与区别是什么？

   > 提示：TDD 可视为 SDD 的一种"执行方式"—— 验收标准用测试来固化；但 SDD 的 spec 还涵盖 TDD 不覆盖的非功能需求与业务背景。
