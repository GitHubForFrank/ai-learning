# 测试策略：常规 + 压力 + 回归

> 版本：1.0 | 定位：高频问题专题 — "SDD 下三类测试怎么挂？跨层怎么验？MCP 怎么用？"

现实工程里不只有"跑一遍单元测试就完事"。一个 task 通常涉及三类测试：**常规测试（单元 / 集成 / E2E）**、**压力测试**、**回归测试**。SDD 把这三类全都挂到 spec 的 AC 上，让它们共享同一个锚点。

---

## 1. 起点：AC 是所有测试的"源"

SDD 的一个关键改变是：**测试不再是"写完代码再补"的事，而是从 spec 起就已定义**。spec §9 验收标准就是测试用例池的源。推荐给 AC 编号区分测试类型：

| 前缀 | 含义 | 对应测试层级 |
|------|------|-------------|
| `AC-N` | 功能 AC | 单元 / 集成测试 |
| `AC-E2E-N` | 端到端 AC | E2E 测试（Playwright / Cypress / 集成套件） |
| `AC-NFR-N` | 非功能 / 性能 AC | 压力 / 基准测试 |

铁律：**测试方法名必须含 AC 编号**（如 `@DisplayName("AC-NFR-1: 100万行 P95 < 500ms")`），CI 失败报告才能直接定位到 spec 章节。

### 1.1 进阶：L1~L4 验收层级

跨栈 / 多仓项目（见 [07 章 §5 多仓 SDD](./07-SDD-跨栈协作-前端+后端+DB.md)）的 AC 还需要按"垂直层级"再分一刀，确保每条 AC 都明确属于哪一层、由谁负责跑：

| 层级 | 名字 | 谁拥有 | 跑在哪 | TC 编号前缀 |
|---|---|---|---|---|
| L1 | 单元（Unit） | 实现层 owner（前后端各自） | `mvn test` / `npm test` | `TC-U-*` |
| L2 | 集成（Integration） | 后端 owner | testcontainers / 内嵌 H2 | `TC-IT-*` |
| L3 | 前端（FE 单层） | 前端 owner | Vitest / Jest + jsdom | `TC-FE-*` |
| L4 | 端到端（E2E） | spec owner / QA | Playwright / Cypress（真后端 + 真 DB） | `TC-E2E-*` |

**铁律**：每条 AC 必须**至少**对应一层 TC；端到端 AC（`AC-E2E-*`）必须**至少**有 L4，不允许只用 L1+L2 凑数（07 章 §3）。

### 1.2 进阶：三维追溯矩阵 BR × 模块 × TC

仅有 AC ↔ TC 的一对多映射，看不出"哪条业务规则覆盖了哪些模块、哪些模块还有空白"。实战项目（`04-SDD 实战项目/sdd-02-task-fullstack/task-spec/04-验收标准.md`）使用**三维矩阵**作为 spec §9 的活文档结构，把追溯链显式化：

```
                     模块（FE-01 / FE-02 / BE-01 / BE-02 / DB-01 ...）
              ┌────────┬────────┬────────┬────────┬────────┐
BR-01 登录    │ TC-FE-1│ ——     │ TC-IT-1│ ——     │ ——     │
BR-02 验码    │ TC-FE-2│ TC-E2E-1│ TC-IT-2│ TC-IT-3│ TC-U-5 │
BR-03 限流    │ ——     │ ——     │ TC-IT-4│ ——     │ ——     │
BR-04 ...     │  ...   │  ...   │  ...   │  ...   │  ...   │
              └────────┴────────┴────────┴────────┴────────┘
```

- **行**：业务规则编号（BR-* 来自 spec §3 / §4）
- **列**：模块编号（FE-* / BE-* / DB-* 来自 spec §8 模块拆分）
- **单元格**：覆盖该 BR×模块的 TC 编号清单（可多个，可显式 `——` 标记"本期不覆盖"）

矩阵的价值：

- **单元格空白 = 覆盖率漏洞**，review 时一眼看到
- **某 BR 行没有 L4 单元格 = 缺端到端验证**，通常是 spec 写得太抽象
- **某模块列没有任何 TC = 死代码或未实现**，应回 spec 砍掉模块或补实现

矩阵维护责任（多仓场景）：

| 矩阵单元的变化 | 维护者 |
|---|---|
| 新增 BR 行 | spec PR 提交者，同步加行 |
| 模块列填入新 TC 编号 | 对应代码仓 PR 提交者，回到 spec 仓追加（可在 Epic 收尾 PR 中批量） |
| 废弃 TC | spec PR 在矩阵中标删除线，**编号不复用** |

> 矩阵是"活文档"，但**只住在 spec 仓**。代码仓里测试代码的 TC-* 注释是**引用**，不是真源——发生不一致时以 spec 仓为准。

## 2. 常规测试（单元 / 集成 / E2E）

这部分 SDD 只加一条约束：**每条 AC-N / AC-E2E-N 至少对应一个测试**。其余与行业通行做法无异。

关键点：

- AC 粒度要与测试粒度匹配——AC 写得太粗（"登录功能正常"），测试也跟着粗，等于白写
- 集成 AC 要明确依赖（DB / 外部服务），子 spec 里列清楚 testcontainers / mock 选型
- E2E AC 必须端到端（见 07 章 §3）——不能用一堆单元测试凑数

## 3. 压力测试的 SDD 写法

压力测试最常见的失败模式：

- spec §4 只写"快"、"稳定"、"高并发" → 无法机械验证
- 压测独立于 spec → 每次跑都要临时讨论指标，争论不休
- 没有基线 → 压出了数但不知道是改善还是回归

SDD 的改造方案有 3 步：

### 3.1 spec §4 非功能需求必须量化

不合格 → 合格 的改写示例：

| 不合格 | 合格 |
|--------|------|
| "响应要快" | P95 < 500ms，P99 < 1s |
| "支持高并发" | 1000 QPS 持续 10 分钟，错误率 < 0.1% |
| "内存友好" | 峰值堆内存 < 1GB（JVM `-Xmx` 监控） |
| "大数据量下也能跑" | 100 万行 fixture 下上述指标成立 |

### 3.2 §9 里写成可机械验证的 AC-NFR

```
[AC-NFR-1] 100 万行 user 表 fixture 下，查询接口 P95 < 500ms（JMH，3 次平均）
[AC-NFR-2] 1000 QPS 持续 10 分钟，错误率 < 0.1%（Gatling 场景 scenario-login）
[AC-NFR-3] 上述 NFR-2 场景下峰值堆内存 < 1GB
```

每条 AC-NFR 必须四要素齐全：**指标 + fixture 规模 + 工具 + 判定方式**，缺一个都会引发"到底算不算通过"的扯皮。

### 3.3 基线管理

- spec 首次 `[APPROVED]` 时跑一次压测作为基线，归档到 `docs/specs/NNNN/perf-baseline.json`
- 每次 release 前重跑并对比基线，偏差 >10% 必须在 PR 描述中解释
- 基线数据与 spec 同 repo、同版本号管理，不要放外部系统

**何时必须跑压测**：

- spec 涉及热路径（查询、写入、接口）变更
- spec §4 任何指标修改
- release 前
- 线上疑似性能问题排查（用于复现 / 缩小范围）

**何时可以跳过**：

- spec 只改文案、配置、UI 样式
- 纯内部重构且有性能基线回归测试兜底

**工具选型参考**（按技术栈）：

| 栈 | 微基准 | 接口压测 | 数据库 |
|----|--------|----------|--------|
| Java | JMH | Gatling / JMeter | sysbench |
| Node / Python | —— | k6 / Locust | pgbench |
| Go | testing.B | k6 / vegeta | sysbench |

## 4. 回归测试：AC 池的持续管理

核心思路：**老 spec 的 AC 永远是回归测试池的主力来源**。

**AC 池的生命周期**：

| spec 状态 | AC 是否入池 | 说明 |
|----------|-------------|------|
| `[DRAFT]` | 否 | spec 未冻结，AC 会变 |
| `[APPROVED]` / `[RELEASED]` | **是** | 每次 CI 必跑 |
| `[DEPRECATED]` | 移除 | 功能已下线，留着只会变噪音 |
| `[RELEASED] v2.0 破坏性升级` | 精选保留 | v1.0 中向后兼容的 AC 保留、被破坏的显式清理 |

**跨 spec 的回归影响评估**

建议在新 spec 的 §8 风险评估（或单列一节）中写明：

```
## 对老 spec 的回归影响
- spec 0003 的 AC-1 ~ AC-3 必须继续通过（本次变更不应破坏登录主路径）
- spec 0005 的 AC-NFR-2 可能下降 5% 以内，超出需重评审
```

CI 失败时，这份清单能立刻告诉你"是哪个老功能挂了 / 是否预期内"。

**CI 集成建议**：

- **功能回归**（AC-N / AC-E2E-N）：每个 PR 必跑，挂了不能合
- **性能回归**（AC-NFR-N）：每 PR 跑成本太高，推荐 nightly 或 release 前跑
- 测试命名含 AC 编号 → 失败报告直接指向 spec 章节
- 破坏性变更（v2.0）PR 必须在描述中列出"清理的 AC 编号清单"

**常见反模式**：

- **AC 池只增不减**：废弃的 spec 的 AC 还在跑，成为失败噪音
- **性能 AC 与功能 AC 混跑**：功能测试被拖慢，开发者开始跳过本地测试
- **破坏性变更没清理老 AC**：CI 红得莫名其妙，调试者找半天
- **只靠代码覆盖率做回归**：覆盖率高不等于行为正确；AC 才是行为契约

## 5. 跨层回归：前端 + 后端 + DB 的联合验证

一次任务完成后的回归测试，不该停在"前端点一下能跳转"。典型的跨层漏测是 **UI 绿了 / API 返回 200，但数据库被静默写坏**——这种失败最隐蔽、最常见，也是线上事故的主要来源。这一节把 07 章跨栈结构和 §4 回归 AC 池接起来，给出"三层都要验"的具体约定。

### 5.1 三类断言必须齐全

任何一条 `AC-E2E-N` 应当同时涵盖三类断言，缺一条就属于"半吊子 E2E"：

| 断言类型 | 验证什么 | 漏掉的典型后果 |
|---------|---------|----------------|
| **UI 断言** | 前端视觉 / 交互状态 | 只看肉眼效果，内部数据可能已错 |
| **接口断言** | API 响应码 + 响应体关键字段 | 接口绿但 DB 没写 / 重复写 / 写错表 |
| **DB 或副作用断言** | 表记录变化、事件消息、外部调用 | 后端返回成功但状态未持久化 |

**铁律**：AC 在 spec §9 中必须**同时写出三类期望**，不能只写一类。

### 5.2 AC 的跨层"三段式"写法

在 07 章 §3 端到端 AC 基础上扩展：

```
[AC-E2E-N] <用户动作描述>
前端: <UI 应呈现什么状态>
后端: <哪个接口 + 状态码 + 响应体关键字段>
数据: <哪张表 / 哪些字段 / 期望值 / 不应存在的记录>
```

反例 —— **不可机械验证**：

```
[AC-E2E-1] 点击"领取优惠券"按钮后页面提示成功。
```

正例 —— **可机械验证**：

```
[AC-E2E-1] 用户在活动页点击"领取优惠券"
前端: 弹窗"领取成功",按钮置灰
后端: POST /coupons/claim 返回 200,body.couponId 非空
数据: user_coupon 表新增 1 条 (user_id=X, coupon_id=Y, status='UNUSED')
       activity_stock 表 remaining 字段减 1
       coupon_event 表新增 1 条 type='CLAIM'
```

后者每一行都能直接翻译成断言代码；前者永远写不出一个稳定的测试。

### 5.3 DB 断言的四种实现模式

| 模式 | 适用场景 | 优点 | 坑 |
|------|---------|------|----|
| **测试内查询 + 逐字段断言** | 常规功能 | 直观、易写 | 字段多时代码冗长 |
| **行级快照对比** | 复杂写入链路（多表） | 一次性覆盖所有字段 | 需 mask 时间戳 / 随机 ID |
| **Golden Master / DB Dump 对比** | 报表、ETL、数据迁移 | 能发现意外字段变化 | 易被"看似合理的小 diff"淹没 |
| **事件 / Outbox 断言** | CQRS / 事件驱动架构 | 验证副作用链路完整 | 需要独立事件 fixture |

选哪种取决于 spec §6 数据模型的复杂度，不是工具偏好。单表写入用第 1 种，跨 3 张以上表考虑第 2/3 种。

### 5.4 测试 DB 的选型与隔离

回归测试的 DB 必须满足三条，缺一条就会在某个灰暗的 Friday 下午咬你：

1. **真实数据库引擎**（不是 H2 / 内存模拟）——否则 SQL 方言差异会掩盖线上 bug
2. **每个测试可隔离**（事务回滚 / 独立容器 / 独立 schema）
3. **schema 从迁移脚本加载**（不是手写建表）——和生产同源

主流隔离方式（按强度）：

- **事务回滚**（最快）——单连接 rollback；不适合多连接 / 异步场景
- **Testcontainers 每套测试一个容器**（推荐常规）——进程级隔离，慢但稳
- **每测试独立 schema**（折中）——速度和隔离兼顾
- **固定共享库 + 清理脚本**（不推荐）——测试间污染是噩梦的起点

**铁律**：回归测试**禁止连接任何共享环境**（dev / staging）**的数据库**。共享库的不确定状态会让失败报告变成谜语。

### 5.5 Fixture 数据的来源

| 来源 | 适用 | 注意 |
|------|-----|------|
| **迁移脚本 + 工厂代码**（推荐） | 大多数场景 | 工厂与业务实体绑定，版本一致 |
| **SQL dump 加载** | 接近真实分布的复杂场景 | dump 必须脱敏 + 入库前 schema diff |
| **API 预热**（前置 case 写入） | 流程串联、幂等验证 | 依赖顺序，失败时排查复杂 |

**禁用**：直接从生产库导数据作 fixture——合规、脱敏、体积三重风险。

### 5.6 一个场景的完整回归示意

以"用户修改邮箱"为例，跨三层的回归应包含：

```
前端层（Playwright / Cypress）
  └─ 打开设置页 → 改邮箱 → 点提交 → 断言 toast "修改成功"

接口层（集成测试 / API 测试）
  └─ POST /users/me/email 返回 200
     + 响应 body.email 为新值
     + 旧 access token 已失效（下一次调用 401）

数据层（DB 断言）
  └─ user 表 email = 新值, updated_at ≈ now
     user_email_history 表新增 1 行（old, new, changed_at）
     未产生任何孤儿 session 记录

事件层（若有 MQ）
  └─ user.email_changed 事件发布一次, payload 与 contracts/events.json 一致
```

**任意一层挂，都算回归失败**。"前端绿 + 后端绿 + DB 红"是最隐蔽的线上事故起点，而这种事故靠只跑 UI 自动化是测不出来的。

## 6. MCP 在 SDD 测试中的角色与边界

经常被问：**"要不要上 MCP 让 AI 直接查 DB 做回归？"** 直接回答：**MCP 是开发期辅助工具，不是回归测试本身**。混淆两者是很多团队上 MCP 后反而让测试质量下降的根因。

### 6.1 MCP 能做什么 vs 不能做什么

| 能做（开发 / 调试期） | 不能做（CI / 回归期） |
|-----|-----|
| 让 Claude 直接查测试库，验证"刚写入的数据对不对" | 替代 CI 自动化测试——AI 不稳定、不可重放 |
| 失败用例排查："哪个字段和 AC 预期不符" | 当作回归断言源——AI 没主动查的字段不会被告警 |
| 评审 PR 时对照 schema 变更与 AC | 替代 testcontainers / fixture 体系 |
| 让 AI 读迁移历史辅助写下一份迁移脚本 | 对生产库做任何事 |

**核心认知**：

- **回归测试的事实源是测试框架 + 断言代码**——版本化、可重放、CI 能跑
- **MCP 是 AI 的眼睛**——减少人在 DB GUI、日志平台、代码之间来回切换的摩擦

一个常见的本末倒置：团队把"让 Claude 通过 MCP 查一下数据是否写对"当作验收动作，于是不写测试代码。这种"验收"**下一轮 AI 会话就忘光**，回归归零。

### 6.2 什么时候值得接入 DB MCP

按 ROI 从高到低：

1. **测试失败时的快速定位**——让 AI 对比"当前表状态"与 "AC 期望"，秒级指向错行 / 错字段
2. **Spec 考古**（10 章 §5）——让 AI 读 schema + 采样数据，辅助反推字段语义
3. **评审迁移脚本**——让 AI 读当前 schema + 新迁移，识别破坏性变更
4. **压测结果解读**——让 AI 查 slow query log、索引使用情况
5. **新人 onboarding**——让 AI 结合 spec + 真实数据做讲解

无以下诉求时，DB MCP 的 ROI 一般：

- 项目数据模型简单、表少
- 团队已有成熟的 DB GUI 工具使用习惯
- 对数据合规要求高，接入审批成本 > 收益

### 6.3 DB MCP 接入的安全约定（项目级约束文件必写）

一旦决定上 DB MCP，以下条目**必须**写入项目级约束文件（`AGENTS.md` / `CLAUDE.md` / `.cursorrules` 等），否则就是给自己埋雷：

- **只连测试 / 本地库，严禁连生产或共享 staging**
- **默认只读**（read-only user）；写权限仅限特定测试脚手架场景，必须在约束文件中显式列白名单
- **禁止 `DROP` / `TRUNCATE` / 无 WHERE 的 `DELETE` / `UPDATE`**——在 MCP server 层白名单，**不是靠提示词**
- **敏感字段**（手机号 / 邮箱 / 身份证等）**必须先脱敏再进上下文**
- **不得把 MCP 查询结果写回 spec 作为事实源**——spec 的事实源是 `contracts/schema.sql`，不是某次查询的快照
- **每次 session 只查当前任务相关的表**，避免 AI "漫游式" dump 全库

**铁律**：**"AI 能查到"不等于"AI 应该查到"**。最小权限原则在 MCP 接入上比一般服务更重要——AI 可能在未被明确授权的情况下扩展查询。

### 6.4 MCP 在回归流程里的正确位置

一次回归的典型工作流：

```
1. 改动触发 CI
2. CI 跑自动化测试（单元 / 集成 / E2E / AC-NFR）    ← 真正的回归
3. 结果：绿  → 合入
        红  → 失败报告自动进 PR
4. 排查阶段（开发或 AI 介入）：
   ↓
   此时 MCP 出场：让 AI 读失败报告 + 查测试库真实状态 + 对照 AC 预期 → 定位根因
5. 修复 → 回到步骤 1
```

MCP 应出现在**步骤 4**，不是步骤 2。弄反了就是用 AI 代替 CI——目前不该这么干。

### 6.5 没有 MCP 也能做好跨层回归

诚实讲：MCP 对**已经把跨层 AC 和测试写好的团队**是效率放大器；对**还没写好测试的团队**是遮羞布。推荐次序：

```
1. 先把 §5 跨层 AC 约定落到 spec 模板里
2. 先把 testcontainers / fixture 体系搭起来，DB 断言写得顺
3. 先让 CI 能稳定跑三层断言
4. 再考虑接入 DB MCP 做开发期加速
```

顺序反了——先上 MCP 期待它"兜底"，通常会得到一个光鲜的 demo + 一个腐烂的测试套件。

---

## 7. 测试金字塔对照 AC 类型

| 层级 | 对应 AC | 代表工具 | 跑的频率 | 通常占比 |
|------|---------|---------|----------|----------|
| 单元 | AC-N（大多数） | JUnit / pytest / go test | 每次 PR | 70% |
| 集成 | AC-N（涉及 IO / DB） | JUnit + testcontainers | 每次 PR | 20% |
| E2E | AC-E2E-N | Playwright / Cypress | 每 PR（快）+ nightly（慢） | 8% |
| 性能 | AC-NFR-N | JMH / Gatling / k6 | nightly + release 前 | 2% |

占比不是硬指标，但跨层级失衡（如 E2E 占比超过 30%）通常意味着**下层 AC 没写清楚，被迫用 E2E 兜底**——应该回去补细化。

---

## 8. Spec 模板的扩展建议

如果你的场景对测试要求高，可以把 03 章 spec 模板的 §9 验收标准细化为 4 小节：

- **§9.1 功能 AC（AC-N）**——所有 spec 必填
- **§9.2 性能 AC（AC-NFR-N）**——热路径 / 性能敏感场景必填，要含指标 + fixture + 工具
- **§9.3 端到端 AC（AC-E2E-N）**——跨栈场景必填（见 07 章 §3）
- **§9.4 回归影响评估**——会动老代码 / 老功能时必填，列出要保持通过的老 AC 编号

纯逻辑小 feature §9.1 就够；上面 4 小节按场景按需启用，别强制所有 spec 都写全。

---

## 9. 诚实结论

SDD 能**约束"AC ↔ 测试"的一一对应关系**，让三类测试共享同一锚点。但它：

- 不替代你做容量规划（多少 QPS 才算够，SDD 决定不了）
- 不替代你选压测工具和 fixture 策略
- 不自动生成性能基线数据
- 不保证你的 CI 跑得动所有 AC（大项目必须分层调度）
- **不接受 MCP 代替 CI 做回归**——MCP 是开发期加速器，不是事实源

SDD 带来的最大改变不是"解决测试问题"，而是让**功能 / 性能 / 回归三类测试不再散落在不同人的脑子里**，全部收拢到 spec §9 里统一管理、统一版本、统一追溯。
