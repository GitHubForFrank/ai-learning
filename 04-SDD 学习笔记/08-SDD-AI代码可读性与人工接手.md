# AI 代码可读性与人工接手

> 版本：1.0 | 定位：高频问题专题 — "AI 写的代码人能看懂吗？"

这是一个**高频被问、但 SDD 单独解决不了**的问题。诚实面对它比给出过度乐观的答案更负责任。

---

## 1. AI 代码的"弯弯绕绕"从哪来

常见失真模式：

- **过度防御**：到处 try/catch、空指针检查、冗余 validation，明明是内部调用
- **过度抽象**：20 行就能写完的逻辑被抽成三层接口 + 工厂 + 策略
- **风格漂移**：同项目不同会话生成的代码命名 / 结构不一致
- **长函数**：把业务逻辑、IO、日志、兜底全塞一个方法
- **隐式依赖**：用了某库的冷门 API，没有注释说明为什么
- **"正确但难读"**：能跑、能过测试，但读起来像谜题

这些问题 **spec 管不到**——spec 约束"做什么"，不约束"怎么实现"。

---

## 2. SDD 能帮到的 vs 帮不到的

| 维度 | SDD 能不能帮 | 备注 |
|------|-------------|------|
| **理解"为什么有这段代码"** | ✅ 能 | 读 spec §1–3 即可 |
| **理解"这段代码的预期边界"** | ✅ 能 | 读 AC 对应测试 |
| **理解"为什么选这个技术"** | ✅ 能 | 读 ADR / spec §7 |
| **命名一致性** | ❌ 不能 | 要 linter + AGENTS.md / CLAUDE.md |
| **控制过度抽象 / 过度防御** | ❌ 不能 | 要 AGENTS.md / CLAUDE.md + review |
| **函数长度 / 嵌套深度** | ❌ 不能 | 要 linter + review |
| **项目内代码复用** | ❌ 不能 | 要 review + 示例模式 |

**误区**：很多团队以为写了 spec 就解决了可读性。实际 SDD 只解决"需求层可读性"，**实现层可读性要靠其他机制**。

---

## 3. 保障 AI 代码可读性的 5 层防线

| 层 | 机制 | 解决什么 |
|---|------|----------|
| 1 | **AGENTS.md / CLAUDE.md 编码约束** | 命名规范、禁用过度抽象、文件长度上限 |
| 2 | **Linter / Formatter** | 机械化统一缩进、命名、import 顺序 |
| 3 | **Spec + AC 测试** | 需求层可读性兜底，永远能回到"要做什么" |
| 4 | **人工 Code Review** | SDD 不免除 review，AI 代码尤其需要人审 |
| 5 | **ADR / 决策注释** | 记录关键"为什么"，防未来踩已踩过的坑 |

**漏掉其中任何一层，AI 代码都会变成"新人接手噩梦"**。很多团队只做第 3 层就以为够了，这是典型误区。

---

## 4. 项目级约束文件必须写死的几条（针对可读性）

下列条目直接决定 AI 代码的可读性，必须在项目级约束文件（`AGENTS.md` / `CLAUDE.md` / `.cursorrules` 等）顶部写死（否则每次会话都要口头重申，成本高得多）：

- 不要为假想的未来需求做抽象；三行相似代码 > 一个过早的抽象
- 不要在内部代码加防御性校验；只在系统边界（用户输入、外部 API）校验
- 函数超过 N 行或嵌套超过 M 层时，必须停下来与我讨论拆分
- 命名沿用项目已有模式（附指向示例文件的路径）
- 新增依赖前必须问
- 注释只写"为什么"，不写"做什么"（代码自身表达"做什么"）
- 不改与当前任务无关的代码（no drive-by refactor）

---

## 5. 新人接手 AI 代码的正确阅读顺序

**严禁上来就读代码**。推荐顺序：

```
1. 读 spec 主文档        → 理解"这个功能要做什么"
2. 读 AC 对应测试用例    → 理解"功能的预期行为边界"
3. 读 ADR（若有）        → 理解"关键决策的为什么"
4. 再读代码              → 代码变成 spec + AC 的"实现注脚"
```

这个顺序让"弯弯绕绕"问题被大幅缓解——读者带着**已知的意图**去读代码，不再需要从代码反推意图。

配套动作：

- spec 末尾列出"实现文件清单 + 对应章节"，充当导航索引
- 代码注释引用 `spec NNNN §X.Y`，可一键跳回 spec
- PR 描述写明 `Implements spec NNNN v1.0`

---

## 6. 什么时候必须让人重写 AI 代码

出现以下信号时，不要修补，**直接让人重写或让 AI 彻底重写**：

| 信号 | 根因 | 处理 |
|------|------|------|
| 同一需求让 AI 改 ≥3 次还跑偏 | spec 不够清晰 | 回去改 spec，不要继续让 AI 猜 |
| 代码过测试但读起来像谜题 | 测试覆盖不充分 / 实现过于聪明 | 人来理清逻辑、改写实现、补测试 |
| 新人连 spec + 代码都读不懂 | 实现方案本身有问题 | 重做，并反思 spec 是否也要改 |
| AI 代码有大量不必要抽象 / 防御 | AGENTS.md / CLAUDE.md 没写死约束 | 先补 AGENTS.md / CLAUDE.md，再让 AI 重生成 |

---

## 7. 诚实结论

**SDD 不能完全解决 AI 代码可读性问题，但它提供了一张"可回退的安全网"**：

- 如果代码难懂，至少还能读 spec
- 如果 spec 和代码冲突，spec 优先、代码重写（见 04 章 Q3）
- 如果连 spec 都没有，就彻底回到"代码考古"状态

SDD 最大的价值不是让 AI 写出干净代码，而是让**未来的维护者永远有一份高于代码层的"原始意图"可参照**。干净代码要靠 **AGENTS.md / CLAUDE.md + linter + review** 的组合拳，SDD 是其中一块，不是全部。

---

## 8. 思考与练习

1. **接手题**：找一段 AI 写的、3 个月前的代码（不是你写的最好），按 §5 的顺序读一遍。哪一步缺失了（无 spec？无 AC？无 ADR？）？补这一步要多少时间？
2. **审查题**：对照 §1 的 6 种失真模式，扫一遍最近一次 AI 生成的 PR，命中几条？哪条 AGENTS.md / CLAUDE.md 条目本可以避免？
3. **改进题**：在你团队的 `AGENTS.md / CLAUDE.md` 里挑一条没写、但 §4 列出的可读性硬约束。加上去之后让 AI 重新生成同一段代码，对比可读性差异。
