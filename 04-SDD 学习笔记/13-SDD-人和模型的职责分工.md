# 人和模型的职责分工

> 版本：1.0 | 定位：高频问题专题 — "SDD 下哪些事人必须做？哪些可以放心交给 AI？边界在哪？"

> 与 12 章的关系：12 章讲**模型 ↔ 模型**的档位分工（用哪档模型）；本章讲**人 ↔ 模型**的角色分工（谁来做什么）。两章正交，可独立读。

AI 越强，团队越容易出现两种极端反应：

- **过度代理**："反正 AI 写得也行，我不细看了" → spec 漂移、契约失守、责任真空
- **过度保留**："AI 写的我都不信，全部我自己来" → 失去 SDD 收益，回到 Vibe Coding 之前的低效

SDD 的人机协作哲学只有一句话：**决策权永远在人，执行权可以下放给 AI；中间的灰色地带按"是否能客观验证"来判**。

---

## 1. 一个核心原则：决策权 vs 执行权分离

| 类别 | 例子 | 谁做 |
|------|------|------|
| **决策权** | 选哪种技术栈、是否上线、AC 算不算通过、是否破坏性变更 | **永远是人** |
| **执行权** | 把决策翻译成 spec / 代码 / 测试 / 文档 | **可以下放 AI** |
| **决策的输入** | 调研方案、列出 trade-off、模拟反例 | **AI 可代劳，人审用** |

**铁律**：AI 可以**为决策提供材料**（"以下是三种方案各自的优劣"），但**不能替人拍板**。任何"我没想法你看着办"的指令，都会让责任无主。

---

## 2. SDD 六阶段的人机分工矩阵

| 阶段 | 人必做 | AI 主导 | 人 + AI 协作 | AI 绝对禁区 |
|------|--------|---------|-------------|------------|
| **1. 需求分析** | 提原始需求、回答澄清问、拍板 | 列澄清问题、识别歧义、生成需求摘要 | 反例验证、边界探查 | ❌ 替用户做需求决策 |
| **2. Spec 生成** | 审 spec 是否反映本意 | 按模板填字段、生成 AC 草案 | 风险评估、技术选型对比 | ❌ 加用户没同意的功能 |
| **3. Spec 评审** | 最终 `[APPROVED]` | 扮演杠精 / 安全 / 新人独立审、汇总意见 | 修订冲突意见 | ❌ 自我审批 spec |
| **4. 实现编码** | 审 PR、reject 不合理实现 | 按 spec 翻译为代码、按 `AGENTS.md / CLAUDE.md` 控风格 | 复杂模块拆分讨论 | ❌ 改 spec 未覆盖的代码 |
| **5. 验证测试** | 验收 AC 真的反映期望 | 写测试代码、跑测试、报失败 | 性能基线诊断 | ❌ 改 AC 让测试通过 |
| **6. 归档沉淀** | confirm 版本号、release | 写 CHANGELOG、更新 spec 索引 | 跨 spec 影响梳理 | ❌ 擅改老 spec 状态 |

**速记**：**人定方向 + 把关，AI 出力 + 复核**。

---

## 3. 各阶段详解

### 3.1 需求分析

**人必做**：

- 把原始需求"说出口"——AI 不会无中生有
- 回答 AI 的澄清问题（每个 yes/no 都是决策）
- 拍板 trade-off（如"作者按 email 还是 name"——AI 可以列利弊，不能替你选）

**AI 主导**：

- 主动列出**用户没明说但必须澄清**的 5–10 个问题
- 用反例 / 边界场景检验用户的初版表述
- 把模糊词（"快"、"稳"、"差不多"）逐个落地为可量化的描述
- 生成"需求摘要"作为下一阶段的输入

**典型反模式**：用户说"AI 你看着办" → AI 顺着自己的训练偏见做决策 → 后续返工根因不可追溯。

---

### 3.2 Spec 生成

**人必做**：

- 通读生成的 spec，确认每一条都符合自己想要的行为
- 标记 `[DRAFT]` → `[APPROVED]` 是人的动作，不是 AI 的

**AI 主导**：

- 按 9 模块模板填空
- 生成 AC 草案（人再裁剪）
- 用 RFC 2119 关键词（MUST / SHOULD / MAY）标注优先级

**灰色协作**：

- §7 技术方案：AI 列候选 + 利弊，人选定
- §8 风险评估：AI 头脑风暴风险，人筛选并评估可能性

**AI 绝对禁区**：

- 在 spec 里加用户没同意的功能（"我顺便加了个 X，应该有用"）
- 把"未来可能用到"写进 §3 功能范围
- 自行升降功能的优先级

---

### 3.3 Spec 评审

**人必做**：

- 最终签字 `[APPROVED]`
- 接受或驳回 AI 提出的修改意见
- 决定哪些意见进入 spec、哪些归档为 ADR、哪些丢弃

**AI 主导**：

- 扮演**杠精 / 安全专家 / 新人** 三个独立角色审 spec（详见 12 章 §4.3）
- 汇总意见、去重、按严重度排序
- 生成评审报告

**铁律**：

- 评审会话**必须独立于生成会话**（最好换模型）
- AI 提出的意见**不直接回写 spec**，必须经人确认
- "AI 都说没问题" ≠ "spec 通过"——人 review 不可省

---

### 3.4 实现编码

**人必做**：

- 审 PR：代码是否真的实现了 spec、是否引入 spec 外行为
- reject 不合理实现并讲清楚原因
- 处理 AI 卡住的疑难（连续 ≥3 次跑偏先怀疑 spec，详见 12 章 §4.4）

**AI 主导**：

- 按 spec 写代码（每个类 / 函数引用 spec 章节号）
- 按 `AGENTS.md / CLAUDE.md` 控制代码风格
- 主动给测试方法名加 AC 编号（`@DisplayName("AC-3: ...")`）

**AI 绝对禁区**：

- **No drive-by refactor**：不改与当前任务无关的代码
- 不在 spec 未覆盖处自由发挥（必须反问）
- 不擅自加新依赖（必须先问）
- 不"为了让测试通过"修改 AC

---

### 3.5 验证测试

**人必做**：

- 验收：每条 AC 是否真的反映了你想要的行为（AI 写的测试可能"对着错的 AC 测对了"）
- 性能基线的最终判定（"5% 偏差能不能接受"是 trade-off）
- 决定破坏性变更要清理哪些老 AC

**AI 主导**：

- 把每条 AC 翻译成测试代码
- 跑测试、汇报失败
- 排查失败时的现场（结合日志、DB 状态等）

**灰色协作**：

- 跨层 E2E 断言设计（AI 起草三段式断言，人 confirm 期望值）
- 性能基线诊断（AI 查 slow query、人结论）

**AI 绝对禁区**：

- **改 AC 让测试通过**——这是 SDD 最严重的违规
- 用更宽松的断言"绕过"失败
- 偷偷 skip / disable 失败用例

---

### 3.6 归档沉淀

**人必做**：

- confirm 版本号（`v0.x` → `v1.0` 是决策）
- release 审批

**AI 主导**：

- 写 CHANGELOG
- 更新 `docs/specs/README.md` 索引
- 把 spec 状态从 `[APPROVED]` 改 `[RELEASED]`（在人确认后由 AI 执行）

**AI 绝对禁区**：

- 擅自修改老 spec 的 `[APPROVED]` / `[RELEASED]` 状态
- 把多个 spec 合并归档（每个 spec 独立追溯）

## 4. 决策树：这件事该谁做？

```
是否涉及 trade-off？─── 是 ──→ 人（AI 可列材料）
        │否
        ▼
是否需要"承担后果"？─── 是 ──→ 人（commit author、reviewer、approver）
        │否
        ▼
是否模式化重复？─── 是 ──→ AI 主导
        │否
        ▼
能否被客观验证（测试 / linter / schema）？─── 是 ──→ AI 主导，人复核
        │否
        ▼
默认：人主导，AI 辅助
```

## 5. 人必须保留的职责（不可代理清单）

| 类别 | 具体动作 | 为什么不能代理 |
|------|---------|---------------|
| **业务决策** | 选 scope、定优先级、做 trade-off | AI 不承担业务后果 |
| **验收** | spec `[APPROVED]`、release | 责任要落到具体的人 |
| **Code Review** | 审 PR、批准合入 | review 的价值 = 第二个有责任的脑子 |
| **契约变更** | API / DB / 事件契约的发起与审批 | 跨人 / 跨团队约束，单方面改是协作灾难 |
| **安全 review** | 权限 / 注入 / 数据泄漏的最终判定 | 安全错误难自动发现 |
| **与人沟通** | 和产品 / 客户 / 团队的对话 | AI 不能代你承诺 |
| **责任承担** | 故障复盘、合规问询、客户解释 | 法律和组织层面只认人 |

**铁律**：上面任意一条**让 AI 替你做了**，出事时责任真空——这是 SDD 比 Vibe Coding 还危险的姿势。

---

## 6. AI 可以独立完成的职责（可委托清单）

| 类别 | 例子 | 风险控制 |
|------|------|---------|
| **模板化生成** | 按 9 模块填 spec、AC 草案、测试骨架 | 人通读一遍即可 |
| **模式重构** | rename、import 整理、提取常量 | linter + 人 review diff |
| **摘要 / 索引** | CHANGELOG、PR 描述、spec 索引更新 | 自动化 + 抽查 |
| **代码翻译** | spec → 代码、AC → 测试、契约 → DTO | 测试覆盖 + review |
| **一致性检查** | spec ↔ 代码漂移、契约 ↔ 实现差异 | 检查报告进 PR |
| **失败定位** | 测试失败时对比期望与实际 | 人 confirm 根因 |
| **资料准备** | 调研技术选型、列出 trade-off | 人做最终决策 |

## 7. 灰色地带：必须人 + AI 协作的事

下列事情**让人单干慢、让 AI 单干错**，必须协作：

| 任务 | AI 做什么 | 人做什么 |
|------|----------|----------|
| **跨栈实现** | 三端各自的代码、契约同步检查 | 协调三端节奏、决定契约谁先动 |
| **Brownfield 考古** | 读代码、读 PR、列 WHAT | 补 WHY、判定哪些是有意设计 vs 历史包袱 |
| **性能基线诊断** | 查 slow query、采样、聚合数据 | 决定是优化还是改容量规划 |
| **ADR 撰写** | 起草、列候选、列利弊 | 拍板、签名、负责 |
| **架构选型** | 多模型并行出方案（详见 12 章 §5.4） | 裁决、记录、长期负责 |
| **Spec 大重构** | 重组结构、提示影响范围 | 决定保留 / 废弃 / 迁移 |

**协作铁律**：交付物的**作者署名是人**，AI 是工具——这关系到事故复盘时的责任落地。

## 8. 反模式：两类极端 + 一类边界模糊

### 8.1 人过度代理给 AI

| 症状 | 后果 | 对治 |
|------|------|------|
| "我没主意，你看着办" | AI 顺着训练偏见做决策，方向跑偏 | 强制人先列 trade-off 再让 AI 补充 |
| AI 一次写完整 PR 不审就合 | 引入 spec 外功能、隐藏 bug | PR diff 强制人逐文件 review |
| AI 评审通过 = 不再人审 | 自我蒙蔽，AI 见不到自己的盲区 | 人 review 不可省，独立于 AI 评审 |
| AI 写完测试就信覆盖率 | "对着错的 AC 测对了" | 人验收 AC 与意图的一致性 |
| 让 AI 改 AC 来通过测试 | SDD 最严重违规——契约被悄悄改 | `AGENTS.md / CLAUDE.md` 写死："AC 只读，不许改" |

---

### 8.2 人过度保留

| 症状 | 后果 | 对治 |
|------|------|------|
| 凡事先自己做完再让 AI "检查" | 失去 SDD 大半收益，AI 沦为 IDE 补全 | 把 AI 放进生成环节而非验证环节 |
| AI 给的方案全部推翻自己重写 | 浪费 token + 人力，团队疲惫 | 接受 AI 70% 输出 + 修订 30%，比 100% 自写快 |
| 不信 AI 写的测试，全部手写 | 测试覆盖率低、回归薄弱 | 让 AI 写、人审，比人手写更高产 |
| 把 AI 当 IDE 自动补全 | 完全没用上 SDD 的对话式优势 | 学 04 章的提示词模板，按阶段用 AI |

---

### 8.3 边界模糊

| 症状 | 风险 | 对治 |
|------|------|------|
| spec `[APPROVED]` 不明确是谁批的 | 出事追溯不到决策者 | spec 顶部必填 `Approved-by:` |
| AI 改了 spec 但没显式说 | 团队拿着新 spec 干，原作者还以为是旧版 | spec 改动走独立 PR + 显式 commit |
| 不知道这次 PR 是 AI 写的还是人写的 | review 强度无法分级（AI 代码要更严） | PR 模板必填"AI 参与度" |
| 会话历史里 AI 说"我同意你的判断" | 给人造成"已审过"的错觉，实则没人审 | 区分 AI 复述 ≠ 人 approve |

---

## 9. 人机协作的 5 条铁律

1. **决策必有人**——任何 trade-off、scope 变更、`[APPROVED]` 都要落到具体的人头上
2. **AI 不能审 AI 自己写的代码**——评审独立会话、最好换模型，详见 12 章 §4.3
3. **AC 是人定的契约**——AI 永远不能改 AC 让测试通过；只能改代码，或回去找人改 AC
4. **契约变更必须人发起**——OpenAPI / schema.sql / events.json 的修改 PR 由人提出，AI 不主动改
5. **责任落到人头上**——commit author、PR reviewer、release approver 都是人；AI 是工具，不是责任主体

---

## 10. 人机协作的最小工程化约定

下面这套约定能把上面的原则落到 PR / Git / 工具层，建议写入 `AGENTS.md / CLAUDE.md`：

- spec 顶部必填字段：`Author:`、`Approved-by:`（必须是人名 / 邮箱）
- PR 模板必填："AI 参与度"（None / Co-author / Primary author）
- Commit message 区分：纯 AI 改动用 `Co-Authored-By:` 标明
- AI 代码 PR review 标准 ≥ 人代码（强度 1.5 倍：必看测试、必看 spec ↔ 代码引用）
- spec 状态变更（`[APPROVED]`、`[RELEASED]`、`[DEPRECATED]`）必须独立 commit、人发起
- 契约文件改动**禁止**与实现改动同 PR——契约 PR 由人提，先合入

## 11. 思考与练习

1. **审计题**：扫一遍最近 5 个 PR，标注每个的"AI 参与度"。哪些应该让 AI 做更多？哪些 AI 做过头了？
2. **追责题**：你团队最近一次线上故障，事后追溯能精确指向"谁做了哪个决策"吗？还是有"AI 那时建议这么改的"这种模糊归因？
3. **边界题**：§5 不可代理清单里，你团队哪一条最容易被悄悄越界？典型场景是什么？
4. **重分配题**：选一个你正在做的 task，按本章矩阵把动作拆成"人 / AI / 协作"三栏。当前实际分工和理想分工差几条？
5. **铁律题**：§9 的 5 条铁律里，哪一条对你团队最难落地？阻力来自工具、流程、还是文化？
