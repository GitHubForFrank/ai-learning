# 模型职责分工与边界点

> 版本：1.0 | 定位：高频问题专题 — "SDD 各阶段该用哪档模型？什么时候升降档？"

SDD 六阶段对模型能力的要求差异极大：澄清需要"杠精脑"、写代码需要"模式匹配脑"、归档需要"摘要脑"。**用一个模型贯穿全程，要么贵得离谱、要么质量打折**。这一章给出按阶段、按职责的模型选择框架，以及跨档切换的边界点。

> 本章对模型只按"能力档"分类，不绑定具体厂商。出现的具体模型名只是示例，核心是**能力 → 阶段 → 模型档**的映射逻辑。

---

## 1. 为什么必须做模型分工

| 反例 | 代价 |
|------|------|
| 全程用最贵的重推理模型（Opus / GPT-5 / Gemini Pro） | 单 task 成本翻 5–10 倍；需求澄清、改文案、跑 lint 全在烧钱 |
| 全程用最便宜的快速模型（Haiku / nano / Flash） | spec 出现幻觉、评审抓不到歧义、跨栈协作失控 |
| 中途换模型但不交接 context | 新模型重新猜意图，等于把前面投入的澄清 token 全废掉 |
| 把同一会话用到底不切档 | 上下文越滚越脏，重推理模型也救不回来（详见 06 章 §7） |

**核心原则**：**让最贵的模型只做最贵的事**。决策、对抗性评审、跨栈推理留给重推理档；模式化生成、样板代码、摘要交给平衡 / 快速档。

## 2. 三档模型能力定位（厂商无关）

| 档位 | 典型代表（仅示例） | 核心能力 | 单价 / 速度 | 在 SDD 中的角色 |
|------|---------|---------|-------------|----------------|
| **重推理档** | Claude Opus 4.7 / GPT-5 / Gemini 2.5 Pro / DeepSeek-R1 | 长链推理、对抗性挑刺、跨上下文一致性、复杂决策 | 最贵 / 最慢 | "架构师 + 杠精" |
| **平衡档** | Claude Sonnet 4.6 / GPT-5-mini / Gemini Flash | 结构化生成、模式匹配、按模板填空、批量代码 | 中等 | "主力工程师" |
| **快速档** | Claude Haiku 4.5 / GPT-5-nano / Gemini Flash-Lite | 摘要、分类、抽取、单步指令、格式转换 | 最便宜 / 最快 | "实习生 + 文档秘书" |

**判别标准**：

- 一个 task 需要**同时持有 ≥5 个相互依赖的事实**才能给出正确答案 → 重推理档
- 输入输出都是结构化模板，决策点 ≤3 个 → 平衡档
- 单步、确定性、无歧义 → 快速档

## 3. SDD 六阶段 × 模型档位映射

| 阶段 | 主导能力 | 推荐档位 | 不该用什么 |
|------|---------|---------|-----------|
| **1. 需求分析** | 识别歧义、刻意反问、补全边界 | 重推理档 | ❌ 快速档（会跳过歧义直接顺着用户讲） |
| **2. Spec 生成** | 模板填空、领域分类、结构化 | **平衡档** | ❌ 重推理档（杀鸡用牛刀，除非合规 / 跨栈） |
| **3. Spec 评审** | 对抗性挑刺、多视角 | 重推理档 | ❌ 同会话同模型自审（详见 §5） |
| **4. 实现编码** | 模式匹配、按 spec 翻译 | **平衡档** | ❌ 复杂算法 / 分布式协议时用快速档 |
| **5. 验证测试** | AC ↔ 用例映射、边界穷举 | 平衡档 | ❌ 性能基线诊断时仍用平衡档（升重推理） |
| **6. 归档沉淀** | 摘要、CHANGELOG、索引更新 | **快速档** | ❌ 重推理档（纯浪费） |

**速记口诀**：**澄清和评审升档，写码和测试守档，归档降档**。

## 4. 各阶段的模型选择详解

### 4.1 需求分析阶段：升档 to 重推理

**为什么必须升档**：用户给的需求 9 成有歧义。澄清的本质是"识别用户没说出口的隐性约束"——这是典型的多事实交叉推理，快速档会顺着用户的直觉走，错失关键问题。

**这个阶段模型应该做的**：

- 主动列出**用户没明说但必须澄清**的 5–10 个问题
- 用反例 / 边界场景检验用户的初版表述
- 把模糊词（"快"、"稳"、"差不多"）逐个落地

**这个阶段模型不该做的**：

- 直接生成 spec（还没拿到完整需求就生成 = 把猜的东西沉淀为契约）
- 替用户拍板（决策权属于人）

---

### 4.2 Spec 生成阶段：守档 in 平衡

**为什么用平衡档够了**：spec 是模板填空——9 个固定模块、字段类型固定、AC 格式固定。结构化能力是平衡档的甜蜜区。

**例外升档场景**：

- 跨栈 spec（主 spec + 契约 + 子 spec，详见 07 章 §2）
- 合规 / 安全敏感场景，AC 必须穷举
- 需要做技术选型对比（§7 技术方案）

**铁律**：spec 生成的会话**不要继承需求澄清会话的全部 context**。让澄清结果归档到一份"需求摘要"，新开会话载入摘要 + 模板，避免上下文污染。

---

### 4.3 Spec 评审阶段：升档 + 多模型组合

**为什么必须升档**：评审的本质是**对抗性思考**。模型要主动找漏洞，不能"看完夸两句"。

**为什么必须换模型 / 换会话**：

- 同一模型刚生成的 spec，自审会有"自己写的总觉得对"的偏向
- 同一会话上下文里有自己的生成痕迹，会被自己骗
- **铁律**：评审会话必须独立开，最好换模型

**推荐组合**：让 3 个独立会话扮演不同角色：

| 角色 | 模型档 | 关注点 |
|------|--------|--------|
| 杠精 | 重推理档 | 找漏洞、找歧义、找未覆盖边界 |
| 安全专家 | 重推理档 | 注入、权限、数据泄漏、合规 |
| 新人 | 平衡档 | 哪里读不懂 = 哪里 spec 写得不清 |

跨厂商组合更佳：让不同厂商模型独立审，能识别出单厂商训练偏差导致的盲区。

---

### 4.4 实现编码阶段：守档 in 平衡，谨慎升档

**为什么平衡档是甜蜜区**：spec 已经把"做什么"定死了，写代码变成"按 spec 翻译成代码"——模式匹配任务，不需要重推理。

**升档信号**：

- AI 连续 ≥3 次按 spec 写出跑偏代码 → 不是模型问题，多半 spec 不够细，回去改 spec
- 涉及并发 / 分布式协议 / 复杂算法 → 升重推理档
- 跨多个文件的协调改动（rename、跨模块重构） → 升重推理档

**降档信号**：

- 纯样板代码（CRUD、DTO 转换、配置文件） → 降快速档
- 单个函数体内的逻辑填充 → 快速档够用

**铁律**：**不要在同一会话频繁切档**——每次切档都意味着新 context，最好按"模块 / PR"分会话。

---

### 4.5 验证测试阶段：守档 in 平衡

测试代码本质上也是模式化生成（"AC-N → 一个测试方法"），平衡档稳定。

**例外**：

- 性能基线异常诊断、慢查询定位 → 升重推理档（多事实交叉）
- 跨层 E2E 断言设计（详见 09 章 §5） → 升重推理档（要在 UI / API / DB / 事件四个维度同时持有期望）

---

### 4.6 归档阶段：降档 to 快速

CHANGELOG 撰写、spec 索引更新、PR 描述生成都是抽取 + 摘要任务，快速档秒杀。

**铁律**：归档让快速档跑，**省下来的 token 全部投入到下一轮的需求澄清**——这是最划算的预算重分配。

## 5. 多模型协作的四种拓扑

### 5.1 单模型贯穿（baseline）

- **形态**：从澄清到归档全用一个模型一个会话
- **何时合适**：mini-spec、个人脚本、无团队协作
- **代价**：贵 + 上下文越滚越脏

### 5.2 主从模式（推荐起点）

- **形态**：重推理档**只在关键节点出场**（澄清、评审、跨栈决策），其余交给平衡档批量执行
- **何时合适**：大多数中型 feature
- **关键**：主从交接必须通过**冻结的 spec 文件**而非对话历史

### 5.3 角色专家组（多 agent 拓扑）

- **形态**：每个 SDD 阶段一个独立 agent / 会话，每个 agent 独立选模型
- **何时合适**：跨栈大需求、合规场景
- **关键**：agent 之间通过**契约文件 + spec 冻结版本**通信，不共享 raw context
- **配套**：sub-agent / 多会话隔离（Claude Code 的 sub-agent、Cursor 的多 chat、Aider 的多 session 都可，详见 §8）

### 5.4 选举 / 裁决（高风险决策）

- **形态**：N 个不同厂商模型各出一份方案，重推理档**裁决**
- **何时合适**：架构选型、破坏性变更、安全敏感设计
- **代价**：贵 ≥3 倍，只在关键决策点用
- **价值**：能识别出单一模型的系统性偏见

## 6. 升档 / 降档的边界信号

### 6.1 必须升档的信号

- AI 连续 ≥3 次给出跑偏答案（先怀疑 spec，再怀疑模型）
- 任务跨 ≥3 个文件 / 模块 / 子系统
- 涉及并发、分布式、安全、合规
- 用户开始烦躁、反复纠正同一类错误
- spec 评审 / 架构决策节点

### 6.2 应该降档的信号

- 任务变成"按模板填空"
- spec 已锁死，剩下纯翻译工作
- 重复批量操作（rename、import 整理、格式化）
- 归档、摘要、索引类任务

### 6.3 不该升档的伪信号

- "这次比较重要" → 重要 ≠ 复杂；spec 锁死后照样平衡档
- "我不放心" → 不放心应该补 review 流程，不是升档
- "看起来很难" → 让模型先列改动计划，再判断难度

## 7. 全模型通用边界（任何档位都不该越过）

无论用哪档模型，以下事**永远不该让 AI 单独做**：

| 边界 | 为什么 | 谁来做 |
|------|--------|--------|
| **拍板决策** | 决策需要承担后果，AI 不承担 | 人 |
| **替代 code review** | review 的价值在于"另一个有责任的脑子"，不是工具 | 人（哪怕 AI 也参与） |
| **改契约不通知** | 契约是跨人 / 跨团队约束，单方面改是协作灾难 | 人发起契约 PR |
| **替代 CI / 回归测试** | AI 不稳定、不可重放（详见 09 章 §6） | 自动化测试框架 |
| **跨 spec 边界自由发挥** | spec 没覆盖的 = 必须反问，不是猜 | 提示词强制约束 + 人审 |
| **改生产环境** | 哪怕 AI 写得对，人要在场审批 | 人 |

## 8. 与 sub-agent / 多会话机制的对接

不同工具的 sub-agent / 多会话机制都能承载 SDD 阶段分工。下表以 Claude Code 的 sub-agent 类型为例（其他工具按等价能力对应即可：Cursor 的 Composer + 多 chat、Aider 的多 session、Continue 的多 model 配置、ChatGPT 的多 thread）：

| SDD 阶段 / 任务 | 推荐 sub-agent / 会话角色 | 推荐模型档 |
|----------|---------------|-----------|
| 澄清前的代码侦察、找现有 spec | `Explore` 类（轻量 read-only 调研） | 平衡档 |
| 复杂改动的实施计划 | `Plan` 类（不写代码，只产计划） | 重推理档 |
| Spec 评审（多角色） | 独立 `general-purpose` 会话 × N（最好换模型） | 重推理档 |
| 批量改动 / 跨文件 rename | `general-purpose` | 平衡档 |
| 单文件实现 | 主会话直接做 | 平衡档 |

**铁律**：

- sub-agent / 子会话完成后**只返回结论**给主会话，不传染 raw context
- 子会话用什么模型不要"传染"主会话 —— 主会话升降档要单独决策
- 跨厂商 / 跨档评审必须开**全新会话**，不能让父 context 漏进来

## 9. 跨厂商组合的实务边界

如果团队同时用多家厂商模型，要注意以下差异：

| 维度 | Claude | OpenAI | Gemini | 影响 |
|------|--------|--------|--------|------|
| **Prompt Caching** | 显式控制（cache_control） | 自动（隐式）+ 部分显式 | Context Caching API | Claude 长 spec 复用 ROI 最高；其他厂商需测算 |
| **上下文窗口** | 200K（部分 1M） | 视模型 | 1M+ | 超大 spec 适合 Gemini；常规 SDD Claude 够用 |
| **结构化输出** | XML 友好 | JSON / Schema 强 | JSON 强 | 子 spec 互通时统一用 JSON / YAML |
| **工具调用稳定性** | 强 | 强 | 中 | 多步 spec 编排 Claude / GPT 优先 |
| **数据合规边界** | 各厂商企业版差异 | —— | —— | 敏感代码必须先确认数据驻留策略 |

**跨厂商协作的最低约定**：

- spec / 契约文件用 **Markdown + JSON / YAML 块**，避免厂商专属格式
- 提示词避免依赖单一厂商的特殊 token（如 Claude 的 `<thinking>` 在其他模型行为不同）
- 跨厂商裁决会话不要包含原模型的输出 context（否则裁决偏向先发言者）

## 10. 反模式

| 反模式 | 症状 | 对治 |
|------|------|------|
| **全场最贵** | Opus 跑 lint / 写 CHANGELOG / 改注释 | 设档位下限：归档 / 格式类一律走快速档 |
| **全场最便宜** | Haiku 写 spec、做架构决策 | 设档位上限：澄清 / 评审 / 架构必须重推理档 |
| **同会话评审自己** | 让生成 spec 的会话审自己 | 评审强制开新会话，最好换模型 |
| **频繁切档** | 同一会话内反复切模型 | 按"任务 / PR"分会话，会话内固定档位 |
| **多 agent 各持 spec 副本** | 每个 agent 自己读了一遍 spec 然后漂移 | 引用同一 spec 文件版本号，不传 raw 内容 |
| **跨档不交接** | 升档 / 降档时把对话历史一股脑丢过去 | 通过冻结的 spec / 摘要文件交接，不传对话 |
| **裁决偏向先发言者** | 让重推理档读了 A 模型答案再读 B 模型答案 | 裁决会话要并列对比、不带顺序偏好 |

## 11. 一张速查图：怎么选模型

```
当前任务是什么类型？
│
├─ 决策 / 评审 / 架构 / 跨栈 ───────→ 重推理档（必须）
│
├─ 按 spec 翻译 / 模板填空 / 批量代码 ─→ 平衡档（默认）
│   │
│   └─ 连续 ≥3 次跑偏？ ─── 是 ──→ 先回去改 spec；仍不行才升档
│
├─ 摘要 / 索引 / CHANGELOG / 格式化 ──→ 快速档
│
└─ 不确定 ───→ 默认平衡档，观察输出再决定升降
```

## 12. 思考与练习

1. **审计题**：扫一遍你最近一周的 AI 使用记录，按本章档位分类。哪一类最浪费（高档干低档活）？哪一类最危险（低档干高档活）？
2. **拓扑题**：你当前最大的 SDD 任务采用 §5 的哪种拓扑？如果换一种拓扑，瓶颈在工具、上下文交接、还是协作流程？
3. **边界题**：§7 列出的"全模型通用边界"中，你团队最容易越界的是哪一条？典型场景是什么？
4. **裁决题**：模拟一次架构选型，让两个不同厂商的模型各出方案，再让重推理档裁决。裁决质量是否优于单模型自检？把过程归档作为团队 ADR 的产出物。
