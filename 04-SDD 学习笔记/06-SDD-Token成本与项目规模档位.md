# Token 成本与项目规模档位

> 版本：1.0 | 定位：高频问题专题 — "SDD 会不会很费 token？"

很多人第一次看到 SDD 的反应是："每次都要把 `AGENTS.md / CLAUDE.md` + spec 塞进上下文，token 岂不是烧得很快？"这个担忧**部分成立**，但不是看单次会话就能得出结论。这一章把成本一次性算清楚，并给出按规模分档的判据。

---

## 1. 项目规模档位：先决定要不要重武器

一套规矩硬套不同场景就是官僚。推荐按规模分档：

| 规模 | spec 形态 | 评审 | 工具 | 归档 |
|------|-----------|------|------|------|
| **个人脚本 (< 100 行)** | 10 行 `README` 顶部注释 | 自审 | 无 | Git history 即归档 |
| **小团队 feature (1–2 人周)** | mini-spec（3–5 节） | 1 人 review | PR 模板 | CHANGELOG 引用 |
| **中型需求 (1 月+)** | 完整 9 模块 spec | ≥ 2 人 + AI 杠精 | CI 校验 + ADR | 独立归档 |
| **跨团队系统 (半年+)** | 主 spec + 子 spec 拆分 | 架构委员会 | 强 CI 门禁 | 专门的 docs 仓库 |

**判断依据**：写 spec 的时间应在**总工时的 5–20%** 之间。过低 → 不够严谨；过高 → 过度设计。

---

## 2. 直觉 vs 实际

- **直觉**：SDD 每轮对话都加载 spec，单次 token 一定高于 Vibe Coding。
- **实际**：**单次会话 token 确实更高，但全生命周期 token 总量未必更高**。

关键是把视角从"一次对话"放大到"一个功能从需求到维护的完整链路"。

---

## 3. 全生命周期成本结构对比

| 阶段 | Vibe Coding | SDD |
|------|-------------|-----|
| 需求澄清 | 低（隐藏在一轮轮 "其实我还要 X" 里） | 中（一次性做透，集中消耗） |
| 初次实现 | 低 | 高（带着 spec 生成） |
| 返工 / 跑偏重写 | **高**（反复打补丁，上下文越滚越长） | 低（spec 锁定方向，一次到位） |
| 二次维护 | 高（要重读代码猜当初意图） | 低（直接让 AI 读 spec） |
| 跨人协作 | 高（每个新人都要重新"教" AI） | 低（spec 一份多人复用） |
| **合计** | **隐性累积，容易超支** | **前期集中、后期衰减** |

Vibe Coding 便宜的只是**第一轮**——从第 3 次返工开始，它的累计 token 通常就已经超过同等需求走一遍 SDD。

---

## 4. Spec 是一种"有损到无损"的上下文压缩

把 spec 理解为**对话历史的无损压缩**：

- 一份 1500 字的 spec ≈ 50 轮澄清对话浓缩后的结论
- 下次会话只需 `@docs/specs/NNNN.md`，即可恢复完整决策边界
- 而纯对话上下文在超长时会被截断 / 压缩 / 遗忘，是**有损**的

换句话说，spec 是**用一次性 token 投入，换无限次低成本复用**。

---

## 5. Token 优化策略（实战清单）

按性价比从高到低：

1. **启用 Prompt Caching**
   多数厂商 API 都支持上下文缓存（Anthropic 显式 `cache_control`、OpenAI 自动、Google `Context Caching`），把 `AGENTS.md / CLAUDE.md` + 稳定 spec 放在 cached prefix，命中后成本约为未命中 1/10。这条是所有策略里 ROI 最高的。
2. **分阶段开会话，不要一条龙**
   需求澄清、spec 评审、代码实现拆成 3 个会话，每个会话只载入必要章节。一条龙会话越往后上下文越臃肿。
3. **引用文件，不要粘贴内容**
   让 AI 自己 `Read docs/specs/NNNN.md`，不要把 spec 正文贴进 prompt。文件读取只在需要时发生，比粘贴式 prompt 更省。
4. **分层加载 spec 章节**
   需求阶段看 §1–3，实现阶段看 §5–9，验收阶段看 §9。很少需要一次性全读。
5. **AGENTS.md / CLAUDE.md 精简到"只有硬约束"**
   约定、禁令、流程→放 `AGENTS.md / CLAUDE.md`；教程、解释、示例→放 README / docs。前者每次都进上下文，后者按需加载。
6. **子 Agent 隔离跑评审**
   让 spec 评审 / 安全评审在独立上下文里跑，不污染主会话。等于把重任务从"主线 token"中切出去。
7. **规模分档决定 spec 厚度**
   个人脚本用 10 行 mini-spec，不要硬套 9 模块。见第 1 节。
8. **版本稳定期集中产出代码**
   spec 一旦 `[APPROVED]`，尽量在同一稳定版本下批量生成实现；每改一次 spec 会使 cache 失效。

---

## 6. 什么场景 SDD 的 token 投入反而不划算

- **一次性脚本 / 个人快速 PoC**：重写成本 < 写 spec 成本
- **纯探索性实验**：边界都没想清楚，写出的 spec 很快会变"上一版化石"
- **<10 行的独立小改动**：上 spec 属于过度设计

**判断准则**：若预计后续**至少还有 2 次**对这块代码的编辑 / 扩展，或**至少 2 个人**会协作，SDD 通常就能回本。

---

## 7. 一个常被忽视的收益：防止"对话越滚越贵"

Vibe Coding 的隐形成本在于**上下文污染**：同一个长会话里堆满了被推翻的方案、中途丢弃的代码、用户的改口。每一轮推理都要把这些噪声重新过一遍。

SDD 通过"需求 → spec → 实现"的阶段切分，让每个新会话从**干净的 spec** 起步——这是单次 token 算账时看不见、却实实在在省下的成本。

---

## 8. Token 之外：SDD 的其他成本维度

token 只是最显眼的一条成本轴。SDD 真实的账单由多条成本共同构成，不同团队的"瓶颈成本项"往往不是 token：

| 维度 | 体现 | 容易被忽视的点 |
|------|------|-----------------|
| **人力工时** | 写 spec、review spec、维护版本号 | 比 AI token 贵得多；1 小时工程师时间 ≈ 海量 token |
| **学习曲线** | 团队从 Vibe 切到 SDD 的适应期 | 前 2 周效率下降是必然，不要以此否定 SDD |
| **基础设施** | PR 模板、CI 校验、spec 索引、ADR 工具 | 小团队手工可行；大团队不工具化 SDD 会退化 |
| **Context Window 占用** | spec + `AGENTS.md / CLAUDE.md` 吃掉上下文配额 | 不是钱的问题，是"**注意力预算**"——context 越挤，模型长程推理质量越降 |
| **文档债务** | spec 与代码不同步后反向误导 | 未维护的过时 spec **比没有 spec 更危险**，会同时欺骗人和 AI |
| **决策冻结成本** | spec approved 后改动阻力变大 | 对稳定需求是收益；对探索性需求是枷锁 |
| **认知负担** | 脑中同时维护 spec 和代码两份心智模型 | 小改动时负担尤其明显；用规模分档缓解 |
| **组织 / 文化成本** | 推动团队习惯变革的软性阻力 | 技术问题易解，人的习惯难改；常被低估 |
| **沉没 / 迁移成本** | 存量无 spec 代码补 spec 的工时 | 多数团队选"只给新功能写 spec"的双轨，接受短期混乱 |
| **流程延迟** | 需求到第一次可交付的时间被拉长 | TTM（time-to-market）敏感的场景要专门评估 |

## 9. 实操建议：怎么算这本总账

1. **算 3 个月账，不算单次账**
   不要对比"一次 Vibe vs 一次 SDD"，要对比 3 个月后两种团队状态的差异：返工次数、新人 onboarding 速度、缺陷密度、AI 输出一致性。
2. **先试点，再推广**
   选一个小团队 / 单模块做 2–4 周试点，用数据说话。跳过试点直接全员推 SDD 是组织级冒险。
3. **警惕文档债务**
   **宁可退档到 mini-spec，也不要留一堆过时 spec**。过时 spec 比没有 spec 更糟——它让未来的人和 AI 都被错误信息带偏。
4. **主动管理 context 占用**
   spec 超过 ~3000 字就该考虑拆子 spec；`AGENTS.md / CLAUDE.md` 超过 ~500 行要清理冗余条目。上下文塞爆后 AI 质量下降的损失，远大于 token 钱。
5. **按需求稳定度选档位**
   - MVP / 探索性项目：允许 spec 频繁迭代，版本号放松
   - 合规 / 对外 API / 多人协作：严守 spec-first，冻结成本就是它的价值
6. **把 SDD 成本当套利看**
   SDD 的本质是在"**今天多花点成本**"和"**未来少踩坑**"之间做套利。在需求稳定、协作规模大、维护周期长的场景最划算；做一次性脚本或单人快开发时反而是负收益。

```
这块代码后续还会改 ≥ 2 次？ ───── 否 ──→ 别上 SDD（或只写 10 行 mini-spec）
          │是
          ▼
有 ≥ 2 人会接触？ ───── 否 ──→ mini-spec 即可
          │是
          ▼
需求边界相对稳定？ ───── 否 ──→ 先做探索、等边界收敛再写 spec
          │是
          ▼
维护周期 ≥ 3 个月？ ───── 否 ──→ mini-spec 即可
          │是
          ▼
   上完整 9 模块 SDD
```

## 11. 思考与练习

1. **测算题**：估算你团队上一次 Vibe Coding 的"返工总轮数 × 单轮平均 token"。换算成同等需求若走 SDD，在哪个返工次数上盈亏平衡？
2. **诊断题**：你现在的 `AGENTS.md / CLAUDE.md` 有多少行？如果超过 500 行，哪些条目可以下沉到 README / docs？
3. **决策题**：用第 10 节的决策图过一遍你手头的 3 个任务，哪些应该上完整 SDD、哪些 mini-spec、哪些直接 Vibe？
