# Skill Prompt 设计原则

> 版本：1.0 | 定位：Prompt 工程核心方法论

---

## 1. 为什么 Prompt 质量决定 Skill 质量

Skill 的执行效果 90% 取决于 Prompt 的质量。同样的功能需求，优秀的 Prompt 能让 Agent 精准执行，而糟糕的 Prompt 会导致：

- 输出格式不稳定，每次结果不一致
- Agent 误解意图，执行错误操作
- 遗漏关键步骤，输出不完整
- 无法处理边界情况

---

## 2. 核心设计原则

### 原则一：角色先行（Role First）

在 Prompt 开头明确 Agent 的角色定位，建立执行上下文。

```markdown
# 不好的写法
分析这段代码的性能问题。

# 好的写法
你是一位专注于 JVM 性能优化的 Java 高级工程师，
拥有10年以上的系统调优经验。请分析以下代码的性能问题。
```

**角色声明的要素：**
- 专业领域（Java/前端/数据库等）
- 经验层次（初级/高级/专家）
- 关注视角（安全/性能/可维护性等）

---

### 原则二：指令原子化（Atomic Instructions）

将复杂任务拆解为**清晰的原子步骤**，避免歧义。

```markdown
# 不好的写法（模糊，让 Agent 自己理解）
帮我审查这段代码并改进它。

# 好的写法（明确步骤）
对以下代码进行审查，**严格按照以下步骤执行**：

Step 1: 列出所有潜在的 Bug（包括空指针、边界越界、并发问题）
Step 2: 标注违反 SOLID 原则的地方
Step 3: 识别性能瓶颈（重点关注循环和数据库访问）
Step 4: 对每个问题给出具体的修复建议和代码示例
Step 5: 总结最高优先级的3个改进点
```

---

### 原则三：约束输出格式（Output Constraints）

明确定义输出的结构、格式和风格，确保结果可预期。

```markdown
# 不好的写法（输出无法预测）
总结这次会议的重点。

# 好的写法（明确输出格式）
总结以下会议记录，**严格按照以下格式输出**：

## 会议摘要
[1-3句话概括会议主题]

## 决策事项
| 决策 | 负责人 | 截止日期 |
|------|--------|---------|
| ... | ... | ... |

## 行动项
- [ ] [具体任务] — @[负责人] — [截止日期]

## 下次会议
[时间、议程]

**注意：不要添加上述格式以外的内容**
```

---

### 原则四：边界声明（Boundary Declaration）

明确告知 Agent 哪些事情**不做**，防止过度执行。

```markdown
# 不好的写法（容易产生过度行为）
帮我优化这段代码。

# 好的写法（明确边界）
对以下代码进行**性能优化**，要求：

✅ 允许的操作：
- 优化算法复杂度
- 减少不必要的内存分配
- 改善数据库查询效率

❌ 禁止的操作：
- 不修改函数签名（保持 API 兼容性）
- 不引入新的外部依赖
- 不改变代码的业务逻辑
- 不做纯格式化/重命名的修改
```

---

### 原则五：提供示例（Few-Shot Examples）

对于格式复杂或理解难度高的任务，提供具体的输入/输出示例。

```markdown
# Skill：生成 API 错误码注释

生成符合以下格式的 API 错误码注释。

**示例输入：**
```java
public class UserErrorCode {
    public static final int USER_NOT_FOUND = 4001;
    public static final int USER_ALREADY_EXISTS = 4002;
}
```

**示例输出：**
```java
public class UserErrorCode {
    /**
     * 用户不存在
     * HTTP 状态码：404
     * 触发场景：通过 ID 或邮箱查询用户时，用户记录不存在
     * 建议处理：前端提示"用户不存在，请检查输入"
     */
    public static final int USER_NOT_FOUND = 4001;
    
    /**
     * 用户已存在
     * HTTP 状态码：409
     * 触发场景：注册时邮箱或用户名已被占用
     * 建议处理：前端提示"该邮箱已注册，请直接登录"
     */
    public static final int USER_ALREADY_EXISTS = 4002;
}
```

现在按照上述格式，处理以下代码：$ARGUMENTS
```

---

### 原则六：描述字段的战略重要性

`description` 字段是 Skill 的"索引"，直接影响 Agent 的自动匹配能力。

```yaml
# 弱 description（无法帮助 Agent 准确匹配意图）
description: 代码审查工具

# 强 description（明确触发条件、使用场景、能力边界）
description: |
  当用户需要对代码进行 Code Review、质量检查、安全审查时使用。
  适用于：完成编码后提交前的质量把关、PR 审查、重构前的现状评估。
  支持语言：Java、Python、TypeScript、Go。
  会输出：问题清单（按严重程度分级）+ 改进建议 + 修复代码示例。
  不适用于：纯格式化需求（使用 /format）、性能专项分析（使用 /perf-analyze）。
```

## 3. 常见 Prompt 反模式

### 反模式一：指令模糊性

```markdown
# ❌ 模糊指令
帮我写好一点

# ✅ 具体指令
将以下文本重写为：
- 面向技术决策者（CTO 级别）的语言风格
- 突出商业价值而非技术细节
- 字数控制在 200 字以内
- 使用主动语态
```

### 反模式二：过度宽泛的范围

```markdown
# ❌ 范围过大，Agent 不知如何取舍
全面分析这个项目

# ✅ 明确关注点
分析这个项目的**依赖安全性**，具体检查：
1. package.json 中的直接依赖是否有已知 CVE
2. 是否有过期超过2年的依赖
3. 是否有不再维护的包
```

### 反模式三：缺乏错误处理指导

```markdown
# ❌ 未说明异常情况处理
读取配置文件并解析

# ✅ 包含异常处理指导
读取配置文件并解析，按以下规则处理异常：
- 文件不存在：输出警告并使用默认值继续
- JSON 格式错误：输出具体错误位置，停止执行并提示修复方法
- 字段缺失：列出所有必填字段，标注哪些缺失
```

### 反模式四：角色和任务不匹配

```markdown
# ❌ 角色定位与任务要求不匹配
你是一个初级程序员，请帮我设计高并发系统架构

# ✅ 角色与任务对齐
你是一位拥有大规模分布式系统设计经验的高级架构师，
曾主导过日均亿级请求系统的设计与优化。
请帮我设计以下高并发场景的系统架构...
```

## 4. Prompt 调试与优化方法

### 4.1 黄金测试三角

对每个 Skill 进行以下三类测试：

```
1. 正常场景测试：标准输入是否产生期望输出
2. 边界场景测试：极端输入（空值、超大值、特殊字符）
3. 对抗场景测试：恶意或错误输入是否被正确处理
```

### 4.2 迭代优化流程

```
写初稿 → 测试执行 → 发现问题 → 定位原因 → 针对性修改 → 回归测试
    ↑                                                           │
    └───────────────────────────────────────────────────────────┘
```

**常见问题定位：**

| 症状 | 可能原因 | 修复方向 |
|------|---------|---------|
| 输出格式不稳定 | 格式约束不够明确 | 增加"严格按照以下格式" + 示例 |
| 遗漏关键步骤 | 步骤描述不完整 | 拆分为有序步骤列表 |
| 执行了额外操作 | 缺少禁止声明 | 添加"禁止做以下操作"条款 |
| 无法处理边界情况 | 未考虑异常路径 | 添加异常场景处理说明 |
| Agent 误解意图 | description 不准确 | 重写 description，增加场景描述 |

### 4.3 A/B 对比测试

对关键 Skill 保存多个版本进行对比：

```
skill-commit-v1.md   → 简单版本
skill-commit-v2.md   → 增加格式约束
skill-commit-v3.md   → 增加示例

测试相同场景，对比输出质量，选择最优版本
```

## 5. Prompt 模板库

### 5.1 分析类 Prompt 模板

```markdown
你是一位[专业角色]，专注于[专业领域]。

请对以下[分析对象]进行全面分析：

**分析维度：**
1. [维度1]：[具体关注点]
2. [维度2]：[具体关注点]
3. [维度3]：[具体关注点]

**输出格式：**
## 分析摘要
[3句话总结]

## 详细分析
### [维度1]
...

## 关键结论
[优先级排列的结论列表]

**约束：**
- 基于事实进行分析，不做无根据的猜测
- 每个结论必须有对应的证据
```

### 5.2 生成类 Prompt 模板

```markdown
你是一位[专业角色]。

根据以下输入生成[目标产物]：

**输入：**
$ARGUMENTS

**生成要求：**
- [要求1]
- [要求2]
- [要求3]

**输出格式：**
[具体格式定义]

**示例：**
输入：[示例输入]
输出：[示例输出]

**禁止：**
- [禁止行为1]
- [禁止行为2]
```

### 5.3 检查类 Prompt 模板

```markdown
你是一位[专业角色]，负责[检查职责]。

对以下[检查对象]进行检查，**逐项评估**以下标准：

| 检查项 | 标准 | 优先级 |
|--------|------|--------|
| [检查项1] | [具体标准] | P0/P1/P2 |
| [检查项2] | [具体标准] | P0/P1/P2 |

**输出格式：**
对于每个发现的问题，输出：
- 问题描述（一句话）
- 严重程度（Critical/Warning/Info）
- 位置（文件:行号 或 具体位置）
- 修复建议（具体操作步骤）

**通过标准：** 无 Critical 级别问题，Warning 问题已知晓。
```
