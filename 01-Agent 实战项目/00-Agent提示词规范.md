# Agent 提示词规范

> 版本：1.0 | 适用范围：本目录下所有 Agent 提示词文件
> 立项依据：本仓库 `00-提示词工程/` 反模式与角色规范，结合实战目录的批量自检结论沉淀

---

## 1. 概述

本规范定义本目录下所有 Agent 提示词文件的结构骨架、写作约束与红线清单，确保新增/修改的 Agent 提示词不再重复以下五类系统性问题：

1. 「禁用指令」纯否定反模式（违反 LLM「正向激活」原则）
2. 同模板克隆扩散（多语言/多平台 Agent 95% 雷同）
3. 工具与平台名硬编码（绑定具体厂商，丧失通用性）
4. 敏感领域无职权边界声明（财/法/税/安/数/人事）
5. 段落规则数量超阈值（>10 条触发选择性忽略）

新增 Agent 必须按本规范完成自检；存量文件按 P0→P2 顺序整改。

---

## 2. 标准 8 段式骨架

每个 Agent 提示词文件**固定 8 段**，顺序如下：

| 段落 | 状态 | 字数/条目 | 必含要点 |
|------|------|-----------|----------|
| 1. 智能体概述 | ✅ 必选 | 80–150 字 | 一句话定位 + 核心能力 + 适用场景 |
| 2. 角色定义 | ✅ 必选 | 4 条固定字段 | 核心定位 / 专业领域 / 工作风格 / 目标导向 |
| 3. 核心职责 | ✅ 必选 | **≤ 5 条** | 每条须可对应到「输出标准」中的具体产出 |
| 4. 行为准则 | ✅ 必选 | **≤ 6 条** | 每条须可指导决策，不与其他条互斥 |
| 5. 工作流程 | ✅ 必选 | **≤ 7 步** | 步骤须可串联从输入到输出标准 |
| 6. 输出标准 | ✅ 必选 | 3–6 条 | 必须列具体格式 / 维度 / 验收点 |
| 7. 职权边界与禁用指令 | ✅ 必选 | 见 §4 | 正向 + 反例配对，禁纯否定堆砌 |
| 8. 提示词示例 | ✅ 必选 | 见 §6 | 核心提示词 + ≥ 1 组完整 Few-shot |

> 总条目数（§3+§4+§5）建议控制在 **≤ 18 条**。超过即视为规则堆砌反模式。

---

## 3. 角色与任务匹配（红线）

| 红线 | 反例 | 正例 |
|------|------|------|
| 角色与任务领域错配 | "你是诗人，请写 SQL" | "你是数据库工程师，请写 SQL" |
| 空洞拔高 | "世界最强 AI"、"无所不能" | 给出"X 年经验 + 具体技术栈" |
| 借名人抬轿 | "你是 Linus Torvalds" | 描述其岗位特征而非具名 |
| 同时多个冲突角色 | "既是 A 又是 B 同时还是 C" | 一个主角色，必要时分阶段切换 |

## 4. 「职权边界与禁用指令」段（最易翻车，单独成节）

### 4.1 写法范式：正向 + 反例配对

**反模式**（清一色"不得 X"）：
```
- 不得提交未经验证的代码
- 不得忽略错误处理
- 不得违反编码规范
- 不得编写存在安全漏洞的代码
```

**正确范式**（每条配对正向行为）：
```
- 提交前应运行单元测试与静态检查；测试不通过时退回上一步并说明原因
- 所有外部输入须显式校验；遇异常分支应明确处理或显式上抛
- 应遵循目标语言的官方风格指南（如 PEP 8 / Effective Java）
- 对涉及鉴权、注入、序列化的代码应给出安全检查结论
```

> 原则：**用"做什么"代替"不做什么"**。必须否定时，搭配一个正向描述。

### 4.2 敏感领域必含「职权边界」声明

涉及以下领域的 Agent，**必须**在本段开头加一条边界声明，模板如下：

| 领域 | 必含边界声明 |
|------|-------------|
| 财务 / 税务 | "输出仅作参考；正式核算 / 申报须由持证会计师 / 税务师审核确认" |
| 法务 / 合规 | "意见仅作参考；正式法律意见以执业律师签字文件为准" |
| 人事 / 招聘 / 考勤 | "建议供 HR 决策参考；不替代最终人事决策；遇法规冲突应转交法务" |
| 安全 / 渗透 / 审计 | "未获书面授权前，不得对生产系统执行任何主动扫描或利用；不输出可直接利用的攻击 payload" |
| 数据 / AI / 风控 | "PII 与受限数据须脱敏后处理；模型输出不直接驱动金额、信贷、医疗等高风险决策" |

> 写在第 7 段最前一条，**不要藏在末尾**——重要边界要前置。

## 5. 工具与平台中立性

### 5.1 红线
- 角色名 / 角色定位 / 核心职责中**禁止**写入具体厂商或产品名
- 行为准则 / 工作流程中如必须提及，应使用抽象类（见词表）

### 5.2 抽象词表（可继续扩展）

| 抽象类 | 替代示例 |
|--------|----------|
| 主流编程语言 | 替代 "Java / Python / VUE" |
| 主流 ETL/编排工具 | 替代 "Airflow / dbt / Spark" |
| 主流向量库 | 替代 "Pinecone / Qdrant / Milvus" |
| 主流 LLM 服务 | 替代 "OpenAI / Claude / 文心" |
| 主流 BI 工具 | 替代 "Tableau / Power BI / 帆软" |
| 主流社交渠道 | 替代 "微信 / 微博 / 小红书 / 抖音" |
| 主流 CI/CD 平台 | 替代 "Jenkins / GitLab CI / GitHub Actions" |
| 主流低代码/工作流 | 替代 "Coze / N8N / Dify" |

### 5.3 同模板克隆判定

当多个 Agent 仅替换语言/平台名而 ≥ 80% 雷同时，**必须**抽通用层：

```
00-{类别}-Agent-{角色}（通用）.md      —— 通用 70% 内容
01-{类别}-Agent-{角色} {平台 A}.md     —— 仅含 ≤ 30% 平台特化
02-{类别}-Agent-{角色} {平台 B}.md     —— 仅含 ≤ 30% 平台特化
```

平台特化文件应在「智能体概述」首句声明 *"本文件为「{角色}（通用）」的 {平台} 特化扩展，通用准则不再赘述"*。

## 6. 「提示词示例」段：必含完整 Few-shot

仅给"核心提示词 + 占位符辅助提示词"**不算合格**。每个 Agent 必须至少给出 1 组完整 Few-shot：

```markdown
### Few-shot 示例

**输入**：
{真实业务场景下的用户输入，含上下文与诉求}

**期望输出**：
{严格符合本文件「输出标准」的完整产出}
```

辅助提示词中的 `[占位符]` **不能替代** Few-shot；占位符是模板复用工具，Few-shot 是行为锚定。

---

## 7. 段间一致性约束

- **职责 ↔ 输出**：每条「核心职责」必须对应至少一条「输出标准」
- **职责 ↔ 流程**：每条「核心职责」必须可由「工作流程」中的 ≥ 1 步执行得到
- **行为准则 ↔ 禁用指令**：每条「行为准则」的反面应能在「禁用指令」中找到对应的正反配对，反之亦然
- **避免重复**：同一思想不在多段重复表述（如「保密」既出现在准则又出现在禁用指令中），应择一段表述

---

## 8. 自检清单（提交前必过）

新增或修改 Agent 提示词文件后，提交前依次自查：

- [ ] 8 段齐全，顺序符合 §2，无段落缺失或自创段落
- [ ] §3 / §4 / §5 各段条目数未超上限（5 / 6 / 7），合计 ≤ 18 条
- [ ] §7「禁用指令」每条均为「正向 + 反例配对」形式，无纯"不得 X"堆砌
- [ ] 敏感领域 Agent 在 §7 首条已加「职权边界」声明
- [ ] 角色定位 / 职责 / 流程未硬编码具体厂商或产品名（按 §5.2 词表抽象化）
- [ ] 同类 Agent 跨语言/平台变体雷同度 ≥ 80% 时已抽出「（通用）」文件
- [ ] §8 至少 1 组完整 `输入 → 期望输出` Few-shot
- [ ] 每条「核心职责」可在「输出标准」找到对应产出（§7 一致性约束）
- [ ] 全文无"优化 / 提升 / 确保 / 改进"等开放词未列具体维度的情形
- [ ] 全文无空洞拔高（"世界最强"、借名人）、无角色与任务领域错配

## 9. 整改优先级（适用于存量文件）

| 优先级 | 触发条件 | 对应改造 |
|--------|----------|----------|
| **P0** | 敏感领域无职权边界声明 | 按 §4.2 模板补充首条 |
| **P0** | 「禁用指令」全部为纯否定 | 按 §4.1 改为正反配对 |
| **P1** | 多语言/多平台同模板克隆 | 抽「（通用）」文件，特化文件瘦身 |
| **P1** | 工具/平台名硬编码 | 按 §5.2 词表抽象化 |
| **P2** | 段落条目超上限 | 同类合并、可选项下沉到 Few-shot |
| **P2** | 缺完整 Few-shot | 按 §6 补 ≥ 1 组 输入→输出 |
